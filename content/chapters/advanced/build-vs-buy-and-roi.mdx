# Build vs Buy & ROI — ตัดสินใจลงทุน AI อย่างฉลาด

---

มีอยู่วันหนึ่ง CEO เรียก meeting ด่วน เปิดประตูมาด้วยคำถามเดียว: "ทำไมเราต้องเสียเงินสร้าง AI เอง ในเมื่อ ChatGPT มันก็ทำได้แล้ว?" สองเดือนถัดมา CTO ของบริษัทอื่นถาม platform vendor ว่า "ทำไมเราต้องจ่ายคุณล้านนึง ในเมื่อเราจ้าง engineer มาสร้างเองก็ได้?" — สองคำถามนี้ฟังดูตรงข้ามกัน แต่เป็นอาการของปัญหาเดียวกัน: คนส่วนใหญ่มองการลงทุน AI เป็นเรื่องขาวดำ ทั้งที่ความจริงมันเป็นสเปกตรัม

บทนี้คือบทสุดท้ายของ curriculum ระดับ Advanced เราจะรวมทุกอย่างที่เรียนมา — Vision, Translation, Judgment — มาใช้ในการตัดสินใจจริงที่มีเม็ดเงินแขวนอยู่

## Part 1: Build-Buy Spectrum — ไม่ใช่แค่ 2 ตัวเลือก

ก่อนอื่น ต้องทลายความเชื่อเรื่องนี้ก่อน: "Build vs Buy" ไม่ใช่การเลือกระหว่าง 2 สิ่ง มันเป็น **สเปกตรัม** ที่มีอย่างน้อย 5 จุดหยุด

<!-- IMAGE: แผนภาพ spectrum จาก "Use API as-is" ทางซ้ายไปถึง "Train from scratch" ทางขวา พร้อม cost และ flexibility scale -->

### จุดที่ 1: Use API As-Is (ซื้อ 100%)

คุณเปิด OpenAI API หรือ Claude API แล้วส่ง prompt ไปตรงๆ ไม่มีการปรับแต่งอะไร

**เหมาะเมื่อ:** ต้องการ prototype เร็ว, use case ทั่วไป (สรุปเอกสาร, แปลภาษา, ตอบคำถาม), ทีมไม่มี ML expertise

**ราคา:** ตาม token — ถูก แต่ scale ขึ้นเรื่อยๆ ตาม usage

**ตัวอย่างบริษัทที่ใช้:** 80% ของ startups ในช่วง 2 ปีแรก

### จุดที่ 2: API + Prompt Engineering + RAG (ซื้อ + customize เล็กน้อย)

ยังใช้ API ของคนอื่น แต่เพิ่ม system prompt ที่ออกแบบมาดี + ต่อกับ vector database ที่มีข้อมูลบริษัทของคุณ

**เหมาะเมื่อ:** ต้องการ domain knowledge เฉพาะ (เช่น นโยบายบริษัท, คู่มือสินค้า), แต่ยังไม่อยากลงทุนใหญ่

**ราคา:** ค่า API + ค่า vector DB (Pinecone, Weaviate) + ชั่วโมง engineer 1-2 คน

**ตัวอย่าง:** Chatbot ตอบคำถาม HR ที่รู้ policy ของบริษัทคุณโดยเฉพาะ

### จุดที่ 3: Fine-Tune Model (ซื้อ + customize กลาง)

เอา base model ของ OpenAI / Mistral / Llama มา fine-tune ด้วยข้อมูลของคุณ model เรียนรู้ tone, format, และ domain knowledge ที่เฉพาะเจาะจงมากขึ้น

**เหมาะเมื่อ:** ต้องการ output ที่มี style เฉพาะ (เช่น เขียนสัญญาแบบที่กฎหมายของบริษัทกำหนด), มีข้อมูล labeled 500-10,000 ตัวอย่าง, ใช้งาน volume สูงจน RAG เริ่มช้า

**ราคา:** ค่า training (OpenAI fine-tuning ราคาต่อ token) + ค่า inference ที่สูงกว่าเล็กน้อย

**ข้อระวัง:** ข้อมูลที่ส่งไป fine-tune จะออกไปอยู่บน server ของ vendor

### จุดที่ 4: Self-Host Open Model (สร้างเอง + ซื้อ infrastructure)

Download open-source model (Llama 3, Mistral, DeepSeek) มารัน server ของตัวเอง ไม่มี data ออกไปข้างนอก

**เหมาะเมื่อ:** data sensitive มาก (healthcare, finance, legal), ต้องการ compliance ที่ vendor ทำให้ไม่ได้, ใช้งาน volume สูงมากจน API cost แซง hosting cost

**ราคา:** GPU server (A100 ลำเดียวอาจ $2-4/ชั่วโมง บน cloud) + DevOps engineer ดูแล + ต้นทุน latency ที่อาจสูงกว่า

**ตัวอย่าง:** โรงพยาบาลที่ต้องการ AI วิเคราะห์ผล lab แต่ข้อมูลผู้ป่วยออกไปข้างนอกไม่ได้เลย

### จุดที่ 5: Train from Scratch (สร้างเอง 100%)

สร้าง model ตั้งแต่ต้น เก็บข้อมูลเอง, ออกแบบ architecture เอง, train เอง

**เหมาะเมื่อ:** เป็น Google, OpenAI, หรือ Anthropic. ไม่ใช่ใครก็ได้

**ราคา:** หลัก 10-100 ล้านเหรียญสหรัฐ + ทีม PhD 20-50 คน + เวลา 1-3 ปี

**สำหรับ 99.9% ของบริษัท:** ข้ามตัวเลือกนี้ไปก่อน

> 💡 **Key insight:** การเลือก Build vs Buy ที่ดีไม่ใช่การเลือก "สุดโต่งซ้ายหรือขวา" มันคือการเลือกจุดที่เหมาะบน spectrum โดยดูจาก use case, ทีม, งบ, และ timeline — แล้วอาจเลื่อนขึ้นไปทาง build มากขึ้นเมื่อ prove value ได้แล้ว

## Part 2: Decision Framework — ถามอะไรก่อนตัดสินใจ

มี framework ที่ดีมากชื่อ **Wardley Mapping** ที่ช่วยตัดสินใจว่าอะไรควร build เอง อะไรควร buy ซึ่งหัวใจของมันคือคำถามเดียว:

**"สิ่งนี้เป็น core competency ของเราหรือเปล่า?"**

<!-- IMAGE: 2x2 matrix แกน X = "เป็น competitive advantage ของบริษัท?" แกน Y = "ความพร้อมของทีม?" พร้อม 4 quadrant strategies -->

### ตัดสินใจด้วย 5 คำถาม

**คำถามที่ 1: มันเป็น competitive advantage ของเราจริงๆ ไหม?**

ถ้า AI recommendation engine คือสิ่งที่ทำให้ผู้ใช้เลือกเราแทนคู่แข่ง — นั่นคือ core competency ควรสร้างเอง

ถ้า AI สรุป meeting notes คือ feature เสริมที่ทุกคนทำได้ — นั่นคือ context ซื้อ vendor มาใช้เลย

**คำถามที่ 2: Data ของเราเป็น moat ไหม?**

ถ้าเราเป็นบริษัทประกันที่มี claims data 10 ล้านรายการในช่วง 20 ปี และข้อมูลนั้นทำให้ fraud detection แม่นกว่าคนอื่น — นั้นคือ reason to build เพราะ vendor ไม่มีข้อมูลนี้

ถ้าเราใช้ข้อมูลทั่วไป (เช่น รีวิวสินค้าจาก Amazon) — ไม่มี data moat, ซื้อ API ดีกว่า

**คำถามที่ 3: ทีมพร้อมไหม?**

| สิ่งที่ต้องมีเพื่อ build | ถ้าไม่มี |
|---|---|
| ML Engineer / Data Scientist 2+ คน | ซื้อก่อน, build ทีมทีหลัง |
| Data pipeline ที่ clean | เริ่มด้วย API + RAG แทน fine-tune |
| MLOps / monitoring | self-host ยังไม่เหมาะ |
| Budget สำหรับ iteration | ต้องการ proof of concept ก่อน |

**คำถามที่ 4: Time-to-market สำคัญแค่ไหน?**

Build จาก scratch อาจใช้เวลา 6-18 เดือน ในตลาดที่เปลี่ยนเร็ว นั่นอาจหมายถึงการเสียโอกาส ซื้อ API แล้ว launch ใน 2 เดือนอาจคุ้มกว่ามาก แม้ cost per unit จะสูงกว่า

**คำถามที่ 5: Data sensitivity อยู่ระดับไหน?**

```python
# Framework ง่ายๆ ประเมิน data sensitivity
def assess_data_sensitivity(data_type: str) -> str:
    """
    ประเมินว่าข้อมูลแบบนี้ส่งออกไปข้างนอกได้ไหม
    """
    # ข้อมูลที่ห้ามออกไปนอก infra เลย
    high_risk = [
        "ข้อมูลสุขภาพผู้ป่วย",     # PDPA, HIPAA
        "ข้อมูลการเงินส่วนตัว",      # ข้อมูลบัญชี, รายได้
        "ข้อมูลลับทางการค้า",        # สูตรสินค้า, ราคา cost
        "ข้อมูลทางกฎหมาย",          # คดี, สัญญาลับ
    ]
    
    # ข้อมูลที่ระวังแต่ส่งได้ถ้า vendor ผ่าน compliance
    medium_risk = [
        "ข้อมูลลูกค้าทั่วไป",        # ชื่อ, อีเมล (ต้องเช็ค DPA)
        "ข้อมูล internal ops",       # process, workflow
        "ข้อมูล aggregate",          # ไม่ระบุตัวตน
    ]
    
    # ข้อมูลที่ส่งออกได้สบาย
    low_risk = [
        "ข้อมูล public",            # ข่าว, เอกสาร public
        "ข้อมูล product catalog",   # รายละเอียดสินค้า
        "ข้อมูล marketing",         # campaign copy, content
    ]
    
    if data_type in high_risk:
        return "Self-host only — ห้ามใช้ external API"
    elif data_type in medium_risk:
        return "ต้องเช็ค vendor compliance ก่อน (SOC2, ISO27001)"
    else:
        return "ใช้ API ได้เลย ไม่มีปัญหา"
```

## Part 3: TCO Analysis — ต้นทุนจริงที่คนมักลืม

TCO ย่อมาจาก Total Cost of Ownership — ต้นทุนรวมทั้งหมดตลอดอายุการใช้งาน

ปัญหาคือ คนมักคิดแค่ต้นทุน "ชัดๆ" เช่น ค่า API หรือค่า software license แต่ลืม "hidden costs" ที่บางทีใหญ่กว่าเยอะ

<!-- IMAGE: Iceberg diagram — ส่วนที่เห็น (API cost, license) เป็นส่วนเล็กด้านบน ส่วนที่จม (maintenance, talent, opportunity cost) เป็นก้อนใหญ่ด้านล่าง -->

### Hidden Costs ที่คนมักลืม

**ถ้าเลือก Build:**

| ต้นทุน | ที่มักถูก underestimate |
|---|---|
| Compute (GPU/CPU) | ราคา scale ตาม load — ช่วง peak อาจแพงกว่าที่คิด 3-5x |
| Data labeling | ถ้าต้องการ 10,000 ตัวอย่าง ราคา $0.05-0.50 ต่อชิ้น = $500-5,000 |
| ML talent | Senior ML Engineer ในไทย เงินเดือน 150,000-300,000 บาท/เดือน |
| Maintenance | 20-30% ของ development cost ต่อปี (model drift, data updates) |
| Infrastructure | Monitoring, logging, CI/CD pipeline, security patching |
| Opportunity cost | ทีมที่ build AI ไม่ได้ build feature อื่น |

**ถ้าเลือก Buy:**

| ต้นทุน | ที่มักถูก underestimate |
|---|---|
| Integration effort | ใช้เวลา engineer 1-3 เดือนในการต่อ API เข้า system |
| Vendor lock-in | ถ้า vendor ขึ้นราคา 3x คุณย้ายได้แค่ไหน? |
| Customization ceiling | มีสิ่งที่ vendor ทำให้ไม่ได้ และคุณอาจสร้างเองไม่ได้ |
| Data egress fees | บางบริษัทคิดค่า export ข้อมูลออก |
| Compliance overhead | ต้อง audit vendor ทุกปี = เวลาทีม legal/security |

### ตัวอย่างการคำนวณ TCO จริง

สมมติบริษัทมีพนักงาน 500 คน อยากสร้าง AI ช่วยตอบคำถาม HR

**Option A: ซื้อ HR chatbot SaaS (เช่น Guru, Leena AI)**
- ค่า license: 500 users × $20/user/เดือน = $10,000/เดือน = $120,000/ปี
- ค่า implementation: $15,000 (one-time, vendor ช่วย)
- ค่า maintenance: ต่ำมาก (vendor ดูแล)
- **TCO ปีแรก:** ~$135,000

**Option B: Build เอง (API + RAG + custom UI)**
- Engineer 1 คน × 3 เดือน = 3 × 150,000 = 450,000 บาท (~$13,000)
- ค่า OpenAI API (ประมาณ 1,000 คำถาม/วัน × $0.01/คำถาม × 365 วัน) = $3,650/ปี
- ค่า vector DB (Pinecone Starter): $70/เดือน = $840/ปี
- ค่า maintenance engineer (0.25 FTE): 150,000 × 0.25 × 12 = 450,000 บาท (~$13,000/ปี)
- **TCO ปีแรก:** ~$30,000

**แต่รอก่อน — Option B มี hidden costs:**
- ถ้า model มี error และพนักงานได้รับ HR info ผิด — ใครรับผิดชอบ?
- เมื่อ HR policy เปลี่ยน ใคร update ข้อมูลใน RAG?
- ถ้า engineer คนนั้นลาออก ใครดูแลต่อ?

> 💡 **Key insight:** TCO ที่ถูกต้องต้องรวม "risk-adjusted cost" ด้วย — โอกาสที่ระบบจะล้มเหลว × ต้นทุนของความล้มเหลวนั้น บางทีนี่คือตัวเลขที่ใหญ่ที่สุดในสมการ

## Part 4: ROI Calculation — วัดกำไรอย่างรอบด้าน

ROI (Return on Investment) คือ: (ผลตอบแทน - ต้นทุน) / ต้นทุน × 100%

แต่ "ผลตอบแทน" ของ AI มีหลายชั้น คนมักวัดแค่ชั้นแรก แล้วสงสัยว่าทำไม business case ดูไม่น่าสนใจ

<!-- IMAGE: Pyramid diagram แบ่งชั้น ROI: Direct (ล่างสุด), Operational, Strategic (บนสุด) -->

### 3 ชั้นของ AI ROI

**ชั้น 1: Direct Value (วัดง่ายที่สุด)**

- **Cost savings:** ลด headcount, ลด error rate, ลดเวลาทำงาน
- **Revenue increase:** conversion rate ดีขึ้น, upsell ได้มากขึ้น, ลูกค้าใหม่จาก feature ใหม่

ตัวอย่าง: AI ช่วยตรวจสอบ invoice อัตโนมัติ ลดเวลาจาก 10 นาที/ใบ เหลือ 30 วินาที/ใบ ทีม 5 คนประมวล 500 ใบ/วัน ประหยัดได้ 4,166 ชั่วโมง/ปี

**ชั้น 2: Operational Value (วัดได้แต่ต้องเชื่อมโยง)**

- **Speed:** product ออก market เร็วขึ้นกี่เดือน? มูลค่าของเดือนที่เพิ่มมาคืออะไร?
- **Quality:** error rate ลดลง → return rate ลด → customer satisfaction สูงขึ้น
- **Employee satisfaction:** คนทำงานน่าเบื่อน้อยลง → turnover ลด → ประหยัด recruiting cost

**ชั้น 3: Strategic Value (วัดยากที่สุด แต่บางทีใหญ่ที่สุด)**

- **Competitive moat:** ถ้า AI ทำให้เราเร็วกว่า/ดีกว่าคู่แข่ง 6 เดือน นั้นมีค่าเท่าไหร่?
- **Data flywheel:** การใช้ AI สะสม data → data ทำให้ AI ดีขึ้น → ดีขึ้นดึงดูดผู้ใช้มากขึ้น
- **Optionality:** ถ้าเรา build platform แล้วมี use case ใหม่เกิดขึ้น ต้นทุน marginal ต่ำมาก

### สูตร ROI Calculation แบบ Practical

```python
def calculate_ai_roi(
    # ต้นทุน
    development_cost: float,      # บาท - ต้นทุน build/implement
    annual_operating_cost: float,  # บาท/ปี - ค่าดำเนินงาน
    years: int = 3,               # ระยะเวลาประเมิน
    
    # ผลตอบแทนชั้น 1: Direct
    annual_cost_savings: float = 0,    # บาท/ปี - ต้นทุนที่ลดได้
    annual_revenue_increase: float = 0, # บาท/ปี - รายได้ที่เพิ่มขึ้น
    
    # ผลตอบแทนชั้น 2: Operational (ประเมินเป็น % ของ direct)
    speed_value_multiplier: float = 1.0,  # 1.2 = +20% จากความเร็ว
    quality_value_multiplier: float = 1.0, # 1.1 = +10% จากคุณภาพ
    
    # Risk adjustment
    success_probability: float = 0.7,  # 70% โอกาสสำเร็จ
) -> dict:
    """
    คำนวณ ROI ของ AI project แบบ 3 ปี
    รวม risk adjustment เพื่อให้ตัวเลขน่าเชื่อถือมากขึ้น
    """
    
    # ต้นทุนรวม 3 ปี
    total_cost = development_cost + (annual_operating_cost * years)
    
    # ผลตอบแทนต่อปี (direct)
    annual_direct_return = annual_cost_savings + annual_revenue_increase
    
    # ปรับด้วย operational multipliers
    annual_adjusted_return = (
        annual_direct_return 
        * speed_value_multiplier 
        * quality_value_multiplier
    )
    
    # ผลตอบแทนรวม 3 ปี (ปีแรก conservative เพราะยังรันระบบ)
    total_return = (
        annual_adjusted_return * 0.5 +  # ปีที่ 1: ได้ครึ่ง (ramp up)
        annual_adjusted_return * 1.0 +  # ปีที่ 2: full
        annual_adjusted_return * 1.2    # ปีที่ 3: มักดีขึ้นอีก
    )
    
    # Risk-adjusted return
    risk_adjusted_return = total_return * success_probability
    
    # คำนวณ ROI และ Payback Period
    net_benefit = risk_adjusted_return - total_cost
    roi_percent = (net_benefit / total_cost) * 100
    
    # Payback period (เดือน)
    monthly_return = annual_adjusted_return / 12
    if monthly_return > 0:
        payback_months = (development_cost + annual_operating_cost) / monthly_return
    else:
        payback_months = float('inf')
    
    return {
        "total_investment": f"{total_cost:,.0f} บาท",
        "risk_adjusted_return": f"{risk_adjusted_return:,.0f} บาท",
        "net_benefit": f"{net_benefit:,.0f} บาท",
        "roi_3_year": f"{roi_percent:.1f}%",
        "payback_period": f"{payback_months:.1f} เดือน",
        "recommendation": (
            "Go ahead" if roi_percent > 100 and payback_months < 18
            else "Pilot first" if roi_percent > 50
            else "Reconsider scope"
        )
    }

# ตัวอย่างใช้งาน: AI ช่วยทีม customer support
result = calculate_ai_roi(
    development_cost=500_000,       # 500K บาท build + implement
    annual_operating_cost=120_000,  # 120K บาท/ปี ค่า API + maintenance
    annual_cost_savings=800_000,    # ลด headcount 2 คน หรือ redirect งาน
    annual_revenue_increase=200_000, # CSAT ดีขึ้น → retention ดีขึ้น
    speed_value_multiplier=1.15,    # ตอบเร็วขึ้น 40%
    success_probability=0.75,
)

print(result)
# Output:

# risk_adjusted_return: 2,475,000 บาท
# net_benefit: 1,615,000 บาท
# roi_3_year: 187.8%

```

## Part 6: The "Start with API, Graduate to Custom" Pattern

นี่คือ strategy ที่ smart teams ใช้มากที่สุด — และที่ผมเห็นว่า work ได้จริงในทางปฏิบัติ

แทนที่จะต้องตัดสินใจทันทีว่าจะ build หรือ buy ให้ใช้ **Progressive Investment Approach**

<!-- IMAGE: Timeline แสดง 4 phases: Validate → Optimize → Scale → Own พร้อม investment level และ expected outcome ในแต่ละ phase -->

### 4 Phases ของ Progressive Investment

**Phase 1: Validate (เดือน 1-2)**

ใช้ API as-is, zero customization เป้าหมายเดียวคือพิสูจน์ว่า use case นี้ work จริง

- งบ: ต่ำมาก (ค่า API + 1-2 สัปดาห์ engineer)
- เป้าหมาย: 30 users ทดลองใช้, ได้ feedback จริง
- ตัดสินใจต่อจากนี้ด้วย: "User ใช้จริงไหม? ปัญหาหายไปไหม?"

**Phase 2: Optimize (เดือน 3-4)**

ถ้า Phase 1 pass ต่อด้วย prompt engineering + RAG ปรับให้เหมาะกับ use case ของเรามากขึ้น

- งบ: ปานกลาง (engineer 1 คนเต็มเวลา + vector DB)
- เป้าหมาย: performance ดีขึ้น 30-50%, cost per query ลดลง
- ตัดสินใจต่อจากนี้ด้วย: "ROI เริ่มชัดขึ้นไหม? volume สูงพอที่จะ justify custom model?"

**Phase 3: Scale (เดือน 5-8)**

ถ้า volume สูงและ ROI ชัด พิจารณา fine-tuning หรือ self-hosting

- งบ: สูง (ML engineer + infra + data labeling)
- เป้าหมาย: cost per query ลดลง 60-80% จาก Phase 2, latency ดีขึ้น
- ตัดสินใจต่อจากนี้ด้วย: "เราต้องการ capability ที่ vendor ทำไม่ได้ไหม?"

**Phase 4: Own (ถ้าถึงจุดนั้น)**

เฉพาะถ้า use case นี้เป็น core product ของบริษัท — พิจารณา self-host หรือ custom architecture

- งบ: สูงมาก (team ML 3-5 คน)
- เป้าหมาย: เป็น competitive moat ที่คู่แข่ง replicate ไม่ได้

> 💡 **Key insight:** ทุก phase มี "exit ramp" — ถ้าผลลัพธ์ไม่ดี สามารถหยุดได้โดยไม่เสียหนักมาก นี่คือสิ่งที่ต่างจากการ commit ทุกอย่างตั้งแต่วันแรก

### Pilot Program Design — หัวใจของการ prove value

Pilot ที่ดีต้องมี 4 สิ่ง:

1. **Clear success criteria ก่อนเริ่ม** — ตกลงกันว่า metric ไหนจะ "pass" เช่น "accuracy > 85%" หรือ "NPS เพิ่มขึ้น 10 points"

3. **Real users, real data** — ไม่ใช่ internal test เท่านั้น ต้องมี user จริงใช้งานจริง

4. **Time-boxed** — กำหนดวันสิ้นสุดชัดเจน (6-8 สัปดาห์) ไม่ใช่ pilot ที่ลากไปเรื่อยๆ

## Part 8: Common Pitfalls —罠 ที่คนมักตก

ก่อนจบ ขอพูดถึงความผิดพลาดที่พบบ่อยที่สุดในการตัดสินใจ AI investment

<!-- IMAGE: Warning signs / pitfall icons พร้อม label แต่ละข้อ -->

### Pitfall 1: Over-Building เพราะ "ทำเองดีกว่า"

บริษัทหลายแห่งใช้เวลา 6 เดือนสร้าง custom ML pipeline เพราะ "ไม่อยากพึ่ง vendor" ในขณะที่ API ของ vendor ทำงานได้ดีกว่า 90% ของที่พวกเขาสร้าง และออก market ได้เร็วกว่า 5 เดือน

**สัญญาณอันตราย:** Engineer พูดว่า "แต่ถ้าเราสร้างเอง เราจะ control ได้ทุกอย่าง" — control ไม่ใช่เป้าหมาย value คือเป้าหมาย

### Pitfall 2: Premature Optimization

สร้าง infrastructure ระดับ Google Scale ตั้งแต่วันแรก ทั้งที่ยังไม่รู้ว่า users จะมาจริงไหม

**กฎ:** อย่า optimize ก่อนที่จะ know ว่า thing ที่ optimize อยู่นั้น matter จริง

### Pitfall 3: Vendor Lock-in โดยไม่รู้ตัว

ใช้ proprietary format ของ vendor สำหรับทุกอย่าง — ข้อมูล, model, workflow เมื่อ vendor ขึ้นราคา 3x หรือเลิกกิจการ ย้ายออกยากมาก

**วิธีป้องกัน:** เก็บ business logic ไว้ใน layer ของเราเอง ให้ vendor เป็นแค่ "plug-in" ที่เปลี่ยนได้

### Pitfall 4: "AI Washing" — เรียกทุกอย่างว่า AI

บางบริษัทเรียก rule-based automation ว่า "AI" เพื่อให้ดูน่าประทับใจ ปัญหาคือเมื่อผู้บริหารถามถึง ROI ของ "AI initiative" จะได้ตัวเลขรวมที่ทำให้ภาพไม่ชัด

**วิธีป้องกัน:** แยก tracking ระหว่าง automation project, ML-based project, และ generative AI project ให้ชัดเจน

### Pitfall 5: ลืม Change Management

ซื้อ AI tool แพงมากแต่ user adoption ต่ำ เพราะไม่ได้ training, ไม่ได้ communicate ว่าจะเปลี่ยนวิธีทำงานอย่างไร, หรือ incentive structure ไม่ได้เปลี่ยนตาม

**กฎ:** งบ 20% ของ AI project ควรเป็น change management และ training — ไม่ใช่แค่ technology

### Pitfall 6: วัด Success ผิด Metric

วัดว่า model accuracy 95% แต่ไม่วัดว่า business impact จริงเป็นยังไง บางที model ที่ accuracy 80% แต่ user trust และใช้งานจริง ดีกว่า model 95% ที่ nobody uses

> 💡 **Key insight:** AI project ที่ดีที่สุดไม่ใช่ที่มี technology ดีที่สุด แต่คือที่สร้าง behavior change จริงในองค์กร และ behavior change นั้นสร้าง business value ที่วัดได้

## สรุป — Capstone ของ Advanced Curriculum

เราเดินทางมาไกลมากตลอด curriculum ระดับ Advanced นี้ ตั้งแต่การเข้าใจ architecture ของ AI systems ไปจนถึง RAG, fine-tuning, evaluation frameworks, production deployment, safety, และวันนี้ — การตัดสินใจลงทุนอย่างฉลาด

<!-- IMAGE: Timeline/roadmap แสดง journey ของ curriculum ทั้งหมด ไฮไลต์บทนี้เป็นจุดสุดท้าย -->

ถ้าจะสรุป 3 สิ่งที่สำคัญที่สุดจากบทนี้:

**หนึ่ง: Build vs Buy ไม่ใช่ binary choice**

มันเป็นสเปกตรัม และจุดที่เหมาะสมขึ้นอยู่กับ use case, ทีม, งบ, timeline, และ data sensitivity เริ่มจาก API แล้วค่อยขยับมาทาง build เมื่อ prove value ได้แล้ว

**สอง: TCO และ ROI ต้องครอบคลุม "hidden" ทั้งหมด**

ต้นทุนจริงมักสูงกว่าที่เห็น ผลตอบแทนจริงมักมีมากชั้นกว่าที่คำนวณ การทำ analysis ที่ incomplete ทำให้ตัดสินใจผิดได้ทั้งสองทิศทาง

**สาม: Judgment คือ skill สูงสุดของ AI work**

Vision คือการเห็นว่า AI ทำอะไรได้ Translation คือการเอา AI ไปใส่ใน workflow จริง แต่ Judgment คือการรู้ว่า "เมื่อไหร่ควรทำ อะไรควรทำ และทำแค่ไหนพอ" — นั่นคือสิ่งที่แยก AI practitioner ที่สร้าง value จริงออกจากคนที่แค่รู้ว่า AI มีอยู่

ทักษะ Judgment นี้ไม่ได้มาจากการอ่านอย่างเดียว มาจากการ practice ตัดสินใจจริง ทำ pilot จริง วัดผลจริง และเรียนรู้จากทั้งความสำเร็จและความผิดพลาด

### Quick Reference: Build vs Buy Decision Checklist

ใช้ checklist นี้ทุกครั้งที่ต้องตัดสินใจ:

```
□ ระบุชัดว่านี่คือ core competency หรือ context?
□ Data ของเรามี sensitivity ระดับไหน?
□ ทีมมี capability พร้อมสำหรับ build ไหม?
□ Time-to-market สำคัญแค่ไหนสำหรับ use case นี้?
□ คำนวณ TCO ครบทั้ง direct และ hidden costs?
□ ROI ครอบคลุม 3 ชั้น (direct, operational, strategic)?
□ มี success criteria ชัดเจนก่อนเริ่ม pilot?
□ มีแผน exit/migration ถ้าเลือก vendor?
□ Change management อยู่ใน plan ไหม?
□ ผู้บริหาร sign-off บน metrics ที่ตกลงกัน?
```
