# Machine Learning คืออะไร — เมื่อโปรแกรมเรียนรู้แทนที่จะแค่ทำตามคำสั่ง

ถ้าคุณเปิด Gmail แล้วเห็นว่า spam ไม่ค่อยรั่วเข้า inbox — นั่นไม่ใช่เพราะมีคนนั่งอ่านอีเมลทุกฉบับแล้วคัดให้คุณ. และมันก็ไม่ได้เกิดจากโปรแกรมที่เขียน rule ไว้ว่า "ถ้ามีคำว่า 'congratulations you have won' ให้ทิ้ง." rule แบบนั้นสแปมเมอร์แก้ทัน.

สิ่งที่อยู่เบื้องหลัง Gmail spam filter, ตัวแนะนำ Netflix, และ ChatGPT ทุกตัวที่คุณใช้ คือแนวคิดเดียวกัน — **machine learning**.

บทนี้จะอธิบายว่า machine learning คืออะไร ทำงานยังไง และทำไมมันถึงเปลี่ยนโลกซอฟต์แวร์ไปตลอดกาล. ถ้าคุณเคยอ่านบท[AI คืออะไร](/foundations/what-is-ai) มาแล้ว ก็พร้อมแล้ว. ถ้ายังไม่ได้อ่าน ลองผ่านบทนั้นก่อนสั้น ๆ ก็ดี แต่ไม่ได้จำเป็นต้องอ่านก่อนก็เข้าใจได้.

เมื่ออ่านจบบทนี้ คุณจะเข้าใจว่าทำไม ChatGPT ตอบได้บางเรื่องแต่ผิดบางเรื่อง, ทำไม AI ต้องการ data จำนวนมาก, และเมื่อไหร่ควรไว้ใจ output ของ AI เมื่อไหร่ควรตรวจสอบ. ความเข้าใจพวกนี้ใช้ได้ทันทีไม่ว่าคุณจะทำงานสายไหน.

## โปรแกรมทั่วไปทำงานยังไง

ก่อนจะเข้าใจ ML ต้องเข้าใจก่อนว่าโปรแกรมทั่วไปทำงานยังไง. เพราะ ML คือการพลิกแนวคิดนั้นโดยตรง.

ถ้าคุณอยากสร้างโปรแกรมคำนวณภาษีเงินได้ คุณเปิดคู่มือกรมสรรพากร เขียนตามนั้นทุกข้อ: ถ้ารายได้ไม่เกิน 150,000 บาทได้รับการยกเว้น, ถ้ารายได้ 150,001–300,000 เสีย 5%, ถ้ารายได้ 300,001–500,000 เสีย 10% — และต่อไปตามอัตราที่กำหนด. โปรแกรมทำตาม rule นั้นทุกครั้ง. ถูกต้อง คาดเดาได้ อธิบายได้.

สูตรของโปรแกรมแบบนี้คือ:

> **Data + Rules → คำตอบ**

คุณใส่ข้อมูล (รายได้ของคน) เข้าไป โปรแกรมใช้ rule ที่คุณเขียนไว้คำนวณ ได้คำตอบออกมา.

นี่เป็นวิธีที่ทำงานได้ดีมาหลายสิบปี และยังใช้ได้ดีสำหรับงานจำนวนมาก. ระบบธนาคาร, ระบบบัญชี, ระบบ ERP ที่บริษัทใช้ทุกวันยังทำงานด้วย traditional programming เป็นส่วนใหญ่ — และนั่นก็ถูกต้องแล้ว.

แต่มันมีขีดจำกัดที่สำคัญมาก: **มนุษย์ต้องรู้ rule ก่อนถึงจะเขียนโปรแกรมได้**.

และงานหลายอย่างในชีวิตจริง — rule มันอยู่ที่ไหน?

ถ้าคุณอยากเขียนโปรแกรมแยกรูปแมวออกจากรูปหมา คุณจะเขียน rule ว่าอะไร? "หูตั้ง = แมว, หูตก = หมา" ไม่ทำงาน เพราะหมาหลายสายพันธุ์หูตั้ง. "จมูกเล็ก = แมว" ก็ไม่ — แมวบางตัวจมูกใหญ่. "ตาแบบนี้ = แมว" ก็ยัง — แสงต่างกัน มุมต่างกัน ตาก็ดูต่างกัน. Rule ที่ครอบคลุมทุกกรณีเขียนได้แทบไม่ได้เลย.

หรือถ้าอยากเขียนโปรแกรมที่ "เข้าใจ" ว่าประโยคไทยมีความหมายว่าอะไร — คุณจะเริ่มยังไง? ภาษาไม่ได้มี rule ตายตัวที่เขียนเป็น if-else ได้.

ปัญหาทั้งสองแบบนี้ — แยกรูปภาพ, เข้าใจภาษา — คือประเภทของงานที่ machine learning ถูกสร้างมาเพื่อแก้.

## ML พลิกสูตรนั้น

แทนที่จะบอกโปรแกรมว่า "นี่คือ rule" — ML บอกโปรแกรมว่า "นี่คือตัวอย่าง แล้วหา rule เอง."

สูตรเปลี่ยนเป็น:

> **Data + คำตอบที่ถูกต้อง → Rules (เรียกว่า "model")**

ใน spam filter แบบ ML แทนที่จะเขียน rule ว่า "email แบบไหนคือ spam" — เราเอา email 100,000 ฉบับที่ติด label ว่า "spam" หรือ "ไม่ใช่ spam" ป้อนให้ algorithm. Algorithm หา pattern เอง. pattern ที่มันพบอาจซับซ้อนกว่า rule ที่คนเขียนได้มาก — เช่น "email ที่ใช้คำแสดงความเร่งด่วนร่วมกับการขอข้อมูลบัตรเครดิต แต่ส่งมาตอนดึก จากโดเมนที่จดทะเบียนเมื่อสัปดาห์ที่แล้ว" — pattern ที่ไม่มีทางเขียน rule ไว้ครบได้.

พอได้ model มาแล้ว ไว้ใช้งาน:

> **Data ใหม่ + Model → คำตอบที่คาดการณ์**

email ฉบับใหม่เข้ามา ใส่เข้า model ได้คำตอบว่า spam หรือไม่. ทั้งหมดนี้เกิดขึ้นใน milliseconds โดยที่ไม่มีคนเกี่ยวข้องเลย.

> **Key insight:** ML ไม่ได้ฉลาดกว่าโปรแกรมทั่วไปในทุกเรื่อง. มันเก่งเฉพาะงานที่ "ทำได้แต่ articulate rule ไม่ออก" — และแลกมาด้วยความต้องการ data จำนวนมาก และการยอมรับว่า model อาจผิดได้บ้าง.

## Mental Model ที่ใช้งานได้

คิดแบบนี้: ML คือ **"สอนด้วยตัวอย่างแทนสอนด้วยกฎ"**.

สอนเด็กให้แยกแมวกับหมา — คุณไม่ได้นั่งอธิบายว่า "หูยาวกี่เซนติเมตร จมูกกว้างแค่ไหน ขาหน้ายาวกี่สัดส่วนเมื่อเทียบกับลำตัว." คุณชี้รูปแล้วบอก "นี่หมา นี่แมว" ซ้ำหลายร้อยครั้ง แล้วเด็กก็แยกได้เองโดยที่ไม่สามารถบอก rule ได้ชัด.

ML ทำแบบเดียวกัน แค่ต้องการตัวอย่างมากกว่าเด็กมาก — บางงานต้องการหลายล้านตัวอย่าง.

analogy นี้ใช้ได้ดีในแง่ intuition แต่มีจุดที่ต่างกัน: เด็กเรียนรู้จากตัวอย่างน้อยกว่ามาก เพราะสมองมนุษย์มี prior knowledge มหาศาลที่สะสมมาจากวิวัฒนาการ. ML ต้องการตัวอย่างมากกว่าเพราะเริ่มจากศูนย์ — ไม่มี prior ใด ๆ ยกเว้น architecture ที่ออกแบบไว้. นี่คือเหตุผลที่ LLM ต้องใช้ text หลายล้านล้านคำจากอินเทอร์เน็ต แทนที่จะ "เรียนรู้" จากตัวอย่างสั้น ๆ แบบเด็กทำได้.

ผลตามมาสามข้อที่สำคัญ:

**ข้อแรก — ML ต้องการ data จำนวนมาก.** ตัวอย่างน้อย model เรียนรู้ pattern ได้ไม่ดีพอ เหมือนเด็กที่เห็นแมวแค่สองตัวแล้วบอกว่าเข้าใจแมวทุกตัวในโลกแล้ว. บาง ML system ต้องการ data หลายสิบล้านตัวอย่างก่อนจะทำงานได้ดี.

**ข้อสอง — ML ดีเท่า data ที่ป้อนให้.** ถ้า data มีความลำเอียงหรือผิดพลาด model ก็จะมีความลำเอียงหรือผิดพลาดตามไปด้วย ไม่มีข้อยกเว้น. ถ้า data ที่ใช้ train spam filter เป็น email ภาษาอังกฤษทั้งหมด มันก็จะทำงานได้แย่กับ spam ภาษาไทย.

**ข้อสาม — ML ไม่ "เข้าใจ" ในแบบมนุษย์.** มันหา pattern ทางสถิติที่ได้ผลกับตัวอย่างที่เห็นมา และคาดหวังว่า pattern นั้นจะยังได้ผลกับข้อมูลใหม่ด้วย. ถ้า pattern ในโลกเปลี่ยนไป model ก็จะตกหลัง. และ model อาจ "เข้าใจผิด" ในลักษณะที่มนุษย์ไม่คาดคิด เช่น spam filter ที่เรียนรู้ว่า "email ที่ยาวเกิน 500 คำ = ไม่ใช่ spam" เพราะใน training data บังเอิญเป็นแบบนั้น.

## เมื่อไหร่ใช้ ML เมื่อไหร่ไม่ต้องใช้

ML ไม่ใช่คำตอบสำหรับทุกปัญหา. มีงานที่ ML เหมาะ และมีงานที่ if-else ธรรมดาดีกว่ามาก.

**ใช้ ML เมื่อ:**

มี pattern ใน data แต่ articulate rule ออกมาเป็นโค้ดได้ยาก — เช่น แยกรูปภาพ, เข้าใจภาษาธรรมชาติ, ตรวจจับการฉ้อโกง. หรือเมื่อ pattern เปลี่ยนตามเวลา — สแปมเมอร์ปรับวิธีใหม่ทุกสัปดาห์ แต่ถ้ามี labeled data ใหม่ก็แค่ re-train โดยไม่ต้องเขียน rule ใหม่.

หรือเมื่อต้องทำงานในสเกลที่มนุษย์ทำได้แต่ทำไม่ไหว — อ่าน X-ray 10,000 ใบต่อวัน, ตรวจสอบ transaction 100 ล้านครั้งต่อวัน.

**ไม่ต้องใช้ ML เมื่อ:**

Rule ชัดเจน ตายตัว — คำนวณภาษี, validate รูปแบบอีเมล, คำนวณดอกเบี้ยเงินกู้. if-else ตรง ๆ แม่นยำกว่า debug ง่ายกว่า อธิบายได้ชัดกว่า. ใช้ ML กับงานพวกนี้แทบจะเป็นการสร้างปัญหาเพิ่มโดยไม่จำเป็น.

หรือเมื่อ data น้อยมาก — ถ้ามีตัวอย่างแค่ 10-50 ตัว intuition ของมนุษย์หรือ rule ธรรมดาดีกว่า ML แน่นอน.

หรือเมื่อต้องการ explainability สูง — ถ้าถูกถามว่า "ทำไมปฏิเสธสินเชื่อลูกค้าคนนี้" และต้องตอบได้ชัดเจนทุก case, ML แบบ black-box อาจไม่เหมาะ.

> **Key insight:** บริษัทเทคโนโลยีหลายแห่งใช้ทั้งสองแนวทางในระบบเดียวกัน — rule-based เป็นด่านแรกสำหรับ case ที่ชัด + ML สำหรับ case ที่ซับซ้อน + human review สำหรับ borderline. ไม่มีคำตอบว่าอันไหนดีกว่าเสมอ.

## ML อยู่ตรงไหนใน AI

จากแผนภาพ 4 วงกลมที่อธิบายไว้ในบท [AI คืออะไร](/foundations/what-is-ai) (AI ⊃ ML ⊃ Deep Learning ⊃ LLM) — ML คือวงที่สองจากนอก. ไม่ใช่ทุก AI เป็น ML เพราะ expert system ยุค 1980s เป็น AI ที่ทำตาม rule ที่มนุษย์เขียน ไม่ใช่ระบบที่เรียนรู้จาก data. แต่ตั้งแต่ประมาณปี 2010 ML ครอบงำ AI ในทางปฏิบัติมากจนคนทั่วไปใช้ทั้งสองคำสลับกัน.

เพราะฉะนั้น เมื่อคุณคุยกับ ChatGPT — คุณกำลังใช้ LLM ซึ่งเป็น deep learning ซึ่งเป็น ML ซึ่งเป็น AI. ซ้อนกันเป็นชั้น ๆ. สิ่งที่บทนี้จะเจาะคือชั้น ML — กลไกที่ทำให้ระบบทุกชั้นด้านในทำงานได้.

## ML มีกี่แบบ

เมื่อเข้าใจภาพรวมแล้ว ขั้นต่อไปคือรู้ว่า ML ไม่ได้มีแบบเดียว. มันแบ่งตามวิธีที่ algorithm ได้รับ feedback เป็นสามแบบหลัก — แต่ละแบบออกแบบมาสำหรับประเภทของปัญหาที่ต่างกัน.

### แบบที่หนึ่ง — Supervised Learning

"Supervised" แปลตรงตัวว่า "มีผู้ดูแล" — หมายถึง data ที่ใช้ train มาพร้อม **label** (คำตอบที่ถูกต้อง) กำกับไว้แล้ว. algorithm ดู input กับ output ที่ถูกต้อง แล้วพยายามเรียนรู้ความสัมพันธ์ระหว่างทั้งสอง.

ตัวอย่างที่เข้าใจง่าย:

- รูปภาพ 50,000 รูป ทุกรูปมีป้าย "แมว" หรือ "หมา" → algorithm เรียนรู้แยกแมวกับหมา
- อีเมล 1 ล้านฉบับ ทุกฉบับมีป้าย "spam" หรือ "ไม่ใช่ spam" → algorithm เรียนรู้กรอง spam
- ข้อมูลบ้าน 200,000 หลัง (พื้นที่, ทำเล, จำนวนห้อง) พร้อมราคาขายจริง → algorithm เรียนรู้ทำนายราคา

สังเกตว่าทุกตัวอย่างมีสิ่งเดียวกัน: input ที่รู้ + output ที่ถูกต้องที่รู้. algorithm พยายามหาความสัมพันธ์ระหว่างสองสิ่งนี้.

ข้อดีของ supervised learning คือประเมินผลง่าย — เอา prediction ไปเทียบกับ label ที่รู้อยู่แล้ว ได้ตัวเลข accuracy ออกมาตรงไปตรงมา. แต่ข้อเสียคือต้องการ **labeled data** ซึ่งมักแพง เพราะต้องให้มนุษย์นั่ง annotate ด้วยมือ. บางงาน เช่น annotation ภาพทางการแพทย์ ต้องใช้ผู้เชี่ยวชาญซึ่งราคาสูงมาก.

Supervised learning ใช้บ่อยที่สุดในงาน production จริง เพราะประเมินผลตรงไปตรงมา, อธิบายได้ว่าดีหรือไม่ดีแค่ไหน, และ iterate ได้เร็ว. ถ้าเริ่มใช้ ML ในองค์กร supervised learning มักเป็นจุดเริ่มต้นที่ดีที่สุด.

### แบบที่สอง — Unsupervised Learning

"Unsupervised" หมายถึง data ที่ใช้ train **ไม่มี label**. algorithm ได้รับแค่ input แล้วพยายามหาโครงสร้างใน data เอง โดยไม่มีใครบอกว่า "ถูก" หรือ "ผิด" คืออะไร.

ลองนึกภาพว่าคุณมี data การซื้อของลูกค้า 500,000 คน และอยากรู้ว่ามีลูกค้ากี่กลุ่มที่มีพฤติกรรมต่างกัน. คุณไม่รู้ว่าควรมีกี่กลุ่ม ไม่รู้ว่าแต่ละกลุ่มควรเรียกว่าอะไร — แค่อยากให้ algorithm หาโครงสร้างที่มีอยู่จริงใน data ออกมาให้.

งานแบบนี้เรียกว่า **clustering** และเป็น unsupervised learning แบบที่พบบ่อยที่สุด.

อีกตัวอย่างหนึ่ง: ระบบตรวจจับธุรกรรมผิดปกติในธนาคาร ที่เรียนรู้ว่า "ธุรกรรมปกติ" มีหน้าตายังไง แล้ว flag อะไรก็ตามที่หลุดออกจาก pattern นั้น — โดยไม่ต้องมีใครบอกล่วงหน้าว่า fraud มีหน้าตายังไง.

ประเมินผล unsupervised learning ยากกว่า supervised เพราะไม่มี "คำตอบที่ถูก" ให้เปรียบเทียบ. คำถามที่ว่า "algorithm แบ่งลูกค้าได้ดีไหม" บางครั้งต้องให้มนุษย์ไปดูผลแล้วตัดสินเอง — ดูว่า cluster ที่ได้ make sense ในธุรกิจหรือเปล่า ไม่ใช่แค่ตัวเลข mathematical.

### แบบที่สาม — Reinforcement Learning

แบบนี้ต่างออกไปจากสองแบบแรกอย่างชัดเจน.

แทนที่จะดู data แล้วเรียนรู้ — reinforcement learning มี "agent" ที่ **ลงมือทำ** ใน environment จริง ได้รับ reward หรือ penalty กลับมา แล้วค่อย ๆ เรียนรู้ว่าควรทำอะไรในสถานการณ์ไหนเพื่อให้ได้ reward สูงสุดในระยะยาว.

นึกถึงการสอนหมาให้ทำท่า "นั่ง" — ไม่มี labeled data ไม่มี rule ที่เขียนให้. แค่ทุกครั้งที่หมานั่งถูกต้องก็ได้ขนมหนึ่งชิ้น. หมาค่อย ๆ เรียนรู้ว่าการนั่งเมื่อเจ้าของพูดคำนี้นำไปสู่ขนม — แม้ไม่มีใครบอก "rule" ให้เลย.

ตัวอย่างที่โด่งดังที่สุดคือ AlphaGo ของ Google DeepMind ที่เรียนรู้เล่นหมากล้อมจากการเล่นกับตัวเองหลายล้านเกม โดยได้รับ reward เมื่อชนะ และสามารถเอาชนะแชมป์โลกมนุษย์ได้ในปี 2016 — ทั้งที่ไม่มีมนุษย์คนไหนสอน "ท่าที่ดี" ให้เลย. มันค้นพบท่าใหม่ที่มนุษย์ไม่เคยคิดถึงมาก่อนด้วย.

reinforcement learning ยังอยู่เบื้องหลังการปรับจูน ChatGPT ให้ตอบในแบบที่มนุษย์พอใจ — กระบวนการนี้เรียกว่า RLHF (Reinforcement Learning from Human Feedback) ซึ่งจะอธิบายเพิ่มเติมในบท[LLM คืออะไร](/foundations/what-is-an-llm).

สิ่งที่ต่างจาก supervised learning: ไม่มี "คำตอบที่ถูก" ให้ทันที — feedback คือ reward ที่อาจมาช้ามาก (ต้องเล่นหมากล้อม 200 ท่าถึงรู้ว่าชนะหรือแพ้).

### ลองทำดู

เปิด ChatGPT แล้วถามว่า: "ผมมีข้อมูลลูกค้า 10,000 คน แต่ไม่มี label อะไรเลย อยากรู้ว่าลูกค้าแบ่งเป็นกลุ่มไหนได้บ้าง — ควรใช้ ML แบบไหน?"

ดูว่า ChatGPT อธิบาย unsupervised learning ยังไง แล้วเทียบกับที่คุณอ่านมา. ถ้ามันพูดถึง "clustering" หรือ "k-means" หรือ "segmentation" แสดงว่าคุณตามทัน. ถ้ามันตอบผิดหรือสับสน ลองบอกมันว่าผิดตรงไหน แล้วดูว่ามันปรับได้ไหม.

## เลือก paradigm ให้ถูก — ตัวอย่างสามสถานการณ์จริง

แค่รู้ว่า ML มีสามแบบยังไม่พอ — ต้องรู้ว่าสถานการณ์แบบไหนควรใช้แบบไหน. ลองดูสามสถานการณ์จริงที่อธิบายความแตกต่างได้ชัดเจน.

**สถานการณ์ที่หนึ่ง — ระบบกรองความคิดเห็นลูกค้า**

บริษัทมี review ลูกค้า 500,000 ข้อความ และอยากให้ระบบแยกอัตโนมัติว่า review ไหน "บวก" ไหน "ลบ" ไหน "กลาง". เหตุผลที่ต้องการระบบนี้คือมีคนอ่าน review ไม่ทันเพื่อตอบสนองต่อลูกค้าที่ไม่พอใจอย่างรวดเร็ว.

นี่คือ **supervised learning** ชัดเจน. เหตุผล: รู้ว่า output ควรเป็นอะไร ("บวก/ลบ/กลาง") และสามารถสุ่ม review มาให้มนุษย์ label 10,000-20,000 ข้อความได้. แค่นั้นก็เพียงพอสำหรับ train model classifier ที่ทำงานได้ดีพอ.

ตัวอย่าง input: "สินค้ามาช้ามากและหีบห่อพัง". label: "ลบ".
ตัวอย่าง input: "บริการดีมากส่งเร็วกว่ากำหนด". label: "บวก".

**สถานการณ์ที่สอง — ค้นหา pattern ในข้อมูลผู้ป่วย**

โรงพยาบาลมีข้อมูลผู้ป่วย 100,000 ราย — อายุ, ผลเลือด, ยาที่ใช้, ประวัติโรค — และอยากรู้ว่ามีกลุ่มผู้ป่วยที่มีลักษณะคล้ายกันกลุ่มไหนบ้าง เพื่อพัฒนา treatment protocol ที่ตรงกลุ่มมากขึ้น.

นี่คือ **unsupervised learning** (clustering) เหตุผล: ไม่รู้ล่วงหน้าว่าควรมีกลุ่มไหนบ้าง ไม่มี label "กลุ่ม A" หรือ "กลุ่ม B" ให้. algorithm จะหาโครงสร้างที่มีอยู่จริงใน data ออกมา แล้วค่อยให้แพทย์ตีความว่าแต่ละกลุ่มคือใครและน่าจะตอบสนองต่อการรักษาแบบใด.

**สถานการณ์ที่สาม — สร้าง agent ที่เล่นเกม**

บริษัทเกมอยากสร้าง AI ที่เล่นเกมกลยุทธ์กับผู้เล่นได้ในระดับที่ท้าทาย โดยไม่มีคนเขียน "ท่าที่ดี" ไว้ให้ล่วงหน้า.

นี่คือ **reinforcement learning** เหตุผล: ไม่มี "คำตอบที่ถูก" สำหรับแต่ละท่า — รู้แค่ว่าชนะหรือแพ้ตอนจบเกม. AI เรียนรู้ว่าท่าไหนนำไปสู่การชนะในระยะยาวจากการเล่นหลายล้านเกม.

สังเกตความแตกต่างสำคัญระหว่างสามสถานการณ์: supervised learning ต้องการ label ที่ชัดเจนสำหรับทุก input ก่อนเริ่ม train, unsupervised learning ไม่ต้องการ label เลยแต่ก็ไม่รู้ว่า output ที่ถูก "ต้องเป็น" อะไร, reinforcement learning ไม่ต้องการ label per-step แต่ต้องการ environment ที่ agent ลงมือทำได้จริงและรับ feedback กลับมา.

| สถานการณ์ | Paradigm | เหตุผล |
| --- | --- | --- |
| วิเคราะห์ sentiment ของ review | Supervised | มี label (บวก/ลบ/กลาง) ที่กำหนดได้ชัด |
| หากลุ่มผู้ป่วยที่คล้ายกัน | Unsupervised | ไม่รู้ล่วงหน้าว่ากลุ่มไหนมีอยู่บ้าง |
| สร้าง AI เล่นเกมกลยุทธ์ | Reinforcement | feedback (ชนะ/แพ้) มาช้า ไม่มีคำตอบ per-step |

> **Key insight:** ในงานจริง ไม่ใช่ทุกปัญหามีคำตอบชัดเจนว่าต้องใช้แบบไหน. บางครั้งต้องผสมหลายแนวทาง — rule-based สำหรับ case ชัด + ML สำหรับ case ซับซ้อน + human review สำหรับ borderline ที่ ML ยังไม่มั่นใจ.

## กลไกที่ทำให้ ML ทำงานได้จริง

ถึงตอนนี้คุณรู้แล้วว่า ML "หา rule จาก data" แต่คำถามถัดมาคือ — **หายังไง?** algorithm ค้นหา pattern ยังไงกันแน่?

คำตอบสั้น ๆ คือ: ผ่าน training data, การวัด error, และการปรับ parameter ซ้ำ ๆ หลายล้านรอบ. รายละเอียดลึก ๆ อยู่ในบท[โมเดลเรียนรู้ยังไง](/foundations/how-models-learn) แต่ก่อนไปถึงตรงนั้น ต้องเข้าใจแนวคิดพื้นฐานสองอย่างก่อน: training/test split และ generalization.

### Training Data กับ Test Data

สมมติคุณ train model spam filter จาก email 100,000 ฉบับ และ model ทำได้ถูก 99% บน email ชุดนั้น — ฟังดูดีมาก แต่นั่นอาจหมายความว่า model แค่ "จำ" email ทั้ง 100,000 ฉบับนั้นได้ขึ้นใจ แล้วพอเจอ email ฉบับที่ 100,001 ที่ไม่เคยเห็น มันก็งงทันที.

นี่คือเหตุผลที่ใน ML เราต้องแบ่ง data ออกเป็นสองส่วน (หรือสามส่วน):

- **Training data** — data ที่ model เรียนรู้จาก (เช่น 70-80% ของทั้งหมด)
- **Test data** — data ที่เก็บไว้ วัด performance หลัง train เสร็จ โดยที่ model ไม่เคยเห็นตอน train (เช่น 20-30%)

model ที่ได้ 99% บน training data แต่ได้ 65% บน test data บอกอะไรบางอย่างที่ชัดเจนมาก: มัน "จำ" แต่ไม่ได้ "เรียนรู้."

ตัวเลขที่สำคัญคือ performance บน **test data** เสมอ.

### Generalization — เป้าหมายแท้จริงของ ML

เป้าหมายของ ML ไม่ใช่การทำได้ดีบน training data. เป้าหมายคือ **generalization** — ความสามารถทำได้ดีบน data ใหม่ที่ model ไม่เคยเห็นมาก่อน.

model ที่ generalize ดีคือ model ที่จับ pattern ที่เป็นจริงในโลก ไม่ใช่ pattern บังเอิญในชุด training data.

ตัวอย่าง: spam filter ที่ generalize ดีจะกรอง spam จากสแปมเมอร์ใหม่ที่ไม่เคยเห็นมาก่อนได้. spam filter ที่ generalize แย่จะทำได้เฉพาะกับสแปมเมอร์เดิม ๆ ที่เคยเห็นตอน train.

ลองนึกถึงแพทย์ที่วินิจฉัยโรคได้แม่นแม้เจอคนไข้หน้าตาใหม่ เพราะเข้าใจ mechanism ของโรค ไม่ใช่แค่จำ case ที่เคยเห็น. กับแพทย์ที่จำ case ได้มากแต่ไม่เข้าใจ mechanism — เจอ case ใหม่ก็สับสน. ML ที่ generalize ดีเหมือนแพทย์คนแรก — จับ pattern ที่จริง ไม่ใช่รายละเอียดสุ่มของตัวอย่าง.

analogy นี้ใช้ได้ดีแค่บางส่วน: แพทย์ "เข้าใจ" ในแบบที่ ML ไม่เคยเข้าใจ. ML แค่ minimize error บน training data ซ้ำ ๆ จนได้ parameter ที่ generalize ได้ดี.

> **Key insight:** ใน ML พูดถึง "accuracy" โดยไม่บอกว่าวัดบน training set หรือ test set ตัวเลขนั้นแทบไม่มีความหมาย. นักพัฒนาที่รายงาน training accuracy เพียงอย่างเดียว กำลังบอกตัวเลขที่อาจทำให้เข้าใจผิดโดยสิ้นเชิง.

### Overfitting — เมื่อโมเดลฉลาดเกินพอดี

**Overfitting** คือเมื่อ model จำ training data ได้ดีเกินไปจนเรียน noise และรายละเอียดสุ่มของชุดนั้นติดมาด้วย ทำให้ทำงานบน data ใหม่ได้แย่.

analogy ที่ดีที่สุด: นักเรียนที่ท่อง keyword ข้อสอบเก่าได้ครบทุกข้อ แต่ไม่เข้าใจเนื้อหา. เจอข้อสอบใหม่ที่เปลี่ยน wording นิดเดียวก็ทำไม่ได้. analogy นี้ใช้ได้ดีมาก แต่ข้อที่ต่างกันคือนักเรียนรู้ตัวว่าท่อง แต่ model ไม่รู้ตัว — มันแค่ minimize error บน training data โดยไม่รู้ว่ากำลัง overfit.

ตัวอย่างที่จับต้องได้: สมมติ train model ทำนายคะแนนสอบนักเรียนจาก feature 50 อย่าง — ชั่วโมงอ่านหนังสือ, ชั่วโมงนอน, จำนวนวิชา, **สีของกระเป๋า, วันเกิด, สีชอบ** ฯลฯ. ถ้า train บน data 30 คน model ขนาดใหญ่จะ "จำ" ได้ว่านักเรียนที่กระเป๋าสีฟ้าในชุดนั้นได้คะแนน 85 และใช้ pattern นั้นทำนายต่อ — ทั้งที่สีกระเป๋าไม่มีความสัมพันธ์กับคะแนนสอบจริง ๆ. ใช้กับนักเรียนคนที่ 31 ก็จะผิด.

อาการ: training accuracy สูง แต่ test accuracy ต่ำกว่าอย่างเห็นได้ชัด.

วิธีป้องกัน overfitting มีหลายทาง: เพิ่ม training data, ตัด feature ที่ไม่เกี่ยวข้องออก, ใช้ model ขนาดเล็กลง — หรือใช้เทคนิคที่เรียกว่า **regularization** ซึ่งจะอธิบายในบท[โมเดลเรียนรู้ยังไง](/foundations/how-models-learn).

**Underfitting** คือตรงข้าม — model เล็กหรือง่ายเกินไปจนจับ pattern ที่มีอยู่จริงไม่ได้. ทำได้แย่ทั้งบน training และ test data. เปรียบกับนักเรียนที่ขี้เกียจเกินไป ไม่ยอมเรียนอะไรเลย — ข้อสอบชุดไหนก็ผ่านไม่ได้. ในยุคที่ model ใหญ่ขึ้นเรื่อย ๆ อย่างปัจจุบัน ปัญหานี้พบน้อยกว่า overfitting มาก.

**ขั้นที่หนึ่ง — เตรียม data.** เก็บ, ทำความสะอาด, ติด label (ถ้าจำเป็น), แบ่งเป็น train/test. ขั้นนี้ใช้เวลามากที่สุดในงานจริง — บางโปรเจกต์ใช้เวลา 70-80% ของเวลาทั้งหมดอยู่ที่ขั้นนี้เพียงขั้นเดียว. ถ้า data ไม่ดี ไม่ว่าจะใช้ algorithm เก่งแค่ไหนก็ไม่ช่วย.

**ขั้นที่สอง — เลือก algorithm.** Linear model, decision tree, neural network, หรืออะไรก็ตามที่เหมาะกับปัญหาและปริมาณ data. ไม่มี algorithm ที่ดีที่สุดสำหรับทุกปัญหา — เลือกตามงาน.

**ขั้นที่สาม — Train.** ป้อน training data ให้ algorithm ปรับ parameter ภายใน ซ้ำ ๆ หลายรอบ. รายละเอียดว่าปรับยังไงอยู่ในบทถัดไป. ขั้นนี้ใช้ computing resource มากที่สุด — train LLM ขนาดใหญ่อาจใช้ GPU cluster หลายพันตัว รันหลายสัปดาห์ต่อเนื่อง.

**ขั้นที่สี่ — Evaluate.** วัด performance บน test data. ถ้าดีพอก็ไปต่อ ถ้าไม่ก็วนกลับไปขั้นต้น. การวนกลับอาจหมายถึงเก็บ data เพิ่ม, เปลี่ยน algorithm, หรือ tune hyperparameter.

**ขั้นที่ห้า — Deploy.** นำ model ไปใช้กับ data จริงในโลก. monitor performance ต่อเนื่อง เพราะ pattern ในโลกเปลี่ยน — model spam filter ที่ดีเยี่ยมในปี 2023 อาจแย่ลงในปี 2025 ถ้าสแปมเมอร์ปรับวิธีใหม่.

วงจรนี้ไม่ได้จบที่ขั้นห้า. ML เป็น iterative process — ไม่ใช่สร้างครั้งเดียวแล้วพักผ่อน. บริษัทใหญ่ ๆ re-train model สำคัญบ่อยครั้งมาก — บางทีทุกไม่กี่วัน.

### ทำไมขั้น "เตรียม data" ถึงใช้เวลา 70-80%

คนที่เพิ่งเริ่มเรียน ML มักคิดว่าขั้นที่ใช้เวลาและความพยายามมากที่สุดคือการเลือก algorithm หรือ tune model. ความจริงในงานจริงตรงข้ามกัน.

"Garbage in, garbage out" เป็นคำที่ใช้บ่อยในวงการ ML เพราะจริงมาก. ถ้า data ที่ป้อนเข้าไปมีปัญหา ไม่ว่าจะใช้ algorithm เก่งแค่ไหนก็ไม่ช่วย. และในโลกจริง data มักมีปัญหาเสมอ:

- **Missing values** — record บางอันไม่มีค่าในบาง field
- **Inconsistent format** — วันที่บางอันเป็น "01/05/2025" บางอันเป็น "1 พฤษภาคม 2568"
- **Labeling errors** — มนุษย์ที่ label data ไม่สม่ำเสมอ — คนหนึ่ง label "กลาง" คนอีกคน label "ลบ" สำหรับข้อความเดียวกัน
- **Class imbalance** — fraud detection ที่มี fraud แค่ 0.1% ของทั้งหมด model อาจ "เรียนรู้" ที่จะบอกว่าทุกอย่างปกติ และยังได้ 99.9% accuracy โดยไม่จับ fraud ได้เลย

การ clean และ validate data อย่างดีก่อน train คือสิ่งที่แยกระบบ ML ที่ work จากที่ไม่ work ในงานจริง.

### ลองทำดู

เปิด ChatGPT แล้วอธิบายสถานการณ์นี้:

"เพื่อนบอกว่า train model ทำนายราคาบ้านจาก data 100,000 หลัง และได้ accuracy 99% แต่พอ deploy ใช้งานจริงทำนายผิดบ่อยมาก ปัญหาคืออะไรได้บ้าง และควรตรวจสอบอะไร?"

ChatGPT ควรพูดถึง training vs test data, overfitting, และความเป็นไปได้ที่ distribution ของ data จริงต่างจาก training data. ถ้ามันอธิบายประเด็นเหล่านี้ได้ แสดงว่าคุณเข้าใจแนวคิดพื้นฐานของ ML แล้วในระดับที่ใช้งานได้จริง.

## ความเข้าใจผิดที่พบบ่อย — และทำไมถึงสำคัญ

หลายความเข้าใจผิดเกี่ยวกับ ML ไม่ใช่แค่เรื่องวิชาการ — มันส่งผลต่อการตัดสินใจจริง ๆ ว่าควรใช้ AI เมื่อไหร่และอย่างไร.

**"ML กับ AI คือสิ่งเดียวกัน"**

ไม่ใช่. ML เป็นส่วนหนึ่งของ AI. AI ครอบคลุมแนวทางอื่นด้วย เช่น rule-based system, search algorithm ในยุคเก่า. แต่เพราะ ML ครอบงำ AI เกือบทั้งหมดในทางปฏิบัติตั้งแต่ประมาณปี 2010 คนจึงใช้ทั้งสองคำแทนกันบ่อย ซึ่งพอรับได้ในบริบทสนทนาทั่วไป.

**"ML ทุกแบบใช้ neural network หรือ deep learning"**

ไม่จริง. Neural network เป็นแค่หนึ่งใน algorithm ของ ML. ในงาน production จริงจำนวนมาก — โดยเฉพาะงานที่ data เป็นตาราง (spreadsheet) ขนาดไม่ใหญ่มาก — algorithm แบบเก่าอย่าง random forest หรือ gradient boosting ยังทำงานได้ดีกว่า neural network บางครั้ง และยังเร็วกว่า ถูกกว่า และ debug ง่ายกว่าด้วย.

**"Model accuracy 99% บน training data คือ model ที่ดี"**

ถ้าไม่ได้วัดบน test data ตัวเลขนี้แทบไม่มีความหมาย. model ที่จำ training set ได้ 100% อาจ generalize ได้แย่มาก. นี่เป็นหนึ่งในข้อผิดพลาดที่พบบ่อยที่สุดเมื่อคนเริ่มใช้ ML.

**"ML เข้าใจสิ่งที่มันทำ"**

ไม่. ML หา pattern ทางสถิติที่ minimize error บน training data. มันไม่มี "ความเข้าใจ" ในแบบที่มนุษย์มี. นี่เป็นเหตุผลหนึ่งที่ทำให้ AI ล้มเหลวในลักษณะที่ดูไม่สมเหตุสมผล เมื่อเจอ input ที่ต่างไปจาก training data. spam filter ที่ "เข้าใจ" ว่า spam คืออะไรในความหมายมนุษย์จะไม่มีวันเกิดขึ้น — มันแค่รู้จัก pattern ที่เคยเห็น.

**"ยิ่งใช้ feature เยอะ ML ยิ่งดี"**

ไม่จริง. feature เกินจำเป็นเพิ่มความเสี่ยง overfitting โดยตรง เพราะ model อาจเรียนรู้ความสัมพันธ์ปลอมระหว่าง feature ที่ไม่เกี่ยวข้องกับ output จริง ๆ. การเลือก feature ที่เกี่ยวข้องจริง ๆ (feature engineering) สำคัญพอ ๆ กับการเลือก algorithm.

## Self-Supervised Learning — แบบที่ต้องรู้ในยุค LLM

มีแบบที่สี่ที่ไม่ค่อยถูกพูดถึงในตำราเก่า แต่สำคัญมากในยุคนี้: **self-supervised learning**.

แทนที่จะต้องการ label จากมนุษย์เหมือน supervised learning — self-supervised learning ใช้โครงสร้างของ data เองมาสร้าง label โดยอัตโนมัติ. ตัวอย่างที่ชัดที่สุดคือการ train LLM: เอาข้อความมาตัดคำท้ายออก ให้ model ทายคำที่หายไป แล้วใช้คำจริงในข้อความนั้นเองเป็นเฉลย. กลไกนี้เรียกว่า **next-token prediction** ซึ่งเราจะลงรายละเอียดเต็ม ๆ ในบท [LLM คืออะไร](/foundations/what-is-an-llm) ที่จะมาถึง.

ประเด็นสำคัญที่ต้องจับในตอนนี้: เพราะ text สร้าง label ให้ตัวเองได้ LLM จึงใช้ text จากอินเทอร์เน็ตหลายล้านล้านคำมา train ได้โดยไม่ต้องจ้างคน label — ถ้าต้อง label ด้วยมือ ปริมาณ data ขนาดนั้นไม่มีวันทำเสร็จในอายุมนุษย์. self-supervised จึงเป็นกุญแจที่ปลดล็อก LLM ยุคปัจจุบัน และยังใช้กับรูปภาพและเสียงได้ด้วย (ให้ model ทายส่วนที่ถูกบังหรือหายไป).

## ตัวอย่างเปรียบเทียบ Rule-based กับ ML จากงานจริง

เพื่อให้เห็นภาพชัดขึ้น ลองดูงานเดียวกันที่ทำด้วยสองวิธี: กรองข้อความที่มีเนื้อหาไม่เหมาะสมในระบบ comment ของแพลตฟอร์มออนไลน์.

### วิธีแบบ Traditional Programming

สมมติทีมพัฒนาเริ่มจากสิ่งที่ตรงไปตรงมาที่สุด:

**Version 1 — Block list ง่าย ๆ**

เก็บรายการคำที่ไม่เหมาะสม 200 คำ เขียน function ตรวจว่า comment มีคำในรายการนั้นหรือไม่. ถ้ามีก็ block.

ใช้ได้ดีช่วงแรก. แต่แล้ว:
- ผู้ใช้เริ่มเขียน "ใ@หม้" หรือ "ไ0หม" แทน สะกดผิดแต่อ่านออก
- ผู้ใช้ใช้คำเดียวกันในบริบทวิชาการหรือบทวิจารณ์หนังสือ — โดน block ทั้งที่ไม่ได้ตั้งใจไม่ดี
- สแปมเมอร์เปลี่ยนคำใหม่ทุกสัปดาห์

**Version 2 — rule ซับซ้อนขึ้น**

เพิ่ม rule หลายร้อยข้อ: pattern ของการสะกดผิดตั้งใจ, บริบทที่ควร allow, sender ที่ verified แล้ว ฯลฯ.

รายการ rule ยาวขึ้นเรื่อย ๆ จนใครก็ตามที่เข้ามาใหม่ในทีมไม่รู้ว่า rule ทั้งหมดมีอะไรบ้าง. ทุกครั้งที่แก้ rule ใหม่อาจทำให้ rule เดิมพัง. ความเร็วในการ update ช้าลงเรื่อย ๆ.

### วิธีแบบ Machine Learning

**Version ML — Classifier**

รวบรวม comment 100,000 ข้อความที่ moderator เคยตัดสินแล้ว — แต่ละข้อความมี label "ไม่เหมาะสม" หรือ "ปกติ".

train text classifier. model เรียนรู้ pattern เอง รวมถึง:
- การสะกดผิดตั้งใจ (มันเห็น pattern แบบนี้ใน training data ที่ label ว่าไม่เหมาะสม)
- บริบทของประโยค ไม่ใช่แค่คำเดียว
- combination ของคำที่รวมกันแล้วมีความหมายต่างไป

ข้อดีที่ได้จริง:
- จับ variation ได้กว่า: "ใ@หม้" ถูก catch เพราะมันเห็น pattern คล้ายกันใน training data
- บริบทช่วย: "ศึกษาเรื่องนี้ในด้านวิชาการ" ไม่โดน flag เพราะบริบทต่างกัน
- เมื่อ pattern ใหม่เกิดขึ้น แค่เพิ่ม labeled example ใหม่แล้ว re-train

ข้อเสียที่ต้องยอมรับ:
- ต้องการ labeled data จำนวนมาก (100,000 ข้อความมีต้นทุนการ label)
- บางครั้งผิดในแบบที่อธิบายยาก — "ทำไม comment นี้โดน block" ตอบได้ยากกว่า rule ธรรมดา
- ถ้า training data มี bias (เช่น moderator คนหนึ่ง bias ต่อคนบางกลุ่ม) model จะเรียนรู้ bias นั้นด้วย

**บทเรียน:** ระบบ production สมัยใหม่มักผสม — rule-based เป็นด่านแรกสำหรับ case ชัดเจน (blacklist คำหยาบชัด ๆ) + ML สำหรับ case กลาง ๆ + human review สำหรับ borderline.

## ML ล้มเหลวได้ยังไง — และคุณจะรู้ได้ยังไง

เข้าใจ ML แล้ว สิ่งที่สำคัญไม่แพ้กันคือรู้ว่า ML ล้มเหลวในแบบไหนได้บ้าง. เพราะถ้าคุณใช้ AI tools ในงาน การรู้ว่า output ผิดเมื่อไหร่สำคัญมาก.

**ล้มเหลวแบบที่หนึ่ง — Distribution Shift**

Model เรียนรู้จาก data ในช่วงเวลาหนึ่ง แต่ถูกใช้กับ data ในอีกช่วงเวลา. ถ้า pattern เปลี่ยนไป model ก็จะทำได้แย่ลงโดยที่ไม่มีใครสังเกตทันที.

ตัวอย่าง: fraud detection model ที่ train จาก data ก่อน COVID อาจทำงานแย่ลงหลัง COVID เพราะ พฤติกรรมการใช้เงินเปลี่ยนไปมาก. ธุรกรรมที่ "ผิดปกติ" ในปี 2019 (เช่น สั่งของออนไลน์จำนวนมาก) กลายเป็นปกติในปี 2021.

วิธีจับ: monitor performance ของ model บน data ใหม่ต่อเนื่อง ไม่ใช่แค่ตอน deploy.

**ล้มเหลวแบบที่สอง — Data Bias**

ถ้า training data มีความลำเอียง model ก็จะเรียนรู้ความลำเอียงนั้นไปด้วย. นี่ไม่ใช่ bug ที่ fix ได้ด้วยการแก้โค้ด — มันเป็นผลของ data ที่ reflect ความลำเอียงที่มีอยู่ในโลกจริง.

ตัวอย่างที่เกิดขึ้นจริง: ระบบ resume screening ของบริษัทหนึ่งเรียนรู้ว่าคนที่เข้ามาทำงานและทำได้ดีในอดีตส่วนใหญ่เป็นผู้ชาย — เพราะ historical data เป็นแบบนั้น. model เลยเรียนรู้ที่จะ prefer resume ผู้ชาย. ไม่ใช่เพราะ programmer ตั้งใจ แต่เพราะ data สะท้อน bias ที่มีอยู่ในโลกจริง.

อีกตัวอย่าง: medical AI ที่ train บน data ผู้ป่วยส่วนใหญ่เป็นกลุ่มชาติพันธุ์หนึ่ง อาจทำงานได้แย่กว่ากับกลุ่มอื่น เพราะ model ไม่ได้เห็น enough examples ของกลุ่มนั้นตอน train.

วิธีจับ: ตรวจสอบ output ของ model แยกตาม demographic group ว่า performance ต่างกันหรือเปล่า.

**ล้มเหลวแบบที่สาม — Adversarial Examples**

ใน input บางประเภท การเปลี่ยนแปลงเล็กน้อยที่มนุษย์แทบไม่สังเกต สามารถทำให้ model ผิดพลาดอย่างสิ้นเชิง.

ตัวอย่าง: รูปภาพของ stop sign ที่ถูกเพิ่ม sticker เล็กน้อยในจุดเฉพาะ — มนุษย์เห็นว่ายัง stop sign อยู่ แต่ image recognition model อาจ classify เป็น "speed limit 45". ในระบบรถยนต์ autonomous นี่คือความเสี่ยงที่จริงมาก.

อีกตัวอย่างที่ใกล้ตัวกว่า: prompt injection ใน LLM — การแอบซ่อน instruction ไว้ใน text ที่ดูเหมือน innocuous ทำให้ LLM ทำตามคำสั่งซ่อนนั้นแทนคำสั่งผู้ใช้จริง. นี่เป็นปัญหาด้าน security ที่ active research ยังทำอยู่ในปี 2026.

สาม pattern นี้บอกอะไร: ML ไม่ใช่ระบบที่ทำงานถูกต้องตลอดเวลา. มันต้องการการ monitor ต่อเนื่อง และต้องมีมนุษย์ในวง (human in the loop) สำหรับงานที่ผิดพลาดแล้วมีผลร้ายแรง.

สิ่งนี้ไม่ได้หมายความว่า ML ไม่ดี — มันหมายความว่า ML เป็นเครื่องมือที่ต้องใช้ด้วยความเข้าใจ. เหมือนมีดในครัว — มีดไม่ใช่สิ่งอันตราย แต่การใช้โดยไม่ระวังอันตราย. คนที่เข้าใจว่า ML ล้มเหลวได้ยังไงจะใช้มันได้ดีกว่า และรู้ว่าเมื่อไหร่ไม่ควรไว้วางใจ output ของมัน.

## ML กับชีวิตประจำวันของคุณในปี 2026

ถ้าคิดว่า ML เป็นเรื่องของวิศวกรหรือนักวิทยาศาสตร์ข้อมูลเท่านั้น — ลองดูว่ามันอยู่ที่ไหนในชีวิตคุณแล้วบ้าง:

**แอปที่คุณใช้ทุกวัน:**
- YouTube แนะนำวิดีโอถัดไปจาก ML ที่เรียนรู้พฤติกรรมการดูของคุณ — ใช้ supervised learning ที่ train จาก data ว่าคนที่ดูวิดีโอแบบ A มักดูวิดีโอแบบ B ต่อ
- Grab กำหนดราคาและจับคู่คนขับจาก ML ที่วิเคราะห์ traffic, อุปสงค์, และตำแหน่งแบบ real-time — ประมาณ ETA ด้วย supervised learning ที่ train จาก GPS ของผู้ขับหลายล้านคน
- Shopee/Lazada แนะนำสินค้าจาก ML ที่ดูประวัติการซื้อและค้นหา — ผสม supervised (predict ว่าสินค้าไหนน่าจะซื้อ) และ unsupervised (clustering ลูกค้าที่มีพฤติกรรมคล้ายกัน)
- Google Maps ประมาณเวลาเดินทางจาก ML ที่วิเคราะห์ข้อมูล GPS ของผู้ใช้หลายล้านคนพร้อมกัน — ไม่ได้ใช้กฎจราจรอย่างเดียว แต่เรียนรู้จาก pattern จริงว่าช่วงไหนถนนไหนติด

**งานที่คุณทำในออฟฟิศ:**
- ถ้าคุณอ่านอีเมล 50 ฉบับต่อวันแทนที่จะเป็น 200 ฉบับ — spam filter ML ช่วยคุณอยู่
- ถ้าคุณใช้ ChatGPT ช่วยเขียนอีเมลหรือสรุปเอกสาร — นั่นคือ ML (ในรูปแบบ LLM ที่ถูก train ด้วย self-supervised learning บน text หลายล้านล้านคำ)
- ถ้าคุณเคยได้รับ SMS แจ้งว่า "ธุรกรรมผิดปกติ" ก่อนที่คุณจะทำอะไร — fraud detection ML วิเคราะห์ pattern การใช้เงินของคุณและจับความผิดปกติออกมา
- ถ้าคุณพิมพ์ใน Google แล้ว Google รู้ว่าคุณกำลังจะพิมพ์อะไรต่อ — autocomplete ใช้ supervised learning ที่ train จาก data ว่าคนมักพิมพ์อะไรต่อ

สิ่งเหล่านี้เกิดขึ้นโดยที่คุณไม่ต้องรู้สึกตัว. ML เป็น infrastructure ที่อยู่เบื้องหลังซอฟต์แวร์สมัยใหม่เกือบทั้งหมด.

> **Key insight:** ทักษะที่สำคัญในยุคนี้ไม่ใช่การเขียน ML algorithm — แต่เป็นการรู้ว่าเมื่อไหร่ควรไว้ใจ output ของ ML เมื่อไหร่ควรตั้งคำถาม และเมื่อไหร่ต้องตรวจสอบด้วยตัวเองก่อนตัดสินใจ. คนที่รู้จัก ML ในระดับนี้ใช้ AI tools ได้ดีกว่าคนที่ไม่รู้มาก.

## คำถามที่คนมักถามเมื่อเริ่มเข้าใจ ML

เมื่อเข้าใจแนวคิดพื้นฐานแล้ว มักมีคำถามตามมาสองสามข้อที่น่าตอบไว้ก่อนไปต่อ.

**"ถ้า ML ต้องการ data มาก แล้ว startup เล็ก ๆ จะใช้ได้ไหม?"**

ใช้ได้ แต่ strategy ต่างกัน. ทางหนึ่งคือ fine-tune model ที่มีคนสร้างไว้แล้ว เช่น fine-tune LLM ด้วย data เฉพาะทาง แทนที่จะ train ตั้งแต่ต้น. อีกทางคือใช้ transfer learning — เอา model ที่ train บน data ขนาดใหญ่มาแล้ว แล้ว fine-tune บน data เล็กของตัวเอง. หรือเลือกปัญหาที่ต้องการ data น้อยกว่า — supervised learning สำหรับ task เฉพาะทางไม่จำเป็นต้องใช้ล้านตัวอย่างเสมอ ถ้า task ชัดเจนพอ.

**"ML ต้องเขียนโค้ดไหม?"**

ถ้าหมายถึงใช้ ML ในงาน — ไม่จำเป็น. ChatGPT, Claude, Gemini คือ ML ที่ใช้ผ่าน interface เว็บ ไม่ต้องเขียนโค้ดเลย. ถ้าหมายถึงสร้าง ML system ใหม่ตั้งแต่ต้น — ต้องใช้ Python และ library เช่น scikit-learn, PyTorch. แต่นั่นคือขั้นของ ML engineer/data scientist ซึ่งต่างจากการ "เข้าใจ ML" ที่บทนี้กำลังสร้าง.

**"ML กับ AI ที่กำลัง hype อยู่มีความสัมพันธ์กันยังไง?"**

AI ที่ hype ในปี 2024-2026 ส่วนใหญ่คือ LLM ซึ่งเป็น deep learning ซึ่งเป็น ML. ความ hype มาจากความสามารถของ LLM ที่ก้าวกระโดดขึ้นมาก — แต่รากฐานคือ ML แนวคิดเดิมที่มีมาตั้งแต่หลายสิบปีก่อน ประกอบกับ data มหาศาล, คอมพิวเตอร์ที่ทรงพลังกว่าเดิมมาก, และการค้นพบ architecture ใหม่ (transformer) ที่ทำให้ self-supervised learning บน text ทำงานได้ดีขึ้นอย่างก้าวกระโดด.

สิ่งที่ไม่เปลี่ยนแม้ AI จะก้าวหน้าแค่ไหน: แนวคิดพื้นฐานของ ML ที่อ่านในบทนี้ — data-in-rules-out, overfitting, generalization, training/test split — ยังเป็นรากฐานที่ทุก AI system ยึดอยู่. เข้าใจสิ่งเหล่านี้ คุณก็เข้าใจ AI ในระดับที่ลึกกว่าคนส่วนใหญ่แล้ว.

ความสามารถในการอ่านข่าว AI แล้ว evaluate ได้ว่า claim อะไรน่าเชื่อถือหรือ overhype เกินไป เป็นทักษะที่มีค่ามากในยุคที่ AI headlines เปลี่ยนทุกสัปดาห์. และมันเริ่มจากการเข้าใจพื้นฐานอย่างที่คุณเพิ่งอ่านมา.

## ทดสอบตัวเอง — อ่านเสร็จแล้วตอบได้ไหม

อ่านบทนี้จบแล้ว ลองทดสอบตัวเองด้วยคำถามสั้น ๆ. ไม่ต้องเขียนตอบ แค่คิดในหัวว่าจะตอบอะไร:

**คำถามที่หนึ่ง:** บอกความแตกต่างระหว่าง traditional programming กับ ML ในหนึ่งประโยค โดยใช้คำว่า "rule" และ "data" ทั้งคู่.

**คำถามที่สอง:** มีสามปัญหา: (ก) คำนวณเงินเดือนหักภาษี (ข) แยกรูปภาพสุนัขออกจากรูปภาพแมว (ค) หาว่า customer ของร้านแบ่งเป็นกลุ่มไหนได้บ้างโดยไม่รู้ล่วงหน้าว่ามีกี่กลุ่ม. แต่ละปัญหาควรใช้แนวทางไหน (traditional programming / supervised / unsupervised)?

**คำถามที่สาม:** model ที่ได้ training accuracy 98% แต่ test accuracy 60% มีปัญหาอะไร? และจะแก้ยังไงได้บ้าง?

**คำถามที่สี่:** ChatGPT ใช้ ML แบบไหน และทำไม self-supervised learning ถึงสำคัญมากสำหรับ LLM?

ถ้าตอบได้สี่ข้อ แสดงว่าคุณเข้าใจ ML ในระดับที่อ่านข่าว AI ได้รู้เรื่อง ใช้เครื่องมือ AI ได้อย่างมีวิจารณญาณ และสื่อสารกับทีมที่ทำงานกับ ML ได้แล้ว.

ถ้าตอบได้ไม่ครบ ไม่เป็นไร — ลองอ่านส่วนที่ไม่แน่ใจอีกครั้ง หรือลองถาม ChatGPT ให้อธิบายซ้ำด้วยตัวอย่างใหม่. ความเข้าใจ ML ไม่ได้เกิดจากการอ่านครั้งเดียว — มันเกิดจากการเห็นแนวคิดซ้ำ ๆ ในหลายบริบท.

และถ้าตอบได้ครบ บทถัดไปจะพาไปลึกกว่านี้อีกชั้นหนึ่ง — เปิดกลไกข้างในที่ทำให้ ML "เรียนรู้" ได้จริง.

## ลองทำดู — สำรวจ ML ใน ChatGPT แบบ hands-on

บทนี้ให้ทฤษฎีมามากพอสมควร ถึงเวลา ground มันกับของจริงบ้าง.

เปิด ChatGPT หรือ Claude แล้วลองทำตามลำดับนี้:

**ขั้นที่หนึ่ง:** พิมพ์ว่า "อธิบาย machine learning ให้เด็กอายุ 10 ขวบเข้าใจ" แล้วดูว่ามันอธิบายยังไง. เทียบกับที่คุณอ่านมา — มันพูดถึง supervised, unsupervised, reinforcement ไหม? พูดถึง "เรียนรู้จากตัวอย่าง" ไหม?

**ขั้นที่สอง:** ถามต่อว่า "แล้ว ChatGPT ตัวเองใช้ ML แบบไหน?" แล้วดูว่ามันรู้จัก self-supervised learning กับ RLHF ไหม.

**ขั้นที่สาม:** ถามคำถามนี้: "ถ้าผมอยากสร้าง spam filter สำหรับ email ภาษาไทยในองค์กร ควรเริ่มยังไง? ต้องการ data เท่าไหร่?" ดูว่ามันแนะนำ supervised learning และพูดถึง labeled data ไหม.

ถ้ามันตอบถูกต้องครบทั้งสาม แสดงว่าคุณเข้าใจ ML ในระดับที่ตรวจสอบ AI ได้แล้ว — ไม่ใช่แค่ใช้มันตาบอด.

## สรุป — สิ่งที่คุณควรพาออกไปจากบทนี้

Machine learning คือการพลิกสูตรโปรแกรมดั้งเดิม: แทนที่มนุษย์จะเขียน rule ให้คอมพิวเตอร์ทำตาม — ใน ML เราให้ตัวอย่างและคำตอบที่ถูกต้อง แล้วให้ algorithm ค้นหา rule เอง.

สูตรเดิมคือ Data + Rules → Answer.
สูตร ML คือ Data + Answers → Rules (model) ซึ่งนำมาใช้งานแบบ Model + New Data → Predicted Answers.

แนวคิดนี้อยู่เบื้องหลังเกือบทุก AI ที่คุณใช้ในชีวิตประจำวัน — ตั้งแต่ spam filter ใน Gmail ไปจนถึง ChatGPT ที่คุณคุยด้วย. ความแตกต่างจากโปรแกรมทั่วไปไม่ใช่แค่วิธีทำงาน แต่ยังเป็นประเภทของงานที่ทำได้ — งานที่มนุษย์ทำได้แต่ articulate rule ไม่ออก.

ML มีสามแบบหลักตามวิธีรับ feedback: supervised ที่เรียนจาก label ชัดเจน, unsupervised ที่หาโครงสร้างใน data เอง, และ reinforcement ที่เรียนจาก reward สะสมระยะยาว. บวกกับ self-supervised ที่สำคัญอย่างยิ่งในยุค LLM เพราะเป็นกลไกที่ทำให้ train บน text หลายล้านล้านคำได้โดยไม่ต้อง label ด้วยมือ.

generalization คือเป้าหมายที่แท้จริง — ไม่ใช่ทำได้ดีบน data ที่เคยเห็น แต่ทำได้ดีบน data ที่ยังไม่เคยเห็น. overfitting เป็นความเสี่ยงหลักที่ต้องระวัง. และ data quality มักสำคัญกว่า algorithm ที่เลือก.

ML ไม่ใช่ silver bullet. มีงานที่ทำได้ดีมาก และมีงานที่ if-else ธรรมดาเหมาะกว่า. และ model ทุกตัวมีโอกาสล้มเหลวจาก distribution shift, data bias, และ adversarial examples — ต้องมีมนุษย์ monitor ต่อเนื่องเสมอ.

คนที่ใช้ AI ได้ดีในยุคนี้ไม่ใช่คนที่ไว้ใจ AI ตาบอด และไม่ใช่คนที่ปฏิเสธ AI ตาบอดเช่นกัน — แต่เป็นคนที่รู้ว่า AI ทำงานยังไง เก่งเรื่องอะไร และพลาดได้ที่ไหน. นั่นคือสิ่งที่บทนี้สร้างขึ้นมาให้คุณ.

ทั้งหมดนี้ยังเป็นภาพรวมในระดับ "รู้ว่ามีอะไร." คำถามที่ค้างอยู่คือ: algorithm "ปรับ parameter" ยังไงกันแน่ในตอน training? มันรู้ได้ยังไงว่าควรเปลี่ยนค่าไหน เปลี่ยนเท่าไหร่?

นั่นคือสิ่งที่บทถัดไปจะเปิดกล่องให้คุณดู — กลไกที่เรียกว่า gradient descent ที่อยู่เบื้องหลัง "การเรียนรู้" ของ ML ทุกแบบ. เมื่อเข้าใจตรงนั้น คุณจะเข้าใจว่าทำไม LLM ถึงเก่งได้ในระดับที่เป็นอยู่ และยังมีขีดจำกัดอะไรที่เปลี่ยนได้ยากอยู่.
