# โมเดล AI เรียนรู้ได้ยังไง — เปิดกลไก training ทีละขั้น

ChatGPT รู้เรื่องกฎหมายภาษี รู้จักกวีนิพนธ์สมัยอยุธยา รู้ว่าสูตรก๋วยเตี๋ยวเรือแท้ ๆ ใช้เครื่องอะไร. ไม่มีทีมวิศวกรนั่งป้อนความรู้พวกนี้ให้ทีละข้อ.

แล้วมันรู้ได้ยังไง?

คำตอบสั้น ๆ คือ: มันไม่ได้ "รู้" — มันฝึก. และการฝึกของมันเป็นกระบวนการที่เข้าใจได้จริง ๆ ไม่ใช่เวทมนตร์. ถ้าคุณเข้าใจกลไกนี้ คุณจะรู้ว่าทำไม AI ถึงเก่งบางเรื่อง อ่อนบางเรื่อง และทำไมการสร้าง AI ตัวใหม่ถึงแพงเท่าการสร้างโรงไฟฟ้า.

บทนี้เป็นบทที่อาจฟังดูเทคนิคที่สุดในบรรดาบทแรก ๆ ของหลักสูตร. แต่ถ้าอ่านจบแล้วยังพอเล่าได้ว่า "training คืออะไร" ในภาษาของตัวเอง — คุณจะเข้าใจ AI ในระดับที่คนส่วนใหญ่ไม่เข้าใจ.

ถ้ายังไม่แน่ใจ ลองอ่านซ้ำแค่สองส่วน: "ขั้นที่ 1 — วัดว่าผิดมากแค่ไหน" และ "ขั้นที่ 2 — รู้ว่าควรปรับไปทางไหน". สองส่วนนั้นคือแกนกลางของทุกอย่าง. ส่วนที่เหลือเป็นรายละเอียดที่ขยายออกมา.

## ทำไมต้อง "ฝึก" ไม่ใช่ "โปรแกรม"

ลองนึกถึงโปรแกรมทั่วไป เช่นแอปคิดเงิน. นักพัฒนาเขียนกฎทุกข้อลงไปตรง ๆ: ถ้าซื้อ 3 ชิ้นลด 10%, ถ้าเป็นสมาชิกลดเพิ่ม 5% ฯลฯ. โปรแกรมทำตามกฎเป๊ะ ไม่มีการ "เรียนรู้" อะไรเพิ่ม.

ทีนี้ลองนึกว่าคุณอยากสร้างโปรแกรมที่ "แยกแยะแมวกับสุนัขในรูปภาพ". จะเขียนกฎยังไง? "ถ้ามีหู 2 ข้างรูปสามเหลี่ยม = แมว"? กฎแบบนั้นพังทันทีที่เจอสุนัขพันธุ์สปิตซ์. "ถ้ามีขนฟูสีส้ม = แมว"? ก็พังเพราะมีแมวสีดำ.

ยิ่งนึกกฎมากขึ้น ยิ่งเจอข้อยกเว้นมากขึ้น. เพราะรูปภาพแมวในโลกนี้มีหลายล้านแบบ ไม่มีกฎชุดไหนที่ครอบคลุมได้หมด.

งานบางประเภทเขียนกฎไม่ได้ — เพราะมีตัวแปรมากเกินไป. ต้องให้โปรแกรม "เรียนรู้" จากตัวอย่างจำนวนมากแทน. นั่นคือหัวใจของ **machine learning**: แทนที่จะเขียนกฎ ให้ข้อมูลตัวอย่างพร้อมคำตอบ แล้วให้โปรแกรมหากฎเอง โดยการปรับตัวเลขภายในซ้ำแล้วซ้ำเล่าจนทายถูก.

ตัวอย่างที่เห็นภาพที่สุด: เอารูปแมว 1 ล้านรูปกับรูปสุนัข 1 ล้านรูป บอกโมเดลว่าแต่ละรูปคืออะไร แล้วให้โมเดลปรับตัวเองจนแยกได้แม่นยำ 99%. โมเดลไม่ได้ "รู้" ว่าแมวมีหูแหลม — แต่มันพบว่า pattern ในตัวเลขของภาพ (ซึ่งเราเรียกกัน) "สัมพันธ์กับคำว่าแมว" จากการทายถูกมานับล้านครั้ง.

> 💡 **Key insight:** กระบวนการเรียนรู้ของ AI ไม่ใช่การ "ป้อนความรู้" แต่เป็นการ "ปรับตัวเลข". ทุกอย่างที่ ChatGPT รู้มาจากการปรับตัวเลขภายในซ้ำแล้วซ้ำเล่าจนทายถูกบ่อยขึ้น — ไม่มีใครนั่งสอนมันทีละเรื่อง.

## ตัวเลขภายในโมเดลคืออะไร

โมเดล AI ทุกตัว — รวมถึง [LLM](/foundations/what-is-an-llm) ที่เราจะเจาะลึกในบทท้าย ๆ ของ Level นี้ — แท้จริงแล้วคือสมการคณิตศาสตร์ขนาดใหญ่. สมการนั้นมีตัวเลขภายในเรียกว่า **parameter** (หรือเรียกอีกชื่อว่า weight).

ลองนึกภาพง่าย ๆ ก่อน: สมมติมีสมการที่ง่ายที่สุดเท่าที่เป็นไปได้ คือเส้นตรง `y = wx + b`.

- `x` คือข้อมูลที่ใส่เข้าไป
- `y` คือคำตอบที่ออกมา
- `w` และ `b` คือ parameter — ตัวเลขที่กำหนดว่าเส้นตรงนี้ชันแค่ไหน ตัดแกนที่ไหน

ถ้าเราอยากให้สมการนี้ทายน้ำหนักของคนจากส่วนสูง: `x` = ส่วนสูง (เซนติเมตร), `y` = น้ำหนัก (กิโลกรัม). แต่ตอนเริ่มต้น เราไม่รู้ว่า `w` ควรเป็นเท่าไหร่ หรือ `b` ควรเป็นเท่าไหร่.

กระบวนการ training คือการหาค่า `w` และ `b` ที่ดีที่สุด. ทำยังไง? ลองผิดลองถูก — แต่ลองแบบมีหลักการ ไม่ใช่สุ่มเดา.

ตอนนี้สมการนี้มีแค่ 2 parameter. LLM ก็ใช้หลักการเดียวกัน — แต่แทนที่จะมี parameter 2 ตัว มีหลายพันล้านตัว. หลักการเหมือนกันหมด ต่างกันแค่สเกล.

ทำไม LLM ต้องมี parameter มากขนาดนั้น? เพราะ "งาน" ที่ต้องทำซับซ้อนกว่าทายน้ำหนักจากส่วนสูงอย่างมาก. โมเดลต้องจำ pattern ของภาษาหลายภาษา ความรู้เรื่องต่าง ๆ นับล้านหัวข้อ ความสัมพันธ์ระหว่างแนวคิด และอีกมากมาย. parameter แต่ละตัวเก็บ "ความรู้" เล็ก ๆ ชิ้นหนึ่ง รวมกันหลายพันล้านชิ้นจึงได้ความสามารถที่เห็น.

เปรียบเทียบได้กับสมองมนุษย์ที่มีเซลล์ประสาทประมาณ 86,000 ล้านเซลล์ และ connection ระหว่างเซลล์มากกว่านั้นหลายเท่า — แต่เปรียบเทียบนี้ใช้ได้แค่ระดับนึง ความคล้ายกันระหว่าง neural network กับสมองจริงเป็นแค่อุปมาหลวม ๆ ไม่ใช่ความเหมือนกันในเชิงกลไก.

## ขั้นที่ 1 — วัดว่าผิดมากแค่ไหน

ก่อน train ได้ต้องตอบคำถามนี้ก่อน: "ผิด" วัดยังไง?

ถ้าไม่มีตัวเลขบอกว่าผิดแค่ไหน โมเดลก็ไม่รู้จะปรับไปทางไหน. เหมือนนักกีฬายิงธนูที่ปิดตาแล้วยิงไปเรื่อย ๆ โดยไม่มีใครบอกว่าลูกธนูตกที่ไหน — จะฝึกยังไงให้แม่นขึ้น?

นี่คือหน้าที่ของ **loss function** — ฟังก์ชันที่รับ "คำตอบที่โมเดลทาย" กับ "คำตอบที่ถูกต้องจริง ๆ" แล้วคืนตัวเลขตัวเดียว: **loss** หรือ "ค่าความผิดพลาด".

loss มากหมายถึงโมเดลทายได้แย่. loss น้อยหมายถึงโมเดลทายได้ดี. เป้าหมายของ training คือทำให้ loss ต่ำที่สุดเท่าที่ทำได้.

### ดูตัวอย่างจริง

สมมติเรากำลัง train โมเดลบนข้อมูล 3 จุด:
- x = 1, คำตอบจริง y = 2
- x = 2, คำตอบจริง y = 4
- x = 3, คำตอบจริง y = 7

ตอนเริ่มต้น สุ่มค่า parameter: `w = 0.5`, `b = 0`. โมเดลทายได้:
- x = 1 → ทาย `0.5 × 1 + 0 = 0.5` (คำตอบจริงคือ 2 ผิดไป 1.5)
- x = 2 → ทาย `0.5 × 2 + 0 = 1.0` (คำตอบจริงคือ 4 ผิดไป 3.0)
- x = 3 → ทาย `0.5 × 3 + 0 = 1.5` (คำตอบจริงคือ 7 ผิดไป 5.5)

ทำไมต้องยกกำลังสองก่อนบวก? เพราะถ้าบวกตรง ๆ ค่าผิดที่เป็นบวกกับลบจะหักล้างกัน. การยกกำลังสองทำให้ค่าผิดทุกตัวเป็นบวกเสมอ และยัง "ลงโทษ" ความผิดขนาดใหญ่หนักกว่าความผิดเล็ก ๆ.

loss = `(1.5² + 3.0² + 5.5²) ÷ 3 = (2.25 + 9 + 30.25) ÷ 3 ≈ 13.8`

เลขใหญ่มาก — แปลว่าแย่มาก. แต่ตอนนี้โมเดลมีตัวเลขบอกว่า "แย่แค่ไหน" แล้ว. นั่นคือจุดเริ่มต้นของการปรับปรุง.

loss function แบบนี้เรียกว่า **mean squared error (MSE)** — ใช้บ่อยกับงานทายตัวเลข. สำหรับงานแยกประเภท เช่น spam vs ไม่ spam หรือ LLM ที่ทายคำถัดไป จะใช้ loss function ที่ชื่อว่า **cross-entropy loss** — แนวคิดต่างออกไปเล็กน้อย แต่หลักการเดียวกัน: ตัวเลขเดียวที่บอกว่าโมเดลผิดมากแค่ไหน.

> 💡 **Key insight:** loss function คือการแปล "เป้าหมายของมนุษย์" ให้เป็น "ตัวเลขที่โมเดลเข้าใจ". ถ้าออกแบบ loss ผิด โมเดลก็จะเก่งในสิ่งผิด — เพราะมันทำแค่หนึ่งอย่าง คือลด loss ให้ต่ำที่สุด ไม่ว่า loss นั้นจะวัดอะไร. โมเดลไม่รู้ว่า "เป้าหมายจริง ๆ" ของมนุษย์คืออะไร — มันเห็นแค่เลข.

## ขั้นที่ 2 — รู้ว่าควรปรับไปทางไหน

ตอนนี้รู้แล้วว่า loss = 13.8. แต่จะปรับ `w` และ `b` ยังไง? เพิ่มหรือลด? เท่าไหร่?

ถ้าสุ่มปรับก็จะใช้เวลาเป็นล้านปี. ต้องมีวิธีที่ฉลาดกว่านั้น.

ที่นี่ต้องใช้แนวคิดที่ชื่อว่า **gradient** (อ่านว่า กราดิเอนต์).

gradient คือ "ความชัน" ของ loss เทียบกับ parameter แต่ละตัว. มันบอกว่า: "ถ้าเพิ่ม `w` ขึ้นนิดหนึ่ง loss จะเพิ่มหรือลด? มากแค่ไหน?"

### ภาพภูเขาในความมืด

นึกภาพแบบนี้:

คุณยืนอยู่บนภูเขาในตอนกลางคืน มองไม่เห็นอะไรเลย แต่คุณรู้สึกความลาดเอียงของพื้นใต้เท้าได้. เป้าหมายคือลงไปถึงหุบเขาข้างล่าง — จุดที่ต่ำที่สุด — เพราะในปัญหา training "ความสูง" คือ loss ยิ่งต่ำยิ่งดี.

วิธีที่สมเหตุสมผลที่สุดคือ: ทุกก้าว รู้สึกว่าพื้นลาดไปทางไหน แล้วเดินไปทางนั้น. ก้าวเล็ก ๆ ซ้ำ ๆ จะพาคุณลงไปถึงข้างล่าง.

นี่คือ **gradient descent** — "เดินลงตามความชัน".

<!-- IMAGE: ภาพภูเขา 3D ที่มีคน (หรือลูกบอล) กำลังเดินลงจากยอดเขาไปยังหุบเขา มีลูกศรเล็ก ๆ แสดงทิศทางของแต่ละก้าว และเส้นทางคดเคี้ยวที่ลงมาเรื่อย ๆ มีป้ายที่ยอดเขาเขียนว่า "เริ่มต้น — loss สูง" และป้ายที่หุบเขาเขียนว่า "เป้าหมาย — loss ต่ำ"
Style: illustration
Text on image: "เริ่มต้น (loss สูง)" / "เป้าหมาย (loss ต่ำ)" / "แต่ละก้าว = 1 การปรับ parameter"
-->

กลับมาที่ตัวอย่าง: หลังคำนวณ gradient แล้วพบว่า "เพิ่ม `w` ขึ้น loss จะลด" และ "เพิ่ม `b` ขึ้น loss ก็จะลด". โมเดลจึงปรับ:

- `w = 0.5 + (ก้าวเล็ก ๆ) ≈ 0.7`
- `b = 0 + (ก้าวเล็ก ๆ) ≈ 0.15`

คำนวณ loss ใหม่ — ลดลงแล้ว. ยังไม่ดีพอ แต่ดีกว่าก่อน. ทำซ้ำหลายพันรอบ แต่ละรอบ loss ลดลงทีละนิด จนในที่สุด `w` และ `b` ลู่เข้าค่าที่ดีที่สุด (สำหรับ data นี้ประมาณ `w ≈ 2.5`, `b ≈ -0.67`).

### ข้อจำกัดที่ต้องรู้ของ gradient descent

ภาพภูเขาใช้ได้ดีในการอธิบาย แต่มีจุดที่ต้องระวัง:

ในภูเขา 2 มิติ "จุดต่ำสุด" ชัดเจน. แต่ในโมเดลที่มี parameter พันล้านตัว ภูเขาอยู่ใน "มิติพันล้าน" — มีหลุมตื้น ๆ เต็มไปหมด บางหลุมดีบางหลุมแย่. gradient descent ไม่รับประกันว่าจะหาจุดต่ำสุดจริง ๆ ได้เสมอ — แค่หาจุดต่ำสุดที่ "ดีพอ" ในบริเวณที่เดินไปถึง.

ในทางปฏิบัติสำหรับ neural network ขนาดใหญ่ พบว่าจุดต่ำสุดที่หาได้มักดีพอสำหรับงานจริง — แม้จะไม่ใช่จุดต่ำสุดสากล.

## ขนาดก้าว — ใหญ่เกินก็พัง เล็กเกินก็ไม่ถึงไหน

มีตัวแปรสำคัญตัวหนึ่งที่ต้องตัดสินใจก่อน train: ก้าวแต่ละก้าวจะใหญ่แค่ไหน? ในวงการเรียกว่า **learning rate**.

ลองนึกถึงการยิงธนู: ถ้าปรับมุมครั้งละมาก ๆ คุณจะกระโดดข้ามเป้าจากฝั่งหนึ่งไปอีกฝั่งตลอด ไม่โดนเป้าสักที. ถ้าปรับครั้งละน้อยเกินไป ต้องยิงหลายพันครั้งกว่าจะโดน และยังเสี่ยง "ติด" อยู่ในตำแหน่งที่ดูเหมือนดีแต่ไม่ใช่ที่ดีที่สุด.

ผลของ learning rate แต่ละแบบ:

- **ใหญ่เกิน** → แต่ละก้าวกระโดดข้ามหุบเขาไปอีกฝั่ง โมเดลแกว่งไปมา loss ไม่ลดลงจริง
- **เล็กเกิน** → ใช้เวลาเป็นล้านล้านรอบกว่าจะถึงหุบเขา หรือ "ติด" อยู่ที่ลุ่มเล็ก ๆ ที่ไม่ใช่จุดต่ำสุดจริง
- **พอดี** → ลงเขาสม่ำเสมอ ถึงจุดดีในเวลาที่เหมาะสม

ในทางปฏิบัติ นักวิจัย AI ส่วนใหญ่ใช้ learning rate ที่ปรับตัวได้อัตโนมัติ — เริ่มใหญ่ตอนต้น ค่อย ๆ เล็กลงเมื่อโมเดลใกล้จุดต่ำสุด. algorithm ที่นิยมใช้ปัจจุบันชื่อ **Adam** ซึ่งทำสิ่งนี้อัตโนมัติโดยที่วิศวกรไม่ต้องปรับมือทุกก้าว.

Learning rate เป็นหนึ่งใน **hyperparameter** — ตัวเลขที่นักวิจัยกำหนดก่อน training เริ่ม แตกต่างจาก parameter ภายในโมเดลที่ปรับตัวเองระหว่าง training. hyperparameter อื่น ๆ ที่พบบ่อย ได้แก่ batch size, จำนวนชั้นของโมเดล, และขนาดของ embedding.

การเลือก hyperparameter ที่ดีเป็นทั้งศาสตร์และศิลป์ ส่วนหนึ่งมาจากประสบการณ์ ส่วนหนึ่งมาจากการทดลองหลายครั้ง (เรียกว่า hyperparameter search) และส่วนหนึ่งมาจากงานวิจัยสาธารณะที่บอกว่า "สำหรับโมเดลขนาดนี้ ค่าแถว ๆ นี้มักได้ผลดี".

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วลองพิมพ์ประโยคนี้:

_"ถ้าฉันเปิดร้านขายน้ำผลไม้ ราคาขายที่ดีที่สุดควรเป็นเท่าไหร่ ถ้าต้นทุนต่อแก้วคือ 15 บาท"_

AI จะตอบโดยประมาณการจากข้อมูลที่เคยเห็นมา ไม่ได้คำนวณตรง ๆ เหมือนเครื่องคิดเลข. ลองถามต่อว่า "คุณมั่นใจแค่ไหนในคำตอบนี้?" แล้วดูว่ามันตอบว่าอะไร — และลองเปรียบเทียบกับการถามคำถามเดิมในรูปแบบต่าง ๆ ดูว่าคำตอบเปลี่ยนไหม.

สิ่งที่เกิดขึ้นอยู่เบื้องหลัง: โมเดลไม่ได้ "คิด" เหมือนมนุษย์ แต่กำลังทำ pattern matching จาก training data — หาว่าในบริบทแบบนี้ คำตอบที่ "ดูสมเหตุสมผล" ตาม loss function ที่ถูกออกแบบมาคืออะไร.

## ขั้นที่ 3 — ทำซ้ำล้านครั้งด้วย data จำนวนมหาศาล

สองขั้นแรก (วัด loss, คำนวณ gradient) เป็นหลักการ. แต่ในทางปฏิบัติ training จริง ๆ ทำงานยังไง?

ถ้าโมเดลมี parameter พันล้านตัว และ training data มีข้อความหลายล้านล้านคำ — จะคำนวณ loss จาก data ทั้งหมดพร้อมกันในทุกก้าวไม่ได้ ต้องใช้ memory เป็นร้อยเท่าของที่มี. แม้แต่ datacenter ขนาดใหญ่ก็ทำไม่ได้.

วิธีแก้คือ: แบ่ง data ออกเป็นกลุ่มเล็ก ๆ เรียกว่า **batch** (เช่น 256 ตัวอย่างต่อครั้ง) แล้วคำนวณ gradient จาก batch นั้น ปรับ parameter ปรับเสร็จก็เดินหน้าไป batch ถัดไป.

### ศัพท์ที่เจอบ่อยในบทความ AI

ถ้าเคยอ่านบทความเกี่ยวกับการ train AI แล้วเจอคำพวกนี้ นี่คือความหมาย:

- **Batch** — กลุ่มข้อมูลเล็ก ๆ ที่ใช้คำนวณ gradient ในแต่ละก้าว (เช่น 256 หรือ 1,024 ตัวอย่าง)
- **Iteration หรือ Step** — 1 ครั้งของ "คำนวณ loss → คำนวณ gradient → ปรับ parameter" บน 1 batch
- **Epoch** — 1 รอบที่ผ่าน training data ทั้งหมดครบทุกตัวอย่าง

ตัวอย่างให้เห็นสเกล: ถ้า training data มี 10 ล้านตัวอย่าง และ batch size = 256 ก็จะมีประมาณ 39,000 iteration ต่อ 1 epoch. LLM ขนาดใหญ่อาจต้อง train หลาย epoch รวมกันเป็นหลายล้าน iteration.

ลองคิดเล่น ๆ: ถ้า 1 iteration ใช้เวลา 1 วินาที (ซึ่งในความเป็นจริงต้องใช้ GPU หลายพัน) — 1 ล้าน iteration ใช้เวลา 11 วัน. แต่ที่จริง iteration หนึ่งบน hardware สมัยใหม่ใช้เวลาเป็นมิลลิวินาที ทำให้ทำได้หลายล้าน iteration ต่อวัน.

ใน LLM pretraining training data คือข้อความที่ถูกตัดเป็น token — แต่ละ iteration โมเดลได้รับ sequence ของ token และพยายามทายว่า token ถัดไปคืออะไร. ถ้าทายถูก loss ต่ำ. ถ้าทายผิด loss สูง. ทำซ้ำกับ token ล้านล้านตัว — นั่นคือ pretraining.

นี่คือสาเหตุที่โมเดลรู้ว่า "สวัสดี" มักตามมาด้วยคำทักทาย รู้ว่า "กรุงเทพมหานคร" มักปรากฏในบริบทของเมืองหลวงประเทศไทย และรู้ว่าถ้าข้อความเริ่มต้นด้วย "ขอโทษ" มักตามมาด้วยคำขอโทษหรือคำอธิบาย — ทุกอย่างนี้มาจากการทายคำถัดไปซ้ำแล้วซ้ำเล่าจนเชี่ยวชาญ.

> 💡 **Key insight:** การใช้ batch สุ่มแทนที่จะใช้ data ทั้งหมดทุกครั้ง ไม่ได้เป็นแค่การ "ประหยัด memory". มันยังช่วยให้โมเดลไม่ติดอยู่ในจุดต่ำสุดที่ตื้น เพราะ noise ในแต่ละ batch ช่วยผลักโมเดลให้หลุดออกไปหาจุดต่ำสุดที่ดีกว่าได้. เป็นบั๊กที่กลายเป็น feature.

## กลไกที่ทำให้ปรับ parameter พันล้านตัวได้จริง

คุณอาจสงสัยว่า: ถ้าโมเดลมี parameter พันล้านตัว แต่ loss มีแค่ตัวเลขเดียว — จะรู้ได้ยังไงว่า parameter ตัวไหนต้องปรับ? และปรับทิศไหน?

ถ้าต้องทดลองปรับทีละตัว จะต้องทำ "ปรับ 1 ตัว → วัด loss → ปรับคืน → ปรับตัวถัดไป → วัด loss → ..." เป็นพันล้านครั้งต่อ 1 iteration — ช้าจนใช้งานไม่ได้แน่นอน.

คำตอบคือ: มี algorithm ที่ชื่อว่า **backpropagation** (เรียกย่อว่า backprop) ที่คำนวณ gradient ของ parameter ทุกตัวพร้อมกันในครั้งเดียว.

### Backprop ทำงานยังไง

Backprop ทำงานโดยเริ่มจาก loss ที่คำนวณได้ตอนท้าย แล้ว "ถอยหลัง" ผ่านโมเดลทีละชั้น โดยอาศัยคณิตศาสตร์เรื่อง chain rule (กฎลูกโซ่) — คำนวณว่า parameter แต่ละตัวในแต่ละชั้นมีส่วน "รับผิดชอบ" ต่อ error ตอนท้ายมากแค่ไหน.

อุปมาแบบไทย ๆ: เหมือนทีมฟุตบอลที่แพ้เกม. โค้ชต้อง "ถอยหลัง" วิเคราะห์ว่าใครรับผิดชอบอะไร:

- กองหน้ายิงพลาดตอนต้นเกม ทำให้ต้องรุกมาก ทำให้กองหลังเหนื่อย?
- กองกลางส่งบอลผิด ทำให้เกิดช่องว่าง?
- กองหลังพลาดทำให้โดนยิงประตู?

ทุกผู้เล่นมีส่วนรับผิดชอบในสัดส่วนต่างกัน โค้ชต้องประเมินให้ถูกก่อนจะบอกได้ว่าใครต้องฝึกอะไรเพิ่ม.

Backprop ทำเรื่องเดียวกันกับ parameter ทุกตัวในโมเดล — คำนวณส่วนรับผิดชอบของแต่ละตัวต่อ loss อย่างมีประสิทธิภาพ โดยใช้กฎลูกโซ่ของ calculus.

<!-- IMAGE: ภาพโมเดล neural network แบบง่าย มีชั้น 4 ชั้น มีลูกศร 2 ทิศทาง — ลูกศรไปข้างหน้า (forward pass) สีน้ำเงิน label ว่า "1. ส่งข้อมูลไปข้างหน้า คำนวณ loss" และลูกศรถอยหลัง (backward pass) สีส้ม label ว่า "2. ถอยหลัง แจกความรับผิดชอบ" มีป้ายที่จุดสุดท้ายเขียนว่า "loss" และป้ายที่ชั้นแรกเขียนว่า "parameter ถูกปรับที่นี่"

-->

ที่ต้องรู้ในระดับนี้แค่: **backprop คือสิ่งที่ทำให้ gradient descent ทำงานกับโมเดลที่มี parameter เป็นพันล้านตัวได้จริง** โดยไม่ต้องทดลองปรับทีละตัว.

## Data ที่ใช้สอน — ยิ่งเยอะไม่ได้แปลว่าดีเสมอ

กระบวนการ training ต้องการ 3 สิ่ง: โมเดล (สถาปัตยกรรม), data (ข้อมูลฝึก), และ compute (กำลังคำนวณ — GPU). ในบรรดาสามอย่างนี้ data มักเป็นสิ่งที่ถูกมองข้ามมากที่สุด เพราะฟังดูน่าตื่นเต้นน้อยกว่า.

ก่อนปี 2022 ความเชื่อกระแสหลักในวงการ AI คือ "ยิ่งข้อมูลเยอะยิ่งดี" เพราะโมเดลมีตัวอย่างให้เรียนมากขึ้น. แต่หลังจากนั้นงานวิจัยเริ่มแสดงหลักฐานว่า: **หลังจากจุดหนึ่ง คุณภาพของ data สำคัญกว่าปริมาณ**.

Microsoft ทดลองสร้างโมเดลชุด Phi — โมเดลขนาดเล็กที่ฝึกบน data คุณภาพสูงสไตล์ตำราเรียนแทนที่จะใช้ข้อความดิบจาก internet. ผลคือ Phi สู้โมเดลใหญ่กว่าหลายเท่าที่ฝึกบน data จำนวนมหาศาลแต่ดิบ ๆ ได้ใน benchmark หลายตัว.

ทำไม? เพราะโมเดลไม่ได้ "อ่านเข้าใจ" data — มันลด loss. ถ้า data เต็มไปด้วยข้อมูลผิด ซ้ำ ๆ หรือ bias — loss ที่โมเดลพยายามลดก็ถูก distort ตามไปด้วย.

นึกถึงการสอนเด็ก: ถ้าคุณสอนเด็กโดยใช้ตำราที่มีข้อมูลผิดครึ่งหนึ่ง เด็กก็จะเรียนทั้งถูกทั้งผิดปนกัน. AI ไม่ต่างกัน ยกเว้นมันไม่มีความสงสัยหรือตั้งคำถามว่า "ข้อมูลนี้สมเหตุสมผลไหม?" — มันแค่ลด loss.

### ผลที่ตามมาจาก data คุณภาพต่ำ

ทุกปัญหาที่คุณเห็นใน AI สามารถ trace กลับไปที่ data ได้:

- **ข้อมูลผิดใน data** → โมเดลตอบผิดมั่นใจ (ที่เรียกกันว่า hallucination — เพราะ training data มีคนเขียนข้อมูลผิดไว้ แล้วโมเดลก็เรียนรู้ว่าคำตอบที่ดูน่าเชื่อถือในบริบทนั้นคืออะไร)
- **ความเอนเอียงใน data** → ถ้า training data มีตัวอย่างของกลุ่มคนหนึ่งมากกว่าอีกกลุ่ม โมเดลก็จะ bias ตาม เพราะมันลด loss บน data ที่มีให้ ไม่ได้คิดว่า data ยุติธรรมหรือเปล่า
- **ข้อมูลซ้ำมากเกิน** → โมเดลอาจ "ท่อง" pattern นั้นแทนที่จะเรียนรู้ pattern ทั่วไป
- **Cutoff ของ data** → ข้อมูลที่เกิดขึ้นหลังจากที่โมเดลหยุด train โมเดลไม่รู้เลย — นั่นคือสาเหตุที่ ChatGPT ไม่รู้ข่าวเมื่อวานถ้าไม่ได้เปิด web search

> 💡 **Key insight:** โมเดลไม่รู้ว่าอะไรถูกหรือผิดในความหมายของมนุษย์ — มันรู้แค่อะไรทำให้ loss ต่ำ. ถ้า data ที่ใช้สอนมีปัญหา โมเดลที่ออกมาก็จะมีปัญหาตาม ไม่ว่าจะใช้ algorithm ดีแค่ไหน.

### ลองทำดู

ทดสอบขีดจำกัดของ data ที่โมเดลเคยเห็น:

เปิด ChatGPT แล้วถามเรื่องข่าวที่เกิดขึ้นเมื่อสัปดาห์ที่แล้ว — เลือกข่าวที่คุณรู้จริง ๆ ว่าเกิดอะไรขึ้น. แล้วลองถามเรื่องเดิมอีกครั้งโดยเปิด web search ด้วย (ไอคอน search ในหน้า ChatGPT).

ดูว่าคำตอบต่างกันยังไง. ตัวแรกตอบจาก training data เดิมที่ถูก freeze ไว้ ตัวที่สองค้นข้อมูลจาก internet จริง ๆ แล้วนำมาตอบ.

ลองถามว่า "ข้อมูลของคุณถูก cut off เมื่อไหร่?" แล้วดูว่าโมเดลรู้ตัวเองไหมว่า training data ของมันสิ้นสุดตอนไหน.

## ทำไม training LLM ถึงแพงเท่าสร้างโรงไฟฟ้า

ทุกอย่างที่คุณอ่านมาเป็นหลักการที่ใช้กับโมเดลทุกขนาด ตั้งแต่โมเดลที่รันบนโทรศัพท์มือถือไปจนถึง GPT-4. แต่ LLM ระดับ frontier มีขนาดที่ทำให้ตัวเลขพวกนี้กลายเป็นอะไรที่ยากจะจินตนาการ.

ลองดูว่าทุกอย่างรวมกันยังไง:

**Parameter จำนวนมหาศาล.** LLM ระดับ frontier มี parameter ตั้งแต่หลักหมื่นล้านถึงล้านล้านตัว. ทุก iteration ต้องคำนวณ gradient ของ parameter ทุกตัว ทำ backprop ผ่านทุกชั้น.

**Training data ขนาดมหึมา.** GPT-4 ถูก pretrain บนข้อความหลายล้านล้านคำ (trillions of tokens) — มากกว่าที่มนุษย์คนหนึ่งจะอ่านได้ในชีวิตหลายพันชาติรวมกัน. การผ่านข้อมูลทั้งหมดนี้แม้แค่รอบเดียวก็มี iteration นับล้าน.

**GPU หลายพันตัวทำงานพร้อมกัน.** Iteration เดียวต้องกระจายงานข้าม GPU/TPU หลายพันตัวพร้อมกัน ผ่าน network ความเร็วสูงพิเศษที่ออกแบบมาเฉพาะ. การประสานงานระดับนี้เป็นปัญหาวิศวกรรมที่ซับซ้อนมากในตัวเอง นอกจากเรื่องของ algorithm.

**ใช้เวลาเป็นเดือน.** แม้จะมี GPU หลายพันตัว training รอบหนึ่งใช้เวลาหลายสัปดาห์ถึงหลายเดือน.

รวมทุกอย่างแล้ว การ pretrain LLM ระดับ frontier รอบหนึ่งใช้เงินในหลักร้อยล้านดอลลาร์. นี่คือเหตุผลที่การสร้าง LLM ใหม่ตั้งแต่ต้น (pretraining) กระจุกอยู่ในไม่กี่บริษัท — OpenAI, Anthropic, Google DeepMind, Meta — เพราะคนอื่นไม่มี compute capacity ระดับนั้น.

ส่วน fine-tuning (ปรับโมเดลที่ train มาแล้วให้เก่งงานเฉพาะ) ต้นทุนต่างออกไปมาก — นักวิจัยในมหาวิทยาลัยหรือ startup เล็ก ๆ ทำได้. ถ้าบริษัทของคุณอยากสร้าง AI ที่รู้เรื่องเฉพาะของบริษัท เช่นรู้เรื่อง process ภายใน รู้เรื่อง product catalog หรือตอบภาษาและ tone ที่บริษัทต้องการ — ทางเลือกที่ประหยัดกว่าคือ fine-tuning บน base model ที่มีอยู่แล้ว หรือใช้ RAG (Retrieval Augmented Generation) ซึ่งเป็นเทคนิคอื่น. ทั้งหมดนี้จะได้เรียนในบทที่ว่าด้วยการใช้ AI ในองค์กร.

## สิ่งที่ผิดพลาดระหว่าง training ได้

Training ไม่ได้ราบรื่นเสมอ. แม้แต่ lab ระดับ frontier ที่มีวิศวกรหลายร้อยคนก็เจอปัญหาระหว่าง train. รู้จักปัญหาเหล่านี้ช่วยให้เข้าใจว่าทำไม AI บางตัวถึงทำงานแปลก ๆ.

**Overfitting — จำแทนที่จะเรียน.** บางครั้งโมเดลลด loss บน training data ได้ดีมาก แต่พอเจอข้อมูลใหม่ที่ไม่เคยเห็น กลับตอบได้แย่. เหมือนนักเรียนที่ท่องเฉลยข้อสอบเก่าแต่ทำข้อสอบข้อใหม่ไม่ได้ เพราะจำ "คำตอบ" ไม่ได้จำ "วิธีคิด".

**Training instability — โมเดลพังกลางทาง.** บางครั้ง loss ที่กำลังลดอยู่ดี ๆ กระโดดขึ้นสูงกะทันหัน หรือโมเดลให้ output ที่ไร้ความหมายไปเลย. นี่คือเหตุผลที่ lab ขนาดใหญ่ต้อง monitor training ตลอดเวลา มี dashboard ดู loss แบบ real-time และเซฟ checkpoint ทุก ๆ หลายชั่วโมง เพื่อ rollback ได้ถ้าพัง.

**ความเอนเอียงจาก data.** ดังที่พูดไปแล้ว ถ้า data ไม่สมดุล โมเดลก็จะ bias ตาม.

**Vanishing gradient — สัญญาณหายไปก่อนถึงชั้นต้น.** ในโมเดลที่มีหลายชั้นมาก การ backprop ย้อนกลับไปหลาย ๆ ชั้นอาจทำให้ gradient เล็กลงเรื่อย ๆ จนชั้นต้น ๆ แทบไม่ได้รับสัญญาณว่าต้องปรับอะไร. เหมือนโทรศัพท์สายเดิมที่เสียงเบาลงทุกครั้งที่ผ่านตู้สาขา — ปลายสายแทบไม่ได้ยินอะไร.

ปัญหานี้ชื่อ **vanishing gradient** และเป็นหนึ่งในเหตุผลที่สถาปัตยกรรมโมเดลสมัยใหม่มีเทคนิคพิเศษอย่าง residual connections (ทางลัดที่ให้สัญญาณข้ามชั้นได้โดยตรง) และ layer normalization. Transformer ซึ่งเป็นสถาปัตยกรรมหลักของ LLM สมัยใหม่ได้รับการออกแบบมาให้จัดการปัญหานี้ได้ดีกว่าสถาปัตยกรรมรุ่นก่อน — เรื่องนี้จะได้เรียนรายละเอียดในบท Deep Learning และ Neural Network ที่จะตามมา.

> 💡 **Key insight:** โมเดล AI ที่ตอบแปลก ๆ หรือมี bias ไม่ใช่เพราะ "โมเดลชั่วร้าย" หรือ "ตั้งใจโกหก" — แต่เพราะกระบวนการ training มีข้อจำกัด หรือ data ที่ใช้สอนมีปัญหา. ทุกพฤติกรรมของโมเดลสามารถ trace กลับไปหากระบวนการ training ได้เสมอ.

## Training ไม่ได้เกิดครั้งเดียว

ความเข้าใจผิดที่พบบ่อยคือคิดว่า "AI ถูกสร้างขึ้นครั้งเดียวแล้วก็ใช้ได้เลย" เหมือนเครื่องจักรที่ประกอบเสร็จแล้วส่งออกโรงงาน. จริง ๆ แล้ว ChatGPT หรือ Claude ที่คุณใช้อยู่เป็นผลของ training หลาย stage ต่อกัน.

### Stage 1 — Pretraining: เรียนจากทุกอย่างที่เขียนบนอินเทอร์เน็ต

โมเดลฝึกบนข้อความจำนวนมหาศาลจาก internet — หนังสือ บทความ โค้ด Wikipedia บทสนทนา ฯลฯ. เป้าหมายเดียวคือ: "ทายคำถัดไปให้ถูก". loss function วัดว่าทายผิดมากแค่ไหน.

ขั้นนี้แพงที่สุดและใช้เวลานานที่สุด. ผลลัพธ์คือโมเดลที่รู้เรื่องราวต่าง ๆ มากมาย แต่ยังไม่ได้ "ดี" ในแง่ของการเป็น assistant ที่มีประโยชน์ — มันอาจตอบแบบ stream of consciousness ยาว ๆ ไม่ตรงประเด็น หรือเขียนเรื่องอันตรายโดยไม่ลังเล.

### Stage 2 — Fine-tuning: ฝึกให้เป็น assistant

นำโมเดลจาก Stage 1 มาฝึกต่อบน data ที่เฉพาะเจาะจงกว่า — เช่น ตัวอย่างบทสนทนาระหว่างมนุษย์กับ AI ที่มีคุณภาพ ที่มนุษย์เขียนขึ้นว่า "ถ้าถามแบบนี้ คำตอบที่ดีคืออะไร". ต้นทุนต่ำกว่า Stage 1 มาก.

หลังจาก fine-tuning โมเดลเริ่มตอบตรงประเด็นขึ้น รู้ว่าควรถามกลับเมื่อคำถามไม่ชัด และเริ่มมี "บุคลิก" ของ assistant.

### Stage 3 — Alignment: ทำให้ปลอดภัยและมีประโยชน์

ปรับโมเดลให้ตอบในแบบที่มนุษย์ถือว่า "ดี" — ไม่พูดเรื่องอันตราย ตอบตรงประเด็น ยอมรับเมื่อไม่รู้ ไม่แต่งเรื่อง. ขั้นนี้ใช้ feedback จากมนุษย์จริง ๆ เป็น signal ในการ train.

กระบวนการที่นิยมใช้ชื่อว่า **RLHF** (Reinforcement Learning from Human Feedback) หรือ **DPO** (Direct Preference Optimization) — มนุษย์เปรียบเทียบคำตอบสองแบบแล้วบอกว่าอันไหนดีกว่า โมเดลเรียนรู้จาก preference นั้น.

โมเดลที่คุณคุยด้วยทุกวันเป็นผลของทั้ง 3 stage รวมกัน.

## ตัวอย่างที่ 1 — สอนให้กรองสแปม

ลองดูกระบวนการ training ในตัวอย่างที่จับต้องได้กว่า: สอนโมเดลให้แยกแยะว่า email ไหนสแปม ไหนไม่ใช่.

**สิ่งที่ต้องเตรียม:**

- Training data: email 1 ล้านฉบับที่มีคนช่วย label ไว้แล้วว่า "สแปม" หรือ "ไม่สแปม"
- โมเดล: neural network เล็ก ๆ ที่รับ feature ของ email (เช่น คำที่ปรากฏ ความถี่ของคำ ชื่อผู้ส่ง) แล้วออก probability ว่าเป็นสแปมแค่ไหน
- Loss function: วัดว่าเมื่อโมเดลตอบผิดและมั่นใจสูง ต้องถูกลงโทษหนัก (เรียกว่า cross-entropy loss)

**กระบวนการ:**

รอบที่ 1: สุ่ม parameter ตั้งต้น. โมเดลทายว่า email ทุกฉบับไม่ใช่สแปม (เพราะไม่รู้อะไรเลย). loss สูงมาก เพราะสแปม 400,000 ฉบับถูกทายผิดหมด.

รอบที่ 2-100: gradient descent ปรับ parameter. โมเดลเริ่มสังเกตว่าคำว่า "winner", "click here", "limited offer" ปรากฏบ่อยใน email ที่ label ว่าสแปม. parameter ที่เกี่ยวกับคำพวกนี้ถูกปรับให้ "เพิ่มน้ำหนักไปทาง spam" มากขึ้น.

หลังหลายพัน epoch: โมเดลแยกสแปมได้ถูกต้อง 95%+. parameter ภายในถูกปรับให้คำที่บ่งบอกสแปมเพิ่มน้ำหนักไปทาง spam และคำธรรมดาเป็นกลาง.

สิ่งที่สำคัญ: โมเดลไม่ได้ "เข้าใจ" ว่าทำไม email แบบนั้นถึงเป็นสแปม — มันแค่หา pattern จาก feature ที่ลด loss ได้ดีที่สุด. ถ้า training data มีสแปมภาษาไทยน้อยมาก โมเดลก็จะกรองสแปมภาษาไทยได้แย่กว่าภาษาอังกฤษ — เพราะมันไม่เคยเห็นตัวอย่างเพียงพอ.

## ตัวอย่างที่ 2 — LLM เรียนรู้ภาษาไทยยังไง

LLM อย่าง ChatGPT เรียนรู้ภาษาไทยได้ยังไง ทั้งที่ทีมสร้างส่วนใหญ่อยู่ในอเมริกาและไม่ได้ตั้งใจสอนภาษาไทยโดยตรง?

คำตอบคืออยู่ใน training data. เมื่อ pretraining data มีข้อความภาษาไทยจากอินเทอร์เน็ต — Wikipedia ภาษาไทย, เว็บไซต์ข่าวไทย, เว็บบอร์ด Pantip, บทความต่าง ๆ — โมเดลก็เรียนรู้ pattern ของภาษาไทยโดยอัตโนมัติ ผ่านการพยายามทายคำถัดไปใน text ภาษาไทยซ้ำแล้วซ้ำเล่า.

loss function เดียวกัน — "ทายคำถัดไปให้ถูก" — ทำงานเหมือนกันกับทุกภาษาในชุด training data.

แต่ภาษาไทยมีน้อยกว่าภาษาอังกฤษในอินเทอร์เน็ตมาก. ผลคือ:

- โมเดลเห็นตัวอย่างภาษาอังกฤษมากกว่าภาษาไทยนับร้อยเท่า
- parameter ถูก "ปรับให้เก่งภาษาอังกฤษ" มากกว่า
- เวลาถามเรื่องที่มีข้อมูลไทยน้อย โมเดลอาจตอบได้แย่กว่า หรือตอบภาษาไทยแต่ใช้ความรู้จากภาษาอังกฤษมา "แปล"

นี่คืออีกตัวอย่างที่แสดงว่า คุณภาพและปริมาณของ data มีผลโดยตรงต่อสิ่งที่โมเดลทำได้.

## ภูมิทัศน์ของ loss — ทำไมไม่มี "คำตอบที่ถูกเสมอ"

กลับไปที่ภาพภูเขาสักครั้ง. ในตัวอย่างง่าย ๆ อย่างเส้นตรง `y = wx + b` ที่มี parameter 2 ตัว "ภูเขา" เป็นแค่พื้นผิว 3 มิติ — ค่อนข้างราบเรียบ มีหุบเขาชัดเจน.

แต่ LLM ที่มี parameter หลายพันล้านตัว ภูมิทัศน์นั้นอยู่ใน "มิติพันล้าน". ไม่มีมนุษย์คนไหนจินตนาการออก และคณิตศาสตร์เกี่ยวกับมันก็ยังเป็นหัวข้อวิจัยที่ไม่ได้ข้อสรุปทุกอย่าง.

สิ่งที่รู้แน่ ๆ คือ:

**มีหุบเขาหลายแห่ง ไม่ใช่แห่งเดียว.** ในสมการที่ซับซ้อน มีหลายค่าของ parameter ที่ให้ loss ต่ำพอ ๆ กัน. โมเดลสองตัวที่ train ด้วย data เดียวกันแต่เริ่มต้น parameter ต่างกัน อาจลงเอยที่หุบเขาคนละแห่ง และให้คำตอบที่ต่างกันเล็กน้อยในบางกรณี.

**ไม่ต้องถึงจุดต่ำสุดสากล.** งานวิจัยพบว่าสำหรับ neural network ขนาดใหญ่ "จุดดีพอ" มักเพียงพอ — ไม่ต้องหาจุดต่ำสุดจริง ๆ ของ loss landscape ทั้งหมด. และในมิติสูงมาก local minimum ที่แย่มักพบได้น้อยกว่าที่คาด.

**Noise ใน training ช่วยได้.** การใช้ mini-batch สุ่ม ทำให้แต่ละก้าวมี noise ตามมา noise นี้บางครั้งช่วยให้โมเดลหลุดออกจาก local minimum ที่ตื้น ๆ และหาจุดที่ดีกว่าได้ — เป็นตัวอย่างที่ "ข้อผิดพลาด" กลายเป็นข้อดี.

> 💡 **Key insight:** ไม่มี "โมเดลที่ถูกต้องที่สุด" สำหรับ task ใดก็ตาม — มีแต่โมเดลที่ "ดีพอสำหรับงานที่ต้องการ" กระบวนการ training เป็นการค้นหาในพื้นที่ที่ใหญ่โตและซับซ้อนเกินจะเข้าใจได้ทั้งหมด.

## ลองทำดู — สำรวจ loss ด้วยตัวเอง

ไม่ต้องเขียนโค้ด ไม่ต้องติดตั้งอะไร. ลองทดลองสังเกต loss function ผ่าน AI โดยตรง:

เปิด ChatGPT หรือ Claude แล้วทดลองพิมพ์ข้อความที่ขึ้นต้นแบบนี้ทีละแบบ:

1. "วันนี้อากาศ..."
2. "สวัสดีปีใหม่ ขอให้..."
3. "ราคา iPhone ล่าสุดคือ..."
4. "ใครเขียนนิยายเรื่อง สี่แผ่นดิน คือ..."

ดูว่า AI ต่อประโยคให้ยังไง. สังเกตว่าประโยคไหนมันต่อได้ "ชัวร์" ประโยคไหนมันลังเลหรือบอกว่าไม่แน่ใจ.

สิ่งที่คุณกำลังสังเกตคือ: ในบริบทที่ training data มีตัวอย่างชัดเจนมาก (เช่นสำนวนไทยที่ใช้บ่อย) โมเดลจะต่อได้แน่น. ในบริบทที่ข้อมูลน้อยหรือเปลี่ยนเร็ว (เช่นราคาสินค้าปัจจุบัน) โมเดลจะไม่แน่ใจหรืออาจตอบผิด.

นั่นคือ loss landscape ที่คุณกำลัง "สำรวจ" อยู่ผ่านหน้าต่างบทสนทนา.

## ความเชื่อมโยงกับสิ่งที่คุณเห็นในชีวิตประจำวัน

ทุกพฤติกรรมของ AI ที่คุณเจอในชีวิตจริงมีรากมาจากกลไก training ที่อธิบายไปทั้งหมด. ลองเชื่อมดู:

**ทำไม ChatGPT บางครั้งบอกว่า "ขอโทษ ฉันไม่รู้เรื่องนี้"**

เพราะ fine-tuning และ RLHF (Stage 2, Stage 3) ได้สอนให้โมเดล "ยอมรับเมื่อไม่แน่ใจ" มีตัวอย่างในชุด training ว่า "เมื่อ user ถามเรื่องที่ไม่รู้ คำตอบที่ดีคือบอกว่าไม่รู้ ไม่ใช่แต่งขึ้นมา" และ RLHF ให้คะแนนสูงกับพฤติกรรมนี้.

**ทำไม AI บางครั้งตอบอย่างมั่นใจแต่ผิด (hallucination)**

เพราะ training ในระดับ pretraining สอนแค่ "ทายคำถัดไปที่ดูน่าเชื่อถือที่สุด" — ไม่ได้สอนว่า "ตรวจสอบว่าข้อเท็จจริงก่อนตอบ". loss function ของ pretraining ไม่ได้วัด "ความถูกต้องของข้อเท็จจริง" — วัดแค่ "ความน่าจะเป็นของคำถัดไป". โมเดลที่ดูเหมือนมั่นใจคือโมเดลที่ความน่าจะเป็นของคำถัดไปมีตัวเดียวชนะขาด.

**ทำไม AI ตอบได้ดีขึ้นเมื่อถามเป็นภาษาอังกฤษ**

เพราะ training data มีภาษาอังกฤษมากกว่าภาษาอื่นอย่างมีนัยสำคัญ. parameter ถูกปรับให้ "เชี่ยวชาญ" ภาษาอังกฤษมากกว่าในสัดส่วนที่ตรงกับ data ที่เห็น.

**ทำไม AI รุ่นใหม่ (GPT-4o, Claude 4) ดีกว่ารุ่นเก่า**

เพราะ training recipe ดีขึ้น data คุณภาพสูงขึ้น และมี alignment technique ที่ซับซ้อนขึ้น — ไม่ใช่แค่ "โมเดลใหญ่ขึ้น" เสมอไป.

## ความเข้าใจผิดที่พบบ่อย — เคลียร์ก่อนไปต่อ

หลาย ๆ คำถามที่คนมักถามเกี่ยวกับ AI มาจากความเข้าใจผิดเรื่อง training. เรื่องพวกนี้ไม่ใช่เรื่องเล็กน้อย — ความเข้าใจผิดในระดับนี้ทำให้คนใช้ AI ผิดวัตถุประสงค์ คาดหวังผิด หรือไว้ใจมากเกินไปในสถานการณ์ที่ไม่ควร. ขอเคลียร์ 4 ข้อที่พบบ่อยที่สุด:

**"AI เรียนรู้แบบเดียวกับมนุษย์"** — ไม่ใช่. มนุษย์เรียนรู้จากตัวอย่างไม่กี่ตัวอย่าง เข้าใจแนวคิด และ generalize ได้กว้าง. โมเดล AI ต้องการตัวอย่างหลายพันล้านตัวอย่าง และกระบวนการทั้งหมดเป็น optimization ของตัวเลข — ไม่มีการ "เข้าใจ" ในแบบที่มนุษย์เข้าใจ. เด็กอายุ 2 ขวบเห็นแมว 5 ตัวก็รู้จักแมวทุกตัวในโลก แต่โมเดล AI ต้องเห็นภาพแมวหลายล้านภาพ — และยังอาจผิดพลาดเมื่อเจอแมวในมุมแปลก ๆ ที่ไม่เคยเห็น.

**"ยิ่งโมเดลใหญ่และข้อมูลเยอะ ผลก็ดีเสมอ"** — มีความสัมพันธ์แบบนั้นอยู่จริง แต่ไม่ใช่เสมอไป. งานวิจัยเรื่อง Chinchilla scaling laws แสดงว่าโมเดลหลายตัวที่ปล่อยก่อนปี 2022 ใหญ่เกินไปเมื่อเทียบกับ data ที่ใช้ train — อีกนัยคือควรใช้ compute เพิ่ม data ให้มากขึ้นแทนที่จะเพิ่ม parameter. ROI ของการเพิ่มขนาดลดลงเรื่อย ๆ และ data quality รวมถึง training recipe มีผลเทียบเท่า scale.

**"Training คือเหตุการณ์ครั้งเดียว"** — อย่างที่อธิบายไปแล้ว ChatGPT หรือ Claude ที่คุณใช้เป็นผลของหลาย stage. และแต่ละ stage ปรับ "บุคลิก" ของโมเดลไปในทางต่างกัน.

**"AI เรียนรู้ระหว่างที่คุณคุย"** — ความเชื่อนี้พบบ่อยมาก. ความจริงคือ ChatGPT หรือ Claude ที่คุณคุยด้วยทุกวัน **ไม่ได้เรียนรู้จากบทสนทนาของคุณ**. parameter ของโมเดลถูก freeze ไว้หลัง training. สิ่งที่โมเดล "จำ" ในบทสนทนาหนึ่งคือข้อมูลใน context window (หน่วยความจำระยะสั้น) ไม่ใช่การ update parameter จริง ๆ. เมื่อบทสนทนาจบ โมเดลไม่ได้เก่งขึ้น.

## ต้นทุน training ที่ไม่ใช่แค่เงิน

เวลาพูดถึงต้นทุน training LLM มักพูดถึงเงิน แต่มีต้นทุนอีกสองอย่างที่มักถูกมองข้าม:

### พลังงานและ carbon footprint

GPU หลายพันตัวที่รันตลอดเวลาหลายเดือนใช้ไฟฟ้ามหาศาล. การ train GPT-4 ครั้งเดียวคาดว่าปล่อย CO2 เทียบเท่ากับรถยนต์หลายร้อยคันวิ่งตลอดปี. เรื่องนี้กำลังถูกพูดถึงในวงการมากขึ้น และบริษัท AI ขนาดใหญ่กำลังลงทุนใน renewable energy เพื่อลด footprint.

ในระดับ inference (ใช้งาน) — ทุกครั้งที่คุณถาม ChatGPT หนึ่งคำถาม ก็ใช้ไฟฟ้าประมาณ 10 เท่าของการ Google Search ธรรมดา. ยังไม่ใช่ตัวเลขใหญ่มากสำหรับผู้ใช้รายบุคคล แต่รวมกันหลายร้อยล้านคนต่อวันก็เป็นเรื่องที่ต้องคิด.

### เวลาของมนุษย์สำหรับ data labeling

ในกระบวนการ alignment (Stage 3) มนุษย์หลายพันคนต้องอ่านคำตอบของโมเดลและตัดสินว่าอันไหนดีกว่า. งาน labeling data ประเภทนี้มักถูก outsource ไปยังประเทศที่ค่าแรงต่ำกว่า — เป็นประเด็นที่มีการถกเถียงเรื่องความเป็นธรรมในวงการ AI ethics.

ข้อมูลส่วนหนึ่งของ training data ก็มาจากงานของมนุษย์ที่ถูกนำมาใช้โดยไม่ได้รับความยินยอมหรือค่าตอบแทน — เช่น งานศิลปะ งานเขียน โค้ดที่นักพัฒนาเผยแพร่ใน GitHub. นี่คือหนึ่งในข้อถกเถียงทางกฎหมายและจริยธรรมที่ยังไม่ได้ข้อสรุปในปี 2026.

## เมื่อคุณเข้าใจ training คุณเข้าใจ AI

บทนี้เป็นหนึ่งในบทที่สำคัญที่สุดในหลักสูตรนี้ เพราะกลไก training เป็นรากฐานของทุกอย่างที่เกี่ยวกับ AI ที่คุณจะได้เรียนต่อไป.

ความสามารถของ AI — มาจาก training data และ loss function ที่ออกแบบมาดี.

ข้อจำกัดของ AI — hallucination, bias, ไม่รู้เรื่องใหม่ ล้วนมีรากมาจาก training.

ราคาของ AI — แพงเพราะ compute ที่ต้องใช้ใน training ไม่ใช่เพราะ inference แพง.

ความแตกต่างระหว่างโมเดล — มาจากสถาปัตยกรรม, data, และ training recipe ไม่ใช่แค่ขนาด.

ครั้งต่อไปที่คุณอ่านข่าวว่า "บริษัท X เปิดตัวโมเดลใหม่ที่ดีกว่าเดิม" คุณจะรู้แล้วว่าหมายความว่าอะไร: กระบวนการ training ได้รับการปรับปรุงในด้านใดด้านหนึ่ง — ไม่ว่าจะเป็น data ที่ดีขึ้น architecture ที่ฉลาดขึ้น หรือ alignment technique ที่ซับซ้อนขึ้น.

## ลองทำดู — วัด AI ด้วยสายตาของ training engineer

ครั้งต่อไปที่คุณใช้ ChatGPT หรือ Claude ลองตั้งคำถามกับตัวเองในใจ 3 ข้อนี้:

**1. คำตอบนี้มาจาก training data ชัดเจน หรือโมเดลกำลัง "สร้าง" คำตอบ?**

ถ้าถามเรื่องที่มีคำตอบชัดเจนในอินเทอร์เน็ตหลายล้านหน้า (เช่น "เมืองหลวงของฝรั่งเศสคืออะไร") โมเดลมักตอบถูกเพราะ training data ชัดเจน.

ถ้าถามเรื่องที่ "สร้างใหม่" (เช่น "เขียนเรื่องสั้นเกี่ยวกับแมวนักสืบ") โมเดลต้อง "สร้าง" โดยอาศัย pattern ที่เคยเห็น — ผลลัพธ์อาจดีหรือแย่ขึ้นอยู่กับว่า training data มีตัวอย่างที่ใกล้เคียงมากน้อยแค่ไหน.

**2. Loss function ของงานนี้วัดอะไร?**

ถ้าคุณใช้ AI เพื่อสรุปเอกสาร — loss function ที่ใช้ train น่าจะวัดว่า "สรุปได้ครบถ้วนและกระชับไหม". ถ้าใช้ AI เพื่อเขียนโค้ด — loss น่าจะวัดว่า "โค้ดรันได้ไหม". แต่ loss function เหล่านั้นออกแบบโดยมนุษย์ และอาจไม่ตรงกับสิ่งที่คุณต้องการจริง ๆ.

**3. Training data มีตัวอย่างเพียงพอสำหรับงานนี้ไหม?**

งานที่มีตัวอย่างชัดเจนในอินเทอร์เน็ตเยอะ (สรุปบทความ แปลภาษา เขียน email) AI มักทำได้ดี. งานที่ training data น้อย (ข้อมูลเฉพาะบริษัทของคุณ ข่าวที่เพิ่งเกิด ความรู้เฉพาะทางมาก) AI จะทำได้แย่กว่า.

ถามตัวเองสามข้อนี้ก่อนเชื่อคำตอบของ AI ทุกครั้ง.

## สรุป — ภาพรวมทั้งหมดในหนึ่งหน้า

กระบวนการ "เรียนรู้" ของ AI ไม่มีความลึกลับอะไรเลย. มันเป็น optimization ล้วน ๆ — ปรับตัวเลขพันล้านตัวซ้ำแล้วซ้ำเล่าเพื่อให้ทายผิดน้อยลง.

กระบวนการนี้มีสามส่วนหลักที่ทำงานร่วมกัน:

- **Loss function** แปลเป้าหมายของมนุษย์เป็นตัวเลขที่โมเดลมองเห็น. ออกแบบผิด โมเดลก็เก่งในสิ่งผิด.
- **Gradient descent** บอกว่าควรปรับ parameter ไปทิศไหนเพื่อลด loss. เหมือนเดินลงเขาในความมืดทีละก้าว.
- **Training data** คือโจทย์ฝึกทั้งหมด. คุณภาพของ data กำหนดสิ่งที่โมเดลรู้ ไม่รู้ และ bias ที่มีอยู่.
- **Backpropagation** เป็นกลไกที่ทำให้สามอย่างข้างบนทำงานกับโมเดลขนาดจริงที่มี parameter เป็นพันล้านตัวได้.
- **Learning rate** กำหนดว่าแต่ละก้าวที่ปรับ parameter ใหญ่แค่ไหน — ใหญ่เกินพัง เล็กเกินช้า.
- **Mini-batch** ทำให้ training เป็นไปได้ในทางปฏิบัติ และยังช่วยให้โมเดล generalize ได้ดีขึ้นด้วย.

และ ChatGPT / Claude ที่คุณใช้ไม่ได้มาจาก training รอบเดียว แต่เป็นผลของ 3 stage: pretraining บน internet, fine-tuning บนบทสนทนา, และ alignment ด้วย feedback ของมนุษย์.

ถ้าจำอะไรได้แค่หนึ่งอย่างจากบทนี้ ให้จำว่า: **โมเดลไม่รู้จัก "ถูก" หรือ "ผิด" ในความหมายของมนุษย์ — มันรู้แค่อะไรลด loss ได้**. ประโยคเดียวนี้อธิบายได้ทั้งความสามารถและข้อจำกัดของ AI ในทุกสถานการณ์.

เวลาคุณเห็น AI ตอบผิด ตอบแบบ bias หรือตอบในสิ่งที่คุณไม่ได้ถาม — คำถามที่ถูกต้องไม่ใช่ "AI โง่หรือเปล่า" แต่คือ "กระบวนการ training ออกแบบ loss ยังไง และ data ที่ใช้สอนเป็นอย่างไร".

ความเข้าใจตรงนี้จะมีประโยชน์จริง ๆ ในบทถัดไปของหลักสูตร ไม่ว่าจะเป็นเรื่องการเขียน prompt ที่ดี (ที่ต้องรู้ว่าโมเดลตัดสินใจอย่างไร), การใช้ AI ในงานที่มีความสำคัญสูง (ที่ต้องรู้ว่า hallucination เกิดเมื่อไหร่), หรือการเลือกใช้ tool AI ที่เหมาะกับงาน (ที่ต้องเข้าใจว่า training ต่างกันอย่างไร).

ในบทต่อไป เราจะเจาะลึกเรื่อง **neural network** — โครงสร้างจริง ๆ ของโมเดลหลายชั้นที่ training ในบทนี้นำไปปรับ. คุณจะเห็นว่า neuron, layer, และ activation function ประกอบกันเป็นเครื่องที่เรียนรู้ได้อย่างไร. ถ้าคุณเข้าใจ training แล้ว โครงสร้างพวกนี้จะไม่ลึกลับเลย.

คำถามที่บทถัดไปจะตอบ: ถ้าโมเดลตื้น ๆ อย่างเส้นตรง `y = wx + b` ก็ learn จาก data ได้ด้วย gradient descent — ทำไมต้องเพิ่มชั้น? ทำไมต้องซับซ้อน? และทำไม neural network ที่มี 100 ชั้นถึงทำอะไรได้ที่โมเดล 1 ชั้นทำไม่ได้เลย ไม่ว่าจะมี parameter มากแค่ไหน?

## ทบทวนคำศัพท์สำคัญในบทนี้

ศัพท์เหล่านี้จะปรากฏซ้ำในบทต่อ ๆ ไปทั้งหมด:

| คำศัพท์ | ความหมายสั้น ๆ |
|---|---|
| **Parameter / Weight** | ตัวเลขภายในโมเดลที่ถูกปรับระหว่าง training |
| **Loss / Loss function** | ตัวเลขที่วัดว่าโมเดลผิดมากแค่ไหน — training คือการลด loss |
| **Gradient** | ทิศทางที่ loss จะเพิ่มเร็วที่สุดเมื่อปรับ parameter |
| **Gradient descent** | อัลกอริทึมปรับ parameter ในทิศตรงข้าม gradient (เดินลงเขา) |
| **Learning rate** | ขนาดก้าวในแต่ละครั้งที่ปรับ parameter |
| **Backpropagation** | วิธีคำนวณ gradient ของ parameter ทุกตัวพร้อมกันอย่างมีประสิทธิภาพ |
| **Batch** | กลุ่มข้อมูลย่อยที่ใช้คำนวณในแต่ละ iteration |
| **Epoch** | 1 รอบที่ผ่าน training data ครบทุกตัวอย่าง |
| **Overfitting** | โมเดลจำ training data แต่ generalize ไม่ได้ |
| **Pretraining** | Stage 1 — train บน data ขนาดใหญ่ เรียนรู้ language pattern |
| **Fine-tuning** | Stage 2 — ปรับโมเดลที่ train มาแล้วให้เก่งงานเฉพาะ |
| **RLHF** | Stage 3 — ใช้ feedback มนุษย์ปรับโมเดลให้ตอบได้ดีขึ้น |
| **Hyperparameter** | ตัวเลขที่กำหนดก่อน training เช่น learning rate, batch size |
| **Vanishing gradient** | ปัญหาที่ gradient เล็กลงจนชั้นต้นไม่ได้รับสัญญาณ |
| **MSE** | Mean Squared Error — loss function ที่ใช้กับงานทายตัวเลข |
| **Cross-entropy** | Loss function ที่ใช้กับงานแยกประเภทและ LLM |
| **Adam** | Optimizer ที่ปรับ learning rate อัตโนมัติ — ที่นิยมใช้ปัจจุบัน |

ทุกคำข้างบนนี้คุณจะเจอในบทความ AI ทั่วไป ใน LinkedIn ของ VP Engineering บริษัทเทคโนโลยี หรือในข่าว AI ทุกสัปดาห์. ตอนนี้คุณรู้ความหมายจริง ๆ แล้ว ไม่ใช่แค่รู้ชื่อ.

การรู้ว่า "gradient descent คืออะไร" ในระดับที่อธิบายให้คนอื่นฟังได้ ทำให้คุณอยู่ใน 5% ของผู้ใช้ AI ที่เข้าใจว่ากำลังใช้อะไรอยู่จริง ๆ. ส่วนใหญ่ใช้ ChatGPT แบบกล่องดำ — รู้แค่ "ถาม แล้วได้คำตอบ" โดยไม่รู้ว่าทำไมคำตอบนั้นน่าเชื่อถือหรือไม่น่าเชื่อถือ.

ลองทดสอบตัวเอง: ถ้าเพื่อนถามว่า "ทำไม ChatGPT ถึงตอบผิดบางครั้ง?" — คุณอธิบายได้แล้วด้วยภาษาของตัวเองว่า loss function วัดอะไร training data มีข้อจำกัดอะไร และทำไมโมเดลจึงไม่รู้จัก "ถูก" หรือ "ผิด" ในแบบที่มนุษย์รู้จัก. ถ้าอธิบายได้ คุณพร้อมแล้วสำหรับบทถัดไป.
