# Multimodal Input — เมื่อ AI ไม่ได้อ่านแค่ข้อความ

คุณเคยถ่ายรูปใบเสร็จร้านอาหารแล้วลากเข้า ChatGPT บอกว่า "ช่วยดูยอดรวมให้หน่อย" และมันตอบถูกต้องทันทีไหม. หรือเคยอัปโหลด PDF สัญญาแล้วถามว่า "ข้อไหนที่ควรระวัง" แล้วได้คำตอบที่อ่านรู้เรื่องภายในไม่กี่วินาที.

เมื่อสองสามปีก่อน การทำแบบนั้นเป็นเรื่องซับซ้อน ต้องใช้หลายเครื่องมือ หลายขั้นตอน. ตอนนี้มันเป็นเรื่องปกติ.

สิ่งที่เปลี่ยนไปคือ AI เรียนรู้ที่จะรับข้อมูลได้หลายรูปแบบในคราวเดียว ไม่ใช่แค่ข้อความอีกต่อไป. ความสามารถนี้มีชื่อเรียกในวงการ AI ว่า **multimodal input** และเป็นหนึ่งในการเปลี่ยนแปลงที่ส่งผลกับการใช้งานจริงมากที่สุดในช่วงปี 2023-2026.

## ก่อนปี 2023 — AI กับรูปไม่ใช่เรื่องง่าย

ลองนึกภาพว่าคุณทำงานในออฟฟิศ และต้องการให้ AI ช่วยอ่านใบเสร็จที่ถ่ายรูปมา. ขั้นตอนในยุคนั้นคือ:

1. เปิดโปรแกรม OCR (Optical Character Recognition) — โปรแกรมที่ "แปลง" รูปเป็นข้อความ
2. รอให้มันสแกนและสกัด text ออกมา
3. Copy text ที่ได้ไปวางใน ChatGPT
4. แล้วถามคำถาม

ขั้นตอนนี้ยาว มักเสียข้อมูลในการแปลง และถ้าใบเสร็จมีตารางหรือรูปแบบซับซ้อน ข้อความที่ออกมาก็ปนเป ผิดพลาดได้ง่าย.

ตั้งแต่ปี 2023 เป็นต้นมา frontier model รุ่นใหม่ถูกออกแบบมาให้รับรูปได้โดยตรง ไม่ต้องผ่านขั้นตอนแยก. GPT-4V (2023), Gemini 1.0 (2023), Claude 3 (2024) ล้วนออกแบบให้ "เห็น" รูปได้ตั้งแต่ตอนเรียนรู้ (pretraining). พอมาถึงปี 2026 frontier model เกือบทุกตัวเป็น multimodal โดย default แล้ว.

> 💡 **Key insight:** multimodal ไม่ใช่ feature add-on ที่เพิ่มทีหลัง แต่เป็นสิ่งที่โมเดลรุ่นใหม่ถูกสร้างมาพร้อมตั้งแต่ต้น เหมือนความแตกต่างระหว่างโทรศัพท์ที่มีกล้องติดตั้งมาให้กับโทรศัพท์ที่ต้องซื้อกล้องมาต่อเพิ่มเอง

## Multimodal คืออะไร — เปิดก่อน แล้วค่อยตั้งชื่อ

ลองนึกถึงสถานการณ์นี้: คุณส่ง screenshot หน้าจอให้เพื่อน พร้อมพูดว่า "ดูตรงนี้สิ มันแปลกไหม" เพื่อนได้ยินคุณพูดและเห็นรูปพร้อมกัน เขาเชื่อมโยงทั้งสองอย่างและตอบคุณได้ทันที. นั่นคือสิ่งที่ frontier model สมัยใหม่ทำได้แล้ว.

คำว่า **multimodal** ฟังดูเป็นศัพท์เทคนิค แต่แปลตรงตัวได้ง่ายมาก: "multi" = หลาย, "modal" = ช่องทาง. Multimodal input ก็คือโมเดลตัวเดียวที่รับข้อมูลได้หลายช่องทางในการสนทนาเดียวกัน ไม่ใช่รับแค่ข้อความ. ความสำคัญคือ ทุกช่องทางถูกประมวลผลร่วมกันเป็น context เดียว ไม่แยกชิ้น.

ช่องทางที่ frontier model ในปี 2026 รับได้หลัก ๆ มีสี่อย่าง:

- **ข้อความ (text)** — เหมือนเดิม พิมพ์คำถามตามปกติ
- **รูปภาพ (image)** — ลากรูป ถ่ายภาพ หรือ screenshot เข้า chat
- **เอกสาร PDF** — อัปโหลดไฟล์ PDF โดยตรง
- **เสียง (audio)** — อัดเสียงหรืออัปโหลดไฟล์เสียง

บางตัวยังรับ **วิดีโอ** ได้ด้วย แต่ยังมีข้อจำกัดมากกว่าช่องทางอื่น (จะพูดถึงทีหลัง).

สิ่งสำคัญที่ต้องเข้าใจคือ โมเดลไม่ได้แค่ "รับ" ทั้งหมดนั้นแยกกัน แล้วค่อยนำมารวม มันประมวลผลทุกอย่างร่วมกันเป็น context เดียว เหมือนกับว่าคุณนั่งคุยกับคนที่มองเห็นรูปเดียวกับคุณ ได้ยินเสียงเดียวกัน และอ่านเอกสารชุดเดียวกันพร้อมกับฟังที่คุณพูด.

## ข้างในโมเดล — ทุกอย่างกลายเป็น token

อย่างที่บท [LLM คืออะไร](/foundations/what-is-an-llm) อธิบายไว้ โมเดลไม่ได้อ่านข้อความตรง ๆ มันแปลงทุกอย่างเป็นตัวเลขก่อน ผ่านสิ่งที่เรียกว่า token. ถ้ายังไม่ได้อ่าน ลองแวะไปอ่านก่อน แล้วค่อยกลับมา.

Multimodal ทำงานบนหลักการเดียวกัน แต่ขยายออกไป: รูป เสียง และ PDF ก็ถูกแปลงให้กลายเป็น token ชุดหนึ่งเช่นกัน เรียกว่า **visual token** หรือ **audio token**.

ลองนึกภาพแบบนี้: เหมือนคุณทำงานในออฟฟิศที่ทุกคนสื่อสารผ่านสัญลักษณ์ตัวเลข รูปถ่ายก็ต้องถูก "แปล" ให้เป็นสัญลักษณ์ตัวเลขก่อนที่ทุกคนจะคุยได้เข้าใจ เสียงก็เช่นกัน พอทุกอย่างอยู่ในระบบเดียวกัน มันก็ประมวลผลร่วมกันได้.

กระบวนการนี้มีชื่อเรียกในงานวิจัยว่า **vision-language alignment** — ไอเดียหลักมาจากงานวิจัยที่ชื่อว่า CLIP (2021) ซึ่งค้นพบว่าถ้าฝึกโมเดลให้จับคู่รูปกับคำอธิบาย embedding ของรูปและ embedding ของข้อความที่เกี่ยวข้องกันจะ "อยู่ใกล้กัน" ในพื้นที่ทางคณิตศาสตร์. โมเดล multimodal สมัยใหม่ขยายหลักการนี้ออกไปจนรูปและเสียงสามารถ "พูดภาษาเดียวกัน" กับ token ข้อความได้.

ผลในมุมผู้ใช้:
- คุณส่ง "รูป chart ยอดขาย + ข้อความว่า 'เดือนไหนสูงสุด'" เข้าไปพร้อมกันได้
- โมเดลเห็นทั้งรูปและคำถามเป็นส่วนหนึ่งของ context เดียวกัน
- ตอบกลับได้โดยอ้างถึงทั้งข้อมูลในรูปและคำถามของคุณ

เปรียบเทียบนี้ใช้ได้แค่ระดับหนึ่ง ส่วนที่ต่างคือกระบวนการแปลงรูปเป็น token ซับซ้อนกว่าการแปลงข้อความ และยังสูญเสียรายละเอียดบางส่วนระหว่างทาง ซึ่งทำให้เกิด "blind spot" ที่จะพูดถึงต่อไป.

> 💡 **Key insight:** ในมุมโมเดล รูปและเสียงไม่ใช่ input "พิเศษ" — ทุกอย่างคือ token ในที่สุด รูปหนึ่งรูปอาจกลายเป็น 250-1,500 token ขึ้นกับขนาด ซึ่งหมายความว่ามันกิน "พื้นที่" ใน context window มากกว่าข้อความ

## Image Input — เข้าใจก่อนว่าโมเดล "เห็น" อะไร

ก่อนจะบอกว่า image input ทำอะไรได้ดีหรือไม่ได้ ต้องเข้าใจก่อนว่าโมเดล "เห็น" รูปในแบบไหน

### compressed representation คืออะไร — และทำไมถึงสำคัญ

เวลาคุณส่งรูปเข้าโมเดล มีคนบอกว่ามันทำ "compressed representation" ของรูป คำนี้หมายความว่าอะไรกันแน่

ลองนึกถึงการเล่าหนังให้เพื่อนฟัง: คุณไม่ได้เล่าทุก frame ทุกวินาที คุณเลือกเล่าเฉพาะส่วนสำคัญ "ตัวเอกวิ่งหนีจากผู้ร้าย แล้วก็กระโดดข้ามรั้ว" — ได้ใจความโดยไม่ต้องบอกสีรถ จำนวนต้นไม้ หรือตัวเลขบนป้ายถนน. นั่นคือ compressed representation ของหนัง.

โมเดลทำแบบเดียวกันกับรูป: รูปที่คุณส่งเข้าไปจะถูก "บีบ" ผ่าน **vision encoder** ซึ่งเป็นส่วนหนึ่งของโมเดลที่ทำหน้าที่แปลงพิกเซลเป็น visual token จำนวนหนึ่ง. Token เหล่านี้จับ "semantic สำคัญ" ของรูปเอาไว้ เช่น "รูปนี้คือใบเสร็จ มีตัวเลขราคา มีรายการอาหาร" แต่ไม่เก็บพิกเซลทุกตัว.

ทำไมถึงไม่เก็บพิกเซลทุกตัว? เพราะรูปขนาด 1080x1920 pixel = ข้อมูลเกือบ 2 ล้านค่า ถ้าต้องแปลงทั้งหมดเป็น token context window จะเต็มภายในรูปเดียว. การ "บีบ" จึงเป็นสิ่งจำเป็น แต่การบีบก็มาพร้อมกับการสูญเสียรายละเอียด.

### semantic-level vs fine spatial detail — ต่างกันอย่างไร

นี่คือหัวใจของการเข้าใจว่า image input ทำอะไรได้ดีและทำอะไรไม่ได้:

**Semantic-level (ระดับความหมาย):** โมเดลจับได้ดีมาก — รู้ว่ารูปคืออะไร มีอะไรอยู่ มีข้อความอะไรบ้าง บริบทของรูปเป็นแบบไหน. เหมือนคุณบอกว่า "รูปนี้คือตลาดนัด มีแผงขายผัก มีป้ายราคา" ได้ถูกต้องโดยไม่ต้องนับว่ามีแผงกี่แผง.

**Fine spatial detail (รายละเอียดเชิงพิกัด):** อ่อนกว่าชัดเจน — "วัตถุ A อยู่ห่างจากขอบรูปกี่ pixel" ไม่ได้. "มีคนกี่คนในรูป" เริ่มผิดเมื่อเกิน 10 คน. การนับวัตถุจำนวนมากหรือการอ้างอิงตำแหน่งแบบละเอียดทำได้ไม่ดี เพราะข้อมูล spatial ถูก compress ไปมากในระหว่างการสร้าง visual token.

เข้าใจแล้วก็จะเห็นว่าทำไมบางงานถึงทำได้ดี และบางงานถึงพลาด

### งานที่รูปภาพทำได้ดีในปี 2026

**อ่าน text จากรูป (OCR):** ถ่ายป้าย screenshot ใบเสร็จ บัตรนามบัตร ให้โมเดลอ่านข้อความออกมา. ระดับความแม่นยำมักดีกว่า OCR แบบดั้งเดิมในกรณีที่ layout ซับซ้อน เพราะโมเดลใช้ context รอบ ๆ ช่วยตีความ เช่น รู้ว่าตัวเลขข้างหลังเครื่องหมายบาทน่าจะเป็นราคา ไม่ใช่รหัสสินค้า.

**ตัวอย่างชีวิตจริง:** ถ่ายรูปใบเสร็จร้านก๋วยเตี๋ยวที่มี 5 รายการ แล้วส่งพร้อม prompt ว่า "อ่านรายการและราคาออกมาทั้งหมด บอกด้วยว่ายอดรวมถูกต้องไหม" โมเดลจะอ่านรายการ คำนวณยอด และแจ้งถ้าพบความไม่สอดคล้อง.

**อ่าน chart และ diagram:** รูป flowchart กราฟแท่ง กราฟเส้น architecture diagram ดึง insight ออกมาได้ระดับหนึ่ง เช่น "เดือนมีนาคมยอดสูงสุด ส่วนกุมภาพันธ์ต่ำกว่าค่าเฉลี่ย".

**Screenshot → code:** ถ่าย screenshot ของ design หรือ UI แล้วขอ HTML/CSS หรือ React component. ออกมาเป็น scaffolding ที่ใช้งานได้ แม้จะไม่ pixel-perfect.

**เปรียบเทียบ design หลายรูป:** ส่งรูปสองรูปพร้อมกันแล้วถามว่า "รูปไหน balanced กว่า" หรือ "ตัวอักษรในรูปไหนอ่านง่ายกว่า" โมเดลให้ความเห็นเชิง subjective ได้พอควร.

**Document understanding:** ใบเสร็จ invoice บัตรประชาชน (layout) สกัด field ออกเป็น JSON ได้.

**ส่ง screenshot แอปธนาคาร:** ถ่าย screenshot รายการเดินบัญชีแล้วถามว่า "รายการไหนผิดปกติ" หรือ "สรุปค่าใช้จ่ายหมวดอาหารให้หน่อย" ใช้ได้ดีในชีวิตประจำวัน.

### ภาษาไทยในรูป — จุดที่ต้องระวังเป็นพิเศษ

นี่คือสิ่งที่ผู้ใช้ไทยต้องรู้ก่อนใช้งานจริง: **ภาษาไทยในรูปยังอ่อนกว่าภาษาอังกฤษในรูปอย่างเห็นได้ชัดในปี 2026**

เหตุผลเชิงกลไกคือ vision encoder ที่สร้าง visual token ถูก train จากข้อมูลที่ส่วนใหญ่เป็นรูปที่มีข้อความภาษาอังกฤษ. โมเดลจึงจับ pattern ตัวอักษรภาษาอังกฤษได้ดีกว่ามาก ตัวอักษรไทยที่มีวรรณยุกต์ สระบน สระล่าง และรูปแบบที่หลากหลายกว่า ทำให้เกิดข้อผิดพลาดสูงกว่า.

สิ่งที่เกิดได้จริงในการใช้งาน:
- วรรณยุกต์หาย: "ไก่" ถูกอ่านเป็น "กา" เพราะสระและวรรณยุกต์ถูกมองข้าม
- ตัวเลขไทย (๑ ๒ ๓) อ่านสลับกับตัวเลขอารบิก หรืออ่านผิดหมวดเลย
- ข้อความที่พิมพ์บนพื้นหลังสีใกล้เคียงกับตัวอักษรอ่านผิดบ่อยกว่ามาก
- คำย่อหรือภาษาปากที่ไม่ค่อยมีใน training data ทั่วไปอาจถูกเดาผิด

**กรณีทดสอบที่ใช้ประเมินก่อนนำไปใช้จริง:** ลองส่งรูปใบเสร็จหรือเอกสารภาษาไทยที่คุณรู้เนื้อหาอยู่แล้ว แล้วตรวจสอบผล output ว่าโมเดลอ่านถูกต้องแค่ไหน ก่อนที่จะเชื่อผลลัพธ์จากรูปภาษาไทยที่ไม่รู้เนื้อหา.

วิธีรับมือ: ถ้างานต้องการความแม่นยำสูง ให้ review output ก่อนใช้ทุกครั้ง โดยเฉพาะส่วนที่เป็นตัวเลข ชื่อ หรือข้อมูลที่ผิดไม่ได้.

### งานที่รูปภาพยังทำได้ไม่ดี

**นับวัตถุจำนวนมาก:** "ในรูปนี้มีคนกี่คน" — พอเกิน 5-10 ชิ้น โมเดลเริ่มนับผิด. เป็น blind spot ที่เกิดจากวิธีที่รูปถูกแปลงเป็น visual token ซึ่งบีบข้อมูล spatial ไว้ไม่ครบ.

**ตำแหน่งเชิงพิกัดละเอียด:** "วัตถุ A อยู่ห่างจากขอบกี่ pixel" ไม่ได้. "อะไรอยู่ซ้ายมือของอะไร" ตอบได้ระดับหนึ่ง แต่ไม่เป๊ะ.

**รูปเบลอหรือ text ขนาดเล็กมาก:** ถ่ายมาแล้วเบลอ ตัวอักษรเล็กมาก โมเดลอาจอ่านผิดแบบมั่นใจ ระวังเป็นพิเศษกับงานที่ต้องการความถูกต้องสูง.

**ลายมือซับซ้อน:** ลายมือชัดเจนอ่านได้ดี ลายมือสะเปะสะปะหรือเขียนเร็วมาก error rate สูงขึ้นมาก.

> 💡 **Key insight:** โมเดลไม่ได้ "เห็น" รูปเหมือนที่คนเห็น มันแปลงรูปเป็น visual token ที่เป็น compressed representation รายละเอียดบาง pixel สูญหายระหว่างทาง semantic-level (รู้ว่ารูปอะไร) ดีมาก แต่ fine spatial detail (ตำแหน่งเป๊ะ นับชิ้น) ยังอ่อน ถ้าคุณคาดหวังว่าโมเดล "เห็น" เท่าคน จะผิดหวังในงานที่ต้องความละเอียด

## PDF — ทั้งง่ายและซับซ้อนในที่เดียว

PDF เป็น modality ที่น่าสนใจเป็นพิเศษ เพราะ PDF ไม่ใช่ "รูปล้วน ๆ" และไม่ใช่ "text ล้วน ๆ" มันอาจมีทั้งสองอย่างในไฟล์เดียว.

พอคุณอัปโหลด PDF เข้า chat สมัยใหม่ โมเดลทำสองอย่างพร้อมกัน:
1. **ดึง text จาก text layer** — สำหรับ PDF ที่ export จาก Word, Google Docs หรือโปรแกรม เนื้อหาส่วนนี้ถูกส่งเข้าโมเดลโดยตรงเป็นข้อความ
2. **เห็นหน้า PDF เป็นรูป** — สำหรับตาราง กราฟ ส่วนที่ scan มา ที่ text layer ไม่ครอบคลุม

ฟังดูดี แต่มีเรื่องที่ต้องระวัง

### ปัญหาตาราง

ตารางที่ดูสะอาดและเป็นระเบียบใน PDF มักกลายเป็น text ที่ "ปนกัน" เมื่อถูก extract. ลองนึกภาพตารางที่มี 3 column ชื่อ "สินค้า / จำนวน / ราคา" พอถูก extract ออกมาอาจกลายเป็น "สินค้า A 10 500 สินค้า B 5 250" ที่ไม่รู้ว่าตัวเลขไหนเป็น column ไหน.

ตัวอย่างจริง: สัญญาจ้างที่มีตารางรายละเอียดค่าตอบแทน — ถ้าโมเดลอ่านผ่าน text extraction ตัวเลขอาจปนกันจนตีความผิด ถ้าเปลี่ยนให้โมเดล "มอง" PDF เป็นรูปหน้าต่อหน้าแทน ผลมักดีกว่าชัดเจน.

ถ้าตารางสำคัญมาก: ให้โมเดลเห็น PDF เป็นรูป (vision mode) แทน text extraction ผลมักดีกว่าชัดเจน.

### ภาษาไทยใน PDF

PDF ที่สร้างจากโปรแกรมภาษาไทย เช่น เอกสารราชการ สัญญาไทย หรือรายงานที่ออกจาก Microsoft Word ภาษาไทย มักมีปัญหา text layer ที่ encoding ไม่ตรงมาตรฐาน Unicode. ผลคือเมื่อ extract text ออกมาอาจได้อักขระแปลก หรือได้ข้อความที่ตัดคำผิดจนอ่านไม่รู้เรื่อง.

ถ้าเจอปัญหานี้ให้ลอง vision mode (ให้โมเดลมอง PDF เป็นรูปหน้าต่อหน้า) แทน text extraction — มักได้ผลดีกว่าสำหรับ PDF ภาษาไทยที่ผลิตจากซอฟต์แวร์เก่าหรือไม่ได้ออกแบบมาสำหรับการ extract.

### ปัญหา context window

PDF ยาวกิน token เร็ว. paper วิจัย 30 หน้า ≈ 20,000-40,000 token. ถ้า context window ของโมเดลที่คุณใช้มีขีดจำกัด PDF บางส่วนอาจถูกตัดทิ้งโดยที่คุณไม่รู้ตัว.

สัญญาณที่บอกว่าเกิดปัญหานี้: โมเดลสรุปแค่ครึ่งแรกของเอกสาร หรือตอบคำถามเกี่ยวกับหน้าหลัง ๆ ได้ไม่ดี.

นอกจากนั้นยังมีปรากฏการณ์ที่เรียกว่า **"lost in the middle"** — โมเดลมักให้ความสนใจกับตอนต้นและตอนท้ายของ context มากกว่าตรงกลาง ถ้า PDF ยาวมาก ข้อมูลในส่วนกลางอาจถูก "มองข้าม" โดยที่โมเดลไม่ได้บอกคุณ.

วิธีรับมือกับเอกสารยาว: แบ่ง PDF ออกเป็นส่วน ๆ สรุปทีละส่วน แล้วค่อยถามจาก summary แทนการยัดทั้งก้อน. เรื่อง context window ละเอียดกว่านี้ไปดูได้ที่บท [Context Windows](/using-ai/context-windows).

## Audio — ฟังแบบธรรมดากับฟังแบบเข้าใจ ต่างกันมาก

เมื่อพูดถึง audio input มีสองระดับที่ต้องแยกให้ชัด เพราะมักทำให้สับสน:

### ระดับที่ 1 — แปลงเสียงเป็นข้อความ (speech-to-text)

โมเดลฟังเสียงพูดและถอดออกมาเป็น text ตามที่พูด. Whisper ของ OpenAI เป็นตัวอย่างคลาสสิก ปี 2026 คุณภาพการถอดความภาษาไทยดีขึ้นมากในสภาพแวดล้อมเสียงสะอาด ใช้ระดับมืออาชีพได้.

### ภาษาไทยในเสียง — ความท้าทายที่ต้องรู้

ภาษาไทยมีความท้าทายพิเศษสำหรับ speech-to-text หลายชั้น ซึ่งส่งผลต่อความแม่นยำของงานโดยตรง:

**วรรณยุกต์และเสียงคล้ายกัน:** "ไข่" กับ "ใข้" เสียงต่างกันแต่สะกดคล้าย, "มา" กับ "ม้า" กับ "หมา" ความหมายต่างกันโดยสิ้นเชิงแต่เสียงใกล้กัน. โมเดลต้องอาศัยบริบทในการตัดสินว่าคำไหนถูกต้อง ถ้าบริบทไม่ชัด โอกาสผิดสูง.

**สำเนียงภูมิภาค:** สำเนียงใต้ อีสาน และเหนือ มี error rate สูงกว่าสำเนียงกรุงเทพอย่างเห็นได้ชัด เพราะ training data ส่วนใหญ่เป็นสำเนียงกลาง. ถ้าต้องทำงานกับเสียงสำเนียงท้องถิ่น ควรทดสอบก่อนและมีคนตรวจสอบผล.

**คำทับศัพท์ที่พูดสำเนียงไทย:** คำอย่าง "โปรเจกต์" "มีตติ้ง" "เวิร์กช็อป" หรือ "ดีดไลน์" มักถูกถอดความออกมาเป็นภาษาอังกฤษหรือไทยไม่สม่ำเสมอ ขึ้นกับโมเดลที่ใช้.

**ศัพท์เฉพาะวงการ:** คำแพทย์ คำกฎหมาย คำวิศวกรรม ที่ไม่ค่อยมีใน training data ทั่วไปมีโอกาสผิดสูงกว่า เช่น ชื่อยา ชื่อกฎหมายเฉพาะ หรือชื่อแผนกในโรงพยาบาล.

**ผลที่ตามมาในการใช้งานจริง:** ถ้าคุณอัดเสียงประชุมและมีคนพูดสำเนียงอีสาน ใช้คำเฉพาะวงการ หรือพูดเร็วมาก ควรตั้งความคาดหวังว่าจะมีข้อผิดพลาด และต้อง review transcript ก่อนนำไปใช้ต่อ โดยเฉพาะส่วนที่เป็นชื่อคนหรือตัวเลขสำคัญ.

### ระดับที่ 2 — เข้าใจ semantic ของเสียง (audio understanding)

โมเดลตอบคำถามเกี่ยวกับเสียงได้โดยไม่ต้องแปลงเป็น text ก่อน เช่น "speaker คนนี้ฟังดูกังวลไหม", "ในไฟล์นี้มีกี่คนพูด", "เสียงพื้นหลังมาจากที่ไหน".

ความสามารถนี้ใหม่กว่าและยังต่างกันมากระหว่าง provider ใช้ได้ดีกับงานสรุปประชุม วิเคราะห์ tone ของ customer call สกัด action items.

**ตัวอย่างการใช้งานจริง:** บันทึกเสียงการประชุมทีม 30 นาที ส่งเข้า Claude หรือ GPT-4o พร้อม prompt ว่า "ช่วยสรุป: (1) ประเด็นที่คุย (2) decisions ที่ได้ และ (3) action items พร้อมชื่อคนรับผิดชอบ" โมเดลจะ transcribe ภายใน แล้วสรุปออกมาเป็นโครงสร้างที่ใช้งานได้ทันที.

**เมื่อไหรควรใช้อะไร:** ถ้าต้องการแค่ "คำพูดทุกคำที่พูดออกมา" → ใช้ speech-to-text ที่ optimize เฉพาะ (Whisper) ถูกกว่าและเร็วกว่า. ถ้าต้องการเข้าใจ semantic ลึก เช่น sentiment, structure ของการสนทนา → ใช้ frontier model multimodal แต่ค่าใช้จ่ายสูงกว่า.

> 💡 **Key insight:** หลายคนคิดว่า audio input คือ speech-to-text เท่านั้น จริง ๆ คือโมเดล multimodal สมัยใหม่เข้าใจ semantic ของเสียงได้ระดับหนึ่งโดยไม่ต้อง transcribe ก่อน ซึ่งเป็นความสามารถคนละชั้นกัน

## Video — เข้าใจกลไกก่อนเชื่อผลลัพธ์

Video เป็น modality ที่ frontier model รองรับมากขึ้นในปี 2026 แต่มีกลไกข้างในที่ต่างจาก image และ audio อย่างสำคัญ — และถ้าไม่รู้ว่ามันทำงานยังไง จะเชื่อผลลัพธ์ผิดจุดได้ง่ายมาก.

### โมเดลไม่ได้ "ดู" ทุก frame — และนี่คือสิ่งที่ต้องรู้

ลองนึกภาพว่าคุณมอบหมายให้ใครสักคนดูหนัง 2 ชั่วโมง แต่คน ๆ นั้นจะเปิดตาดูแค่ทุก 2 วินาที แล้วหลับตาระหว่างนั้น. เขาได้เห็นภาพรวมของหนัง แต่พลาดรายละเอียดที่เกิดขึ้นในช่วงที่หลับตา. นั่นคือสิ่งที่โมเดลทำกับ video.

เมื่อคุณส่ง video เข้าโมเดล ระบบภายในไม่ได้ประมวลผล video แบบ real-time ทุก frame. มันทำการ **frame sampling** — สุ่มหรือเลือก N frame ต่อวินาที (ส่วนใหญ่ 1-2 frame ต่อวินาที) มาเป็น input.

ผลคือ video 1 นาทีที่มี 1,800 frame (30fps) โมเดลอาจเห็นจริง ๆ แค่ 60-120 frame. ส่วนที่เหลืออีกกว่า 90% ถูก "ข้าม" ไปทั้งหมด.

**สิ่งที่หมายความว่าในการใช้งาน:**
- งานที่ต้องการดูรายละเอียดในช่วงเวลาสั้น เช่น "ท่าทางผิดพลาดตอนไหน" หรือ "อ่านป้ายที่ปรากฏ 1 วินาที" มีโอกาสพลาดสูงเพราะ frame ที่สำคัญอาจอยู่ระหว่าง sampling
- ถ้าผู้พูดในวิดีโอเปลี่ยนความคิดเห็นในช่วง 3 วินาทีแล้วกลับมาพูดเหมือนเดิม โมเดลอาจไม่รู้ว่าเคยมีการเปลี่ยนแปลงนั้น
- timestamp ที่โมเดลอ้างอิงไม่ใช่ timestamp จริงเสมอไป แต่เป็นการ "เดา" จาก frame ที่มองเห็น

### Temporal reasoning ยังอ่อน

ถามว่า "เกิดอะไรขึ้นก่อน-หลัง" ได้ในระดับกว้าง เช่น "video นี้พูดเรื่องอะไรในช่วงครึ่งแรกและครึ่งหลัง" แต่ลำดับเหตุการณ์ละเอียดในช่วงวินาทียังไม่แม่น.

งานที่ต้องการ temporal precision เช่น "ในนาทีที่ 23:47 เขาพูดว่าอะไร" ยังเชื่อถือได้ไม่เต็มที่ เพราะโมเดลอาจไม่ได้ sample frame ช่วงนั้นพอดี.

### เสียงใน video อาจถูกประมวลแยก

บาง provider รวม audio track กับ visual frame บางตัวไม่รวม ถ้าคำพูดใน video สำคัญ ควรถอดความแยกด้วย speech-to-text แล้วส่งเสริมกัน.

สัญญาณที่บอกว่าโมเดลประมวล audio แยก: โมเดลอธิบายสิ่งที่เห็นในภาพแต่ไม่อ้างถึงสิ่งที่พูด หรือตอบคำถามเกี่ยวกับภาพได้แต่ตอบเรื่องเนื้อหาการพูดไม่ได้.

### ใช้ได้ดีกับงานไหน และอะไรยังไม่เหมาะ

Gemini ของ Google เป็น leader ด้าน video input ในปี 2026 ด้วย context window ใหญ่ที่รองรับ video หลายชั่วโมง provider อื่นกำลังตามมา.

**ใช้ได้ดีกับ:**
- สรุปภาพรวมของคลิป tutorial สั้น เช่น "สรุป 5 ขั้นตอนหลักในวิดีโอสอนทำอาหารนี้"
- อธิบายเหตุการณ์ใน security footage ระดับกว้าง เช่น "มีคนเข้าออกกี่รอบในช่วงเช้า"
- หา key moment ในการบรรยายที่ยาว เช่น "ในคลิปนี้ speaker พูดถึงเรื่องราคาตอนไหน"
- ตรวจสอบว่า tutorial ทำตามขั้นตอนถูกหรือเปล่าในระดับภาพรวม

**ยังไม่เหมาะกับ:**
- งานที่ต้อง timestamp เป๊ะ
- อ้างอิง frame ที่แน่นอนหรือตรวจจับรายละเอียดในช่วงเวลาสั้น
- วิดีโอที่มีการเปลี่ยนแปลงความเร็วสูง เช่น การทดลองวิทยาศาสตร์ที่เกิดขึ้นเร็ว

## เขียน Prompt Multimodal ให้ถูกวิธี

การส่งรูปหรือ PDF เข้าโมเดลโดยไม่บอกว่าต้องการอะไรเป็นเรื่องที่คนทำบ่อย และได้ผลลัพธ์ที่ไม่ดีเท่าที่ควร โมเดลจะ "เดา" ว่าคุณต้องการอะไร ซึ่งมักไม่ตรงกับที่คุณคาดหวัง.

หลักการเขียน prompt multimodal ที่ดี:

**บอกชัดว่าต้องการอะไรกับ input.** แทน "ดูรูปนี้" ให้ใช้ "อ่าน text ทั้งหมดในรูปนี้และจัดเป็น JSON โดยมี field: รายการ, จำนวน, ราคา".

**อ้างถึงส่วนเฉพาะของรูปหรือเอกสาร.** "ในตารางหน้า 3", "ที่มุมขวาบนของรูป", "ในกราฟแท่งสีน้ำเงิน" โมเดลจัดการ reference เชิงพื้นที่และหน้าได้พอสมควร.

**ระบุ output format.** "ตอบเป็น JSON ตามนี้...", "ให้ผลเป็น bullet list ไม่เกิน 5 ข้อ" สำคัญพอ ๆ กับ prompt ปกติ.

**ถ้ามีหลาย input บอกความเชื่อมโยง.** "เปรียบเทียบ design A (รูปแรก) กับ design B (รูปที่สอง) ในด้าน visual balance" แทนที่จะส่งสองรูปโดยไม่อธิบาย.

**บอก context ถ้าจำเป็น.** "นี่คือ screenshot จากแอปธนาคารไทย ช่วยอ่านยอดเงินคงเหลือ" ช่วยลด ambiguity.

ตัวอย่างก่อนและหลัง:

| งาน | Prompt คลุมเครือ | Prompt ชัดเจน |
|-----|------------------|----------------|
| อ่านใบเสร็จ | "ดูรูปนี้หน่อย" | "อ่านรายการและราคาจากใบเสร็จในรูป ตอบเป็น JSON: `{items: [], total: }`" |
| สรุปประชุม | "ฟังไฟล์นี้" | "สรุป action items และชื่อคนรับผิดชอบจาก audio การประชุมนี้ ตอบเป็น bullet list" |
| วิเคราะห์ chart | "chart นี้บอกว่าอะไร" | "ใน bar chart นี้ เดือนไหนมียอดสูงสุดและต่ำสุด และแนวโน้ม Q4 เป็นอย่างไร" |

## ลองทำดู — อ่านใบเสร็จหรือ Screenshot ด้วย Structured Prompt

ลองทำแบบนี้กับรูปที่มีข้อความ เช่น ใบเสร็จ ใบแจ้งยอด หรือ screenshot แอปพลิเคชันใดก็ได้:

1. เปิด ChatGPT หรือ Claude
2. ถ่ายรูปหรือ screenshot สิ่งที่ต้องการให้อ่าน (ลองกับใบเสร็จร้านสะดวกซื้อ ใบแจ้งยอดค่าไฟ หรือ screenshot LINE Pay ก็ได้)
3. ส่งรูปพร้อม prompt ต่อไปนี้ (ปรับ field ตามเนื้อหาในรูป):

```
อ่านข้อมูลทั้งหมดจากรูปนี้ และตอบในรูปแบบ JSON ดังนี้:
{
  "items": [{"name": "", "amount": null}],
  "total": null,
  "date": null,
  "unreadable_fields": []
}
ถ้าอ่านข้อมูลใดไม่ออกหรือไม่แน่ใจ ให้ใส่ null และระบุใน unreadable_fields ว่าอ่านไม่ออกเพราะอะไร
```

4. สังเกตผลสามอย่าง: (1) ความแม่นยำ (2) สิ่งที่โมเดลระบุว่าอ่านไม่ออก และ (3) สิ่งที่โมเดลอ่านผิดโดยไม่รู้ตัว

ข้อที่ 3 สำคัญที่สุด — มันบอกว่างานไหนที่ต้อง review ก่อนใช้จริง

**ขั้นต่อไป:** ทดสอบด้วยใบเสร็จภาษาไทยและดูว่ามีรายการที่อ่านผิดหรือไม่ โดยเฉพาะชื่อสินค้าภาษาไทยและตัวเลขราคา นั่นจะบอกคุณว่าควรเชื่อ output ของ image OCR ภาษาไทยได้มากแค่ไหนในบริบทที่คุณใช้งาน.

## ลองทำดู — สรุปประชุมจาก Audio

ถ้าคุณมีการบันทึกเสียงการประชุมหรือ voice note ลองทำดังนี้:

1. เปิด ChatGPT หรือ Claude (ตรวจสอบว่า version ที่ใช้รับ audio file ได้)
2. อัปโหลดไฟล์เสียง ไม่เกิน 10-15 นาทีสำหรับการทดสอบครั้งแรก
3. ใส่ prompt ว่า: "จากการประชุมในไฟล์เสียงนี้ ช่วยสรุป: (1) ประเด็นหลักที่คุย (2) decisions ที่ตัดสินใจแล้ว (3) action items ที่ต้องทำต่อ พร้อมชื่อคนรับผิดชอบถ้ามี ตอบเป็น bullet points"
4. อ่านผลลัพธ์และสังเกตว่าโมเดลสรุปได้ตรงแค่ไหน และพลาดอะไรบ้าง

ถ้าโมเดลสรุปแค่ครึ่งแรกของการประชุม นั่นคือสัญญาณของ context window ที่เต็ม ลองตัดไฟล์ให้สั้นลงก่อน.

**ขั้นต่อไปสำหรับการใช้งานจริง:** ลองเพิ่ม instruction ว่า "ถ้าไม่ได้ยินชัด หรือไม่แน่ใจว่าชื่อคนที่รับผิดชอบคือใคร ให้ใส่เครื่องหมาย [?] แทน" วิธีนี้ช่วยให้คุณรู้ว่าส่วนไหนต้อง verify ก่อนส่งต่อ.

**สิ่งที่ควรตรวจหลังได้ผลลัพธ์:** ชื่อคนในการประชุม (โดยเฉพาะชื่อไทยที่ฟังออกเสียงยาก), ตัวเลขที่พูดถึง เช่น งบประมาณหรือ deadline และ action items ที่ระบุชื่อคนรับผิดชอบ สิ่งเหล่านี้มีโอกาสผิดสูงสุดและมีผลต่อการทำงานโดยตรง.

## Privacy — รูปและเสียงพ่วงข้อมูลที่คุณไม่ตั้งใจส่ง

เมื่อใช้ multimodal input มีเรื่อง privacy ที่ต้องระวังมากกว่าการส่งแค่ข้อความ เพราะรูปและเสียงมักมีข้อมูลที่ซ่อนอยู่ซึ่งคุณอาจไม่ได้ตั้งใจให้ใคร. นี่ไม่ใช่เรื่องทฤษฎี — มันเกิดขึ้นในชีวิตประจำวันของคนที่ใช้ multimodal input ทุกวัน.

### รูปถ่าย — มากกว่าที่ตาเห็น

รูปที่ถ่ายจากสมาร์ตโฟนมักมีสิ่งที่เรียกว่า **EXIF metadata** ฝังอยู่ในไฟล์ ซึ่งรวมถึง:

- **GPS coordinates** — ตำแหน่งที่คุณถ่าย (บ้าน ออฟฟิศ หรือที่ไหนก็ตาม) มักถูกบันทึกไว้โดย default
- **เวลาและวันที่** ที่ถ่ายรูป ถ้าบวกกับ GPS บอกได้ว่าคุณอยู่ที่ไหนเมื่อไหร่
- **รุ่นอุปกรณ์** ที่ใช้ถ่าย

นอกจากนั้นในรูปเองอาจมี: ใบหน้าของคนอื่นที่อยู่ในพื้นหลัง, ป้ายทะเบียนรถ, เอกสารบนโต๊ะที่ปรากฏบางส่วน, หรือข้อมูลที่ไม่ได้ตั้งใจให้เห็น.

**ตัวอย่างที่เกิดได้จริง:** ถ่ายรูปใบเสร็จร้านอาหารเพื่อให้ AI อ่านยอดเงิน แต่รูปนั้นถ่ายที่บ้าน และมี GPS coordinates ฝังอยู่ ข้อมูลตำแหน่งบ้านของคุณก็ถูกส่งไปด้วยโดยไม่รู้ตัว.

หรือลองนึกถึงการถ่ายรูปสลิปโอนเงินเพื่อให้ AI ตรวจสอบตัวเลข ในรูปอาจมีชื่อผู้รับ เลขบัญชี และยอดเงิน ซึ่งเป็นข้อมูลทางการเงินที่ sensitive ถูกส่งออกไปในการ query เดียว.

วิธีป้องกัน: ก่อนส่งรูปที่ถ่ายด้วยโทรศัพท์ ลองใช้แอปที่ strip EXIF data ออก หรือ screenshot รูปนั้นอีกทีแทนการส่งต้นฉบับตรง ๆ (screenshot มักไม่มี GPS metadata ของต้นฉบับ).

### PDF — ข้อมูลที่ซ่อนในไฟล์

PDF ดูเหมือนเอกสารธรรมดา แต่มักมีข้อมูลที่ไม่เห็นด้วยตาเปล่าซ่อนอยู่:

- **ชื่อผู้เขียน** และประวัติการแก้ไขทั้งหมด
- **Comment และ annotation** ที่ซ่อนไว้หรือถูกซ่อนในชั้น layer
- **Watermark** ที่ซ่อนในชั้น layer ล่าง ซึ่งอาจบอกว่าเอกสารนี้ส่งให้ใครโดยเฉพาะ
- เวอร์ชันก่อนหน้าของเอกสารที่ยัง track changes อยู่

กรณีที่ sensitive เป็นพิเศษ: รายงานการประชุมที่มีความเห็นส่วนตัวของคนที่แก้ไขฝังอยู่ใน comment หรือ draft สัญญาที่มีการเจรจาก่อนหน้าอยู่ใน revision history.

ถ้า PDF นั้นเป็นสัญญาหรือเอกสารที่ต้องการความลับ ควรใช้เครื่องมือ sanitize metadata ก่อน หรือ export เป็น PDF ใหม่จาก reader ที่ไม่เก็บ metadata.

### Audio — เสียงของคนอื่นและ PDPA

นี่คือเรื่องที่ละเอียดอ่อนที่สุดในบริบทไทย และมีนัยทางกฎหมายโดยตรง.

ถ้าคุณอัดเสียงการประชุมและมีคนอื่นพูดอยู่ด้วย คุณกำลังส่งเสียงของคนอื่นให้กับบริษัท third party (OpenAI, Anthropic, Google) โดยที่เขาอาจไม่รู้และไม่ได้ยินยอม.

เสียงพูดมี **voice biometric** ที่สามารถระบุตัวตนได้ในทางทฤษฎี ซึ่งถือเป็นข้อมูลส่วนบุคคลที่ sensitive ตาม PDPA (พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562). การส่งเสียงบุคคลอื่นออกไปโดยที่เขาไม่ยินยอมอาจเข้าข่ายการประมวลผลข้อมูลส่วนบุคคลโดยไม่ชอบด้วยกฎหมาย.

**แนวทางปฏิบัติในองค์กร:**
- แจ้งคนในประชุมก่อนว่าจะอัดเสียงและส่งให้ AI ประมวลผล
- ถ้าทำไม่ได้ในทางปฏิบัติ ให้พิมพ์สรุปประชุมเองก่อน แล้วส่งข้อความสรุปนั้นให้ AI แทนไฟล์เสียง
- สำหรับ audio ที่มีข้อมูลลับทางธุรกิจ ตรวจสอบ privacy policy ของ provider ก่อนว่าข้อมูลถูกนำไป train model หรือเก็บไว้นานแค่ไหน

**รูปถ่ายที่มีใบหน้าบุคคลอื่น:** ในทำนองเดียวกัน ถ้าคุณถ่ายรูปในที่สาธารณะแล้วมีใบหน้าคนอื่นติดมาในพื้นหลัง การส่งรูปนั้นให้ AI ประมวลผลก็เป็นการส่งข้อมูลชีวมิติ (biometric) ของบุคคลที่สามออกไป ควรพิจารณา blur ใบหน้าก่อนส่งถ้าเป็นไปได้.

> 💡 **Key insight:** ก่อนอัปโหลดรูปหรือเสียงที่มีคนอื่น ให้ถามตัวเองว่า "ถ้าคนในรูป/เสียงนี้รู้ว่าคุณกำลังทำอยู่ เขาจะโอเคไหม" ถ้าไม่แน่ใจ ให้ blur ใบหน้า ลบ metadata หรือเลือกไม่ใช้ข้อมูลชุดนั้น

### Provider ต่างกัน นโยบายต่างกัน

สิ่งหนึ่งที่ต้องรู้คือ provider แต่ละเจ้าจัดการข้อมูลที่คุณส่งไปต่างกันอย่างมีนัยสำคัญ:

**เรื่องการ training:** บางตัวนำ input ไปใช้ train model ต่อโดย default ถ้าคุณไม่ opt out บางตัวไม่นำไปใช้. ส่วนใหญ่มีหน้า privacy settings ที่ toggle ได้.

**เรื่อง data retention:** บางตัวเก็บ conversation history ไว้เป็นเดือน บางตัวลบหลังจบ session. ถ้าข้อมูลที่ส่งเป็นความลับ เรื่อง retention period สำคัญมาก.

**Enterprise tier:** provider หลัก ๆ มีตัวเลือก enterprise ที่รับประกันว่าข้อมูลไม่ถูกนำไปใช้เพิ่ม ราคาสูงกว่าแต่ให้ความมั่นใจสำหรับงานที่ sensitive.

ถ้างานที่ทำเกี่ยวข้องกับข้อมูลลูกค้าหรือข้อมูลธุรกิจที่ sensitive ควรอ่าน privacy policy ของ provider อย่างน้อยในส่วน "data retention" และ "training data usage" ก่อนใช้งาน.

## Cost — Multimodal ไม่ฟรี และแพงกว่าที่คิด

หลายคนคิดว่าการส่งรูปเพิ่มเข้าไปไม่มีค่าใช้จ่ายเพิ่ม ความเข้าใจนี้ถูกต้องสำหรับการใช้งานผ่าน subscription แบบ flat-rate เช่น ChatGPT Plus หรือ Claude Pro แต่ผิดอย่างสิ้นเชิงสำหรับ API usage และการสร้างโปรแกรม.

### ทำไม multimodal ถึงแพงกว่า

รูป เสียง และ PDF กิน token มากกว่าข้อความที่ "สื่อข้อมูลเดียวกัน" หลายเท่า เพราะกระบวนการแปลงสัญญาณ visual และ audio เป็น token ต้องใช้ token จำนวนมากในการเก็บข้อมูล:

| Input | Token โดยประมาณ (ปี 2026) |
|-------|--------------------------|
| รูป 1 รูป (resolution ปกติ) | 250-1,500 token |
| รูป 1 รูป (HD high-detail mode) | 1,000-4,000 token |
| PDF 1 หน้า (text + visual) | 500-2,000 token |
| Audio 1 นาที | 600-1,500 token |
| ข้อความ 500 คำ (ภาษาอังกฤษ) | ประมาณ 700-800 token |
| ข้อความ 500 คำ (ภาษาไทย) | ประมาณ 1,200-1,500 token |

ตัวเลขเหล่านี้เปลี่ยนไปตาม provider และ pricing tier แต่สัดส่วนให้เห็นภาพ: รูป 1 รูปอาจเทียบเท่ากับข้อความ 250-4,000 คำ ขึ้นกับ resolution และ mode ที่ใช้.

### ตัวอย่างเชิงตัวเลข — ผลกระทบจริงใน production

สมมติว่าคุณสร้างระบบที่รับใบเสร็จจากผู้ใช้ 1,000 คนต่อวัน ใบเสร็จเฉลี่ยใช้ 800 token ต่อรูป ราคา token ที่ $5 ต่อ 1 million token:

- ต้นทุนต่อวัน: 1,000 × 800 × $5/1,000,000 = $4/วัน
- ต้นทุนต่อเดือน: ≈ $120/เดือน
- เทียบกับ text-only ที่ข้อมูลเทียบเท่า (~100 token): $0.50/เดือน

ผลต่างคือ **240 เท่า** ในกรณีนี้. ถ้า resolution สูงขึ้นหรือ input ซับซ้อนขึ้น ตัวเลขเปลี่ยนได้อีกมาก. ระบบที่รับรูปคุณภาพสูงจากผู้ใช้และใช้ HD mode อาจมีต้นทุน 5-10 เท่าของตัวอย่างนี้.

### กลยุทธ์ลด cost โดยไม่เสีย quality

**เลือก resolution ให้เหมาะกับงาน:** ถ้างานคือ "อ่านข้อความจากใบเสร็จ" ไม่จำเป็นต้องใช้ HD mode ที่กิน token 4 เท่า. ใช้ resolution ปกติ (250-1,500 token) เพียงพอสำหรับงาน OCR ส่วนใหญ่.

**ถ้าข้อมูลที่สำคัญอยู่ใน text layer ของ PDF ใช้ text extraction แทน vision mode:** text extraction ราคาถูกกว่ามาก เพราะส่ง text ตรง ๆ ไม่ต้องแปลงรูปหน้า.

**Pre-process รูปก่อนส่ง:** crop ส่วนที่ไม่สำคัญออก ลด resolution ถ้าไม่จำเป็น ช่วยลดจำนวน token ได้มาก.

**ใช้ dedicated speech-to-text สำหรับ audio transcription ล้วน ๆ:** Whisper ถูกกว่า frontier model multimodal หลายเท่า ถ้าต้องการแค่ transcript ไม่ต้องการ semantic understanding ใช้ Whisper แล้วส่ง text ต่อให้ frontier model ทำงานถัดไป.

> 💡 **Key insight:** ถ้าคุณแค่ใช้งานส่วนตัวผ่าน subscription เช่น ChatGPT Plus หรือ Claude Pro ไม่ต้องกังวลเรื่อง cost per token มากนัก แต่ถ้ากำลังสร้างโปรแกรมหรือ workflow อัตโนมัติที่ประมวลผลรูปหรือ audio จำนวนมาก cost เป็นปัจจัยที่ต้องคิดตั้งแต่แรก ก่อนที่ bill จะบอกคุณทีหลัง

## เลือก Modality ให้ถูกงาน — Decision Framework

เมื่อคุณมีข้อมูลหลายรูปแบบและต้องการให้ AI ช่วย คำถามที่เกิดขึ้นคือ "ควรส่งอะไรเข้าไป"

ใช้คำถาม 4 ข้อนี้ช่วยตัดสิน:

**1. ข้อมูลที่สำคัญอยู่ในรูปแบบไหน?**
ถ้า insight สำคัญอยู่ในข้อความ (ที่ copy ออกมาได้) → ส่งเป็น text จะถูกและแม่นกว่า
ถ้า insight สำคัญอยู่ใน layout ของรูป ตาราง หรือ visual → ส่งเป็นรูปหรือ PDF

**2. ต้องการความแม่นยำระดับไหน?**
ถ้า error เล็กน้อยยอมรับได้ → multimodal ตรง ๆ
ถ้าต้องแม่นทุกตัวเลขทุกตัวอักษร → review output ทุกครั้งก่อนใช้

**3. งานซ้ำแค่ครั้งเดียว หรือทำซ้ำหลายครั้ง?**
ครั้งเดียว → ส่งตรง ๆ ได้
ทำซ้ำ → ลงทุน setup pipeline ที่แม่นกว่า เช่น OCR เฉพาะทาง + text processing

**4. ข้อมูลนั้น sensitive แค่ไหน?**
ข้อมูลสาธารณะ → ใช้ได้ทันที
มีชื่อคน ข้อมูลลูกค้า หรือข้อมูลลับ → ประเมิน privacy และ provider policy ก่อน

## ตัวอย่างจริง — Input, Prompt, และสิ่งที่ต้องตรวจ

เข้าใจทฤษฎีแล้ว มาดูตัวอย่างครบวงจรที่คุณนำไปใช้งานได้ทันที

### ตัวอย่างที่ 1 — ใบเสร็จร้านสะดวกซื้อ

**สถานการณ์:** คุณซื้อของที่เซเว่นหลายรายการ และต้องการ log ค่าใช้จ่ายรายวัน

**Input:** รูปถ่ายใบเสร็จเซเว่น (ถ่ายจากโทรศัพท์)

**Prompt ที่ดี:**
```
อ่านรายการสินค้าและราคาจากใบเสร็จนี้ทั้งหมด
ตอบในรูปแบบนี้:
วันที่: [วันที่บนใบเสร็จ]
รายการ:
- [ชื่อสินค้า]: [ราคา] บาท
ยอดรวม: [ยอดรวม] บาท
ยอดที่จ่าย: [จ่ายไป] บาท
เงินทอน: [ทอน] บาท

ถ้าอ่านรายการใดไม่ชัด ให้ระบุ [อ่านไม่ชัด] แทน
```

**ผลที่คาดได้:** โมเดลอ่านรายการได้ดีพอสมควรสำหรับใบเสร็จที่พิมพ์สะอาด

**สิ่งที่ต้องตรวจก่อนใช้:**
- ตัวเลขราคา โดยเฉพาะรายการที่มีสาขาทศนิยม เช่น 45.00 บาท
- ชื่อสินค้าภาษาไทยที่อาจอ่านสลับกัน
- ยอดรวมและเงินทอน — โมเดลมักคำนวณ cross-check ให้ด้วย ซึ่งเป็นประโยชน์

---

### ตัวอย่างที่ 2 — สัญญาเช่าที่พัก (PDF)

**สถานการณ์:** คุณได้รับสัญญาเช่าอพาร์ตเมนต์ PDF 8 หน้า และต้องการรู้ว่ามีเงื่อนไขอะไรที่ควรระวัง

**Input:** ไฟล์ PDF สัญญาเช่า

**Prompt ที่ดี:**
```
อ่านสัญญาเช่านี้และสรุปให้หน่อย:
1. ระยะเวลาเช่าและค่าเช่ารายเดือน
2. เงื่อนไขการยกเลิกสัญญาก่อนกำหนด
3. สิ่งที่ผู้เช่าต้องระวังหรืออาจเป็นภาระ
4. ข้อที่ไม่เป็นมาตรฐานหรือผิดปกติกว่าสัญญาเช่าทั่วไป

ตอบเป็นภาษาไทยธรรมดา ไม่ต้องใช้ภาษากฎหมาย
```

**ผลที่คาดได้:** โมเดลสรุปสาระสำคัญได้ดี โดยเฉพาะถ้า PDF มี text layer สะอาด

**สิ่งที่ต้องตรวจก่อนใช้:**
- ตัวเลขค่าเช่าและค่าปรับ — ต้องตรงกับต้นฉบับ
- วันที่และระยะเวลา — error ตรงนี้มีผลทางกฎหมาย
- เงื่อนไขพิเศษที่โมเดลอาจตีความผิด เพราะภาษากฎหมายไทยมี nuance
- ถ้า PDF มีตาราง ให้ตรวจตารางในต้นฉบับด้วย เพราะอาจ extract ผิด

**ข้อควรระวัง:** สัญญาเช่าเป็นเอกสารที่ sensitive อ่าน privacy policy ของ provider ก่อนส่ง และพิจารณาใช้ account enterprise ถ้าเป็นสัญญาของบริษัท

### ตัวอย่างที่ 3 — ถ่าย screenshot รายการโอนเงิน

**สถานการณ์:** คุณส่งเงินหลายรายการผ่านแอปธนาคาร และต้องการสรุปว่าโอนให้ใครไปเท่าไหรในสัปดาห์นี้

**Input:** screenshot รายการเดินบัญชีจากแอปธนาคาร

**Prompt ที่ดี:**
```
จาก screenshot รายการเดินบัญชีนี้ ช่วยสรุป:
1. รายการโอนออกทั้งหมด (วันที่ / ชื่อผู้รับ / จำนวนเงิน)
2. ยอดรวมที่โอนออกทั้งหมด
3. ถ้ามีรายการที่อ่านไม่ออกหรือไม่ชัด ระบุด้วย

ตอบเป็นตารางถ้าทำได้
```

**ผลที่คาดได้:** ทำได้ดีพอสมควร แต่ชื่อผู้รับอาจถูกตัดทอนหรืออ่านผิดถ้าตัวอักษรเล็ก

**สิ่งที่ต้องตรวจก่อนใช้:**
- ชื่อผู้รับ — โดยเฉพาะชื่อภาษาไทยที่มีพยัญชนะซับซ้อน
- ยอดเงินทุกรายการ — cross-check กับ screenshot ต้นฉบับ
- วันที่ — ถ้ามีหลายรายการในวันเดียว โมเดลอาจสลับลำดับ

**คำเตือนด้าน privacy:** screenshot รายการธนาคารมีข้อมูลเลขบัญชี ยอดเงิน และชื่อบุคคล ก่อนส่งให้ AI ให้ประเมินก่อนว่าข้อมูลนั้น sensitive แค่ไหน และพิจารณา crop ส่วนที่ไม่จำเป็นออกก่อน

## Common Mistakes — ความเข้าใจผิดที่พบบ่อย

**"โมเดล generate รูปได้ทุกตัว"** — ความสามารถในการ "อ่าน/วิเคราะห์" รูปกับ "สร้าง" รูปเป็นคนละชุดกัน. Claude ในช่วงต้นปี 2026 เห็นรูปได้แต่ไม่สร้างรูป. GPT-4o รวมทั้งสองด้านในตัวเดียว. DALL-E, Midjourney คือ image generator ที่ไม่ได้รับ text input แบบ conversation ตรวจสอบให้แน่ใจก่อนใช้งานว่าโมเดลที่คุณใช้ทำอะไรได้บ้าง.

**"ส่ง PDF ยาวแล้วโมเดลอ่านทั้งหมด"** — PDF กิน token เร็ว และพฤติกรรม "lost in the middle" ทำให้โมเดลให้ความสนใจตอนต้นและตอนท้ายของ context มากกว่าตรงกลาง. เอกสารยาวมาก ๆ ควรแบ่งส่ง.

**"Audio input คือ speech-to-text เท่านั้น"** — โมเดล multimodal สมัยใหม่เข้าใจ semantic ของเสียงได้ระดับหนึ่งโดยไม่ต้อง transcribe ก่อน ซึ่งต่างกับ Whisper ที่ทำ transcription อย่างเดียว.

**"โมเดลอ่านภาษาไทยในรูปได้เหมือนอังกฤษ"** — ยังไม่ใช่ในปี 2026 ภาษาไทยในรูปอ่อนกว่าภาษาอังกฤษอย่างเห็นได้ชัด โดยเฉพาะลายมือ ตัวเล็ก หรือสีพื้นหลังซับซ้อน.

**"Multimodal ไม่มีผล privacy เพิ่มกว่า text"** — รูปและเสียงมี data layer ซ่อนอยู่ที่ text ไม่มี ทั้ง EXIF metadata GPS coordinates voice biometric และข้อมูลของบุคคลที่สามที่ติดมาโดยไม่ตั้งใจ.

## สรุป — Multimodal เปิดประตูงานใหม่ แต่ต้องรู้ขีดจำกัด

ความสามารถ multimodal input เปลี่ยนวิธีทำงานกับ AI จาก "พิมพ์แล้วรับคำตอบ" ไปเป็น "ส่งทุกอย่างที่มีแล้วรับความช่วยเหลือ". คุณไม่ต้องแปลงรูปเป็น text ก่อน ไม่ต้อง transcribe เสียงก่อน ไม่ต้อง copy-paste จาก PDF ก่อน โมเดลจัดการให้.

แต่ "จัดการให้" ไม่ได้แปลว่า "ทำได้ทุกอย่างสมบูรณ์แบบ". compressed representation ของรูปทำให้ spatial detail สูญหาย ภาษาไทยในรูปและในเสียงยังอ่อนกว่าอังกฤษ PDF ยาวยังมีปัญหา truncation video ยังอ่อนเรื่อง temporal reasoning และทุก modality พ่วงเรื่อง privacy และ cost ที่ต้องคิดก่อนส่ง.

ผู้ที่ใช้ multimodal ได้ดีในปี 2026 ไม่ใช่คนที่ส่งทุกอย่างเข้าโมเดลแล้วหวังว่าจะได้ผลดี แต่คือคนที่รู้ว่างานไหนเหมาะกับ modality ไหน เขียน prompt ที่บอกชัดว่าต้องการอะไร รู้ว่าควร review ผลลัพธ์ตรงไหนก่อนใช้งานจริง เข้าใจว่า input แต่ละชนิดมี cost และ privacy implication ที่ต่างกัน และระวังเป็นพิเศษกับ Thai-specific issues ที่โมเดลยังอ่อน.

สิ่งที่ทำให้คุณแตกต่างจากผู้ใช้ทั่วไปคือ การรู้จัก "ตรวจ" ผลลัพธ์อย่างมีระบบ ไม่ใช่แค่รับมาใช้ต่อ. รูปที่โมเดลอ่านผิดโดยมั่นใจ เสียงที่ถอดความชื่อคนผิด ตารางใน PDF ที่ตัวเลขปนกัน สิ่งเหล่านี้เกิดได้เสมอ แต่คนที่รู้ว่าต้องตรวจตรงไหนจะจับผิดได้ก่อนที่มันจะกลายเป็นปัญหา.

ในบทต่อไป เราจะขยับจาก "รู้ว่าโมเดลทำอะไรได้" ไปสู่ "รู้ว่าควรใช้ tool ไหนกับงานไหน" ซึ่งเป็นทักษะที่ต่างออกไป เมื่อ AI มีความสามารถหลายอย่างในตัวเดียว การเลือกใช้ให้ถูกจุดคือสิ่งที่แยกผู้ใช้ธรรมดาออกจากผู้ใช้ที่ได้ประโยชน์จริง.

> 💡 **Key insight:** multimodal ไม่ใช่ "ส่งทุกอย่างแล้วรอ" — มันคือการรู้ว่าแต่ละ modality มีจุดแข็งและจุดอ่อนอะไร เลือกให้ถูก เขียน prompt ให้ชัด และตรวจผลลัพธ์ให้ครบก่อนใช้ต่อ
