## พูดสั้นๆ

LLM ให้ output ที่ฟังดูดีเสมอ — แต่ "ฟังดูดี" ไม่เท่ากับ "ดีจริง". การประเมินคุณภาพคือหน้าที่ของผู้ใช้ ไม่ใช่ของโมเดล โดยตรวจ 3 มิติคือ correctness, completeness, fit-to-purpose แล้วปรับความเข้มงวดให้ตรงกับ stakes ของงาน. ทักษะนี้คือสิ่งที่แยก "คนใช้ AI เก่ง" ออกจาก "คนที่โดน AI หลอกแบบมั่นใจ".

LLM ในปี 2026 มีคุณสมบัติอันตรายอย่างหนึ่ง — **output ฟังดูดีเสมอ**. ไม่ว่าจะถูกหรือผิด ตอบตรงคำถามหรือไม่ ภาษาที่ออกมาจะลื่นไหล มีโครงสร้าง ดูมั่นใจ. นี่คือผลข้างเคียงของกระบวนการ training ที่ฝึกให้โมเดล "ตอบเหมือนตัวอย่างที่ดี" — แต่ไม่ได้ฝึกให้ "ตอบเฉพาะตอนรู้จริง" ([hallucination](/using-ai/hallucination)).

ผลคือผู้ใช้ที่ไม่ฝึกประเมิน output จะเข้าใจผิดว่า "ลื่น = ถูก" และส่งงานที่ผิดออกไปโดยไม่รู้ตัว.

**Mental model: AI = freelancer ที่เร็วมาก แต่ unreliable**

นึกถึง LLM เป็นเหมือน contractor ที่ทำงานเร็วมาก สำเนียงดี อ้างอิงคล่อง ราคาถูก — แต่บางครั้งก็แต่งเรื่อง อ้างหนังสือที่ไม่มีจริง หรือทำงานข้ามขั้นตอน. ถ้าจ้าง freelancer แบบนี้ในชีวิตจริง คุณจะไม่ส่งงานเขาออกไปยังลูกค้าโดยตรง — คุณจะ review ก่อน. การใช้ AI ก็เหมือนกัน.

**กรอบการคิด 3 มิติของคุณภาพ:**

1. **Correctness** — ข้อเท็จจริงถูกต้องหรือไม่? ตัวเลข ชื่อ วันที่ citation ทุกชิ้นต้องตรงกับโลกจริง
2. **Completeness** — ครอบคลุมสิ่งที่ถามหรือไม่? AI มักตอบ "ส่วนที่ง่าย" แล้วข้าม "ส่วนที่ยาก"
3. **Fit-to-purpose** — เหมาะกับ context ของงานหรือไม่? audience, tone, length, format ตรงกับที่ต้องใช้

output ที่ผ่านเกณฑ์ทั้ง 3 ข้อคือ output ที่ "พร้อมใช้". ขาดข้อใดข้อหนึ่ง = ต้องแก้ก่อน. งานคุณภาพต่ำที่พบบ่อยที่สุดคือ output ที่ "correct + complete แต่ไม่ fit" — ข้อมูลถูกแต่โทนผิด หรือยาวเกินไป หรือ format ไม่ตรง.

**ระดับการตรวจสอบขึ้นกับ stakes:** drafting ข้อความหาเพื่อน ตรวจเบา ๆ ก็พอ. ส่ง email หาลูกค้า ตรวจกลาง ๆ. เอกสารกฎหมาย/แพทย์/PR ภายนอก ตรวจเข้มหรือไม่ใช้ AI เลย. ผู้ใช้ที่เก่งจะ **คาลิเบรตความเข้มงวดของการตรวจให้พอดี** — ไม่ตรวจน้อยจนปล่อย error ผ่าน และไม่ตรวจเยอะจนเสียประโยชน์เรื่องความเร็วของ AI ไป.

## ลึกขึ้นหน่อย

### 2.1 ทำไม AI ตรวจตัวเองไม่น่าเชื่อ

หลายคนพยายามแก้ปัญหาด้วยการ "ให้ AI ตรวจ output ของ AI" (LLM-as-judge). วิธีนี้ **ใช้ได้บางกรณี แต่ไม่ใช่ทุกกรณี** — และเข้าใจว่าทำไมถึงเป็นทักษะสำคัญ.

LLM ตรวจตัวเองได้ดีในมิติที่เป็น "pattern matching":
- **Style/tone consistency** — โทนเป็นทางการหรือไม่ ใช้ persona ตรงกันทั้งย่อหน้าไหม
- **Format compliance** — output เป็น JSON valid ไหม มีหัวข้อตามที่ขอครบไหม
- **Spelling/grammar** — สะกดถูกไหม ประโยคสมบูรณ์ไหม
- **Self-critique กับ rubric ที่ชัด** — ถ้าให้ criteria เป็น checklist โมเดล apply checklist ได้พอใช้

LLM ตรวจตัวเองไม่น่าเชื่อในมิติที่ต้องอ้าง "ความจริง":
- **Factual correctness** — ถ้าโมเดลแต่ง fact ขึ้นมาในรอบแรก รอบ critique ก็ไม่มีทางรู้ว่าผิด เพราะอ้างจาก weight เดียวกัน
- **เรื่องที่อยู่นอก training** — เหตุการณ์ใหม่ ข้อมูลภายในองค์กร โมเดลไม่มีทางรู้
- **Subjective judgement กับมาตรฐานที่ไม่ได้บอก** — "เขียนดีไหม" สำหรับลูกค้ารายหนึ่งกับอีกรายต้องวัดต่างกัน โมเดลไม่รู้
- **Domain expertise ที่โมเดลขาด** — เรื่องเฉพาะทางที่ training data มีน้อย

นี่คือเหตุผลที่ **ผู้ใช้ที่เก่งจะรู้ว่าเมื่อไหร่ฝาก AI critique ได้ เมื่อไหร่ต้องตรวจเอง** — และคำตอบเกือบทุกครั้งคือ "ตรวจเอง ในส่วนที่เป็น correctness".

### 2.2 เทคนิคตรวจสอบที่เป็นรูปธรรม

**Cold read / read aloud.** อ่าน output ออกเสียง หรือทิ้งไว้ครึ่งชั่วโมงแล้วกลับมาอ่านใหม่. หูและตา "สด" จะจับ awkward phrasing, ความซ้ำซ้อน, ประโยคที่ฟังดูเข้าท่าแต่ไม่มีเนื้อ. เป็นเทคนิคถูกที่สุดและได้ผลที่สุด.

**Spot-check facts.** เลือก specific claim 3-5 ข้อจาก output แล้วยืนยันจากแหล่งอื่น (search, เอกสารต้นฉบับ, expert). ไม่ต้องตรวจทุกข้อ — แค่ตัวอย่างพอ. กฎที่ใช้ได้: **ถ้า spot-check 3 ข้อแล้วผิด 1 ข้อ ให้สันนิษฐานว่ายังมีอีก** และตรวจเพิ่มหรือเริ่มใหม่.

**Citation verification.** ทุก citation ที่ LLM ให้ — paper, URL, หนังสือ, คดี — **ต้องเปิดตรวจทุกอัน**. LLM hallucinate citation บ่อยมาก () — ชื่อ paper ดูถูกต้อง ผู้เขียนดูถูกต้อง แต่ paper ไม่มีอยู่จริง หรือพูดเรื่องอื่น. กฎเหล็ก: **ห้ามอ้าง citation ที่ไม่ได้เปิดอ่านเอง**.

**Number sanity.** ตัวเลขทุกตัวจาก AI — สถิติ ราคา วันที่ ปริมาณ — cross-check. LLM เก่งเดา "ตัวเลขที่ดูสมเหตุสมผล" แต่บ่อยครั้งเป็นการ interpolate จาก pattern ไม่ใช่ recall จากแหล่งจริง (ดูเหตุผลเชิงกลไกใน [what-is-an-llm](/using-ai/what-is-an-llm)).

**Counter-question.** ถามคำถามตรงข้ามหรือคำถามที่เกี่ยวข้องกับ output เดิม. ถ้า AI ตอบว่า "X ทำให้ Y เพิ่มขึ้น" ลองถามใหม่ว่า "Y ทำให้ X เปลี่ยนยังไง" — ถ้าคำตอบ contradict ตัวเอง = สัญญาณว่าโมเดลไม่มีฐานความรู้จริง แค่ generate ตาม context.

**Domain knowledge test.** ในเรื่องที่คุณรู้จริง สแกน output หา "สิ่งที่เฉพาะคนในวงการเท่านั้นจับได้" — ศัพท์ที่ใช้ผิด, ขั้นตอนที่ข้าม, ข้อยกเว้นที่ละเลย. ถ้าจับเจอ 1 จุด ใน area ที่คุณเชี่ยวชาญ = อาจมีอีกหลายจุดใน area ที่คุณไม่เชี่ยวชาญ.

**Adversarial check.** ถามตัวเองว่า "ถ้า critical reader / คู่แข่ง / ทนายฝั่งตรงข้ามอ่านสิ่งนี้ จะหาช่องโจมตีตรงไหน?" — แล้วอุดช่องเหล่านั้นก่อนส่งงาน.

### 2.3 Failure mode เฉพาะของ LLM ที่ต้องสแกนหา

จากกลไก next-token prediction ([what-is-an-llm](/using-ai/what-is-an-llm)) failure pattern มีลักษณะคาดเดาได้:

- **Hallucinated specifics** — ชื่อบุคคล, วันที่, ตัวเลข, citation — ส่วนที่เฉพาะเจาะจงคือส่วนที่เสี่ยงที่สุด
- **Plausible-sounding generics** — ประโยคแบบ "นี่เป็น trend ที่สำคัญ" "การวิเคราะห์ครอบคลุมหลายมิติ" — ฟังดูดีแต่ไม่ได้บอกอะไร เป็น filler
- **Over-confidence** — ขาด uncertainty marker เช่น "ประมาณ", "น่าจะ", "ตามข้อมูลที่มี" — โมเดลมัก confident เกินเหตุ
- **Subtle inversion** — กลับเหตุ-ผล เช่น พูดว่า "การที่ฝนตกทำให้คนใช้ร่ม" สลับเป็น "การใช้ร่มทำให้ฝนตก" — error แบบนี้ดูเล็กแต่บิดความหมายทั้งหมด
- **Missing nuance** — ตอบด้านเดียว ไม่แสดง trade-off ไม่กล่าวถึงข้อยกเว้น
- **Repetition / padding** — ขยายความคิดเดียวให้กลายเป็นหลายย่อหน้า ออกมาดูเยอะแต่เนื้อน้อย
- **Translated idioms** — สำนวนที่แปลตรง ๆ จากภาษาอังกฤษ ฟังในไทยแล้วแปลก ๆ (เช่น "ในตอนท้ายของวัน")
- **Lost in the middle** — ใน context ยาว โมเดลละเลย info ตรงกลาง () — ถ้าใส่เอกสารยาวให้สรุป ต้องตรวจว่ามีส่วนกลางที่ถูกข้ามไหม

### 2.4 ปรับ stakes ให้ตรงกับการตรวจสอบ

**Low stakes (drafting ข้อความสั้น ๆ, brainstorm, ส่วนตัว):**
- อ่านผ่าน ๆ ครั้งเดียว
- ส่งได้ถ้าไม่มี red flag ชัด
- เสียเวลามากกว่านี้คือ over-engineer

**Medium stakes (email งาน, internal report, slide ภายใน):**
- Cold read 1 รอบ
- Spot-check fact ที่ดูเฉพาะเจาะจง 2-3 ข้อ
- ตรวจ tone และ format ให้ตรง audience
- เวลาที่ใช้: ~10-20% ของเวลา generate

**High stakes (PR ภายนอก, legal, medical, hiring, financial, anything published):**
- ตรวจทุก fact ทุก citation ทุก number
- มี 2nd reviewer (มนุษย์)
- พิจารณาว่า AI เหมาะกับงานนี้จริงไหม — บางงานคำตอบคือ "ไม่ใช้ AI เลย"
- กฎเหล็ก: ถ้า error จะทำให้คน/บริษัทเสียหายร้ายแรง ห้ามใช้ AI เป็นด่านสุดท้าย

### 2.5 Red flags ที่บอกว่า "ทิ้งแล้วเริ่มใหม่"

บางครั้ง output แย่พอที่ "แก้" จะใช้เวลามากกว่า "เริ่มใหม่ด้วย prompt ที่ดีกว่า":

- **Output ไม่ตอบคำถาม** — เริ่ม drift ไปทาง topic ใกล้เคียง = prompt ไม่ชัดพอ ปรับ prompt แล้ว generate ใหม่
- **Spot-check เจอ error หลายข้อ** — ถ้า 3 จาก 5 ข้อผิด สันนิษฐานว่าทั้ง output ปนเปื้อน
- **Tone หลุดมาก** — บางครั้งกลับมาเป็น tone ที่ขอไม่ได้ ต้องเริ่มใหม่ด้วย system prompt ที่ชัดขึ้น
- **Format ผิด** — ขอ JSON ได้ prose, ขอ bullet ได้ paragraph — ปกติ generate ใหม่ด้วย instruction ชัด ๆ
- **โมเดลขอโทษและเริ่ม "พยายาม" รอบที่ 3-4** — สัญญาณว่าโมเดลไม่มีคำตอบจริง ถึงเวลาเปลี่ยน approach หรือเปลี่ยนโมเดล

### 2.6 The "second opinion" pattern

วิธีหนึ่งที่ใช้ได้ดี — **ส่ง output ที่ generate จากโมเดลหนึ่งให้อีกโมเดลหนึ่งวิจารณ์**. เช่น ให้ GPT critique output ของ Claude หรือกลับกัน.

ทำไมได้ผล? โมเดลแต่ละตัวมี blind spot ต่างกัน. ถ้าทั้งคู่เห็นด้วย confidence เพิ่ม. ถ้า disagree = สัญญาณว่ามีจุดที่ต้องตรวจมือ.

ข้อจำกัด: ทั้งสองโมเดลอาจผิดทางเดียวกัน (เพราะ training data ทับซ้อน) — จึงไม่ใช่การยืนยันแบบ independent จริง ๆ แต่ก็ดีกว่าใช้โมเดลเดียว.

### 2.7 Calibration: ทักษะที่ compound ตามเวลา

ผู้ใช้ AI ที่เก่งที่สุดมีคุณสมบัติร่วม — เขาเก็บ **mental note ของแต่ละโมเดล**:
- "Claude เก่ง explanation แต่ขี้ปฏิเสธ"
- "GPT-5 เก่ง coding แต่ confidence เกินจริงเรื่อง citation"
- "โมเดลตัวเล็กเร็วและพอใช้กับ format แต่อย่าเชื่อ fact"
- "เรื่อง medical/legal ทั้งโลก LLM ยังไม่ไว้ใจได้"

calibration แบบนี้สร้างไม่ได้ใน 1 วัน — สะสมจากการใช้แล้วสังเกตผลเป็นเดือน ๆ. นี่คือเหตุผลที่ผู้ใช้ระดับ expert มีข้อได้เปรียบมหาศาลเหนือผู้เริ่มต้น — ไม่ใช่ตรง prompt engineering เท่านั้น แต่ตรง **รู้ว่าเมื่อไหร่ไม่ควรเชื่อ AI**.

ในระดับองค์กร calibration นี้คือสิ่งที่จะถูกแทนที่ด้วย **programmatic evaluation** — ดู [evals-and-benchmarks](/building-with-ai/evals-and-benchmarks) ที่ Level 3 ซึ่งทำการประเมินเป็นระบบและ repeatable.

## Worked Examples

### Example 1: ประเมิน research summary
**Prompt:** "สรุปงานวิจัยล่าสุดเรื่อง LLM hallucination เป็น 5 bullet"

**Output ที่ได้ (สมมติ):**
- "งานวิจัยของ Smith et al. (2024) ในวารสาร Nature พบว่า hallucination ลด 73% ด้วย RLHF"
- "Hallucination มีหลายประเภทรวมถึง intrinsic และ extrinsic"
- "RAG ช่วยลด hallucination อย่างมีนัยสำคัญ"
- ฯลฯ

**การประเมิน:**
1. **Citation check** — มี Smith et al. (2024) ใน Nature เรื่องนี้จริงไหม? ค้นใน Google Scholar → ไม่เจอ. Red flag: hallucinated citation.
2. **Number check** — "73%" — มาจากไหน? ไม่มีแหล่ง → สันนิษฐานว่าแต่ง
3. **Generic content** — "ลด hallucination อย่างมีนัยสำคัญ" — ไม่มีตัวเลข ไม่มีเงื่อนไข = filler
4. **Decision** — ทิ้ง output ทั้งหมด, generate ใหม่โดยให้ AI ระบุเฉพาะ paper ที่อยู่ใน knowledge base ของตัวเองและให้ผู้ใช้ตรวจ

### Example 2: ประเมิน email หาลูกค้า
**Prompt:** "เขียน email ขอเลื่อนนัดกับลูกค้าใหญ่อย่างสุภาพ"

**Output ที่ได้:** email สำเนียงดี โครงสร้างครบ — opening, reason, alternative, closing.

**การประเมิน (medium stakes):**
1. **Cold read** — ฟังดูเป็นทางการพอดี ไม่หลุด ผ่าน
2. **Fit-to-purpose** — ลูกค้ารายนี้คุ้นเคยกันแบบกึ่งทางการ → email นี้ formal เกินไปนิด ปรับลง 1 ระดับ
3. **Spot-check** — ไม่มี fact เฉพาะที่ต้องตรวจ
4. **Format** — ผ่าน
5. **Decision** — ใช้ได้ แต่ปรับ tone ก่อนส่ง (ใช้เวลา &lt; 1 นาที)

### Example 3: ประเมิน code ที่ AI เขียน
**Prompt:** "เขียน Python function ที่ parse CSV และ return list ของ dict"

**การประเมิน:**
1. **Run มัน** — syntax error ไหม? run บน input จริง → ผ่าน
2. **Edge cases** — empty file? header เฉพาะ? quote ใน field? ลอง 3 case → fail 1 case (quoted field มี comma)
3. **Domain check** — ใช้ `csv.DictReader` standard library หรือเขียนเอง? ใช้ standard = ดี
4. **Decision** — แก้ edge case ที่ fail แล้วเพิ่ม test ก่อนใช้

## Common Misconceptions

1. **ความเข้าใจผิด:** ถ้า output ฟังดูดีและมั่นใจ แปลว่าน่าจะถูก
   **ความจริง:** ความลื่นของภาษาเป็นผลของกระบวนการ training — ไม่เกี่ยวกับความถูกของเนื้อหา. โมเดลเขียน output ที่ผิดสนิทออกมาด้วยความมั่นใจระดับเดียวกับ output ที่ถูกได้

2. **ความเข้าใจผิด:** ให้ AI ตรวจ output ของ AI ก็เพียงพอ
   **ความจริง:** AI ตรวจตัวเองได้ในมิติ pattern (format, style) แต่ตรวจ correctness ไม่ได้น่าเชื่อ — เพราะถ้ามันมีความรู้ที่ทำให้รู้ว่าผิด มันก็คงไม่ผิดตั้งแต่แรก

3. **ความเข้าใจผิด:** ตรวจทุก output แบบเข้มเสมอจะปลอดภัยที่สุด
   **ความจริง:** การตรวจเข้มทุกครั้งทำลายประโยชน์เรื่องความเร็ว — กลายเป็นใช้ AI ช้ากว่าทำเอง. คาลิเบรตให้ตรงกับ stakes คือทักษะที่ต้องฝึก

4. **ความเข้าใจผิด:** Citation ที่ AI ให้น่าจะถูกประมาณ 90%
   **ความจริง:** อัตรา hallucinate citation สูงมากในโมเดลส่วนใหญ่ — ในบางงานวิจัยพบ > 50% ของ citation ที่ generate ไม่มีอยู่จริงหรืออ้างเนื้อหาผิด (). กฎ default คือ **ไม่เชื่อจนกว่าจะเปิดอ่านเอง**

5. **ความเข้าใจผิด:** ถ้าโมเดลใหม่กว่า/ใหญ่กว่า ก็ตรวจน้อยลงได้
   **ความจริง:** โมเดลใหม่ลด error rate ใน benchmark แต่ failure mode ไม่หายไป — เพียงแค่ "ผิดบ่อยน้อยลง แต่ผิดเนียนกว่าเดิม". การตรวจสำคัญเท่าเดิม อาจสำคัญกว่าเดิมเพราะ error ที่เหลือยากจับขึ้น

## Self-Check

ลองตอบให้ได้โดยไม่ดูข้างบน:

1. **(Recall)** 3 มิติของคุณภาพ output ที่ต้องตรวจคืออะไร?
2. **(Explain)** ทำไม "ให้ AI ตรวจ AI" ถึงเชื่อได้ในเรื่อง format แต่เชื่อไม่ได้เรื่อง correctness? อธิบายเชิงกลไก
3. **(Apply)** สมมติว่า AI สรุปบทความวิจัยมาให้ 6 bullet มี citation 3 ตัว และตัวเลขสถิติ 4 ตัว — คุณจะตรวจอะไรก่อนหลัง ภายในงบเวลา 5 นาที?
4. **(Apply)** เพื่อนใช้ AI เขียน proposal ขอทุนวิจัยภาครัฐ และจะส่งพรุ่งนี้เช้า — คุณจะแนะนำให้ตรวจอะไร ในระดับไหน เพราะอะไร?
5. **(Synthesize)** ถ้าต้องสอนคนใหม่ในทีมเรื่องการประเมิน output ภายใน 10 นาที — คุณจะให้ heuristic 3 ข้อที่สำคัญที่สุดคืออะไร และทำไมเลือก 3 ข้อนี้?
6. **(Synthesize)** อธิบายความสัมพันธ์ระหว่าง [hallucination](/using-ai/hallucination) กับ skill ของการ evaluating ในบทนี้ — บทนี้คือ "วิธีรับมือ" กับ hallucination ในระดับไหน?
