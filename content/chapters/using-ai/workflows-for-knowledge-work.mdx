# Workflows สำหรับ Knowledge Work — AI ช่วยงานจริง ๆ ยังไง

คุณเปิด ChatGPT แล้วพิมพ์ว่า "สรุปรายงานนี้ให้หน่อย" — ได้คำตอบมาแต่รู้สึกว่ามันผิวเผิน. ลองอีกครั้ง "เขียน email ให้ลูกค้าหน่อย" — ได้ draft มา แต่ฟังเหมือนโรบอทเขียน ไม่ใช่คุณ. หลายคนใช้ AI ทุกวันแต่ยังรู้สึกว่า "ได้แค่นี้เองเหรอ".

ปัญหาไม่ได้อยู่ที่ AI ไม่เก่ง ปัญหาอยู่ที่ **วิธีที่คุณส่งงานให้มัน**.

งาน knowledge work ทุกชิ้น ไม่ว่าจะสรุปประชุม ร่าง proposal วิเคราะห์ตัวเลข หรือตอบ email ล้วนมีโครงสร้างที่ซ้ำได้ และมี pattern ของ prompt ที่ทำให้ AI ทำงานได้ดีหรือแย่อย่างคาดเดาได้. บทนี้คือ catalog ของ pattern เหล่านั้น ทั้งหมด 11 workflow พร้อม template ที่คัดลอกไปใช้ได้เลย.

## งาน knowledge work ไม่ได้หลากหลายอย่างที่คิด

ลองนึกถึงงานในสัปดาห์ที่ผ่านมา. บางทีคุณเขียน email หาลูกค้า, สรุปรายงานให้ผู้บริหาร, อ่าน briefing document 3 ฉบับแล้วต้องเขียน recommendation, เตรียม slide สำหรับประชุม, แปล terms and conditions จากอังกฤษเป็นไทย.

ดูเหมือนหลากหลาย แต่จริง ๆ มันคือ pattern เดิม ๆ ในรูปแบบต่างกัน.

"เขียน email หาลูกค้า", "เขียน executive summary", "เขียน LinkedIn post" — ทั้งสามคือ **drafting workflow** เหมือนกัน. input คือ context + goal + audience + tone + length. output คือ text. AI ต้องการ specification ชุดเดียวกันทั้งสาม.

เมื่อคุณเห็นงานผ่านเลนส์นี้ สิ่งที่เกิดขึ้นคือ: เรียนรู้ workflow ครั้งเดียว ใช้กับงานสิบงาน.

> 💡 **Key insight:** งาน knowledge work ส่วนใหญ่ตกใน 11 pattern ที่ทำซ้ำได้ ไม่ใช่งานสร้างสรรค์อันเป็นเอกลักษณ์. ถ้ารู้ว่างานในมือเป็น workflow ใด คุณรู้ทันทีว่าต้อง prompt ยังไง ต้องระวังอะไร และตรวจ output ยังไง.

## Mental model ก่อนเข้าสู่ workflow

ก่อนดู 11 workflow ต้องตั้ง mental model ให้ถูกก่อน.

AI ไม่ใช่ "ผู้ทำงานเก่ง" ที่รับงานแล้วส่งงานได้ทันที. AI คือ **เครื่องสร้าง first draft + เครื่องช่วยคิด** ที่:

- เก่งงานที่มี pattern ชัด เช่น drafting, summarization, editing, translation
- ต้องการ context ที่ดีถึงจะให้ผลดี (ขยะเข้า ขยะออก)
- ไม่น่าเชื่อในข้อเท็จจริงเฉพาะ ใช้สำหรับโครงสร้างและภาษา ไม่ใช่ ground truth
- ทำงานผ่าน iteration ไม่ใช่ get-it-right-first-try

**ผลทางปฏิบัติ:** workflow ส่วนใหญ่ในบทนี้ไม่ใช่ "ส่ง prompt แล้วจบ" แต่เป็นการสร้าง first draft แล้ว iterate. AI ลดเวลาจาก blank page ไปสู่จุดเริ่มต้นที่ดี ส่วนความ "เสร็จสมบูรณ์" ยังคงเป็นงานของมนุษย์.

ถ้ายังไม่ได้อ่านเรื่อง [prompt engineering patterns](/using-ai/prompt-engineering-patterns) และ [hallucination](/using-ai/hallucination) บทนั้นจะช่วยให้ใช้ workflow เหล่านี้ได้ดีขึ้นมาก เพราะแต่ละ workflow ใน catalog นี้สมมติว่าคุณรู้ failure mode พื้นฐานของ AI แล้ว.

## 1. Research synthesis — สังเคราะห์ข้อมูลจากหลายแหล่ง

**เมื่อใช้:** มี source 3-10 อันที่ต้องอ่านและสรุปเป็นความเข้าใจรวม ต้องเขียน briefing / literature review / competitor analysis.

**กลไก:** AI รับ source ดิบทั้งหมด → สังเคราะห์ความเข้าใจที่ coherent ตามคำถามที่ถาม → output ในรูปแบบที่กำหนด.

Pattern ที่ทำงานได้จริง: ส่ง source ทั้งหมดเข้า context + ตั้งคำถามที่ชัด + ระบุ output format + **บังคับให้อ้าง source ทุก claim**.

```
ฉันมีเอกสาร 5 ฉบับด้านล่าง (S1-S5). คำถามที่ต้องการคำตอบ:
"<question>"

ทำสิ่งต่อไปนี้:
1. สังเคราะห์คำตอบเป็น 3 หัวข้อหลัก
2. ทุก claim ต้องมี citation รูปแบบ [S1], [S2]...
3. ระบุจุดที่ source ขัดแย้งกัน
4. ระบุจุดที่ไม่มี source ตอบ — บอกตรง ๆ ว่า "ไม่พบใน source"
5. ความยาว: 600-800 คำ

[S1] ...
[S2] ...
```

**Mode/tool ที่เหมาะ:** Canvas/Artifacts สำหรับ output ยาว, Deep Research mode ถ้า AI ต้องค้นเอง.

**สิ่งที่ต้องระวัง:**

- AI แต่ง citation ที่ไม่มีจริง ใส่ [S2] เฉย ๆ ทั้งที่ S2 ไม่ได้พูดเรื่องนั้น
- AI ผสม knowledge ของตัวเองเข้ามาแทนที่จะอิง source ถ้าเรื่องอ่อนไหวต้องสั่งให้ "ใช้เฉพาะจาก source"

**ตรวจ output:** สุ่ม 3 citation มาเปิด source จริง ถ้าหลุด > 0 แม้ครั้งเดียว ต้อง re-prompt หรือพิจารณาใช้ระบบ RAG.

## 2. Document summarization — สรุปเอกสาร

**เมื่อใช้:** มี meeting note / report / article ยาวเกินจะอ่านเอง ต้องส่งต่อให้คนที่ไม่มีเวลาอ่าน full text.

**กลไก:** ย่อ document ยาวเป็นรูปแบบสั้นที่รักษาโครงสร้างและจุดสำคัญ.

Pattern ที่ทำงานได้: กำหนด **ระดับของ summary** ที่ต้องการ + รักษา structure เดิม + flag uncertainty.

```
สรุปเอกสารด้านล่างเป็น 3 ระดับ:

ระดับ 1 (Executive): 3 bullet points สำหรับ CEO
ระดับ 2 (Manager): 1 paragraph สรุปประเด็นและ action items
ระดับ 3 (Detailed): สรุปทีละ section ตาม structure เดิม

ถ้ามีจุดที่ source ไม่ชัด/ขัดแย้ง — flag ด้วย ⚠

เอกสาร:
"""
<full text>
"""
```

**Mode/tool ที่เหมาะ:** Chat สำหรับ document สั้น ถ้ายาวกว่า context window ต้องตัดเป็น chunk แล้วรวม.

**สิ่งที่ต้องระวัง:**

- "Lost in the middle" — โมเดลให้น้ำหนักตอนต้น/ตอนท้ายมากกว่าตรงกลาง เป็นปรากฏการณ์ที่มีงานวิจัยรองรับ โดยเฉพาะเมื่อ context ยาวมาก
- Summary ที่ "ใส่สิ่งที่ไม่มีใน input" — โมเดลเติม knowledge ทั่วไปเข้ามาโดยไม่แจ้ง

**ตรวจ output:** เปิดเอกสารต้นฉบับ สุ่มหา point ที่สำคัญสุด 3 จุด อยู่ใน summary หรือไม่ ถ้าหายไป > 1 ต้อง re-prompt ระบุ "เน้น X, Y, Z".

> 💡 **Key insight:** สั่ง 3 ระดับพร้อมกันใน prompt เดียวดีกว่าสั่งทีละระดับ เพราะ AI เห็นภาพรวมของ document ครั้งเดียวและรักษา consistency ระหว่างระดับได้ดีกว่า.

## 3. Drafting — ร่างเอกสารใหม่จาก blank page

**เมื่อใช้:** ต้องเริ่มเขียน email / proposal / post / brief / report จากศูนย์ งานที่มี template ชัดอย่าง cover letter, status update, intro email.

**กลไก:** สร้าง first draft โดยให้ context + goal + audience + tone + length.

Prompt ที่ดีสำหรับ drafting ต้องระบุครบ 5 อย่างนี้:

| Element | คืออะไร | ตัวอย่าง |
|---|---|---|
| Context | สถานการณ์ที่เกิดขึ้น | "ลูกค้าแจ้งว่ามีบั๊ก" |
| Goal | อยากให้ผู้อ่านทำอะไรหลังอ่าน | "ไม่ยกเลิกสัญญา" |
| Audience | ใคร รู้อะไรอยู่แล้ว | "ผู้จัดการ IT บริษัทใหญ่" |
| Tone | น้ำเสียงที่ต้องการ | "รับผิดอย่างมืออาชีพ" |
| Length | ความยาวที่ต้องการ | "4-6 ประโยค" |

```
เขียน email โดยมีรายละเอียดดังนี้:

Context: ลูกค้า Fairdee แจ้งว่าระบบยังมีบั๊ก ฉันต้องตอบและขอเวลา 2 สัปดาห์
Goal: ลูกค้าเข้าใจว่าเรากำลังแก้ และไม่ยกเลิกสัญญา
Audience: ผู้จัดการ IT บริษัทใหญ่ ภาษาทางการ
Tone: รับผิดอย่างมืออาชีพ ไม่แก้ตัว
Length: 4-6 ประโยค
Constraints:
- ห้ามใช้คำว่า "ขอโทษ" เกิน 1 ครั้ง
- ต้องระบุ timeline ที่ชัดเจน
- ลงท้ายด้วยการเสนอนัด call
```

**Mode/tool ที่เหมาะ:** Canvas/Artifacts ดีกว่า chat เพราะ output ยาวและต้อง iterate.

**สิ่งที่ต้องระวัง:**

- AI ใช้สำนวน generic ที่ฟัง "AI-flavored" เช่น "I hope this email finds you well"
- AI ใส่ข้อเท็จจริงที่คุณไม่ได้บอก เช่น วันที่ ตัวเลข ชื่อ มักเป็น hallucination ที่อันตราย

**ตรวจ output:** อ่านดัง ๆ ฟังเหมือนคุณเขียนเองหรือไม่ ถ้าไม่ ให้ AI rewrite โดยใส่ตัวอย่างที่คุณเคยเขียนเองประกอบ (few-shot).

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วเลือกงาน drafting ที่คุณต้องทำจริง ๆ ในสัปดาห์นี้ อาจเป็น email ลูกค้า, ข้อความขอนัดหมาย, หรือ caption สำหรับ post.

ก่อนพิมพ์ ลองตอบ 5 คำถามนี้ในหัวก่อน: Context คืออะไร? Goal คืออะไร? Audience เป็นใคร? Tone อยากได้แบบไหน? ความยาวเท่าไหร่?

จากนั้น draft prompt โดยใส่ทั้ง 5 อย่างลงไป แล้วส่ง เทียบกับครั้งที่คุณเคยส่ง prompt แบบสั้น ๆ ดูว่า output ต่างกันยังไง.

## 4. Editing and rewriting — แก้ไขข้อความที่มีอยู่แล้ว

**เมื่อใช้:** มี draft อยู่แล้วแต่ยังไม่พอใจ ต้องปรับให้เหมาะกับ audience ใหม่ หรือตรวจ grammar / สำนวน.

**กลไก:** ปรับ existing text ตามทิศทางที่ชัด เช่น "กระชับลง 30%", "เป็นทางการขึ้น", "ตัดศัพท์เทคนิค".

มีความจริงหนึ่งที่หลายคนยังไม่รู้: **editing workflow มักเชื่อถือได้กว่า drafting workflow** เพราะ AI มี anchor คือข้อความเดิมที่ป้องกันการแต่งเรื่องขึ้นมาเอง. drafting มี degree of freedom เยอะกว่ามาก ผลจึง vary มากกว่า.

```
แก้ paragraph ด้านล่างตามคำสั่ง:
- ลดความยาวลง 30%
- เปลี่ยน tone เป็นทางการขึ้น
- เก็บ technical term ทั้งหมด ห้ามแปล
- เก็บ structure เดิม (เริ่มด้วยอะไร จบด้วยอะไร)

แสดง output 2 version ให้เลือก

ข้อความ:
"""
<text>
"""
```

**Mode/tool ที่เหมาะ:** Canvas โดยใช้ inline edit หรือ chat ที่ส่ง text เป็น code block.

**สิ่งที่ต้องระวัง:**

- AI เปลี่ยนความหมาย ไม่ใช่แค่สำนวน ต้องเช็ค semantic ทุกครั้ง
- "Smoothing" คือ AI ทำให้ทุกอย่างฟังเรียบจน lose nuance ที่สำคัญ

**ตรวจ output:** วาง original กับ edited คู่กัน เปลี่ยนเฉพาะที่ขอ ไม่ใช่ทั่วทั้งข้อความ ความหมายเดิมยังอยู่ครบ.

> 💡 **Key insight:** หลายคนใช้ workflow "เขียน outline เองให้ครบ → ให้ AI rewrite ตามนั้น" ซึ่งได้ผลดีกว่า "บอก AI ให้เขียนจาก scratch" เพราะคุณควบคุมโครงสร้างและ logic ส่วน AI ช่วยภาษาและสำนวน.

## 5. Brainstorming — สร้าง options หลาย ๆ ชุด

**เมื่อใช้:** ต้องตั้งชื่อ product / feature / project หา angle ของบทความ / campaign คิด solution หลายแบบเพื่อ compare.

**กลไก:** สร้าง options หลากหลายเพื่อช่วยตัดสินใจ ไม่ใช่หา "คำตอบที่ถูก".

Pattern ที่ทำงานได้: define problem แคบที่สุดที่ทำได้ + ขอ **N options ที่ต่างกันจริง ๆ** โดยระบุ axis ของความหลากหลาย + ค่อย probe winners ใน turn ถัดไป.

```
ฉันต้องตั้งชื่อ feature ใหม่ของ app บัญชีออนไลน์ — feature นี้ทำให้ผู้ใช้
upload สลิปแล้ว AI จัด category ให้อัตโนมัติ.

ขอชื่อ 15 ชื่อ แบ่งเป็น 3 กลุ่ม:
- กลุ่ม A: ตรง ๆ บอกสิ่งที่ทำ (5 ชื่อ)
- กลุ่ม B: ใช้ metaphor (5 ชื่อ)
- กลุ่ม C: สั้น playful (5 ชื่อ)

แต่ละชื่อตามด้วย 1 ประโยคบอกว่าทำไมเลือกชื่อนี้
```

**Mode/tool ที่เหมาะ:** Chat หลาย session เปิดคู่กันแล้วเปรียบเทียบช่วยได้ เพราะ temperature ของ AI ทำให้แต่ละ session ให้ output ต่างกัน.

**สิ่งที่ต้องระวัง:**

- "Diversity collapse" คือขอ 20 ชื่อ แต่ได้ variation รอบ ๆ idea เดียวกัน 20 แบบ
- AI รีบ evaluate ว่าชื่อไหนดีที่สุดทั้งที่คุณยังอยู่ใน divergent phase

**ตรวจ output:** อย่างน้อย 3 ตัวเลือกควรเป็น direction ที่ "คุณไม่เคยคิดเอง" ถ้าทุกตัวคล้ายกัน เพิ่ม constraint ให้แคบลงใน axis.

## 6. Analysis — วิเคราะห์ข้อมูลและหา pattern

**เมื่อใช้:** มี structured data ที่ต้องเข้าใจ ต้องเปรียบเทียบ options หลายอัน ต้องการ multi-step reasoning.

**กลไก:** ตีความข้อมูล หา pattern เปรียบเทียบ option ระบุข้อสรุป.

Pattern ที่ทำงานได้: ส่ง **structured data** + คำถามเฉพาะ + **thinking framework** เช่น SWOT, decision matrix, root-cause + ใช้ reasoning mode สำหรับงานที่ต้องคิดหลายขั้น.

```
มี data ตารางด้านล่าง (ยอดขาย 6 เดือน 4 product).

วิเคราะห์ด้วย framework:
1. ระบุ trend ของแต่ละ product (เพิ่ม/ลด/คงที่)
2. ระบุ anomaly (เดือนที่ตัวเลขผิดปกติจาก trend)
3. ตั้ง hypothesis 2-3 ข้อที่อธิบาย anomaly
4. ระบุ data ที่ต้องการเพิ่มเพื่อ verify hypothesis

ห้ามใช้ knowledge นอก data นี้

| Month | A | B | C | D |
|...|
```

**Mode/tool ที่เหมาะ:** Reasoning mode เช่น Claude extended thinking หรือ GPT o-series สำหรับงานหลายขั้น Canvas เมื่อต้อง output ยาว.

**สิ่งที่ต้องระวัง:**

- AI คำนวณตัวเลขผิดได้ง่าย เพราะ LLM ทำ math ด้วย pattern matching ไม่ใช่การคำนวณจริง ถ้างานต้องการความแม่นยำให้บังคับใช้ code execution ถ้า tool นั้นมี
- AI สรุปก่อนวิเคราะห์ ลอกคำตอบที่ "น่าจะถูก" ไม่ใช่อิงตัวเลขจริง ใน data

**ตรวจ output:** สุ่มเช็คตัวเลข 3 จุด ทุก hypothesis ต้องอิง data point ในตาราง ไม่ใช่ generic claim.

## 7. Translation and localization — แปลและปรับให้เข้ากับ audience

**เมื่อใช้:** แปลเอกสารทางการ ปรับเอกสารภาษาอังกฤษให้เหมาะกับ audience ไทย เปลี่ยน register จาก academic เป็น conversational.

**กลไก:** แปลระหว่างภาษา / ระหว่างสไตล์ / ระหว่าง audience.

สิ่งที่ต้องใส่ใน prompt: source + target language/register + **glossary** ของศัพท์เฉพาะ + **named entity list** ที่ห้ามแปล เช่น ชื่อบริษัท product technical term.

```
แปลเอกสารด้านล่างจากอังกฤษเป็นไทย โดย:

Target audience: นักพัฒนา software ระดับ senior ในไทย
Register: ทางการแต่ไม่แข็ง (ใช้ "เรา" ไม่ใช่ "ผู้พัฒนา")
Glossary บังคับ:
- "deployment" → "deployment" (ห้ามแปล)
- "rollback" → "rollback"
- "context window" → "context window"
Named entities ห้ามแปล: Claude, GPT, Anthropic, OpenAI

แสดง output 2 รูปแบบ:
1. แปลเต็ม
2. รายการคำที่ตัดสินใจยาก พร้อมเหตุผล

ข้อความ:
"""
<text>
"""
```

**สิ่งที่ต้องระวัง:**

- AI แปลศัพท์เทคนิคแบบ literal ที่คนในวงการไม่ใช้จริง
- AI ตัด nuance ของ source โดยเฉพาะ idiom / tone
- ชื่อเฉพาะถูกแปลผิดหรือ transcribe ผิด

**ตรวจ output (round-trip check):** ส่ง translated text กลับเข้า model ให้แปลกลับเป็นภาษาเดิม ความหมายยังเหมือนต้นฉบับหรือไม่ สำหรับงานสำคัญ ใช้ native speaker review.

## 8. Coding-adjacent — สูตร / regex / SQL / script สำหรับคนไม่ได้เขียนโค้ด

**เมื่อใช้:** คุณรู้ว่าต้องการผลอะไรในเชิงผลลัพธ์ แต่ไม่รู้ syntax งาน one-off ที่ไม่คุ้มจะเรียน language นั้น.

**กลไก:** สร้าง code สั้น ๆ จากคำอธิบายภาษาธรรมชาติ เช่น Excel formula, regex, SQL, shell command.

Pattern ที่ทำงานได้: อธิบายผลที่ต้องการ + **ให้ตัวอย่าง input/output** + ระบุ environment ให้ชัด.

```
ฉันต้องการ Excel formula ที่:
- รับ cell ที่มีข้อความเช่น "2026-05-18 (Mon)"
- คืนค่าเป็นวันของสัปดาห์ภาษาไทย เช่น "จันทร์"

Environment: Excel 365 ภาษาอังกฤษ
ตัวอย่าง:
- Input "2026-05-18 (Mon)" → "จันทร์"
- Input "2026-05-19 (Tue)" → "อังคาร"

ขอ formula 2 แบบ:
1. แบบสั้นที่สุด
2. แบบที่อ่านง่ายแม้จะยาวกว่า
อธิบายแต่ละ part ของ formula สั้น ๆ
```

**สิ่งที่ต้องระวัง:**

- Regex / SQL ที่ "ดูถูก" แต่พลาด edge case ที่สำคัญ
- Function ที่ AI เขียนมาอาจไม่มีอยู่จริงใน version นั้น
- SQL syntax ที่ผสม dialect เช่น PostgreSQL กับ MySQL ปนกัน

**ตรวจ output:** ทดสอบกับ input อย่างน้อย 3 ตัว รวม edge case เช่น empty cell, ตัวเลขยาว, อักขระพิเศษ อย่า paste เข้า production โดยไม่ test.

> 💡 **Key insight:** การให้ตัวอย่าง input/output จริงใน prompt ช่วยได้มากกว่าการอธิบายยาว เพราะ AI เห็น pattern ที่ชัดกว่า.

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วลองส่ง task นี้:

"ฉันมีคอลัมน์ใน Excel ที่เก็บราคาเป็น text เช่น '1,250 บาท' หรือ '850 บาท' ต้องการ formula ที่แปลงให้เป็นตัวเลขที่บวกลบคูณหารได้ Environment: Excel 365 ภาษาไทย ตัวอย่าง: '1,250 บาท' → 1250 / '850 บาท' → 850"

ลอง formula ที่ได้กับ cell จริง แล้วดูว่าทำงานได้หรือมี edge case ที่พลาด.

## 9. Learning on demand — ขอคำอธิบายตามระดับที่ต้องการ

**เมื่อใช้:** เจอ concept ใหม่ในงาน ต้องอธิบายเรื่องของตัวเองให้คนอื่นเข้าใจ เตรียมตัวก่อนประชุม cross-functional.

**กลไก:** ขอให้ AI อธิบายเรื่องที่ไม่รู้ในระดับที่เหมาะกับ background และเป้าหมาย.

Pattern ที่ทำงานได้: ระบุ **ระดับของผู้รับ** + **goal** ว่าทำไมถึงอยากรู้ + **what you already know** เพื่อให้ AI ไม่อธิบายซ้ำสิ่งที่รู้แล้ว.

```
อธิบายเรื่อง "consistent hashing" ให้ฉันฟัง

Background ของฉัน:
- เป็น product manager ไม่ใช่ engineer
- เข้าใจ basic database (rows, indexes)
- ไม่เคยเขียน distributed system

Goal: ต้องคุยกับทีม backend ว่าทำไมเราเลือกใช้ และอะไรคือ tradeoff

Format:
1. อธิบายปัญหาที่มันแก้ (3-4 ประโยค)
2. mental model ที่จำง่าย
3. tradeoff ใหญ่ 2 อย่าง
4. คำที่ฉันควรรู้เพิ่มถ้าจะคุยลึก
```

**สิ่งที่ต้องระวัง:**

- Confident hallucination โดยเฉพาะใน niche topic ที่ AI ไม่มี training data เยอะ
- "Textbook fluff" คืออธิบายยาวแต่ไม่ตอบ goal ที่แท้จริง
- Outdated info ถ้า training cutoff เก่า best practice อาจเปลี่ยนไปแล้ว

**ตรวจ output:** หลังอ่านแล้ว ลอง explain back ให้ตัวเองฟัง 30 วินาที ติดตรงไหน ถาม follow-up สำหรับ topic สำคัญ verify กับ source อื่น.

## 10. Decision support — ช่วยคิดประกอบการตัดสินใจ

**เมื่อใช้:** เลือกระหว่าง options ที่มี dimension หลายชั้น ประเมิน risk ทำ scenario planning.

**กลไก:** AI ทำหน้าที่ thinking partner สำหรับ pros/cons, risk assessment, scenario planning — **ไม่ใช่ผู้ตัดสินใจ**.

Pattern ที่ทำงานได้: frame decision ให้ชัด + ใส่ **context ที่ครบ** เช่น constraint, stakeholder, timeline, value + ขอ **structured analysis** + ระบุชัดว่าคุณจะตัดสินใจเอง.

```
ฉันต้องตัดสินใจระหว่าง 2 option:

Option A: <รายละเอียด>
Option B: <รายละเอียด>

Context: <ทำไมต้องตัดสินใจ, deadline, budget, stakeholder>
Constraint ที่ตายตัว: <list>
Value ที่ฉันให้น้ำหนัก: <list>

ทำสิ่งต่อไปนี้:
1. ระบุ assumption ที่ฉันใช้โดยไม่รู้ตัว
2. ทำ pros/cons ทั้งสอง option ตาม value ฉัน
3. ระบุ risk ที่อาจมองข้าม (สำหรับแต่ละ option)
4. ให้ scenario worst case ของแต่ละ option
5. ระบุ information ที่ถ้ามีเพิ่มจะเปลี่ยน decision

ห้ามบอกฉันว่า "ควรเลือกอันไหน" — ฉันจะตัดสินใจเอง
```

**สิ่งที่ต้องระวัง:**

- AI ลำเอียงไป option ที่ฟังดู safe ไม่ใช่ option ที่ดีที่สุดสำหรับ context คุณ
- AI เห็นด้วยกับ assumption ที่ผิดของคุณ เพราะมันออกแบบมาให้ cooperative ไม่ใช่ adversarial — เรื่องนี้เรียกว่า sycophancy
- AI ปิดบัง risk เพราะกลัวทำให้คุณไม่พอใจ

**ตรวจ output:** อ่าน analysis แล้วเจอ **อย่างน้อย 1 ข้อที่คุณไม่เคยคิดเอง** หรือไม่ ถ้าไม่ AI ไม่ได้เพิ่ม value ให้ re-prompt ว่า "แสดง argument ที่ adversarial ต่อ option ที่ฉันน่าจะเอนเอียง".

## 11. Meeting prep and follow-up — เตรียมและตามผลประชุม

**เมื่อใช้:** ก่อนประชุมสำคัญ ออกแบบ agenda + เตรียม point. หลังประชุม ดึง action item จาก transcript / note.

**กลไก (ก่อนประชุม):** ใส่ context เช่น จุดประสงค์ ผู้เข้าร่วม เวลา ประวัติที่เกี่ยวข้อง → ขอ agenda + talking points.

```
ออกแบบ agenda 30 นาทีสำหรับประชุม

Goal: ตัดสินใจว่าจะใช้ vendor A หรือ B
Participants: CEO, CTO, Head of Product (4 คน)
Prior context: CTO เคยใช้ A มา 2 ปีและไม่ชอบ; CEO ยังไม่รู้ technical detail

Agenda format:
- Topic | Time | Owner | Goal
- ระบุ 2-3 talking points ที่ฉันควรเตรียมในฐานะ Head of Product
- จุดที่คาดว่าจะมี disagreement
```

**กลไก (หลังประชุม):** ส่ง transcript/note → ขอ extracted action items + decisions + open questions.

```
ฉันมี note ประชุมด้านล่าง (อาจมีตรงไหน implicit)

ดึงให้:
1. Decisions ที่ตัดสินใจแล้ว (ใคร, อะไร, when)
2. Action items (assignee, due date, expected output)
3. Open questions ที่ยังไม่ตัดสินใจ
4. จุดที่ note ขัดแย้งกันเอง — flag

Note:
"""
<text>
"""
```

**Mode/tool ที่เหมาะ:** Chat ก่อนประชุม Audio transcription tool + Canvas หลังประชุม.

**สิ่งที่ต้องระวัง:**

- AI สร้าง action item ที่ "ดูเหมือนมีในประชุม" แต่ไม่ได้พูดจริง
- AI ลืม implicit assignee ถ้าไม่ระบุชัดว่าใครเป็นคนทำ
- โทนของ summary บิดเบือนทำให้ดูดีกว่าหรือ drastic กว่าจริง

**ตรวจ output:** เทียบกับ note ของคุณเอง มี item ใน output ที่ "ไม่เคยพูด" หรือ "หาย" ไหม assignee ต้องเช็คทุกคน.

## ตัวอย่างจริงจากการใช้งาน

### Research synthesis ที่ผิดและถูก

**ผิด:**
> "อ่าน paper สามอันนี้แล้วสรุปให้หน่อย"

ปัญหา: ไม่มี question, format, citation requirement. AI จะ generate summary generic ที่ผสมเนื้อหา paper เข้าด้วยกันแบบไม่ชัด และอาจใส่ knowledge ของตัวเองเข้ามาโดยไม่แจ้ง.

**ถูก:**
> "ฉันกำลังเขียน briefing ให้ทีม มี paper 3 อัน [P1][P2][P3] เรื่อง prompt caching คำถาม: 'caching ลด latency ได้กี่เปอร์เซ็นต์ในการใช้งานจริง และมี tradeoff อะไร' Output: 200 คำ ทุก claim มี citation [P1]/[P2]/[P3] ระบุจุดที่ paper ขัดแย้ง จุดที่ไม่มี paper ตอบ บอก 'ไม่พบ'."

ทดสอบ: AI return summary 200 คำที่ทุก claim มี citation. เปิด P1 ตรวจ claim แรก อยู่จริงในหน้า 4. ทำงานได้.

### Drafting ที่ iterate ครบรอบ

**Round 1:** ส่ง prompt ตาม template ข้างต้น ขอ email ขอเลื่อนนัด.
**Output 1:** Email ดีในเชิง structure แต่โทน "AI-flavored" เกินไป ใช้คำว่า "I hope this email finds you well".

**Round 2:**
> "Output ดีในเชิงโครงสร้าง แต่ฟังเป็น AI เกิน นี่คือ email ที่ฉันเคยเขียนเองในสถานการณ์คล้ายกัน: [paste 2 examples] Rewrite output ใหม่ให้ tone เหมือนตัวอย่างของฉัน."

**Output 2:** ใช้ได้ โทนเหมือนคุณเขียน เวลารวม 6 นาที เทียบกับ 25 นาทีถ้าเขียนเองจาก scratch.

**บทเรียน:** drafting แทบไม่เคยจบใน round เดียว budget 2-3 iterations เป็นปกติ.

### Analysis ที่จับ failure ได้

**Setup:** ส่งตาราง sales 6 เดือน 4 product ถาม "ระบุ anomaly".
**Output:** AI ระบุว่า Product B เดือนมีนาคมตก 40% "อาจเป็นเพราะ seasonal effect".

**Verify:** เปิดตารางดู Product B มีนาคม = 1,200 กุมภาพันธ์ = 2,000 ลด 40% จริง ทำงานได้.
แต่ "seasonal effect" คือ knowledge นอก data ที่ AI เดามาเอง ต้อง re-prompt: "อย่าเดาเหตุผล ระบุเพียง pattern ใน data".

**บทเรียน:** constraint "stay within data" เป็นวิธีดักที่ดีสำหรับ analysis workflow.

## ความเข้าใจผิดที่พบบ่อย

**"AI ทำให้งานเสร็จเร็วขึ้น 10 เท่า"**

ในงาน drafting / editing / summarization ที่มี pattern ชัด เร็วขึ้น 2-5 เท่า เป็นไปได้ แต่ในงาน analysis / decision-making ที่ต้อง verify เยอะ อาจเร็วขึ้นเล็กน้อยหรือใช้เวลาเท่าเดิม เวลาที่ประหยัดในการสร้าง first draft เสียไปกับการตรวจสอบ. productivity gain ไม่กระจายเท่ากันในทุก workflow.

**"Prompt ยิ่งยาวยิ่งดี"**

Prompt ที่ดีคือ prompt ที่ **ครบ** ไม่ใช่ยาว ยิ่งยาวเสี่ยง "lost in the middle" คือโมเดลให้น้ำหนักตอนต้น/ท้ายมากกว่าตรงกลาง structure เช่น heading, bullet, code block ช่วยมากกว่า volume.

**"Template สำเร็จรูปบนอินเทอร์เน็ตใช้ได้เลย"**

Template ส่วนใหญ่ optimize ให้ generic case งานจริงต้องการ context เฉพาะของคุณ เช่น บริษัท, audience, style. template ที่ดีต้อง customize 30-50% ในครั้งแรกที่ใช้ การ "save แล้วใช้เลย" ไม่ค่อย work.

**"ถ้า output ไม่ดี แปลว่า AI ไม่เก่ง"**

80%+ ของ output ไม่ดีมาจาก prompt ไม่ครบ ขาด context, audience, format, constraint อย่างใดอย่างหนึ่ง ก่อนสรุปว่า "model ทำไม่ได้" ลองเพิ่ม specification ก่อน model เดิม prompt ที่ดีขึ้น มักให้ผลที่ดีขึ้นมาก.

## ทำไม template ที่ดีคุ้มเสมอ — เรื่องของ compound effect

ก่อนปิดบท มีประเด็นหนึ่งที่สำคัญกว่า workflow ใด ๆ ในบทนี้.

ใช้เวลา 10 นาทีออกแบบ prompt template ดี ๆ สำหรับงานที่ทำซ้ำ แล้ว save เก็บไว้ จะคุ้มภายใน 3-5 ครั้งของการใช้.

คำนวณ: ถ้างาน drafting ใช้เวลาเฉลี่ย 20 นาทีต่อรอบ และ template ที่ดีลดเหลือ 7 นาที = ประหยัด 13 นาทีต่อรอบ ใช้ 5 รอบ = 65 นาที cost ของการสร้าง template = 10 นาที ROI ภายในสัปดาห์เดียวสำหรับงานที่ทำทุกวัน.

แต่ effect ที่ใหญ่กว่าคือ second-order:

**1. Skill accumulation** — ทุกครั้งที่ refine template คุณเข้าใจ workflow ลึกขึ้น เข้าใจว่า variable ไหนสำคัญ ไหนไม่สำคัญ.

**2. Quality consistency** — ผลที่ได้สม่ำเสมอกว่า กระบวนการตรวจสอบ output ก็ง่ายขึ้น เพราะรู้ว่าจะ fail ตรงไหน.

**3. Delegation possibility** — template ที่ดีพอจะส่งต่อให้คนในทีมหรือ junior ได้ productivity gain ขยายเป็นทีม ไม่ใช่แค่ตัวเอง.

**4. System escalation** — เมื่อ template stable แล้ว มันคือ specification สำหรับการ automate ผ่าน API, agent, หรือ workflow tool ซึ่งเป็นสิ่งที่ Level 3 จะพูดถึง.

ทางปฏิบัติ: เก็บ template ใน file system / Notion / Obsidian ที่หาง่าย ตั้งชื่อตาม workflow + use case เปิดอ่าน/แก้ทุกครั้งที่ใช้ ภายใน 2-3 เดือนคุณจะมี personal library ของ prompt ที่ optimize สำหรับงานของคุณโดยเฉพาะ.

นี่คือจุดที่ AI เปลี่ยนจาก "เครื่องมือที่ใช้บางครั้ง" เป็น "infrastructure ของงาน".

## สรุป — อะไรที่ควรเอาออกไปจากบทนี้

งาน knowledge work ส่วนใหญ่ตกใน 11 pattern ที่ทำซ้ำได้ ได้แก่ research synthesis, document summarization, drafting, editing, brainstorming, analysis, translation, coding-adjacent, learning on demand, decision support, และ meeting prep & follow-up.

แต่ละ workflow มีโครงสร้าง prompt, mode/tool ที่เหมาะ, failure mode, และเกณฑ์ตรวจ output ที่แตกต่างกัน การรู้ว่างานในมือเป็น workflow ใด ทำให้คุณรู้ทันทีว่าต้อง prompt ยังไงและระวังอะไร แทนที่จะเดาทุกครั้ง.

และที่สำคัญที่สุด: เวลาที่ใช้ออกแบบ template ที่ดีสำหรับงานที่ทำซ้ำไม่ใช่เวลาที่เสียไป แต่เป็นการลงทุนที่ให้ผลตอบแทนทุกครั้งที่ใช้.

> 💡 **Key insight:** productivity gain ของ AI ไม่ได้มาจากการ "พิมพ์ prompt แล้วหวังว่าจะดี" แต่มาจากการเข้าใจว่างานในมือเป็น workflow ใด แล้วออกแบบ prompt ตาม specification ของ workflow นั้น.

## ในบทต่อไป

ทุก workflow ในบทนี้จบด้วย "ตรวจ output" — แต่ยังพูดได้แค่สั้น ๆ ในแต่ละ workflow.

คำถามที่ยังค้างอยู่คือ: เมื่อ AI ให้ output มาแล้ว คุณรู้ได้ยังไงว่าดีพอ ดีพอหมายความว่าอะไร และเกณฑ์ควรต่างกันยังไงตาม workflow และ stakes ของงานนั้น.

บทต่อไป [Evaluating AI Output](/using-ai/evaluating-ai-output) ตอบคำถามเหล่านั้นโดยตรง พร้อม framework สำหรับตรวจ output ที่ใช้ได้จริงในทุก workflow ที่เพิ่งเรียนมา.
