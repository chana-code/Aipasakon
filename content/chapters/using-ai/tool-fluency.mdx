# Tool Fluency — เลือก mode ที่ใช่ก่อนพิมพ์คำแรก

คุณเคยถาม ChatGPT เรื่องข้อมูลตลาดปีนี้ แล้วได้คำตอบที่ดูดีมาก — จนไปเช็คแล้วพบว่าตัวเลขเป็นของปีที่แล้ว. หรือใช้ AI เขียน document ยาว แล้วต้องก็อปวางทั้งหมดใหม่ทุกครั้งที่อยากแก้แค่ย่อหน้าเดียว.

ปัญหาไม่ได้อยู่ที่ AI ไม่เก่งพอ. ปัญหาคือใช้ผิด mode.

LLM tool ในปี 2026 ไม่ใช่ "กล่องแชตเดียวที่ทำได้ทุกอย่าง" แต่เป็น **platform ที่รวม mode หลายแบบไว้ด้วยกัน** — แต่ละ mode ออกแบบมาสำหรับงานคนละประเภท. คนที่รู้ว่าต้องใช้ mode ไหนตอนไหน คือคนที่ได้คำตอบดีกว่าคนที่จ่ายเงิน Pro plan แต่ใช้ผิด mode อยู่ตลอด.

## Workshop ไม่ใช่ค้อนตัวเดียว

ลองนึกถึงช่างไม้. เขามีค้อน ไขควง เลื่อย สว่าน — เครื่องมือแต่ละชิ้นออกแบบมาต่างกัน. ถามว่า "ค้อนดีกว่าไขควงไหม" ไม่มีความหมายถ้าไม่บอกว่าทำงานอะไร.

LLM tool ก็เป็นแบบนั้น. เปิด ChatGPT ขึ้นมา จะเห็นปุ่ม Voice, Deep Research, Canvas, ตัวเลือก Reasoning model — เปิด Claude จะเห็น Projects, Artifacts, Web Search, Computer Use. แต่ละปุ่มไม่ใช่ของแถม — มันคือ **เครื่องมือคนละชิ้น** ที่ส่งผลต่อพฤติกรรมของ AI, ความเร็ว, ราคา และคุณภาพคำตอบ.

ผู้ใช้ส่วนใหญ่เปิดมาแล้วพิมพ์ในช่อง default เสมอ เพราะมันอยู่ตรงหน้า. ผลคือพอเจองานที่ default chat ไม่เหมาะ ก็ได้คำตอบที่แย่กว่าที่ควรจะได้ — แล้วก็คิดว่า AI ยังไม่ดีพอ.

**Mode-fluency** คือความสามารถในการรู้ว่าต้องหยิบ mode ไหนขึ้นมาก่อนพิมพ์คำแรก. และนั่นคือสิ่งที่แยก power user ออกจาก casual user ได้ชัดกว่าการเลือกว่าจะใช้ ChatGPT หรือ Claude เสียอีก.

> 💡 **Key insight:** mode-fluency ไม่ใช่เรื่อง "ใช้ Claude หรือ ChatGPT ดี" — นั่นคือการเลือก provider. Mode-fluency คือเมื่อเปิด tool ตัวใดตัวหนึ่งขึ้นมาแล้ว ใช้ feature ไหนใน tool นั้น. คนที่ใช้ AI เก่งมักใช้ได้หลาย provider พร้อมกัน เพราะแต่ละ provider มี mode ที่เด่นต่างกัน.

## Basic Chat — mode ที่ทุกคนรู้จัก แต่ไม่ใช่สำหรับทุกงาน

Basic chat คือสิ่งที่ ChatGPT เปิดตัวด้วยในปี 2022 และยังเป็น default ของทุก tool — พิมพ์เข้าไปหนึ่งกล่อง ได้คำตอบกลับมาหนึ่งกล่อง สลับกันไป.

กลไกเบื้องหลังคือ model รับ context ทั้งหมดของการสนทนาที่ผ่านมา แล้ว generate คำตอบต่อ ตามหลัก next-token prediction ที่อธิบายไว้ใน[บท LLM คืออะไร](/foundations/what-is-an-llm). ความเรียบง่ายนี้ทำให้มันเหมาะมากกับงานบางประเภท — และไม่เหมาะกับอีกหลายประเภท.

**เหมาะกับ:** ถามตอบสั้น, ideation, อธิบาย concept, ปรับ wording, แปลภาษา, เขียน reply อีเมลทั่วไป — งานที่ไม่ต้องการข้อมูลภายนอก ไม่ต้องการ output ยาว และไม่ต้องการแก้ไขหลายรอบ.

**ข้อจำกัดที่ต้องรู้:**

ประการแรก — context จะถูก reset ระหว่างการแชต เว้นแต่ provider มี persistent memory อย่าง ChatGPT Memory หรือ Claude Projects. ดังนั้นถ้าคุณปิด tab แล้วเปิดใหม่ AI ไม่รู้จักคุณแล้ว.

ประการที่สอง — ไม่มี citation. โมเดลตอบจาก parametric knowledge อย่างเดียว นั่นคือสิ่งที่มันเคย "อ่าน" ตอน training — ไม่ได้ค้นข้อมูลใหม่. ดังนั้นข้อมูลที่เปลี่ยนแปลงบ่อย เช่น ราคา, ข่าว, ข้อมูลบริษัท — เสี่ยงเก่าหรือผิด.

ประการที่สาม — แชตยาวขึ้น คุณภาพเสื่อม. งานวิจัยพบว่าเมื่อ context ยาวขึ้น โมเดลมักให้ความสนใจส่วนหัวและท้ายมากกว่ากลาง (lost-in-the-middle effect) — ถ้าข้อมูลสำคัญอยู่กลาง session มีโอกาสถูกมองข้าม.

ลองสังเกตในการใช้งานจริง: ถ้าคุณแชตกับ ChatGPT ยาว 30 รอบ แล้วอ้างถึงสิ่งที่บอกไว้รอบที่ 5 — โมเดลมักตอบไม่แม่นยำ หรือ "ลืม" ราวกับไม่เคยพูดถึง. ไม่ใช่เพราะ ChatGPT แย่ แต่เพราะกลไก attention ของ LLM มีขีดจำกัดเรื่องการกระจาย context ยาว ๆ.

สิ่งที่ต้องจำ: basic chat ดีมากสำหรับงานสั้นและงาน one-shot ที่ไม่ต้องการ context สะสม. พอสังเกตว่า session ยาวขึ้นมากหรือต้องใช้ข้อมูลเฉพาะซ้ำ ๆ — เป็นสัญญาณว่าอาจถึงเวลาย้าย mode.

## Long-Context / Document Mode — เมื่อต้องทำงานกับไฟล์

เวลาคุณต้องสรุป PDF รายงาน 80 หน้า หรือวิเคราะห์ contract 30 ข้อ การพิมพ์สรุปเนื้อหาด้วยมือแล้วถามมันไม่ใช่วิธีที่ดี — attach ไฟล์เข้าไปตรง ๆ ดีกว่า.

ChatGPT, Claude และ Gemini ทุกตัวรองรับการ attach ไฟล์ (PDF, docx, txt, code, รูปภาพ). เมื่อ attach ไฟล์ ระบบแปลงเนื้อหาเป็น token แล้วใส่เข้า context window ของโมเดล — การ "เข้าใจไฟล์" ของ AI คือการที่ token ของไฟล์นั้นอยู่ในขอบเขตที่โมเดลเห็น.

Gemini โดดเด่นในเรื่องนี้เป็นพิเศษ — context window ขนาด 1 ล้าน token ขึ้นไปทำให้อ่านหนังสือหลายเล่มในคราวเดียวได้.

**เหมาะกับ:** สรุปรายงาน, วิเคราะห์ contract, ถามคำถามจาก policy document, เปรียบเทียบ proposal หลายฉบับ, อ่านงานวิจัย.

**ข้อจำกัดที่สำคัญ:**

ยิ่งเอกสารยาว คุณภาพการ "อ่าน" ยิ่งไม่สม่ำเสมอ — ส่วนกลางมักได้รับความสนใจน้อยกว่าหัวและท้าย. ดังนั้นถ้าข้อมูลสำคัญอยู่กลางเอกสาร ลองดึงเฉพาะส่วนนั้นมาถามแยกต่างหาก.

วิธีใช้ที่ฉลาดขึ้นอีกขั้น: แทนที่จะ attach ไฟล์ทั้งหมดแล้วถามแบบกว้าง ลองบอก AI ว่าต้องการอะไรอย่างละเอียดก่อน เช่น "อ่าน contract นี้และหาทุกข้อที่เกี่ยวกับ termination clause" — การระบุ scope ชัดทำให้ precision สูงขึ้นมาก.

นอกจากนี้ ถ้าเอกสารมีตาราง เลข หรือ code block — อย่าเชื่อผลคำนวณโดยไม่ตรวจ. โมเดลไม่ได้คำนวณจริง มันทำ pattern-matching กับเลข ซึ่งผิดได้.

> 💡 **Key insight:** Gemini เหมาะกับเอกสารยาวมากกว่า Claude หรือ ChatGPT ในปัจจุบัน เพราะ context window ใหญ่กว่า — แต่ context window ใหญ่ไม่ได้แปลว่าอ่านได้แม่นยำกว่า แค่ตัดทิ้งน้อยกว่า.

## Canvas / Artifacts Mode — เขียนแล้วแก้ได้ไม่ต้องส่งใหม่ทั้งหมด

ลองนึกสถานการณ์นี้: คุณให้ AI เขียน proposal 5 หน้า ได้มาแล้ว. อยากแก้แค่ย่อหน้าที่สาม. ใน basic chat คุณต้องบอกว่า "แก้ย่อหน้าที่สาม" แล้วรับของทั้ง 5 หน้ากลับมาใหม่ทั้งหมด — วนไปเรื่อย ๆ. ทุกรอบกิน token, context บวมขึ้น, และมีโอกาส AI "ลืม" สิ่งที่เคยตั้งใจไว้ตั้งแต่ต้น.

Canvas mode (ChatGPT Canvas, Gemini Canvas) และ Artifacts (Claude) แก้ปัญหานี้ด้วยการแยก document ออกจาก conversation.

กลไกคือระบบสร้าง artifact แยกต่างหากจาก chat history — document ด้านข้างที่ทั้งคุณและ AI แก้ทีละส่วนได้. เมื่อคุณบอกให้แก้ย่อหน้าที่สาม โมเดลรับ artifact เดิม + คำสั่ง → สร้าง version ใหม่เฉพาะส่วนนั้น — ไม่ต้องส่งของทั้งหมดใหม่.

**เหมาะกับ:** เขียน document ยาว, draft proposal หรือ report ที่ต้องแก้หลายรอบ, สร้าง code ที่ต้อง iterate, ทำ HTML/React prototype (Claude Artifacts ทำได้ดีเป็นพิเศษ).

**ข้อจำกัด:** ผูกกับ format ที่ provider รองรับ (markdown, HTML, code) — งานที่ format แปลกออกไปอาจตกกลับเป็น basic chat. และถ้าแก้ไปมากเกินไปโดยไม่ตรวจ artifact อาจ "ลืม" เจตนาเดิมของตัวเอง.

### ลองทำดู

เปิด Claude หรือ ChatGPT ขึ้นมา แล้วพิมพ์ว่า:

> "เขียน proposal สั้น ๆ ความยาว 1 หน้า เสนอให้บริษัทนำ AI มาช่วยงาน customer support — ใน Canvas/Artifacts"

เมื่อได้ draft แล้ว ลองบอกให้แก้เฉพาะ section แรก "ปรับโทนให้เป็นทางการขึ้น" — สังเกตว่า document ด้านข้างเปลี่ยน แต่ส่วนที่ไม่ได้บอกแก้ยังอยู่ครบ.

นั่นคือข้อดีของ canvas mode เมื่อเทียบกับ basic chat.

## Web Search Mode — ข้อมูลสดจากอินเทอร์เน็ต

LLM ทุกตัวมี training cutoff — วันที่มันหยุด "อ่าน" ข้อมูลใหม่. ChatGPT รู้เรื่องที่เกิดก่อน cutoff date เท่านั้น. ดังนั้นถ้าถามว่า "ราคาน้ำมันวันนี้เท่าไหร่" หรือ "บริษัท X ประกาศอะไรไปเมื่อเดือนที่แล้ว" ใน basic chat — คำตอบที่ได้อาจเก่าหรือ hallucinated.

Web search mode แก้ปัญหานี้. Claude with web search, ChatGPT browse, Gemini with Google Search grounding, Perplexity (สร้างทั้งแอปมาสำหรับเรื่องนี้โดยเฉพาะ) — ทั้งหมดนี้ทำงานในรูปแบบเดียวกัน: โมเดลค้น web ระหว่างตอบ แล้วใช้ผลค้นเป็น context พร้อม citation.

ในเชิงกลไก นี่คือ **RAG (Retrieval-Augmented Generation)** — ค้น query, ดึง snippet จากเว็บ, ใส่เข้า context, แล้ว generate คำตอบพร้อมอ้างอิงแหล่งที่มา. (สถาปัตยกรรม RAG เจาะลึกใน Level 3 บท [RAG](/building-with-ai/rag-retrieval-augmented-generation).)

**เหมาะกับ:** เหตุการณ์ล่าสุด, fact-check, ค้นราคาสินค้า/บริการ, หาข่าวบริษัท, ตรวจ changelog ของ software version ล่าสุด.

**ข้อจำกัดที่ต้องรู้:**

Latency สูงกว่า basic chat (5-30 วินาทีต่อ query) เพราะต้องรอค้นเว็บก่อน.

ที่สำคัญกว่า — citation ที่ได้มา **ต้องตรวจสอบเอง**. pattern ที่พบบ่อยคือ link ถูกต้อง แต่เนื้อหาที่ AI สรุปมาไม่ตรงกับที่ source จริง ๆ บอก. อย่าเชื่อ citation โดยไม่เปิด link ตรวจ key claim.

ตัวอย่างที่เกิดจริง: ถาม AI ว่า "บริษัท X ประกาศกำไรไตรมาส 1 ปี 2026 เท่าไหร่" — AI ดึงข่าวมาได้ อ้าง link มา แต่ตัวเลขที่สรุปในคำตอบอาจเป็น headline ของข่าวเก่าในหน้าเดิม ไม่ใช่ตัวเลขที่ถูกต้อง. ข่าวการเงินเป็นกรณีที่ต้องตรวจ source เสมอ.

Perplexity โดดเด่นใน web search mode เพราะ workflow ออกแบบมาสำหรับเรื่องนี้โดยเฉพาะ — citation inline ทุกประโยค ทำให้ตรวจได้ง่ายกว่า ChatGPT หรือ Claude ที่บางครั้ง cite ในรูปเชิงอรรถท้าย.

> 💡 **Key insight:** web search mode ไม่ได้ "ฉลาดขึ้น" — มันแค่ได้ข้อมูลใหม่กว่า. โมเดลยังเดาคำต่อไปตามปกติ แค่มี context เพิ่มจากเว็บ. hallucination ยังเกิดได้ โดยเฉพาะเมื่อ source ต้นทางอยู่หลัง paywall และโมเดลเห็นแค่ headline.

## Deep Research Mode — สำรวจทั้ง field ในครั้งเดียว

Web search mode ค้นหาคำตอบสำหรับคำถามเดียว. แต่ถ้าคุณต้องการทำ market scan, competitive analysis, หรือ literature review ของ field ที่ไม่คุ้น — ค้นทีละคำถามจะช้ามาก.

Deep Research mode (Perplexity Deep Research, ChatGPT Deep Research, Gemini Deep Research, Claude Research) ออกแบบมาสำหรับงานแบบนี้โดยตรง.

กลไกคือ **agentic loop** — โมเดลไม่ได้ค้นครั้งเดียวแล้วตอบ แต่:
1. สร้าง research plan
2. ค้น source แรก
3. อ่าน สังเคราะห์ แล้วตั้ง sub-question
4. ค้นต่อจาก sub-question
5. วนซ้ำจนพอ แล้ว produce รายงานพร้อม citation

ทั้งกระบวนการใช้เวลา 5-30 นาที และค้นได้หลายสิบถึงร้อยแหล่งต่อ query.

**เหมาะกับ:** สรุปสถานะของ field ที่ไม่คุ้น, เปรียบเทียบ vendor/เครื่องมือ, due diligence บริษัท, สำรวจตลาดก่อนตัดสินใจสำคัญ.

สถานการณ์ที่ deep research ให้ ROI สูงมากคืองานที่ถ้าทำเองต้องใช้เวลาหลายชั่วโมง เช่น "ต้องการเลือก CRM สำหรับทีม 20 คน — เปรียบเทียบ Salesforce, HubSpot, Zoho, Monday CRM ในด้านราคา, feature, ความเหมาะกับ SME ไทย" — งานแบบนี้ทำมือเต็มวัน deep research ทำใน 20 นาที.

**ข้อจำกัดที่สำคัญ:**

ใช้เวลานาน — ถ้าแค่อยากรู้ข้อเท็จจริงเดียว web search mode ดีกว่า.

Report ที่ได้มักดู authoritative มาก — ซึ่งอันตราย เพราะคนที่ไม่รู้ field นั้นดีมักเชื่อทั้งหมดโดยไม่ตรวจ. แต่ deep research ยัง hallucinate ได้ โดยเฉพาะ paper วิชาการที่อยู่หลัง paywall.

ถาม vague ได้ vague — คุณภาพ output ขึ้นกับ query แรกที่ใส่เข้าไป.

เทคนิคที่ช่วยได้: ก่อนส่ง query ไป deep research ลองถาม basic chat ก่อนว่า "ช่วยฉันเขียน research query ที่ชัดและครอบคลุมสำหรับ [หัวข้อ]" — ใช้ AI ช่วยตั้งคำถามให้ดีขึ้นก่อนส่ง query ไปรัน.

## Reasoning Mode — เมื่องานต้องการความคิดที่รอบคอบกว่าปกติ

บางครั้งคุณถาม AI เรื่อง logic ของสัญญา หรือขอให้ตรวจ code หา edge case — แล้วได้คำตอบที่ฟังดูถูก แต่พอเช็คเองพบว่าพลาด.

นั่นไม่ใช่เพราะ AI ไม่ฉลาดพอ — แต่เพราะ basic chat model ถูก train มาเพื่อ generate คำตอบที่ดู "น่าจะถูก" ไม่ใช่เพื่อ trace ทุก step อย่างระมัดระวัง.

Reasoning mode แก้ปัญหานี้ด้วยการให้โมเดล "คิดออกเสียง" ใน hidden chain-of-thought ก่อนตอบ — OpenAI o1/o3, Claude Extended Thinking, Gemini Thinking, DeepSeek R1 คือตัวอย่างในปี 2026.

กลไกต่างจาก chat model ปกติ — reasoning model ถูก train (มักด้วย reinforcement learning บนปัญหาที่ verify ได้) ให้สร้าง reasoning chain ก่อน final answer. ผลคือมันหา edge case ที่ chat model มองข้าม แต่ใช้เวลาและต้นทุนสูงกว่า.

**เหมาะกับ:** โจทย์เลข, code debugging ที่ต้องหา logic error, planning หลายขั้นตอน, วิเคราะห์ข้อโต้แย้งที่ซับซ้อน, ตรวจสอบ consistency ของเอกสาร.

**ข้อจำกัดที่สำคัญมาก:**

Reasoning mode **แย่กว่า** basic chat ในงานเขียน creative, brainstorm, สรุปสั้น ๆ, และงาน conversational ทั่วไป. เพราะมันถูก train มาให้คิดเป็นขั้น ผลคือ output แห้ง มีโครงสร้างแข็ง ไม่ลื่นไหล.

Mode ไม่ใช่ ranking — ไม่ได้แปลว่า reasoning mode ฉลาดกว่า chat mode. มันแค่ specialization คนละแบบ.

นอกจากนี้ latency สูงมาก — บาง query ใช้เวลา 30 วินาทีถึงหลายนาที. และราคาต่อ query สูงกว่า เพราะ reasoning chain นับเป็น token ด้วย.

ลองทดสอบด้วยตัวเองเพื่อให้เห็นภาพ: ถามคำถามเดียวกันใน basic chat กับ reasoning mode แล้วเปรียบเทียบ.

- ถาม "เขียน email ขอบคุณหลังการประชุม โทนอบอุ่น" — basic chat จะได้ email ที่อ่านลื่น, reasoning mode จะได้ email ที่ดูเหมือนถูกวิเคราะห์มาก เขียนเป็น step ชัดเจนแต่แข็ง.
- ถาม "ฟังก์ชัน Python นี้มี edge case อะไรบ้าง" — reasoning mode จะค้นหา edge case ได้ครบกว่ามาก.

ความต่างชัดเจน. เลือกให้ตรงงาน.

> 💡 **Key insight:** สัญญาณที่บอกว่าต้องสลับไป reasoning mode คือ "งานนี้ผิดหน่อยเดียวเสียหายมาก" — เช่น logic ของสัญญา, เลขใน financial model, หรือ code ที่จะ deploy จริง. ถ้างานแค่ต้องการคำตอบที่ดูพอใช้ได้ basic chat เร็วกว่าและถูกกว่า.

### ลองทำดู

เปิด Claude หรือ ChatGPT แล้วทดสอบความต่างของ reasoning mode กับ basic chat:

**Step 1:** ใน basic chat ถามว่า

> "ฉันมี list ของตัวเลข เช่น [3, 1, 4, 1, 5, 9, 2, 6]. ฉันอยากหาตัวเลขที่ซ้ำกัน แล้วเอาตัวซ้ำออก เหลือแค่ตัวที่ไม่ซ้ำ เรียงจากน้อยไปมาก. ผลลัพธ์ควรเป็นอะไร?"

**Step 2:** ถามคำถามเดิมใน reasoning mode (Claude Extended Thinking หรือ o1/o3 ใน ChatGPT)

**Step 3:** เปรียบเทียบ — ดูว่า reasoning mode แสดง chain-of-thought ยังไง และผลลัพธ์ต่างกันไหม. สังเกตว่า reasoning mode จะ trace ทีละขั้นชัดเจนกว่า แต่ใช้เวลานานกว่า.

## Voice Mode — AI ในมือถือขณะเดินทาง

เวลาขับรถหรือเดิน บางครั้งอยากคุยกับ AI แต่พิมพ์ไม่ได้ — หรืออยากระดมไอเดียออกเสียงโดยไม่ต้องนั่งหน้าจอ. Voice mode ถูกออกแบบมาสำหรับสถานการณ์แบบนี้.

ChatGPT Advanced Voice Mode, Gemini Live, Claude Voice — ทั้งหมดรับเสียงพูดและตอบกลับเป็นเสียงทันที. โมเดล multimodal ใหม่รับ audio token โดยตรง (ไม่แปลงเป็น text ก่อน) ทำให้รักษาน้ำเสียงและจังหวะได้ — ต่างจาก voice mode รุ่นแรกที่เป็นแค่ speech-to-text → LLM → text-to-speech ซ้อนกัน.

**เหมาะกับ:** ระดมไอเดียขณะเดินทาง, ฝึกภาษาต่างประเทศ, อธิบาย concept ให้เด็ก, brainstorm ที่ไม่ต้องการ output เป็น text.

**ข้อจำกัดที่ต้องรู้:**

ควบคุม output detail ยาก — บอก "ทำเป็น bullet 5 ข้อ" ผ่านเสียงไม่ได้ผลเท่าใน text.

ตรวจสอบความถูกต้องยาก — เสียงผ่านไปเร็ว ถ้าต้องการข้อมูลที่แม่นยำ เช่น เลข, ชื่อเฉพาะ, URL ไม่ควรใช้ voice mode.

Hallucination ใน voice เกิดได้เหมือนกัน แต่ "ดูเหมือนจริง" มากกว่า เพราะน้ำเสียงมั่นใจ.

## Custom GPT / Projects / Gems — ตั้งค่าครั้งเดียว ใช้ได้นาน

สมมติคุณทำงานที่ต้องตอบอีเมลลูกค้าวันละหลายสิบฉบับในสไตล์เดียวกัน. ทุกครั้งที่เปิด ChatGPT ใหม่ คุณต้องพิมพ์บริบทว่าคุณเป็นใคร บริษัทขายอะไร โทนที่ใช้เป็นแบบไหน — แล้วค่อยถามงานจริง.

Custom GPT (ChatGPT), Claude Projects, Gemini Gems แก้ปัญหานี้ด้วยการ save configuration ไว้เป็น "ตัวตน" ที่เรียกใช้ซ้ำได้.

กลไกคือทุกครั้งที่เริ่ม session ใหม่ใน custom assistant ระบบ inject system prompt และ knowledge files (ถ้ามี) เข้า context อัตโนมัติ — เหมือนมีคนบรีฟ AI ให้ทุกครั้งก่อนคุณพิมพ์คำแรก.

**เหมาะกับ:** งานซ้ำบ่อย เช่น เขียนอีเมลในสไตล์ของทีม, ตอบคำถาม customer ตาม FAQ, review document ตาม style guide, งานที่ต้องการบริบทเดิมทุกครั้ง.

**ข้อจำกัด:**

Provider lock-in — Custom GPT ใช้ได้แค่ใน ChatGPT, Project ใช้ได้แค่ใน Claude.

Knowledge file ยังถูกจำกัดด้วย context window — file ใหญ่ถูก chunk และ retrieve บางส่วน ไม่ใช่ทั้งหมด.

Custom GPT/Project ไม่ได้ทำให้โมเดล "ฉลาดขึ้น" ในด้าน domain ของคุณ — มันแค่ inject context. ถ้าโมเดล base ตอบเรื่องนั้นได้แย่ Custom GPT ก็แย่เหมือนกัน.

### ลองทำดู

เปิด Claude แล้วสร้าง Project ใหม่. ใส่ข้อมูลนี้เป็น instructions:

> "คุณเป็น assistant สำหรับทีม [ชื่อทีม]. ตอบเป็นภาษาไทยเสมอ. โทนเป็นมืออาชีพแต่เป็นกันเอง. เมื่อต้องเขียนอีเมล ให้ใช้รูปแบบ: เปิดด้วย context สั้น ๆ → สิ่งที่ต้องการ → ขอบคุณและปิด"

แล้วลอง start conversation ใหม่ใน Project นั้น สังเกตว่า AI รู้บริบทโดยไม่ต้องอธิบายซ้ำ.

## Agent / Computer-Use Mode — AI ที่ลงมือทำแทนคุณ

ทุก mode ที่พูดถึงมา AI ยังแค่ "ตอบ" — คุณยังต้องลงมือทำเอง. Agent mode ต่างออกไป: AI ควบคุม browser หรือ desktop ทำงานหลายขั้นด้วยตัวเอง.

Claude Computer Use, OpenAI Operator, Gemini agentic features — กลไกคือ model + tool execution loop: โมเดลเห็น screenshot หรือ DOM → ตัดสินใจ action ถัดไป → ระบบ execute → ได้ผลลัพธ์กลับมา → วนซ้ำจนเสร็จ.

**เหมาะกับ (ในทางทฤษฎี):** workflow ที่ต้องใช้หลายแอป, copy ข้อมูลระหว่างระบบที่ไม่มี API, repetitive task ที่ต้องเปิดเว็บ.

**ข้อจำกัดที่ต้องเข้าใจก่อนใช้ในปี 2026:**

ยัง brittle มาก — error เล็กในขั้นแรกขยายเป็น error ใหญ่ในขั้นถัดไป (compounding error). ถ้า UI ของเว็บที่ใช้เปลี่ยน แม้แต่ปุ่มที่ย้ายที่ไปไม่กี่เซนติเมตร agent ก็อาจล้มเหลว.

มีความเสี่ยงด้าน security — AI ที่ควบคุม browser ของคุณสามารถเห็นข้อมูลทุกอย่างในหน้าจอ รวมถึงอาจถูก prompt injection จากเว็บที่เปิด (เว็บฝัง instruction ซ่อนไว้เพื่อควบคุม AI).

ต้องนั่งเฝ้า — ในปี 2026 ยังไม่ใช่ "สั่งแล้วปล่อยทิ้งไว้" ได้จริง ๆ. ใช้กับงานที่ถ้าผิดพลาดเสียหายน้อย.

ลองนึกภาพให้ชัด: สั่ง agent ให้เข้าไปในระบบ HR กรอกข้อมูลพนักงานใหม่ 10 คน. ถ้า field ใดใน UI เปลี่ยนชื่อ agent อาจกรอกข้อมูลผิด field โดยไม่รู้ตัว — แล้ว error นั้นขยายต่อทุกขั้น. ราคาของ error นี้สูง.

Rule of thumb ที่ใช้ได้: ใช้ agent mode กับงานที่ (1) ทำมือได้ในเวลาสั้น ๆ แต่ทำซ้ำบ่อยมาก และ (2) ถ้าผิดแก้ได้ง่าย. งานที่ผิดแล้วแก้ยาก — ทำมือดีกว่า.

สำหรับรายละเอียดสถาปัตยกรรมของ agent mode — จะอธิบายลึกกว่านี้ใน Level 3 เรื่อง agents และ orchestration.

## API Mode — ชั้นล่างสุดที่ทุก mode วางอยู่

มี mode หนึ่งที่ไม่ได้อยู่ใน UI ปกติแต่สำคัญมากในการเข้าใจภาพรวม — API mode.

ทุก mode ที่พูดถึงมา ตั้งแต่ basic chat ไปจนถึง deep research — ล้วนเป็น web UI ที่เรียก API เบื้องหลัง. เมื่อคุณกด "Deep Research" ใน ChatGPT, ChatGPT เรียก OpenAI API พร้อม parameter ที่ preset ไว้แล้ว. เมื่อคุณเปิด Claude Artifacts, Claude เรียก Anthropic API พร้อม system prompt และ configuration พิเศษ.

API คือ "ก็อกน้ำตรง" ที่ข้ามตัว UI ทั้งหมด — นักพัฒนาและทีมเทคนิคเรียก model ตรงผ่าน HTTP request. ทำให้ integrate AI เข้า product, batch process ข้อมูล หรือสร้าง assistant ของตัวเองได้.

สำหรับผู้ใช้ทั่วไปในปี 2026, API mode เป็นเรื่องของ Level 3 — ต้องเขียน code, จัดการ rate limit, billing แบบ pay-per-token. รายละเอียดอยู่ใน Level 3 ทั้งหมด.

แต่มีสิ่งหนึ่งที่รู้ตอนนี้จะเป็นประโยชน์: เมื่อเห็น feature ใหม่ใน UI ของ ChatGPT หรือ Claude — มักมี underlying API parameter ที่สอดคล้อง. เข้าใจ API ทำให้เข้าใจว่า mode แต่ละอันต่างกันที่ระดับไหน ไม่ใช่แค่ที่หน้าตาของปุ่ม.

## เรื่องที่คนเข้าใจผิดบ่อย

**"Pro plan มี mode พิเศษ เลยได้คำตอบดีกว่าเสมอ"**

Pro plan ปลดล็อก mode ที่เหมาะกับงานบางประเภท เช่น deep research, reasoning, large context — แต่ถ้าใช้ผิด mode ก็ยังได้คำตอบแย่กว่า free user ที่เลือก mode ถูก. การจ่ายเงินไม่ทดแทน mode-fluency.

**"Reasoning mode ฉลาดกว่า chat mode เลือกอันฉลาดกว่าไว้ก่อนดีกว่า"**

Reasoning mode แย่กว่า chat mode ในงาน creative, brainstorm, summarization สั้น ๆ เพราะถูก train มาให้คิดเป็นขั้น ผล output จึงแห้งและช้า. Mode คือ specialization ไม่ใช่ ranking.

**"Deep Research อ้าง citation มาแล้วเชื่อได้"**

Deep research มักอ้าง source ที่มีจริง แต่เนื้อหาที่สรุปมาอาจไม่ตรงกับที่ source จริง ๆ บอก — โดยเฉพาะ paper วิชาการที่ AI เห็นแค่ abstract แต่สรุปเหมือนอ่านทั้งเล่ม. ต้องเปิด link ตรวจ key claim เสมอ.

**"Agent mode สั่งแล้วไปทำอย่างอื่นได้"**

ในปี 2026 ยังต้องนั่งเฝ้าเกือบตลอด. error ที่ขั้นแรกขยายในขั้นถัดไป และ AI ที่ควบคุม browser เสี่ยง prompt injection จากเว็บที่เปิด.

---

## Decision Framework — เลือก mode ใน 5 วินาที

| งาน | Mode ที่ใช้ |
|---|---|
| ถามตอบสั้น / ideation / reword / แปล | Basic chat |
| ข้อมูลล่าสุด / fact-check / ราคา / ข่าว | Web search |
| สรุปหรือวิเคราะห์เอกสารที่มีอยู่ | Long-context (attach file) |
| เขียน document หรือ code ที่ต้องแก้หลายรอบ | Canvas / Artifacts |
| ทำตอนขับรถ / เดิน / ฝึกพูด | Voice |
| โจทย์เลข, logic, code ซับซ้อน, หา edge case | Reasoning mode |
| สำรวจ field ที่ไม่คุ้น / market scan / competitive analysis | Deep Research |
| งานซ้ำทุกวันที่ต้องการบริบทเดิม | Custom GPT / Project |
| Workflow หลายแอปที่ไม่มี API (ระวัง) | Agent / Computer Use |
| Integrate เข้าระบบ / batch processing | API (Level 3) |

## เมื่อต้องย้าย Mode กลางทาง

บางงานต้องใช้มากกว่าหนึ่ง mode ในลำดับ — เช่น เริ่มที่ deep research เพื่อเก็บข้อมูล ย้ายไป canvas เพื่อเขียน document แล้วใช้ reasoning mode ตรวจ logic ก่อนส่ง.

ปัญหาคือการย้าย mode มักไม่ได้พา context ไปทั้งหมด. บาง tool ย้าย smooth (Claude Artifacts เกิดใน chat session เดิมอยู่แล้ว) แต่บาง tool ต้องเริ่ม session ใหม่ (deep research = session ใหม่เสมอ).

**กลยุทธ์ที่ใช้ได้:**

คิดก่อนว่างานนี้ใหญ่แค่ไหนและต้องใช้ mode ไหน — ถ้าต้องใช้ deep research อยู่แล้ว อย่าเริ่มใน basic chat แล้วค่อยย้าย.

ถ้าต้องย้ายกลางทาง ก่อนย้ายให้ขอ AI "สรุปสิ่งที่เราคุยกันมาเป็น brief สั้น ๆ" แล้วเอา brief นั้นไปเริ่ม session ใหม่ในอีก mode — context ไม่หาย.

> 💡 **Key insight:** งานเดียวอาจต้องใช้หลาย mode ในลำดับ — Deep Research → Canvas → Reasoning review คือ workflow ที่ใช้จริงสำหรับ report สำคัญ. Mode-fluency รวมถึงการวางแผน sequence นี้ล่วงหน้า ไม่ใช่แค่รู้จักแต่ละ mode แบบแยกกัน.

## ตารางเปรียบเทียบ Mode — ภาพรวมในที่เดียว

| Mode | ความเร็ว | ราคา | เหมาะกับ | ไม่เหมาะกับ |
|---|---|---|---|---|
| Basic chat | เร็ว | ต่ำ | งานสั้น ideation แปล | ข้อมูลล่าสุด, งานยาวที่ต้องแก้ |
| Long-context (file) | ปานกลาง | ปานกลาง | สรุป/วิเคราะห์เอกสาร | เอกสารที่ยาวเกิน context window |
| Canvas / Artifacts | ปานกลาง | ปานกลาง | document ที่ต้องแก้หลายรอบ | งานสั้น one-shot |
| Web search | ช้ากว่า chat | ปานกลาง | ข้อมูลล่าสุด fact-check | งานที่ไม่ต้องการข้อมูลใหม่ |
| Deep Research | ช้ามาก | สูง | market scan, literature review | quick lookup, งานสั้น |
| Reasoning | ช้า | สูง | logic, code, math ซับซ้อน | creative, brainstorm, งาน conversational |
| Voice | เรียลไทม์ | ปานกลาง | เดินทาง ฝึกพูด | งานที่ต้องการ precision สูง |
| Custom GPT/Project | เท่า base mode | เท่า base mode | งานซ้ำที่ต้องการบริบทเดิม | งาน one-off ที่ไม่ซ้ำ |
| Agent | ช้ามาก | สูง | workflow หลายแอปที่ซ้ำ | งานสำคัญที่ error แก้ยาก |

## Worked Examples

### เตรียม report บอร์ดบริษัทเรื่องคู่แข่งในตลาด

ถ้าเปิด basic chat แล้วถามว่า "คู่แข่งของเราในตลาด X มีใครบ้างและทำอะไร" — อาจได้คำตอบที่ดูดี แต่ข้อมูลเป็นของช่วง training cutoff และไม่มี citation. ใส่ใน report จริงเสี่ยงมาก.

Mode chain ที่ใช้ได้จริง:
1. **Deep Research** — ขอ "comprehensive scan คู่แข่งในตลาด X ปี 2026 พร้อม citation" (รอ ~15 นาที)
2. **Canvas / Artifacts** — paste output แล้วบอก "ปรับเป็น executive summary 2 หน้าในโทน board document" — แก้ด้านข้างไปมา
3. **Reasoning mode** — "review logic ของ argument ในนี้ — มีจุดไหนที่ assertion ไม่มี evidence รองรับ" — sanity check ก่อนส่ง

### เขียน reply อีเมลลูกค้าที่ไม่พอใจ

Deep research — over-engineering. งานนี้สั้น ไม่ต้องการข้อมูลภายนอก ไม่ต้องการ reasoning ลึก.

Basic chat — ขอ "draft reply สุภาพ ยอมรับปัญหา เสนอ solution X" แล้วแก้ทีละ pass สั้น ๆ. ถ้าทำ pattern นี้บ่อย ย้ายไป Custom Project ตั้ง template ครั้งเดียว ใช้ตลอด.

### ตรวจ code function ที่มี logic bug

Basic chat อาจมองข้าม edge case แล้วตอบ "ดูเหมือนถูก" ทั้งที่ไม่ถูก.

Reasoning mode — paste code แล้วบอก "เช็คทุก edge case ที่อาจทำให้ function นี้ทำงานผิด". โมเดล trace ทุก path อย่างระมัดระวังกว่า. งานวิจัยบน mathematical และ coding benchmark แสดงว่า reasoning model ทำได้ดีกว่า chat model อย่างมีนัยสำคัญในงานประเภทนี้.

### ฟังประชุม Zoom 2 ชั่วโมงแล้วต้องส่ง summary

Voice mode ไม่ใช่ทางเลือกที่ถูกต้อง — Voice ออกแบบมาสำหรับสนทนา ไม่ใช่ transcription.

Mode chain:
1. Transcribe ก่อนด้วยเครื่องมือแยก เช่น Otter, Granola, หรือ Zoom built-in
2. Long-context mode ใน Claude หรือ Gemini — attach transcript แล้วขอสรุปตาม structure ที่ต้องการ
3. ถ้าต้องการ action item list ละเอียด — Canvas แก้ทีละ section

## สรุป — เครื่องมือถูกงานคือครึ่งหนึ่งของคำตอบที่ดี

ทุก mode ที่พูดถึงในบทนี้มีอยู่ในแอปที่คุณใช้อยู่แล้ว — ไม่ต้องสมัครอะไรใหม่ ไม่ต้องเรียนรู้ tool ใหม่. แค่รู้ว่าปุ่มไหนทำอะไร และงานไหนเหมาะกับปุ่มไหน.

Mode-fluency ไม่ใช่เรื่องของเทคนิคซับซ้อน — มันคือการเข้าใจว่าแต่ละ mode ออกแบบมาเพื่ออะไร แล้วหยิบขึ้นมาใช้ให้ตรง. basic chat เหมาะกับงานเร็วสั้น, canvas เหมาะกับงานที่ต้องแก้ไปมา, reasoning mode เหมาะกับงานที่ error เล็กน้อยมีผลใหญ่, deep research เหมาะกับการสำรวจ field ที่ไม่คุ้น.

สิ่งที่ทำให้ผู้ใช้ระดับสูงต่างจากคนทั่วไปไม่ใช่แค่รู้จัก mode มากกว่า — แต่คือ **รู้ว่าต้องหยิบ mode ไหนขึ้นมาก่อนพิมพ์คำแรก** และวางแผน sequence ล่วงหน้าเมื่องานต้องการหลาย mode ต่อกัน.

อีกสิ่งหนึ่งที่ควรรู้: mode ใหม่ออกเรื่อย ๆ — ในปี 2025-2026 ชื่อ feature เปลี่ยนทุกไม่กี่เดือน. Deep Research เพิ่มเข้ามาในหลาย provider ช่วงปลาย 2024 ต้นปี 2025, voice mode กลายเป็น multimodal แบบ native audio ในปี 2024-2025, agent mode ยังอยู่ระหว่างพัฒนา. สิ่งที่อ่านในบทนี้จะ outdated ใน 12 เดือน — แต่กรอบการคิดว่า "mode ไหนเหมาะกับงานแบบไหน" จะยังใช้ได้.

ในบทต่อไป เราจะเอา mode เหล่านี้มาประกอบเป็น workflow ที่ใช้ได้จริงในงาน knowledge work — ตั้งแต่ research เริ่มต้น ไปจนถึง deliverable ที่ส่งให้คนอื่นได้.
