# 11 Pattern สำหรับเขียน Prompt — คู่มือที่เปิดซ้ำได้ตลอดชีวิต

คุณเคยลองพิมพ์คำขอเดิมซ้ำกันหลายรอบ แล้วงง ว่าบางรอบ AI ตอบดีมาก บางรอบตอบออกมาแบน ๆ ไม่มีชีวิต. ความต่างไม่ได้อยู่ที่โชค แต่อยู่ที่ **โครงสร้างของ prompt**.

นักใช้ AI ที่ทำงานกับเครื่องมือพวกนี้จริง ๆ ทุกวันพบว่ามี pattern บางอย่างที่ใช้ซ้ำได้ในงานหลายแบบ ได้ผลสม่ำเสมอ. pattern เหล่านี้ไม่ใช่ magic words ไม่ใช่ "เคล็ดลับลับ" ที่มีคนซ่อนไว้ แต่เป็นวิธีให้ข้อมูลกับโมเดลในรูปแบบที่โมเดล "ประมวลผลได้ดีกว่า".

บทนี้แคตตาล็อก 11 pattern หลัก พร้อมบอกว่าใช้เมื่อไหร่ ใช้ยังไง และที่สำคัญกว่า — ใช้แล้วไม่ได้ผลเมื่อไหร่. เพราะทุก pattern มีเงื่อนไข.

## ทำไม Pattern ถึงทำงานได้

ก่อนเข้าเรื่อง pattern แต่ละตัว ต้องเข้าใจก่อนว่าทำไมมันถึงได้ผล.

ถ้าคุณอ่านบทก่อน ๆ มาแล้ว คุณรู้ว่า LLM ทำงานโดยการ "เดาคำต่อไป" ตามบริบทที่มี. นั่นแปลว่า ยิ่งคุณให้บริบทที่ตรงกับ "โลก" ที่คำตอบที่ดีอยู่ในนั้น มากเท่าไหร่ โมเดลก็จะเดาคำตอบที่ดีได้มากขึ้นเท่านั้น.

ลองนึกแบบนี้: ถ้าคุณถามแค่ว่า "สรุปให้หน่อย" — โมเดลไม่รู้ว่าคุณต้องการสรุปสำหรับใคร ยาวแค่ไหน ใช้โทนไหน ให้ความสำคัญกับส่วนไหน. มันเดาแบบกลาง ๆ ได้ผลกลาง ๆ.

แต่ถ้าคุณบอกว่า "สรุปให้ทีมผู้บริหาร 3 bullet point แต่ละ bullet ไม่เกิน 20 คำ เน้นประเด็นเรื่องงบประมาณและ timeline ข้ามเรื่องเทคนิคไป" — โมเดลรู้ชัดว่าคำตอบที่ดีอยู่ในโลกแบบไหน และเดาเข้าหาโลกนั้น.

> 💡 **กลไกหลัก:** Pattern ไม่ใช่ magic — มันคือวิธีให้ข้อมูลกับโมเดลในรูปแบบที่โมเดล distribute ความน่าจะเป็นไปหาคำตอบที่คุณต้องการได้แม่นยำขึ้น. ถ้าไม่เข้าใจกลไกนี้ คุณใช้ pattern ได้แค่แบบท่อง — แต่ถ้าเข้าใจ คุณดัดแปลงและประกอบ pattern ใหม่ได้เอง.

## Pattern 1 — Role Prompting

### มันคืออะไร

บอกโมเดลให้ "สวมบทบาท" ก่อนทำงาน. ไม่ใช่ให้มันแกล้งทำ — แต่เป็นการบอกว่า "คำตอบที่ฉันต้องการมันอยู่ในโลกของคนแบบนี้".

เวลาคุณบอกว่า "Act as a senior technical writer" โมเดลจะ shift distribution ของคำที่จะ generate ไปใกล้กับสิ่งที่ senior technical writer เขียน — ภาษาจะ precise ขึ้น passive voice ลดลง jargon ที่ไม่จำเป็นจะหาย.

**ใช้เมื่อ:** output ที่ได้ฟังดู generic เกินไป ไม่ตรงกับวงการที่ต้องการ หรือ tone ผิด.

**Template:**
```
Act as a [specific role] with [experience/context].
[Task]
```

**ตัวอย่าง:**
```
Act as a senior technical writer at a SaaS company.
Rewrite the changelog below so it's clear to non-engineers 
but still accurate. Keep under 80 words.
```

**จุดที่หยุดช่วย:**
- Role ที่ generic เกินไป ("act as an expert") ไม่ค่อยช่วย ต้องระบุให้ชัด
- Role ไม่ได้ทำให้โมเดล "รู้มากขึ้น" — มันแค่เปลี่ยน style และ tone. งานที่ต้องการข้อเท็จจริงถูกต้องยังต้องตรวจสอบเหมือนเดิม
- โมเดลรุ่นใหม่ (Claude 4, GPT-4o) มี default style ที่ดีขึ้นมาก role ช่วยน้อยลงสำหรับงานทั่วไป — ยังได้ผลในงานที่ต้องการ voice เฉพาะจริง ๆ

## Pattern 2 — Few-Shot Prompting

### มันคืออะไร

ให้ตัวอย่าง input → output 2-5 คู่ใน prompt แล้วปล่อยให้โมเดล "เดา pattern" แล้วทำซ้ำกับ input ใหม่.

นักวิจัยเรียกมันว่า "in-context learning" — โมเดลไม่ได้ถูก retrain แต่ใช้ตัวอย่างที่ให้มาเป็น mini dataset ใน context window เพื่อ calibrate output ทันที.

**ใช้เมื่อ:**
- format ของ output มี structure ที่ไม่ standard
- งาน classification ที่ category เฉพาะของคุณ — โมเดลไม่รู้จาก training
- zero-shot ให้ผลไม่ stable ตอบได้บ้างไม่ได้บ้าง

**Template:**
```
[Instruction]

Input: [example 1] → Output: [example 1]
Input: [example 2] → Output: [example 2]
Input: [example 3] → Output: [example 3]

Input: [real input] → Output:
```

**ตัวอย่างจริง:** จำแนก sentiment ของ review สินค้าภาษาไทย
```
จำแนก sentiment เป็น positive / negative / neutral

"ร้านนี้อร่อยมาก กลับมาแน่นอน"        → positive
"รอนานมาก แอร์ก็เสีย ไม่กลับมาแล้ว"   → negative
"อาหารธรรมดา ราคาก็พอดี"             → neutral

"ส่งช้าเกินไป แต่รสชาติโอเค"          →
```

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วลองสองแบบนี้กับ review เดียวกัน:

**แบบที่ 1 (zero-shot):**
```
จำแนก sentiment ของ review นี้: "บริการดีมาก แต่ราคาแพงไปหน่อย"
```

**แบบที่ 2 (few-shot):**
```
จำแนก sentiment เป็น positive / mixed / negative

"อร่อยมาก ราคาก็โอเค" → positive
"โอเคนะ แต่ไม่ประทับใจ" → mixed
"แย่มาก ไม่กลับมาแล้ว" → negative

"บริการดีมาก แต่ราคาแพงไปหน่อย" →
```

สังเกตว่า few-shot ทำให้ผลที่ได้ตรงกับ category ที่คุณกำหนดมากขึ้น ไม่ใช่ category ที่โมเดลเดาเอง.

**จุดที่หยุดช่วย:**
- เกิน 5-8 ตัวอย่าง ผลมักไม่ดีขึ้น — แค่เปลืองพื้นที่ใน context window
- ตัวอย่างที่เลือกมา bias ผลแรง ถ้าตัวอย่างทั้งหมดเป็น positive โมเดลจะ lean ไป positive แม้ input ควรเป็น neutral
- งาน reasoning ลึก ๆ few-shot อย่างเดียวไม่พอ ต้องคู่กับ Chain-of-Thought

## Pattern 3 — Chain-of-Thought (CoT)

### มันคืออะไร

บอกโมเดลให้ "คิดออกมาเป็นขั้น" ก่อนตอบคำตอบสุดท้าย. ฟังดูง่าย แต่กลไกข้างหลังไม่ง่าย.

เวลาโมเดลเขียน reasoning ออกมา tokens ของ "กระบวนการคิด" กลายเป็น context ที่โมเดลใช้ในการ generate tokens ต่อไป. เหมือนการ "ทำงานบนกระดาษทด" — แทนที่จะข้ามไปคำตอบเลย ซึ่งบางครั้งทำให้ผิดเพราะ context หมด.

paper ดั้งเดิมโดย Wei et al. (2022) แสดงว่า CoT ช่วย performance ใน math word problems และ multi-step reasoning อย่างมีนัยสำคัญ — แต่ใช้ได้ผลเฉพาะกับโมเดลขนาดใหญ่เท่านั้น.

**ใช้เมื่อ:**
- งาน math, logic, planning, multi-hop reasoning — "ถ้า A แล้ว B แล้ว C ดังนั้น..."
- ต้องการ audit trail — ดูว่าโมเดล reason ผิดตรงไหน
- zero-shot ตอบสุดท้ายผิดบ่อย และสงสัยว่า reasoning พังตรงไหน

**Template:**
```
[คำถาม]

Let's think step by step.
```
หรือแบบที่ชัดกว่า:
```
[คำถาม]

Before answering, list the steps you'll take, 
work through each step, then give the final answer.
```

**ตัวอย่าง:**
```
ร้านขายขนม 3 ร้าน
ร้าน A: ชิ้นละ 12 บาท ลด 10%
ร้าน B: ชิ้นละ 11 บาท ไม่ลด
ร้าน C: ชิ้นละ 15 บาท ลด 30%

ถ้าซื้อ 20 ชิ้น ร้านไหนถูกที่สุด?

Let's think step by step.
```

> 💡 **เรื่องที่ต้องรู้:** "การคิด" ที่โมเดลเขียนออกมาไม่ใช่กระบวนการคิดจริงของโมเดล — มันคือ generated text ที่ดูเหมือนการคิด. งานวิจัย interpretability แสดงว่า reasoning chain บางครั้งไม่ตรงกับ internal computation จริง. ใช้ CoT เพื่อ accuracy ได้ แต่อย่าเชื่อ "เหตุผล" ที่โมเดลให้มาว่านั่นคือสิ่งที่มัน "คิด" จริง ๆ.

**จุดที่หยุดช่วย:**
- โมเดลขนาดเล็ก CoT ช่วยน้อย บางทีแย่ลง — เพราะ "การคิด" ที่ generate ออกมาก็ผิดอยู่ดี
- โมเดลที่มี built-in reasoning mode เช่น Claude Extended Thinking หรือ o-series ของ OpenAI — การพิมพ์ "think step by step" อาจซ้ำซ้อนกับ internal reasoning ที่ทำอยู่แล้ว
- งานที่ไม่ใช่ reasoning เช่น สรุปบทความ CoT เปลือง token โดยไม่ได้ผลดีขึ้น

## Pattern 4 — Self-Consistency

### มันคืออะไร

Run prompt เดียวกันหลายรอบ แล้วเลือกคำตอบที่ "ส่วนใหญ่ตอบเหมือนกัน" (majority vote). เหมือนถามคนหลายคนแล้วเอาเสียงข้างมาก.

ทำไมต้องหลายรอบ? เพราะ LLM ตอบแบบ probabilistic — ถ้า temperature > 0 แต่ละรอบจะได้ผลแตกต่างกันเล็กน้อย. รอบที่ reasoning พังก็มีบ้าง รอบที่ reasoning ดีก็มีบ้าง. ถ้า run 10 รอบและ 8 รอบตอบเหมือนกัน ความน่าจะเป็นที่คำตอบนั้นถูกสูงกว่ารอบเดียว.

**ใช้เมื่อ:**
- งาน reasoning ที่ correctness สำคัญมากและ budget ของเวลา/เงินไม่ใช่ปัญหา
- งาน high-stakes เช่น medical triage หรือ legal classification ที่อยากลด variance
- การทำ benchmark / evaluation ที่ต้องการ noise reduction

**ข้อควรรู้เรื่อง cost:** self-consistency เพิ่ม cost ตามจำนวนรอบตรง ๆ — run 20 รอบ คือจ่ายเงิน 20 เท่า. สำหรับงานที่ต้องการ scale ใหญ่ต้องคิดให้ดี.

**จุดที่หยุดช่วย:**
- งานที่คำตอบเป็น text เปิด เช่น creative writing หรือ summarization — โหวตไม่ได้
- ถ้าโมเดล bias ผิดทาง systematic majority vote ก็ผิดต่อ — self-consistency ลด variance แต่ไม่ลด bias

## Pattern 5 — Structured Output

### มันคืออะไร

บังคับให้โมเดลตอบใน format ที่ code อ่านได้ — JSON, XML, YAML, markdown table, หรือ schema เฉพาะ.

ทำได้สองระดับ: ผ่าน prompt (ขอใน text) กับผ่าน API feature (JSON mode / structured output). ระดับ API ปลอดภัยกว่ามาก เพราะ enforce ที่ระดับ tokenization ไม่ใช่แค่คำขอ.

**ใช้เมื่อ:**
- เอา output ไปใช้ใน code/system อื่น — ต้องการ field ชัดเจน
- ต้องการหลาย field จาก response เดียว
- free-form output ต้อง parse ลำบากและเกิด error บ่อย

**Template:**
```
[Task]

Respond in this exact JSON format:
{
  "field_1": "...",
  "field_2": "...",
  "field_3": "..."
}

Do not include any text outside the JSON.
```

**ตัวอย่าง:** Extract ข้อมูลจาก email ลูกค้า
```
Extract the following from the customer email below.

{
  "customer_name": "...",
  "issue_type": "billing" | "technical" | "feature_request" | "other",
  "urgency": "low" | "medium" | "high",
  "summary": "under 30 words"
}

Do not include any text outside the JSON.

Email: """[email content]"""
```

**จุดที่หยุดช่วย:**
- ผ่าน prompt อย่างเดียว โมเดลยัง hallucinate JSON ที่ parse ไม่ได้ เช่น ลืม comma ใช้ single quote ใส่ markdown fence รอบ JSON. งาน production ต้องใช้ JSON mode หรือ structured output API
- Schema ใหญ่เกิน 20-30 field โมเดลจะลืม field หรือ duplicate ต้องแตกเป็นหลาย call
- Format ที่บังคับเข้มเกินอาจกระทบ quality ของ content — โมเดลใช้ token ไปกับ format มากกว่าเนื้อหา

## Pattern 6 — Self-Critique / Reflection

### มันคืออะไร

ให้โมเดลทำงานสองรอบ — รอบแรกตอบ รอบสองวิจารณ์ตัวเอง แล้วรอบสามแก้ไขตามคำวิจารณ์.

กลไกที่ทำงานอยู่: เวลาโมเดลเขียน critique มันต้องอ่านคำตอบแรกกลับมาใหม่ในฐานะ reviewer — ซึ่งให้ตัวมันเองมี "perspective" อีกแบบหนึ่งที่อาจจับจุดที่ pass-through ในรอบแรก.

**ใช้เมื่อ:**
- งานเขียนที่ต้องการ quality สูง เช่น proposal, marketing copy, หรือ presentation outline
- งานวิเคราะห์ที่ first-pass มักพลาด angle บางอย่าง
- มี budget เวลา/cost สำหรับ 2-3 turn

**Template:**
```
Step 1: [original task]

Step 2: Review your answer above. 
List 3 specific weaknesses or errors.

Step 3: Rewrite fixing those weaknesses.
```

**ตัวอย่าง:**
```
Step 1: Draft a 150-word product launch email for a new Thai payment app.

Step 2: Critique your draft — list 3 weaknesses 
in tone, clarity, or call-to-action. Be specific.

Step 3: Rewrite the email fixing all 3 weaknesses.
```

**จุดที่หยุดช่วย:**
- โมเดลมักวิจารณ์แบบ generic เช่น "could be more engaging" ซึ่งนำไปปรับไม่ได้จริง — แก้โดยบอกให้ critique เฉพาะ dimension ที่สนใจ
- ถ้า first-pass ผิดด้านข้อเท็จจริง self-critique มักไม่จับ เพราะโมเดล believe คำตอบตัวเอง
- Run มากกว่า 3 รอบมักไม่ดีขึ้น — ผลมักจะ converge ไปที่ "safe, bland" version

## Pattern 7 — Decomposition

### มันคืออะไร

แตกงานซับซ้อนเป็น sub-task ย่อย ๆ แล้ว chain output ของ task หนึ่งเป็น input ของอีก task หนึ่ง.

ทำไมต้องแตก? เพราะงานซับซ้อนในคำขอเดียวทำให้โมเดลต้องทำหลายอย่างพร้อมกัน — extract ข้อมูล + analyze + format + summarize ทีเดียว. แต่ละ step มีโอกาส error และ error compound ไปเรื่อย ๆ โดยเราไม่รู้ว่าพังตรงไหน.

แตกเป็น step ทำให้ inspect ได้, debug ได้, และ error ไม่ลามเงียบ ๆ.

**ใช้เมื่อ:**
- งานที่มี multi-step pipeline ชัดเจน
- งานที่ context window จะล้นถ้าทำใน prompt เดียว
- ต้องการ inspect แต่ละ step — เช่น ตรวจสอบก่อนจะส่งต่อ

**ตัวอย่าง:** วิเคราะห์ feedback ลูกค้า 500 รายการ

แทนที่จะพิมพ์ "วิเคราะห์ feedback ทั้งหมดนี้และสรุปให้ฉัน" — แตกเป็น:
```
Call 1: จำแนกแต่ละรายการเป็น category 
        (billing / technical / ux / other)

Call 2 (input = output of Call 1):
        สรุป pain point หลักของแต่ละ category

Call 3 (input = output of Call 2):
        Prioritize ตาม frequency + severity

Call 4 (input = output of Call 3):
        เขียน executive summary 200 คำ
```

**จุดที่หยุดช่วย:**
- งานเล็ก ๆ — decomposition เพิ่ม latency + cost โดยไม่ได้ quality ดีขึ้น
- Error compound ถ้าไม่ใส่ check ระหว่าง step — step 1 ผิด step 2 ก็พลอยผิด และคุณอาจไม่รู้จนถึง step 4
- Agentic decomposition ที่ให้โมเดลแตก task เองยังไม่ stable ในปี 2026 — ควรแตก task เองถ้าเป็นไปได้

## Pattern 8 — Negative Prompting

### มันคืออะไร

ระบุชัดเจนว่า "อย่าทำ" อะไร นอกเหนือจากระบุว่า "ทำ" อะไร.

โมเดลมี default behaviors — ใส่ disclaimer, ขึ้นต้นด้วย "Of course!", ใส่ emoji, ใส่ heading เยอะ, ใส่ prose ก่อน JSON. สิ่งเหล่านี้เป็น pattern ที่โมเดลเรียนมาจาก training data ที่ reinforced พฤติกรรมแบบ "helpful assistant". ถ้าไม่อยากได้ ต้องบอกตรง ๆ.

**ใช้เมื่อ:**
- โมเดลมี default behavior ที่ไม่ต้องการ
- output จะถูก parse — ต้องไม่มี prose ก่อนหรือหลัง JSON
- ต้องบังคับ scope ของคำตอบให้แน่น

**Template:**
```
[Main instruction]

Constraints:
- Do NOT [unwanted behavior 1]
- Do NOT [unwanted behavior 2]
- Start your response with [how it should start]
```

**ตัวอย่าง:**
```
Summarize this article in exactly 3 bullet points.

Constraints:
- Do NOT add "Here is the summary:" or any introduction
- Do NOT use emoji
- Each bullet must be under 20 words
- Start your response with the first bullet
```

**จุดที่หยุดช่วย:**
- Constraint ที่ขัดกับ safety training โมเดลจะไม่ยอมตาม และไม่ควรพยายามให้ตาม
- Constraint ยาวเกิน 10 ข้อ โมเดลจะหลงและทำผิดบางข้อ — concise กว่า exhaustive
- บางครั้ง negation backfires — "don't mention pricing" ทำให้โมเดล obsess กับ pricing. ถ้าเป็นไปได้ให้ rephrase เป็น positive instruction แทน

## Pattern 9 — Delimiter Separation

### มันคืออะไร

ใช้ delimiter เช่น triple quotes, XML tags, หรือ markdown headers แยกส่วนต่าง ๆ ของ prompt อย่างชัดเจน — instruction, context, input, examples, output format.

โมเดลต้องการรู้ว่า "ส่วนไหนคือคำสั่ง" และ "ส่วนไหนคือข้อมูลที่ส่งมาให้ process". ถ้าปนกันในข้อความเดียว โมเดลอาจเดาผิด หรือในกรณีที่ user input อาจมีคำสั่งปลอมปนมา (prompt injection) delimiter ช่วยลดความเสี่ยง.

**ใช้เมื่อ:**
- prompt ยาวกว่า 5 บรรทัด
- มี user-supplied content ปนใน prompt
- มีหลาย section ที่โมเดลต้อง refer ต่างกัน

**Template:** สำหรับ Claude — XML tags ทำงานได้ดีเป็นพิเศษ
```xml
<task>Translate the email below to formal Thai.</task>

<rules>
Use ครับ/ค่ะ register.
Preserve proper nouns in English.
Do not translate code.
</rules>

<email>
{email_content}
</email>
```

**สำหรับ ChatGPT — triple quotes ใช้งานได้ดี:**
```
Summarize the article below in 3 bullet points.

Article:
"""
{article_content}
"""
```

**จุดที่หยุดช่วย:**
- Prompt สั้นมาก (&lt; 3 บรรทัด) — delimiter เปลือง token เปล่า
- Delimiter ที่ collide กับ user content — ถ้า user paste markdown ที่มี triple backtick แล้วคุณใช้ triple backtick เป็น delimiter ด้วย จะสับสน แก้โดยเลือก delimiter ที่ไม่น่าปรากฏใน input
- Delimiter อย่างเดียวไม่ใช่ defense ต่อ prompt injection ได้สมบูรณ์ — แต่ช่วย

## Pattern 10 — Persona / Style Anchoring

### มันคืออะไร

ให้ตัวอย่าง style จริง 1-3 ชิ้นที่ต้องการให้โมเดลเลียนแบบ แทนการอธิบาย style ด้วยคำคุณศัพท์. ต่างจาก few-shot ตรงที่ few-shot เน้น task pattern ส่วน style anchoring เน้น voice และ tone.

ทำไม? เพราะ "punchy, witty, conversational" ในใจคุณกับในใจโมเดล อาจหมายถึงคนละอย่าง. แต่ถ้าคุณ paste ตัวอย่าง 2 บทความที่คุณชอบมาก โมเดลสามารถ extract pattern ของ voice ได้ชัดกว่าคำบรรยาย.

**ใช้เมื่อ:**
- งานเขียนที่ต้องการ brand voice เฉพาะ
- เลียน writing style ของตัวเองเพื่อสร้าง content เพิ่ม
- อธิบาย style ด้วยคำพูดไม่ได้เพราะ style มันละเอียดเกิน

**Template:**
```
Write in the voice exemplified by these samples.

Sample 1:
"""[150-300 word sample]"""

Sample 2:
"""[another sample]"""

Now write [target task] in that same voice.
```

### ลองทำดู

ลองเอา 2 บทความหรือ email ที่คุณชอบ tone ของมัน paste ลงใน prompt แบบด้านบน แล้วบอกให้เขียนเรื่องใหม่ในสไตล์เดียวกัน. เปรียบเทียบกับการพิมพ์ว่า "เขียนแบบ professional แต่ friendly" โดยไม่มีตัวอย่าง — ผลจะต่างกันเห็นได้ชัด.

**จุดที่หยุดช่วย:**
- Sample สั้นเกิน (&lt; 50 คำ) — โมเดลจับ style ไม่ทัน
- Sample เยอะเกิน (> 5 ชิ้น) — โมเดลเริ่ม mix style ออกมาไม่ consistent
- โมเดล chat-tuned บางตัว revert ไป default voice หลังหลาย turn — ต้อง re-anchor ทุกครั้งที่ context ยาว

## Pattern 11 — Refusal Handling

### มันคืออะไร

เมื่อโมเดลปฏิเสธคำขอ สิ่งแรกที่ต้องทำคือแยกระหว่าง **legitimate refusal** กับ **over-refusal**. สองอย่างนี้ต้องการ response ที่ต่างกันสิ้นเชิง.

**Legitimate refusal** — คำขอ harmful จริง: ขอช่วยโกง, สร้างเนื้อหาผิดกฎหมาย, ทำอันตรายคน. กรณีนี้ไม่มีวิธี rephrase ที่ถูกต้อง ต้องออกแบบงานใหม่.

**Over-refusal** — request ที่ legitimate แต่ phrasing ทำให้โมเดลเข้าใจผิด. กรณีนี้ rephrase ได้ และ rephrase ที่ดีคือ "เพิ่ม context ที่ legitimate" ไม่ใช่ "หา loophole".

**วิธีแยก:**
- ถ้าเป็น task ที่มีคนทำงานจริงอยู่ในโลก (security researcher, doctor, teacher) และคุณอธิบาย context จริงได้ — likely over-refusal
- ถ้าต้องพยายาม bypass หลายรอบโดยไม่มี legitimate context เพิ่มขึ้น — likely legitimate refusal

**Template สำหรับ over-refusal:**
```
Context: I'm a [role] working on [legitimate use case].
The reason I need this is [specific reason].
Request: [rephrased, specific request]
```

**ตัวอย่าง:** งาน security awareness training

โมเดลปฏิเสธ: `"Write a phishing email for me"`

Rephrase ที่ถูกต้อง:
```
I'm running security awareness training for a Thai bank.
Show 3 phishing email patterns commonly used to target 
banking customers, with annotations on red flags 
employees should recognize. This is for defensive 
training, not for sending.
```

ความต่างไม่ใช่ "หาคำหลอกโมเดล" — แต่เป็น **เปลี่ยน frame งานให้ตรงกับ context จริง**. ถ้างานจริงคือโจมตี ไม่มี rephrase ใดที่ถูกต้อง.

> 💡 **กฎง่าย ๆ:** "ฉันไม่แน่ใจ" ของโมเดล ไม่ใช่ safety refusal — มันคือ uncertainty signal. เพิ่ม specificity หรือใช้ข้อมูลเพิ่มเติม แทนการ rephrase แบบ bypass.

## วิธีประกอบหลาย Pattern เข้าด้วยกัน (Stacking)

ใน production คุณแทบจะไม่ใช้ pattern เดียว. prompt ที่ใช้งานจริงมักเป็นการ stack 3-5 pattern เข้าด้วยกัน.

ลำดับที่ทำงานได้ดีในงานส่วนใหญ่:

```
1. Identity      — role + style anchoring
2. Structure     — delimiter แยกส่วน
3. Examples      — few-shot ถ้าต้องการ
4. Reasoning     — CoT ถ้าเป็นงาน reasoning
5. Output        — structured output + negative constraints
6. Verification  — self-critique ถ้า budget เอื้อ
```

### ตัวอย่าง Stack จริง: Intent Classifier สำหรับ Chatbot ธนาคาร

งาน: รับข้อความลูกค้าภาษาไทย แล้วตอบว่า intent คืออะไร

```
You are a Thai-language conversation analyst.

<examples>
User: "ขอเช็คยอดบิลเดือนนี้หน่อยค่ะ"  
→ {"intent": "check_balance", "confidence": 0.95}

User: "สวัสดีครับ"                  
→ {"intent": "greeting", "confidence": 0.99}

User: "อันนี้ใช้ยังไง"               
→ {"intent": "help", "confidence": 0.80}
</examples>

Choose intent from: greeting, check_balance, help, complaint, other.
Do NOT add any text outside the JSON.

User: "{user_message}"
```

prompt นี้ stack 4 pattern:
- **Role** (2.1): "Thai-language conversation analyst"
- **Few-shot** (2.2): 3 ตัวอย่างใน XML tag
- **Structured output** (2.5): JSON schema ชัดเจน
- **Negative constraint** (2.8): "Do NOT add any text outside the JSON"

ผล: output เป็น JSON สม่ำเสมอกว่า zero-shot อย่างมีนัยสำคัญในงาน domain-specific

### กฎสำหรับการ Stack

**เริ่มน้อย เพิ่มทีละ pattern** — ทุก pattern เพิ่ม token cost + complexity. ถ้า zero-shot ก็พอ ใช้ zero-shot. ทดสอบก่อนเพิ่ม.

**วัดผลทุกครั้งที่เพิ่ม pattern** — บางครั้ง pattern ทำให้ผลแย่ลง โดยเฉพาะกับโมเดลรุ่นใหม่ที่ trained มาดีแล้ว. อย่าเชื่อว่า "pattern มากกว่า = ดีกว่า" โดยไม่ทดสอบ.

**อย่ามี pattern ที่ขัดกัน** — CoT ("คิดออกมาเป็นขั้น") + strict JSON ("no text outside JSON") ขัดกันถ้าไม่ออกแบบดี ต้องเลือก หรือใส่ reasoning เป็น field ของ JSON เช่น `"reasoning": "..."`.

**System prompt vs user prompt** — pattern พวก role, rules, format ส่วนใหญ่ติดตั้งใน system prompt. task และ input อยู่ใน user prompt. ถ้าคุณใช้แค่หน้าเว็บไม่มี system prompt แยก ให้รวมทุกอย่างไว้ต้น user prompt.

## ความเข้าใจผิดที่พบบ่อย

**"มี magic prompt ที่ทำให้โมเดลตอบดีขึ้นเสมอ"** — ไม่มี. ทุก pattern มี trade-off. CoT ช่วย reasoning แต่เปลือง token สำหรับงานสรุป. Few-shot ช่วย format แต่ bias ผลลัพธ์. ไม่มีอะไรฟรี.

**"โมเดลรุ่นใหม่ทำให้ prompt engineering หมดความสำคัญ"** — ไม่ถูกทั้งหมด. โมเดลรุ่นใหม่ "อภัย" prompt ที่แย่ได้ดีขึ้น zero-shot performance สูงขึ้น แต่ในงาน production ที่ต้องการ consistency, structured output, หรือ domain-specific behavior pattern เหล่านี้ยังจำเป็น. พื้นฐานเพิ่มขึ้นไม่ได้แปลว่า ceiling เพิ่มขึ้นเอง.

**"ยิ่ง prompt ยาว ยิ่งดี"** — ผิด. Prompt ยาวเกินทำให้โมเดลสนใจ instruction หลักลดลง (lost-in-the-middle effect), เสีย token, เพิ่ม latency. กฎคือ "สั้นที่สุดที่ได้ผล" — แต่ละ token ต้อง earn its place.

**"Chain-of-Thought คือโมเดล 'คิดจริง'"** — ไม่ใช่. CoT คือ generated text ที่ดูเหมือนการคิด ไม่ใช่กระบวนการ internal จริงของโมเดล. ใช้ CoT เพื่อ accuracy ได้ แต่อย่าเชื่อ "เหตุผล" ว่าตรงกับสิ่งที่โมเดลทำจริง.

## สรุป — เครื่องมือ ไม่ใช่ recipe

11 pattern ที่เรียนในบทนี้ไม่ใช่รายการของ "เคล็ดลับ" ที่ท่องแล้วจะเก่ง. มันคือ vocabulary ของการสื่อสารกับโมเดล — เหมือนรู้ว่าจะใช้ประโยคแบบไหนในสถานการณ์แบบไหน.

ทุก pattern แก้ปัญหาเฉพาะ: role prompting เปลี่ยน tone, few-shot สอน format, CoT ช่วย reasoning, self-consistency ลด variance, structured output ทำให้ parse ได้, self-critique ปรับ quality, decomposition จัดการความซับซ้อน, negative prompting ตัด default ที่ไม่ต้องการ, delimiter แยกส่วน, style anchoring ถ่ายโอน voice, refusal handling แยก legitimate จาก over-refusal.

ในงานจริง คุณจะ stack หลายตัวเข้าด้วยกัน เริ่มน้อย เพิ่มทีละตัว วัดผลทุกครั้ง. โมเดลที่ดีขึ้นทุกปีทำให้ "พื้น" สูงขึ้น แต่คนที่รู้ pattern ยังได้ผลดีกว่าคนที่ไม่รู้ใน edge case ทุกประเภท.

แต่ยิ่งคุณใช้ pattern เหล่านี้มากขึ้น คุณจะเจอข้อจำกัดสำคัญข้อหนึ่งของ AI ทุกตัว — context window. ทุก pattern ที่เพิ่มเข้าไปใช้พื้นที่ใน context มากขึ้น และ context window นั้นมีขีดจำกัดที่ส่งผลต่อทุกอย่างที่คุณทำ. ในบทต่อไป เราจะมาดูว่า context window ทำงานยังไง ทำไมมันจำกัด และจะบริหารมันยังไงให้ได้ผลดีที่สุด.
