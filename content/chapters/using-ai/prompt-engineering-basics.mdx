# Prompt Engineering Basics — เขียน prompt ให้ AI ทำงานตามที่ต้องการ

คุณพิมพ์ "ช่วยเขียนอีเมลหน่อย" แล้วได้อีเมลทั่วไปที่ฟังดูเหมือนร่างแรกของใครก็ไม่รู้. พิมพ์อีกครั้งในสัปดาห์ถัดมาด้วยข้อความเดิม ได้ผลต่างออกไป. ไม่รู้ว่าเกิดอะไรขึ้น และไม่รู้ว่าต้องพิมพ์อะไรให้ได้ผลที่ต้องการ.

นี่ไม่ใช่ปัญหาของ AI — เป็นปัญหาของวิธีที่เราสื่อสารกับมัน.

บทนี้จะอธิบายกลไกว่า **ทำไม prompt บางอันถึงได้ผลดีกว่า** และให้โครงสร้างที่ใช้ได้จริง แทนการเดาสุ่ม ถ้ายังไม่ได้อ่านเรื่องว่า LLM ทำงานอย่างไร ลองอ่าน [LLM คืออะไร](/foundations/what-is-an-llm) ก่อน เพราะ prompt engineering ทั้งหมดต่อยอดจากกลไกนั้นโดยตรง.

## Prompt คืออะไร — ก่อนพูดถึงวิธีเขียน

เวลาคุณพิมพ์ข้อความใน ChatGPT หรือ Claude แล้วกด enter — ข้อความนั้นทั้งหมดคือ **prompt**.

แค่นั้นเอง. ไม่มีคำจำกัดความที่ซับซ้อนกว่านี้.

แต่มีศัพท์ที่คนมักสับสนสามคำที่ควรแยกออกจากกัน:

**User prompt** — สิ่งที่คุณพิมพ์ใน turn นั้น เช่น "สรุปบทความนี้ให้หน่อย"

**System prompt** — คำสั่งที่ตั้งไว้ก่อนที่คุณจะเริ่มคุย เพื่อกำหนดบุคลิกและพฤติกรรมของโมเดล คนที่ใช้ ChatGPT หรือ Claude บนเว็บไซต์ปกติจะไม่เห็นส่วนนี้ แต่บริษัทที่สร้างแอปโดยใช้ API ตั้งเองได้ เช่น แอปธนาคารที่มี AI chatbot — ข้างหลังมี system prompt อยู่ว่า "คุณคือผู้ช่วยทางการเงิน ตอบเฉพาะเรื่องการเงินเท่านั้น"

**Context** — ทุกอย่างที่อยู่ใน context window ณ ขณะนั้น รวม system prompt, ประวัติการสนทนา, ไฟล์ที่แนบ และ user prompt ล่าสุด โมเดลมองเห็น context ทั้งหมดนี้เป็นก้อนเดียวเวลาตอบ

ดังนั้นเวลาพูดว่า "prompt engineering" ในชีวิตประจำวัน หมายถึงการเขียน user prompt ให้ดีขึ้น แต่ในระดับที่ลึกขึ้น ครอบคลุมทั้งสามส่วน

> 💡 **Key insight:** โมเดลไม่มีความสามารถในการ "ถามกลับ" เมื่อ prompt กำกวม มันจะเดาว่าคุณหมายถึงอะไรแล้วตอบไปเลย — เดาตาม pattern ที่พบบ่อยที่สุดใน training data ซึ่งอาจไม่ใช่สิ่งที่คุณต้องการ

## กลไกที่อยู่เบื้องหลัง — ทำไม specificity ถึงสำคัญ

ก่อนดูว่า prompt ที่ดีมีองค์ประกอบอะไร ต้องเข้าใจก่อนว่าเกิดอะไรขึ้นข้างในเวลา LLM รับ prompt กำกวม

LLM ทำงานด้วยกลไก next-token prediction — มันทำนายคำต่อไปจากคำที่มีอยู่ทั้งหมด. เวลา prompt กำกวม โมเดลมีทางเลือก "คำถัดไปที่สมเหตุสมผล" อยู่มาก — distribution ของความน่าจะเป็นกว้าง. ผลคือ output มีความสุ่มสูง คุณไม่รู้ว่าจะได้อะไรกลับมา.

เวลา prompt จำเพาะ — บอกว่าต้องการอะไร ในรูปแบบไหน โทนแบบไหน สำหรับใคร — แต่ละคำใน prompt ตัด "ทางเลือกที่ไม่ต้องการ" ออกไป distribution แคบลง โมเดล converge เข้าหา output ที่คาดเดาได้มากขึ้น.

ลองนึกถึงตัวอย่างง่าย ๆ: ถ้าพิมพ์ว่า "เขียนเกี่ยวกับ AI" — โมเดลต้องเดาตัวเลือกมหาศาล: บทความ? อีเมล? บทกวี? ภาษาอังกฤษหรือไทย? ยาวแค่ไหน? สำหรับใคร? เทคนิคหรือทั่วไป? คุณได้อะไรมาขึ้นอยู่กับว่า pattern ไหนบ่อยที่สุดใน training data

แต่ถ้าพิมพ์ว่า "เขียนบทความ 300 คำ อธิบายว่า LLM ทำงานอย่างไร สำหรับผู้บริหารที่ไม่มีพื้นฐาน technical ใช้โทนเป็นทางการ ห้ามใช้ศัพท์เทคนิคโดยไม่อธิบาย จบด้วยตัวอย่างใช้งานในธุรกิจ 1 ตัวอย่าง" — โมเดลมีที่ต้อง converge ที่ชัดมาก

> 💡 **Key insight:** prompt ที่ดีไม่ใช่ prompt ที่ยาว — เป็น prompt ที่ตัด degree of freedom ที่โมเดลต้องเดาเอง. ทุกประโยคใน prompt ควรตัดทางเลือก output ที่ไม่ต้องการออกได้อย่างน้อยหนึ่งทาง. ถ้าประโยคไหนทำสิ่งนั้นไม่ได้ ตัดออกได้เลย.

## Mental model ที่ช่วยได้จริง

นึกว่าคุณกำลัง brief **นักเขียน contractor ที่เก่งมาก แต่ไม่รู้จักคุณเลย และถามกลับไม่ได้**. contractor คนนี้:

- มีความรู้กว้างมาก เคยอ่านหนังสือและบทความมหาศาล
- เก่งงานเขียนทุกรูปแบบ แต่ถ้าไม่บอกรูปแบบ จะใช้รูปแบบ default ที่ "ปกติ" ที่สุด
- ถ้าไม่บอกผู้อ่าน จะเขียนให้คนทั่วไป
- ถ้าไม่บอกโทน จะใช้โทนกลาง ๆ
- ถ้าไม่บอกความยาว จะเขียนยาวพอที่เขาคิดว่า "เหมาะสม"
- และถ้าไม่บอกอะไรเลย จะตีความตามสัญชาตญาณที่อ้างอิงจาก "ลูกค้าทั่วไป" ที่เคยเจอ

prompt engineering คือทักษะการเขียน brief ให้ contractor คนนี้ครั้งเดียวจบ — ไม่ต้องรอรอบที่สองเพราะไม่สามารถถามกลับได้.

**จุดที่ analogy นี้ใช้ไม่ได้:** contractor จริงมีประสบการณ์ของตัวเองและจะ push back ถ้า brief แย่. LLM ไม่ push back — มันทำตาม brief เสมอ แม้ brief นั้นจะขัดแย้งกันเองหรือไม่สมเหตุสมผล. ดังนั้นภาระทั้งหมดอยู่ที่คุณในการเขียน brief ให้ดี.

## 5 องค์ประกอบของ prompt ที่ดี

prompt ที่ได้ผลมักประกอบด้วย 5 ส่วน — ไม่ใช่ทุก prompt ต้องมีครบ แต่ยิ่งงานซับซ้อน ยิ่งควรครบ.

### (1) Context — เบื้องหลังว่าทำไมขอสิ่งนี้

บอกให้ชัดว่า:
- คุณเป็นใคร หรืองานนี้คืออะไร
- output จะถูกนำไปใช้ที่ไหน อ่านโดยใคร
- ข้อมูลภายในที่โมเดลไม่รู้ (เช่น เอกสาร ตัวเลข ชื่อลูกค้า)

ตัวอย่างที่ขาด context: "ช่วยเขียน pitch ให้หน่อย"

ตัวอย่างที่มี context: "ฉันเป็น product manager กำลังเตรียม pitch deck ให้นักลงทุน VC ระดับ seed ที่ไม่รู้จักวงการประกันภัยมาก่อน"

Context บอกโมเดลว่าผู้อ่านคือใคร ระดับความรู้ไหน และ output ควรทำงานในบริบทอะไร

### (2) Task instruction — บอกว่าให้ทำอะไรด้วยกริยาที่ชัด

ใช้กริยาตรง ๆ: เขียน, สรุป, แปล, วิเคราะห์, เปรียบเทียบ, จัดเรียง, แก้ไข

หลีกเลี่ยงกริยากำกวม: "ช่วยดู", "ปรับให้ดีขึ้น", "ทำให้ดีกว่านี้"

ตัวอย่าง: "สรุป เอกสารนี้เป็น 5 bullet ความยาวสูงสุด 15 คำต่อ bullet"

สังเกตว่า task instruction ที่ดีบอกทั้ง verb (สรุป), object (เอกสารนี้), format (5 bullet) และ constraint (15 คำต่อ bullet) — ในประโยคเดียว

### (3) Examples — แสดงรูปแบบที่ต้องการด้วยตัวอย่าง

บางงานอธิบายยากเป็นคำพูด แต่แสดงด้วยตัวอย่างได้ง่าย.

เช่น ถ้าต้องการให้เปลี่ยน feedback ที่ตรงไปตรงมาให้สุภาพขึ้น "โดยคงเนื้อหาครบ" — การ explain ว่า "สุภาพ" หมายถึงอะไรอาจใช้เวลานาน แต่ถ้าให้ตัวอย่าง 2-3 คู่ ก่อน/หลัง โมเดลจะจับ pattern ได้ทันที

เทคนิคนี้เรียกว่า **few-shot prompting** — ลงรายละเอียดอยู่ใน [Prompt Engineering Patterns](/using-ai/prompt-engineering-patterns) บทต่อไป

### (4) Output format — บอกรูปแบบที่ต้องการแน่นอน

สิ่งที่ต้องระบุ:
- รูปแบบ: markdown / JSON / bullet / paragraph / table / numbered list
- ความยาว: จำนวนคำ / bullet / ย่อหน้า
- โทน: เป็นทางการ / สบาย ๆ / technical / เรียบง่าย
- ภาษา: ไทย / อังกฤษ / ผสม

ตัวอย่าง: "Output เป็น JSON ที่มี key `summary` (string) และ `key_points` (array of strings) — ไม่ต้องมีคำอธิบายเพิ่ม"

ถ้าไม่ระบุ output format — โมเดลจะใช้รูปแบบ default ที่ "ปกติ" ที่สุดสำหรับ task นั้น ซึ่งส่วนใหญ่คือ paragraph ยาว

### (5) Constraints — บอกสิ่งที่ห้าม

"ห้ามใช้ศัพท์เทคนิค", "ห้ามอ้างอิงตัวเลขที่ไม่มีในเอกสาร", "ห้ามใช้ emoji", "ห้ามยาวเกิน 200 คำ"

Constraints มีประโยชน์มากเมื่อ default behavior ของโมเดลเอนเอียงไปทางที่ไม่ต้องการ เช่น ถ้าต้องการ output ภาษาธรรมชาติแต่โมเดลมักใส่ emoji เอง — การระบุ "ห้ามใช้ emoji" แก้ได้ทันที

สังเกตว่า constraint ที่ดีต้องจับต้องได้และตรวจสอบได้ — "ห้ามยาวเกิน 200 คำ" ตรวจนับได้. "ห้ามใช้ภาษาที่ซับซ้อน" ตรวจยากกว่ามาก เพราะ "ซับซ้อน" ไม่มีนิยามที่ชัดสำหรับโมเดล. ถ้าอยากให้ภาษาง่ายขึ้น ระบุให้ชัดกว่านั้น: "ใช้ประโยคไม่เกิน 20 คำ" หรือ "อธิบายให้คนที่ไม่มีพื้นฐาน technical อ่านเข้าใจได้".

**เคล็ดเล็ก ๆ ที่ใช้ได้จริง:** ถ้าโมเดลมักทำ X ที่คุณไม่ต้องการ แต่บอกยากว่า X คืออะไร — ให้ตัวอย่าง output ที่ดีและตัวอย่าง output ที่ไม่ดีแนบไปใน prompt. โมเดลจะเข้าใจ X ผ่านตัวอย่างได้ดีกว่าคำอธิบาย.

## ดูตัวอย่างจริง — prompt กำกวม vs จำเพาะ

**กรณีที่ 1: อีเมลขอเลื่อนงาน**

**Prompt กำกวม:**
```
เขียน email บอกลูกค้าว่าโปรเจ็คเลื่อน
```

Output ที่ได้: อีเมล generic ที่ใช้กับใครก็ได้ ไม่รู้ว่าเลื่อนนานแค่ไหน ไม่มีเหตุผล โทนกลาง ๆ — ต้องนั่งแก้เองอีกนาน

**Prompt จำเพาะ:**
```
เขียน email ขอเลื่อนวันส่ง deliverable ให้ลูกค้า โดย:
- Context: เป็น software project, deliverable เดิม 15 พ.ค., ต้องเลื่อนเป็น 29 พ.ค.
- เหตุผล: เจอ bug ใน integration กับระบบของลูกค้าที่ต้องใช้เวลาทดสอบเพิ่ม
- โทน: เป็นทางการแต่อบอุ่น แสดงความรับผิดชอบ ไม่โทษลูกค้า
- ความยาว: 150-200 คำ
- ต้องมี: วันที่ใหม่ที่ commit ได้, ข้อเสนอ mitigation (เช่น preview deploy ก่อน)
- ห้าม: ใช้คำว่า "ขออภัย" เกิน 1 ครั้ง, ใส่ emoji
```

Output: อีเมลที่พร้อมใช้ทันที — ไม่ต้อง iterate

ความต่างไม่ใช่ว่า prompt ที่สองยาวกว่า แต่ทุกบรรทัดตัดทางเลือกที่ไม่ต้องการออกได้.

```
Output:
```

โมเดลจับ pattern จาก 2 ตัวอย่างได้ทันที — output คาดเดาได้และมี style ที่สม่ำเสมอ. เทคนิคนี้เรียกว่า few-shot prompting.

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วลองเปรียบเทียบสองแบบนี้:

**รอบ 1 — prompt กำกวม:**
```
ช่วยเขียนสรุปรายงานประจำเดือนให้หน่อย
```

**รอบ 2 — prompt จำเพาะ:**
```
สรุปรายงานประจำเดือนนี้เป็น 4 bullet point โดย:
- เน้นผลลัพธ์ที่วัดได้ (ตัวเลข, %) ไม่ใช่แค่กิจกรรม
- แต่ละ bullet ยาวไม่เกิน 20 คำ
- โทนเป็นทางการ เหมาะอ่านในประชุมผู้บริหาร
- เนื้อหา: [วางข้อมูลของคุณตรงนี้]
```

สังเกต output ทั้งสองรอบ — ต่างกันแค่ไหน และต่างกันตรงไหนบ้าง.

## เทคนิค "Act as a [role]" — ทำงานอย่างไรในทางกลไก

คุณน่าจะเคยเห็นแนะนำให้เริ่ม prompt ว่า "Act as a senior lawyer" หรือ "You are an expert in marketing". เทคนิคนี้ได้ผลจริง — แต่ด้วยเหตุผลที่ต่างจากที่หลายคนคิด.

LLM ถูก train บนข้อความหลากหลายรูปแบบ — บทความวิชาการ, code review, บทสนทนาระหว่างแพทย์กับคนไข้, บล็อก, forum, ฯลฯ. ข้อความแต่ละประเภทมี vocabulary, structure และ conventions ของตัวเอง.

เวลา prompt บอก role — โมเดลจะ **focus subset ของ training distribution ที่ตรงกับ role นั้น**. บอกว่า "Act as a senior data engineer reviewing code" โมเดลจะดึง pattern จาก code review ที่เคยเห็น: วิจารณ์ specific, อ้าง best practice, สนใจ edge case — ไม่ใช่จาก casual chat ทั่วไป.

**แต่มีส่วนที่ใช้ไม่ได้:** role prompt ไม่ทำให้โมเดล "รู้มากขึ้น". ถ้าโมเดลไม่มีความรู้เรื่อง X — บอกให้ "Act as expert in X" แค่ทำให้มัน hallucinate ด้วยน้ำเสียงที่มั่นใจมากขึ้น. ใช้ role prompt เพื่อปรับ style และ structure ของ output ได้ดี — แต่อย่าใช้เพื่อคาดหวังความรู้ที่โมเดลไม่มี.

> 💡 **Key insight:** role prompt เป็นเครื่องมือปรับ "น้ำเสียงและโครงสร้าง" ไม่ใช่เครื่องมือเพิ่ม "ความรู้". บอกว่า "You are an expert financial analyst" จะทำให้ output มีโครงสร้างเหมือน financial report ที่อ่านเจอบ่อย ๆ — แต่ข้อมูลที่ใช้ยังมาจากสิ่งที่โมเดลเคยเห็นเท่านั้น

## วิธี iterate อย่างเป็นระบบ — ไม่ใช่เดาสุ่ม

สิ่งที่แยกคนที่เก่ง prompt engineering กับคนที่ไม่เก่งคือวิธี respond ต่อ output ที่ไม่ตรง. คนไม่เก่งเขียน prompt ใหม่ทั้งอัน — มั่ว ๆ — แล้วหวังว่าครั้งนี้จะดีกว่า. คนที่เก่ง **อ่าน output ก่อน ระบุให้ได้ว่าผิดที่ไหน แล้วแก้แค่ส่วนนั้น**.

Process มาตรฐาน:

1. เขียน prompt ฉบับแรก — ครอบคลุมองค์ประกอบที่จำเป็น แต่ยังไม่ต้องเป๊ะ
2. รัน — ดู output จริง
3. อ่าน output แล้ว **ระบุ "จุดที่ผิด"** ให้ชัด: ผิดรูปแบบ? ผิดเนื้อหา? ขาดข้อมูล? โทนผิด? ยาวเกิน? สั้นเกิน?
4. Map จุดผิดกลับไปที่ส่วนของ prompt: ผิดรูปแบบ → Output format ไม่ชัด / ผิดเนื้อหา → Context ขาด / โทนผิด → Role หรือ Constraints ไม่ชัด
5. แก้เฉพาะส่วนนั้น — **ไม่เขียนใหม่ทั้ง prompt**
6. รันใหม่ ทำซ้ำจนพอใจ

**ตัวอย่าง:**

Prompt v1: "สรุปบทความนี้ให้หน่อย"

Output v1: สรุปยาว 3 ย่อหน้า

จุดที่ผิดที่ระบุได้: ยาวเกินไป, ไม่เป็น bullet, ไม่รู้ว่าเน้นมุมไหน — map กลับไปที่ Output format ไม่ชัด และขาด task constraint

Prompt v2: "สรุปบทความนี้เป็น 5 bullet สั้น ๆ เน้นสิ่งที่เกี่ยวกับ business impact"

Output v2: 5 bullet ตามที่ขอ — แก้ได้ในรอบเดียวเพราะระบุได้ชัดว่า v1 ผิดที่ไหน

ถ้าเดา prompt ใหม่มั่ว ๆ โดยไม่ระบุจุดผิด อาจเขียน prompt v2 ที่ "ยาวขึ้น" แต่ไม่ได้แก้ที่ root cause และได้ output ที่ไม่ต่างกัน

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วลอง iterate 2 รอบนี้:

**รอบ 1 — prompt ตั้งต้น:**
```
อธิบาย machine learning ให้หน่อย
```

อ่าน output ที่ได้ แล้วระบุให้ได้ว่าผิดหรือไม่ตรงตามความต้องการคุณยังไง:
- ยาวหรือสั้นเกินไป?
- technical เกิน หรือ general เกินไป?
- ไม่มีตัวอย่างที่เป็นรูปธรรม?
- รูปแบบไม่ตรง?

**รอบ 2 — แก้เฉพาะส่วนที่ระบุได้:**

เขียน prompt v2 โดยเพิ่มเฉพาะสิ่งที่แก้จุดผิดนั้น อย่าเขียนใหม่ทั้งหมด

เปรียบเทียบ output ทั้งสอง — ความต่างมาจากส่วนไหนของ prompt ที่เปลี่ยนไป.

## ความผิดพลาดที่พบบ่อย — และวิธีแก้

**"ช่วยปรับให้ดีขึ้น"**

คุณรู้ว่า "ดีขึ้น" หมายถึงอะไร แต่โมเดลไม่รู้. ดีขึ้นในมิติไหน — กระชับขึ้น? เป็นทางการขึ้น? มี data สนับสนุนมากขึ้น? ใช้ภาษาง่ายขึ้น? ต้องบอกให้ชัด.

**ขอหลายอย่างใน prompt เดียว**

"สรุป แล้วแปล แล้ววิเคราะห์ความเสี่ยง แล้วเขียน executive summary" — โมเดลทำได้ แต่คุณภาพแต่ละส่วนตก. แบ่งเป็นหลาย turn จะได้ผลดีกว่า ทำทีละขั้นและตรวจสอบก่อนไปต่อ.

**เขียน prompt ยาวเพื่อปลอบใจตัวเอง**

เพิ่ม "please be very careful and accurate" หลายรอบไม่ช่วย — ใช้ constraint ที่จับต้องได้แทน เช่น "ห้ามอ้างอิงตัวเลขที่ไม่ปรากฏใน input" ชัดกว่ามาก

**ไม่ให้ตัวอย่างเมื่อ format แปลก**

บางครั้ง explain ด้วยคำพูดอย่างเดียวไม่เพียงพอ โดยเฉพาะถ้า style ที่ต้องการมีความเฉพาะตัว ตัวอย่าง 2-3 คู่จะชัดกว่าคำอธิบาย 2-3 ย่อหน้า.

**ไม่อ่าน output แต่ลองใหม่ทันที**

นี่คือข้อผิดพลาดที่แพงที่สุด — เพราะเท่ากับ iterate แบบ random search ไม่ใช่ informed search. หยุดอ่านก่อนทุกครั้ง.

> 💡 **Key insight:** คนที่เก่ง prompt engineering ไม่ใช่คนที่ "เขียน prompt สวย" — เป็นคนที่ debug output ได้เร็ว. ทักษะที่สำคัญกว่าการเขียน prompt คือการอ่าน output แล้วระบุ root cause ว่าผิดที่ไหน.

## เมื่อ prompt engineering ตัน — สัญญาณที่ต้องเปลี่ยนวิธี

prompt engineering แก้ปัญหาได้เยอะมาก — แต่ไม่ใช่ทุกปัญหา. มีสัญญาณที่ควรหยุดและถามว่า "prompt engineering ใช่เครื่องมือที่ถูกต้องสำหรับปัญหานี้หรือเปล่า" ก่อน iterate รอบต่อไป:

**ข้อมูลไม่อยู่ในโมเดล**

เช่น เอกสารภายในบริษัท, ข้อมูลหลัง training cutoff ของโมเดล, ข้อมูลเฉพาะองค์กร. ต่อให้ prompt ดีแค่ไหน โมเดลไม่รู้สิ่งที่ไม่เคยเห็น — ต้องใช้ **RAG** (retrieval-augmented generation) หรือ tool ที่ดึงข้อมูลสด.

**ต้องการ output ที่สม่ำเสมอมาก ๆ ในหลายร้อย input**

prompt อย่างเดียวควบคุมไม่อยู่ในระดับนี้ — พิจารณา fine-tuning หรือ structured output schema.

**ต้องการ "guarantee" ที่ prompt ให้ไม่ได้**

เช่น "ห้าม hallucinate" หรือ "ห้ามตอบเรื่อง X เด็ดขาด" — ต้องการ validation layer นอก prompt ไม่ใช่แค่ instruction ใน prompt.

**Task ต้องการเหตุผลที่ซ้อนกันลึกหลายขั้น**

chain-of-thought ช่วยได้ระดับหนึ่ง แต่บางปัญหาต้องการ multi-step agent ที่เรียก tool

หลักทั่วไป: ถ้า iterate 5-10 รอบแล้วยังไม่ได้ผล — หยุดถามก่อนว่า prompt engineering ใช่ปัญหาหรือเปล่า แล้วค่อย iterate รอบที่ 11 ถ้าใช่.

## สิ่งที่คนมักเข้าใจผิด — เจอบ่อยมาก

**"prompt ยิ่งยาว ยิ่งดี"**

ยาวช่วยเมื่อเพิ่ม specificity จริง ๆ แต่ "please please be very careful" หลายรอบไม่ช่วยและบางครั้ง dilute instruction ที่สำคัญ. หลักคือทุกประโยคต้องตัดทางเลือก output ที่ไม่ต้องการออกได้จริง.

**"มี prompt วิเศษที่ใช้ได้กับทุก task"**

prompt template ที่ดีกับ task หนึ่งอาจไม่ดีกับอีก task. "Let's think step by step" ช่วย reasoning บางอย่าง แต่ไม่ช่วย summarization. ไม่มี one-size-fits-all — ต้อง iterate กับ task ตัวเอง.

**"prompt เดียวกันใช้ได้กับทุกโมเดล"**

ChatGPT, Claude, Gemini, Llama ถูก train ด้วยข้อมูลและวิธีต่างกัน — มี default behavior ต่างกัน. prompt ที่ดีกับ Claude อาจต้องปรับเล็กน้อยกับ GPT. เวลาเปลี่ยนโมเดลควรทดสอบ prompt ใหม่เสมอ.

**"ถ้า output ผิด แปลว่าโมเดลไม่เก่งพอ"**

บ่อยครั้งกว่าที่คิด สาเหตุคือ prompt ไม่จำเพาะพอ. ก่อนสรุปว่า "โมเดลไม่ไหว" ลอง iterate prompt 3-5 รอบโดยระบุจุดผิดทุกรอบ — ถ้ายังไม่ได้ค่อยพิจารณาเปลี่ยนโมเดลหรือเปลี่ยนวิธี.

## Prompt engineering ในภาษาไทย — มีอะไรที่ต้องระวังเพิ่ม

ถ้าใช้ AI เป็นภาษาไทยเป็นหลัก มีเรื่องที่ควรรู้เพิ่มจากหลักการทั่วไป

**เรื่อง token และความยาว**

ภาษาไทยใช้ token มากกว่าภาษาอังกฤษประมาณ 2-3 เท่า ซึ่งส่งผลต่อทั้งค่าใช้จ่ายและพื้นที่ context window (ดูรายละเอียดและตารางเปรียบเทียบในบท [LLM ตอบคำถามยังไง](/using-ai/how-llms-respond))

**เรื่องระดับความสุภาพ**

ภาษาไทยมีระดับความสุภาพหลายชั้น — ทางการ, กลาง, สบาย ๆ, formal writing. ถ้าไม่บอก โมเดลมักจะเลือกระดับกลางที่ "ปลอดภัย" — ซึ่งบางครั้งฟังดูแข็งเกินไปสำหรับบางบริบท หรือสบายเกินไปสำหรับเอกสารทางการ. การระบุให้ชัดว่าต้องการโทนแบบไหน (เช่น "ระดับเดียวกับอีเมลถึงผู้บริหารระดับสูง" หรือ "สบาย ๆ เหมือนคุยกับเพื่อนร่วมงาน") ช่วยได้มาก.

**เรื่อง code-switching**

คนไทยที่ใช้ภาษาอังกฤษทางอาชีพมักผสมศัพท์ tech อังกฤษเข้าไปในประโยคไทย — "ช่วย review PR นี้หน่อย" หรือ "สรุป meeting notes ให้เป็น bullet". โมเดลรับ pattern นี้ได้ดีพอสมควร แต่ถ้าต้องการ output ที่ consistent ควรระบุให้ชัดว่าศัพท์ไหนให้คงเป็นภาษาอังกฤษ และศัพท์ไหนให้แปลเป็นไทย.

> 💡 **Key insight:** prompt ภาษาไทยที่ดีต้องระบุ 2 เรื่องที่ prompt ภาษาอังกฤษไม่ต้องระบุ: ระดับความสุภาพที่ต้องการ และนโยบาย code-switching ว่าศัพท์ไหนให้ทับศัพท์ไว้ ศัพท์ไหนให้แปล.

## ตัวอย่างเต็ม — จาก prompt แรกจนถึง output ที่ใช้ได้จริง

ลองดู workflow ตั้งแต่ต้นจนจบในกรณีจริง: ต้องการเขียนอีเมลถึงทีมแจ้งการเปลี่ยนแปลง process ใหม่

**รอบที่ 1 — prompt ตั้งต้น:**
```
เขียนอีเมลแจ้งทีมเรื่อง process ใหม่
```

Output ที่ได้: อีเมลทั่วไป ไม่รู้ว่า process อะไร ใครรับผิดชอบ เริ่มเมื่อไหร่ โทนบอกข่าวล้วน ๆ ไม่มี call to action

**การอ่าน output และระบุจุดผิด:**
- ขาด Context: ไม่รู้ว่า process อะไร
- ขาด Constraints: ไม่รู้โทนที่ต้องการ
- ขาด Output format: ไม่มีโครงสร้างที่ชัด
- ขาด Task: ไม่ชัดว่าต้องการให้ทีม "ทำอะไร" หลังอ่านอีเมล

**รอบที่ 2 — prompt ที่เพิ่ม elements ที่ขาด:**
```
เขียนอีเมลภาษาไทยแจ้งทีม sales 8 คน เรื่อง process รายงานยอดขายรายวันใหม่

Context:
- เดิม: ส่ง Excel ทาง Line กลุ่ม ทุกเย็น 5 โมง
- ใหม่: กรอกใน Google Form ที่ลิงก์นี้ [link] ให้เสร็จก่อน 4 โมงครึ่ง
- เหตุผล: ทีม management ต้องการ real-time dashboard ไม่ใช่ไฟล์แยก
- เริ่ม: จันทร์หน้า (26 พ.ค.)

โทน: เป็นกันเอง อบอุ่น แต่ชัดเจนเรื่อง timeline
ความยาว: ไม่เกิน 150 คำ
ต้องมี: วันเริ่ม, link form, ขอให้ confirm ว่าเข้าใจ
ห้าม: ใช้คำว่า "กรุณา" เกิน 2 ครั้ง, โทนแข็ง, อธิบายยาวเกินไป
```

Output รอบที่ 2: อีเมลที่ใช้ส่งได้ทันที ครบทุกข้อมูลที่จำเป็น โทนตรงตามที่ต้องการ

ความต่างระหว่างรอบ 1 กับรอบ 2 ไม่ใช่ว่า "รู้เคล็ดลับพิเศษ" — แต่เพราะ **อ่าน output ก่อน ระบุจุดผิดได้ชัด แล้วเพิ่มเฉพาะส่วนที่ขาด**.

## prompt engineering เป็น "ทักษะ" — ไม่ใช่ "เคล็ดลับ"

มีคนจำนวนมากมองหา "prompt วิเศษ" ที่ copy ไปใช้แล้วได้ผลดีทันที. แนวคิดนี้ผิดในระดับพื้นฐาน — เพราะ prompt engineering เป็นทักษะ ไม่ใช่ recipe.

ความต่างสำคัญ:

**Recipe** — ทำตามขั้นตอน ได้ผลเดิมทุกครั้ง. ใช้ได้เมื่อ input เหมือนกันทุกครั้ง.

**ทักษะ** — เข้าใจหลักการ adapt ได้กับสถานการณ์ใหม่. ใช้ได้เมื่อ input เปลี่ยนไปทุกครั้ง.

งานจริงที่ต้องใช้ AI มี input ใหม่ทุกวัน — เอกสารใหม่, context ใหม่, ผู้อ่านใหม่, เป้าหมายใหม่. คนที่ copy prompt template แล้วแปลกใจว่าทำไมไม่ได้ผลทุกครั้ง เป็นเพราะใช้ recipe กับงานที่ต้องการทักษะ.

ทักษะที่ต้องฝึก 3 อย่างหลัก:
1. **อ่าน output ออก** — รู้ว่าผิดหรือไม่ตรง และผิดในมิติไหน
2. **Map จุดผิดกลับไปที่ prompt component** — รู้ว่าต้องแก้ที่ไหน
3. **เขียน constraint ที่จับต้องได้** — แปลง "ดีขึ้น" ให้เป็นคำสั่งที่ตรวจสอบได้

สามอย่างนี้เรียนรู้ได้จากการลงมือทำ — ไม่ใช่จากการท่องจำ framework.

### ลองทำดู

เปิด ChatGPT หรือ Claude แล้วทดสอบทักษะ "อ่าน output" นี้:

ส่ง prompt นี้ไป:
```
เขียน job description สำหรับตำแหน่ง marketing manager
```

ได้ output แล้ว — อ่านทั้งหมด แล้วตอบคำถาม 3 ข้อนี้ก่อนส่ง prompt ต่อ:
1. output นี้ตรงกับสิ่งที่คุณต้องการไหม — ถ้าไม่ตรง ผิดในมิติไหน?
2. ถ้าจะแก้ prompt — คุณจะเพิ่มหรือเปลี่ยนส่วนไหน (Context / Task / Format / Constraints)?
3. เขียน prompt v2 โดยแก้เฉพาะส่วนนั้น แล้วดูว่า output เปลี่ยนไปอย่างที่คาดไหม

การทำซ้ำ 5-10 ครั้งกับ task ต่าง ๆ จะฝึก pattern recognition ที่ใช้ได้กับทุก task ใหม่.

## ดูกลไกลึกลงอีกชั้น — ทำไม prompt ชัดถึง converge ได้เร็วกว่า

ส่วนนี้สำหรับคนที่ต้องการเข้าใจกลไกมากกว่าแค่ "เพราะมัน specific กว่า" — ถ้าเนื้อหาก่อนหน้าพอแล้ว ข้ามไปที่สรุปได้เลย.

เวลา LLM รับ prompt มา มันไม่ได้ "อ่าน" แล้ว "คิด" แบบคน. มันทำนาย token ถัดไปจาก distribution ของ probability — แต่ละ token มีโอกาสเป็น output ถัดไปในระดับที่ต่างกัน.

นึกง่าย ๆ แบบนี้: ถ้า context เป็นแค่ "เขียน" — token ถัดไปที่น่าจะเป็น มีหลายร้อยตัวเลือก ("บทความ", "เรื่องราว", "จดหมาย", "รายงาน", ฯลฯ) แต่ละตัวมี probability ใกล้เคียงกัน. โมเดลต้อง "เลือก" จากตัวเลือกกว้างมาก.

เมื่อ context เพิ่มเป็น "เขียนอีเมลภาษาทางการ 150 คำถึงลูกค้า" — token ถัดไปที่น่าจะเป็น "เรียน" หรือ "สวัสดี" หรือ "Dear" ขึ้นมาสูงมาก ตัวเลือกอื่น ๆ ตกไปเกือบหมด. distribution แคบลงมาก — โมเดล converge เร็วกว่า.

กลไกนี้อธิบายสิ่งที่เห็นได้ในทางปฏิบัติหลายอย่าง:

**ทำไม output prompt เดิมถึงต่างกันในแต่ละครั้ง** — เพราะโมเดลไม่ได้เลือก token ที่ "ดีที่สุด" เสมอไป มันสุ่มจาก distribution โดยมี temperature เป็นตัวควบคุม (สูง = สุ่มมาก, ต่ำ = คาดเดาได้มาก). prompt ที่กำกวมมี distribution กว้าง ผลต่างระหว่าง run จึงสูง.

**ทำไม constraint เชิงลบ ("ห้าม X") ถึงช่วยได้** — เพราะมัน cut probability ของ token cluster ที่ไม่ต้องการออก เหลือ space ที่แคบลง. ถ้าสั่ง "ห้ามใช้ passive voice" — โมเดลจะ down-weight pattern ที่เคยเห็นว่า "ประโยค passive" แล้ว up-weight pattern "ประโยค active" แทน.

**ทำไม example ถึงทรงพลังมาก** — เพราะ example คือ "ground truth" ของ output ที่ต้องการ โมเดลเห็น pattern จาก example แล้ว copy style/structure นั้นโดยตรง ไม่ต้องอนุมานจาก description. มันตรง กว่าการอธิบายด้วยคำพูด.

> 💡 **Key insight:** prompt engineering คือการ shape distribution ของ output ก่อนที่โมเดลจะเริ่มสร้าง token แรก. Context กำกวม = distribution กว้าง = output สุ่ม. Context ชัดเจน = distribution แคบ = output คาดเดาได้. นี่คือกลไกเดียวกับที่อธิบายว่าทำไมทุกเทคนิคใน prompt engineering ถึง "ได้ผล" — ทุกอย่างกลับมาที่การตัด output space.

## prompt engineering ใช้กับงานประจำวันยังไง

หลักการฟังดูดี แต่ในชีวิตจริงคนส่วนใหญ่ไม่ได้นั่งเขียน prompt ทุกครั้ง. ต่อไปนี้คือวิธีใช้ให้ได้ประโยชน์จริง ๆ โดยไม่ต้องเสียเวลาทุกครั้ง.

**สร้าง prompt template สำหรับงานที่ทำซ้ำบ่อย**

ถ้ามีงานประจำที่ต้องใช้ AI ซ้ำทุกสัปดาห์ — เช่น สรุป meeting notes, เขียนอีเมลติดตามงาน, ตรวจ draft ของทีม — ลงทุน iterate prompt นั้นครั้งเดียวจนได้ผลดี แล้วบันทึกไว้เป็น template.

ตัวอย่าง template สำหรับสรุป meeting:
```
สรุป meeting notes นี้โดย:
1. สรุปการตัดสินใจหลัก (decision) ในรูป bullet — แต่ละข้อเริ่มด้วย "ตัดสินใจว่า: ..."
2. สรุป action items ในตาราง: | ใคร | ทำอะไร | deadline |
3. สรุปประเด็นที่ยังไม่ได้ข้อสรุป (parking lot) ถ้ามี
4. โทน: กระชับ ไม่ต้องอธิบายบริบทเพิ่ม เป็น reference สำหรับคนที่อยู่ในห้อง

[วาง meeting notes ตรงนี้]
```

template นี้ลงทุนครั้งเดียว ใช้ได้ทุก meeting. ถ้า meeting แต่ละครั้งต่างกัน ก็แค่ปรับ section 1-3 เล็กน้อย ไม่ต้องเขียนใหม่ทั้งหมด.

**ใช้ conversation แทน single prompt**

งานซับซ้อนไม่จำเป็นต้องอยู่ใน prompt เดียว. การแบ่งเป็นหลาย turn ให้โมเดลทำทีละขั้น แล้วตรวจสอบก่อนไปต่อ มักได้ผลดีกว่าการยัดทุกอย่างลงไปครั้งเดียว.

ตัวอย่าง: ต้องการเขียนบทความ 500 คำ
- Turn 1: "ช่วยทำ outline 5 หัวข้อสำหรับบทความเรื่อง [X] กลุ่มเป้าหมาย [Y]"
- ตรวจ outline แก้ถ้าจำเป็น
- Turn 2: "เขียนเต็ม section แรก ตาม outline ที่ตกลงกัน"
- ตรวจ แก้
- Turn 3: ต่อ section ถัดไป

วิธีนี้ให้คุณ review และ steer ระหว่างทาง แทนที่จะรอ output ยาว 500 คำแล้วพบว่าทิศผิดตั้งแต่ต้น.

**รู้ว่าเมื่อไหร่ไม่ต้องเขียน prompt ดีเลย**

ไม่ใช่ทุก task ต้องการ prompt ที่ดีที่สุด. งานที่ output ผิดนิดหน่อยยังใช้ได้ หรืองานที่คุณจะ edit ต่ออยู่แล้ว — prompt คร่าว ๆ ก็พอ. ลงทุน iterate prompt ให้คุ้มกับ task ที่ทำซ้ำบ่อยหรือ output ต้องแม่นมาก เช่น เอกสารสำคัญ, communication กับลูกค้าภายนอก, งานที่คนอื่นจะอ่านโดยตรง.

## สรุป — กลับไปที่หลักการเดียว

ทุกอย่างที่อ่านมาสรุปได้เป็นหลักการเดียว: **LLM ทำงานบน pattern — ยิ่ง context ชัดเจน โมเดล converge เข้าหา output ที่คาดเดาได้มากขึ้น**.

5 องค์ประกอบ (Context, Task, Examples, Format, Constraints) คือ framework สำหรับตัดทางเลือก output ที่ไม่ต้องการออก. คุณไม่จำเป็นต้องใช้ครบทุกองค์ประกอบทุกครั้ง — แต่ทุกครั้งที่ output ไม่ตรง คำตอบมักอยู่ที่ส่วนใดส่วนหนึ่งใน 5 นี้ที่ยังขาดหรือกำกวม.

Process iterate (อ่าน → ระบุจุดผิด → map → แก้เฉพาะส่วน) คือวิธีทำให้การปรับปรุงมีระบบ ไม่ใช่ random. คนที่ใช้เวลา 30 วินาทีอ่าน output ก่อน iterate มักใช้รอบน้อยกว่าคนที่ลองสุ่มแก้ prompt มาก.

และเมื่อ iterate 5-10 รอบแล้วยังไม่ได้ผล — สัญญาณนั้นมักหมายความว่า prompt engineering ไม่ใช่เครื่องมือที่ถูกต้องสำหรับปัญหานั้น ไม่ใช่ว่าต้องเขียน prompt ให้ยาวขึ้นอีก. ข้อมูลที่ต้องการอาจไม่ได้อยู่ในโมเดล หรือ task อาจต้องการ multi-step reasoning ที่เกินขอบเขต prompt เดียว.

บทนี้ให้พื้นฐาน แต่ยังมี pattern ขั้นสูงที่ต่อยอดต่อไปได้: few-shot prompting อย่างละเอียด, chain-of-thought, structured output, prompt chaining และ ReAct pattern — ทั้งหมดนั้นอยู่ใน [Prompt Engineering Patterns](/using-ai/prompt-engineering-patterns) บทถัดไป. ถ้าบทนี้ทำให้รู้สึกว่า "เข้าใจแล้วว่าทำไม" — บทถัดไปจะให้ "toolkit ว่าต้องทำยังไง" ในสถานการณ์ที่ซับซ้อนขึ้น.

## Quick reference — สิ่งที่ควรจำก่อนออกไปใช้จริง

**5 องค์ประกอบ prompt ที่ดี:**

| ส่วน | ถามตัวเองว่า | ตัวอย่าง |
|---|---|---|
| Context | ใคร ทำไม ใช้ที่ไหน | "ฉันเป็น PM เตรียม deck ให้ VC" |
| Task | กริยาที่ชัดคืออะไร | "เขียน / สรุป / วิเคราะห์ / เปรียบเทียบ" |
| Examples | format แปลกอธิบายยากไหม | "ตัวอย่าง before/after 2 คู่" |
| Format | รูปแบบ ความยาว โทน ภาษา | "JSON / 5 bullet / ทางการ / ไทย" |
| Constraints | default ของโมเดลที่ไม่ต้องการคืออะไร | "ห้าม emoji ห้ายาวเกิน 200 คำ" |

**Checklist ก่อนส่ง prompt:**
- [ ] task มีกริยาที่ชัดไหม — ถ้าไม่มี เพิ่มก่อน
- [ ] บอก output format แล้วไหม — ถ้าไม่บอก จะได้ paragraph default
- [ ] context เพียงพอให้โมเดลรู้ว่า output จะถูกใช้ยังไงไหม
- [ ] มี constraint ที่จับต้องได้สำหรับสิ่งที่ไม่ต้องการไหม

**Checklist ก่อน iterate:**
- [ ] อ่าน output ครบแล้วไหม
- [ ] ระบุได้แล้วไหมว่าผิดในมิติไหน (format / content / tone / length)
- [ ] map กลับไปที่ component ของ prompt ได้ไหม
- [ ] แก้เฉพาะส่วนนั้น ไม่เขียนใหม่ทั้ง prompt

**สัญญาณว่าถึงเวลาเปลี่ยนวิธี:**
- iterate มากกว่า 10 รอบแล้วยังไม่ได้
- ข้อมูลที่ต้องการไม่ได้อยู่ใน training data ของโมเดล
- ต้องการ guarantee ที่ instruction ใน prompt ให้ไม่ได้
