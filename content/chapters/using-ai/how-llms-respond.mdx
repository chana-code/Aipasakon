# LLM ตอบคำถามยังไง — กลไกข้างในที่คุณควรรู้

คุณส่งข้อความไปแล้วรอ. สักครู่คำตอบเริ่มปรากฏทีละคำ ราวกับมีคนนั่งพิมพ์อยู่อีกฝั่ง. บางทีถามเรื่องเดิมสองครั้งได้คำตอบต่างกัน บางทีคำตอบหยุดกลางประโยคโดยไม่รู้เหตุผล.

ทั้งหมดนี้ไม่ใช่เรื่องสุ่มสี่สุ่มห้า มีกลไกชัดเจนที่อธิบายทุกอย่างได้. และเมื่อคุณเข้าใจกลไกนี้ คุณจะใช้ AI ได้ฉลาดขึ้นทันที — ไม่ใช่เพราะคุณรู้คำศัพท์เพิ่ม แต่เพราะคุณเข้าใจว่ามันทำงานยังไงจริง ๆ ข้างใน

## เกิดอะไรขึ้นในตอนที่คุณกด Enter

ถ้าคุณเคยอ่านเรื่อง [LLM คืออะไร](/foundations/what-is-an-llm) มาแล้ว คุณรู้ว่า LLM ทำงานบนหลักการเดียว: ทำนายคำถัดไป. บทนั้นอธิบายว่า "LLM คืออะไร" บทนี้จะอธิบายว่า "LLM ตอบยังไง" — กระบวนการที่เกิดขึ้นตั้งแต่คุณกด Enter จนได้คำตอบ

ลองเดินผ่านทีละขั้น. สมมติคุณพิมพ์ว่า "ช่วยแนะนำร้านกาแฟในสยามหน่อย" แล้วกด Enter:

**ขั้นที่ 1 — ข้อความถูกตัดเป็นชิ้นเล็ก ๆ ที่เรียกว่า token**

ก่อนที่โมเดลจะทำอะไรได้ ข้อความของคุณต้องถูกแปลงให้อยู่ในรูปที่คอมพิวเตอร์เข้าใจก่อน. กระบวนการแรกคือตัดข้อความออกเป็นหน่วยเล็ก ๆ แต่ละหน่วยเรียกว่า **token**.

Token ไม่ใช่ตัวอักษร และไม่ใช่คำ แต่อยู่ระหว่างกลาง — ลองนึกว่าเป็น "ชิ้นส่วนของคำ" ที่โมเดลรู้จักเป็นก้อน. ในภาษาอังกฤษ 1 token มักเป็นคำหนึ่งคำหรือส่วนของคำ ในภาษาไทยมักเป็นกลุ่มตัวอักษร 1-3 ตัว เพราะโมเดลส่วนใหญ่ไม่ "รู้จัก" คำไทยเป็นก้อนใหญ่ได้ดีเท่าคำอังกฤษ.

ประโยค "ช่วยแนะนำร้านกาแฟในสยามหน่อย" อาจถูกตัดเป็นประมาณ 15-20 token. ถ้าเป็นภาษาอังกฤษ "Can you recommend a coffee shop in Siam?" อาจแค่ 9 token — ความหมายเดียวกัน แต่ใช้ token ต่างกันมาก เราจะกลับมาพูดถึงเรื่องนี้ในหัวข้อ "Token ภาษาไทยแพงกว่า"

**ขั้นที่ 2 — โมเดลดู token ทั้งหมดที่มี แล้วคำนวณว่า token ถัดไปน่าจะเป็นอะไร**

นี่คือขั้นที่คณิตศาสตร์หลายพันล้านสมการทำงาน. ลองนึกว่ามันเหมือนการ "ชั่งน้ำหนัก" ให้ token ทุกตัวใน vocabulary — token ที่น่าจะมาตรงนี้ได้น้ำหนักมาก token ที่ไม่น่ามาได้น้ำหนักน้อย. ผลรวมน้ำหนักทั้งหมดเท่ากับ 100% เสมอ.

ลองนึกว่าคุณกำลังโยนเหรียญที่มีน้ำหนักไม่เท่ากัน — "ร้าน" หนัก 12 กรัม "มี" หนัก 9 กรัม "สยาม" หนัก 7 กรัม และที่เหลืออีกหมื่นตัวรวมกัน 72 กรัม. เมื่อคว่ำถาด เหรียญที่หนักกว่ามีโอกาสหงายหน้ามากกว่า แต่ก็ไม่ได้ชนะทุกครั้ง.

ผลลัพธ์ของขั้นนี้คือรายการโอกาสของ token ทุกตัวใน vocabulary ซึ่งมีตั้งแต่ 50,000 ถึง 200,000 token. ในวงการเรียกรายการโอกาสชุดนี้ว่า **probability distribution** — คิดง่าย ๆ ว่าเป็น "ตารางความน่าจะเป็น" ของทุกคำที่เป็นไปได้. เช่น:
- token "ร้าน" — 12%
- token "มี" — 9%
- token "สยาม" — 7%
- token "แน" — 5%
- และอีกหลายหมื่นตัวที่เหลือ รวมกันเป็นส่วนที่เหลืออีก 67%

**ขั้นที่ 3 — ระบบ "จั่ว" token หนึ่งตัวจากรายการนั้น**

ไม่ใช่เลือก token ที่น่าจะเป็นที่สุดเสมอไป แต่สุ่มตามน้ำหนัก token ที่มีโอกาส 12% มีโอกาสถูกเลือก 12 ครั้งใน 100 ครั้ง token ที่มี 5% มีโอกาส 5 ครั้ง เป็นต้น. ขั้นนี้เรียกว่า **sampling** และมันคือเหตุผลทั้งหมดที่ output ออกมาไม่เหมือนเดิมทุกครั้ง

**ขั้นที่ 4 — token ที่ได้ถูกส่งขึ้นหน้าจอทันที แล้วต่อท้ายเข้าไปใน input**

คุณเห็นตัวอักษรปรากฏ ขณะเดียวกันโมเดลก็เห็น token ตัวนั้นด้วย. แล้ววนกลับขั้นที่ 2 โดยมีบริบทยาวขึ้น 1 token. ทำแบบนี้ซ้ำไปเรื่อย ๆ จนกว่าโมเดลจะหยุด.

> 💡 **สิ่งที่สำคัญที่สุดจากกระบวนการนี้:** LLM ไม่ได้ "คิดคำตอบทั้งก้อน" ก่อนพิมพ์ มันตัดสินใจทีละ token และทุกการตัดสินใจขึ้นกับทุก token ที่ผ่านมา รวมถึง token ที่มันเพิ่งเลือกไปเอง

## ทำไมถามซ้ำได้คำตอบต่างกัน

นี่คือคำถามที่หลายคนสงสัย และมักคิดว่าเป็น bug หรือว่า AI "ไม่แน่ใจ" ในคำตอบของตัวเอง

ไม่ใช่ทั้งสองอย่าง. มันเกิดจากขั้น sampling ที่อธิบายไปข้างบน

ลองนึกภาพแบบนี้: ก่อนพ่อค้าแม่ค้าตลาดจะเลือกว่าจะพูดอะไรต่อ เขามีตัวเลือกหลายอย่างในหัวที่ล้วนเหมาะสม เช่น จะพูดว่า "อร่อยมากเลยนะ" หรือ "ของสดทุกวันเลย" หรือ "มาบ่อย ๆ นะ" ก็ได้ทั้งนั้น แต่ละครั้งที่ขายของเขาอาจพูดต่างกันเล็กน้อย

LLM คล้ายกันตรงที่มันก็ "มีตัวเลือกหลายอย่าง" พร้อมกัน ทุกตัวมีความน่าจะเป็น ทุกครั้งที่ generate ระบบ "จั่ว" ใหม่จาก probability distribution นั้น

เปรียบเทียบนี้ใช้ได้แค่จุดเดียว ตรงที่มีตัวเลือกหลายอย่างพร้อมกัน ส่วนที่ต่างคือ พ่อค้ามีความรู้สึกและประสบการณ์จริง LLM ไม่มี มันแค่มีตัวเลขที่บอกว่า token ไหนน่าจะมาตรงนี้

สำหรับคำถาม "แนะนำชื่อสุนัขเพศผู้ 1 ชื่อ" ให้ลองส่งไปสาม-สี่ครั้ง คุณจะได้: "แม็กซ์", "โบ", "มอนเต้", "ลีโอ" — คนละคำตอบทุกครั้ง ทั้งที่โมเดลไม่ได้ "คิด" ใหม่จากศูนย์ มันแค่สุ่มจาก probability distribution ที่ชื่อสุนัขหลาย ๆ ตัวมีความน่าจะเป็นใกล้กัน

> 💡 **Key insight:** randomness ของ LLM ไม่ใช่ความไม่แน่นอนในความหมาย "ไม่รู้จะตอบอะไร" แต่เป็น design choice ที่ทำให้คำตอบอ่านแล้วฟังดูเป็นธรรมชาติ ถ้าโมเดลเลือก token ที่น่าจะเป็นที่สุดเสมอ คำตอบจะออกมาซ้ำซากและน่าเบื่อ — ลองนึกถึงโฆษณาที่ใช้คำเดิมซ้ำ ๆ จนไม่มีใครอยากอ่าน

### ลองทำดู — สังเกต randomness ด้วยตัวเอง

เปิด ChatGPT หรือ Claude แล้วพิมพ์คำถามเดิมสองครั้งติดกัน เลือกคำถามที่ตอบได้หลายแบบ เช่น:

> "แนะนำประโยคเปิด email ขอบคุณลูกค้า 1 ประโยค"

กด Enter รอคำตอบ แล้วส่งข้อความเดิมอีกครั้ง

สังเกตว่าได้คำตอบต่างกันไหม ถ้าต่างกัน นั่นคือ sampling กำลังทำงาน. ถ้าเหมือนกันมาก อาจเป็นเพราะ token แรกที่น่าจะเป็นที่สุดมีโอกาสสูงมากจนการสุ่มออกมาตัวเดิมทุกครั้ง

ลองเปลี่ยนคำถามเป็นแบบที่ตอบได้หลายทาง เช่น "เปรียบ AI เหมือนอะไรในชีวิตประจำวัน 1 อย่าง" แล้วส่งซ้ำสี่ห้าครั้ง คุณจะเห็นความแตกต่างชัดกว่า

## สุ่มแค่ไหน — และ temperature คืออะไร

ขั้น sampling ที่อธิบายไปข้างบนไม่ได้ทำงานแบบเดียวกันเสมอ มีตัวปรับที่ชื่อว่า **temperature** — ลองนึกว่าเป็น "ไดอัลควบคุมความกล้าเสี่ยง" ของโมเดล

temperature ต่ำ (ใกล้ 0) หมายถึงโมเดลจะเลือก token ที่มีโอกาสสูงสุดแทบทุกครั้ง คำตอบนิ่ง ทำซ้ำได้ แต่อาจรู้สึกแข็งและซ้ำซาก temperature สูงหมายถึง token ที่มีโอกาสน้อยก็ยังพอมีสิทธิ์ถูกเลือก ทำให้คำตอบหลากหลายและฟังดูเป็นธรรมชาติมากขึ้น แต่ควบคุมยากขึ้น

ChatGPT และ Claude ใช้ค่ากลาง ๆ เป็น default เพราะให้คุณภาพการสนทนาดีที่สุดสำหรับการใช้งานทั่วไป

บท [Temperature คืออะไร](/using-ai/temperature-and-determinism) ที่จะมาถึงใน Level นี้ จะอธิบายกลไกนี้อย่างละเอียด รวมถึงวิธีเลือกค่า temperature ให้เหมาะกับงานแต่ละประเภท

## ทำไมข้อความไหลขึ้นจอทีละคำ

หลายคนคิดว่าที่เห็น ChatGPT หรือ Claude "พิมพ์" ทีละคำเป็นแค่ animation ที่ทีม UX ออกแบบมาให้ดูน่าตื่นเต้น ราวกับมีใครนั่งพิมพ์จริง ๆ

ไม่ใช่. โมเดลสร้าง token ทีละตัวจริง ๆ ที่ฝั่ง server. ไม่มีทางที่จะรู้ token ตัวที่ 10 ก่อนที่จะสร้าง token ตัวที่ 9 เพราะ token ตัวที่ 9 คือส่วนหนึ่งของข้อมูลสำหรับการตัดสินใจ token ตัวที่ 10

เมื่อสร้างแบบนี้อยู่แล้ว ระบบก็ส่ง token ให้คุณเห็นทันทีที่ได้แต่ละตัว แทนที่จะรอจนทั้งคำตอบเสร็จ กระบวนการนี้เรียกว่า **streaming** — ส่งออกมาเป็น "กระแส" ทีละชิ้น เหมือนน้ำที่ไหลต่อเนื่องแทนที่จะรอให้เต็มถังก่อนแล้วค่อยเทออกมาทีเดียว

**ทำไม streaming ถึงสำคัญกับคุณในทางปฏิบัติ:**

ประการแรก คุณรู้สึกว่า AI ตอบเร็ว. เวลาตั้งแต่กด Enter จนเห็นตัวอักษรแรก อาจแค่ 0.5-1 วินาที แม้คำตอบทั้งหมดจะใช้เวลา 15-20 วินาที สมองคนอ่านเวลารอที่รู้สึกว่า "มีอะไรเกิดขึ้น" ต่างจาก "รอนิ่งแล้วพลุ่งออกมา"

ประการที่สอง คุณอ่านไปพร้อมกับที่มัน generate. แทนที่จะนั่งรอเฉย ๆ 15 วินาที คุณอ่านได้เลย 3-4 บรรทัดแรกไปพร้อมกัน ประหยัดเวลาจริง ๆ

ประการที่สาม คุณหยุดกลางคันได้. และนี่คือสิ่งที่คนส่วนใหญ่ไม่รู้ว่าทำได้ และประหยัดได้มากถ้าใช้ API

> 💡 **Key insight:** ถ้าคุณกำลังรอคำตอบยาว ๆ และเห็นจาก 3-4 บรรทัดแรกว่าโทนผิดหรือทิศทางไม่ใช่ กด Stop ทันที คุณจ่ายเฉพาะ token ที่สร้างไปแล้ว ไม่ใช่คำตอบทั้งก้อน แล้วค่อย rewrite prompt ใหม่ ผู้ใช้ที่เชี่ยวชาญ AI ใช้นิสัยนี้ตลอด — อ่านไปพร้อมกับที่ AI เขียน แล้วหยุดทันทีที่รู้ว่าผิดทาง

**สิ่งที่ streaming ไม่ใช่:**

streaming ไม่ได้แปลว่าโมเดล "คิดสด ๆ ออกหน้ากล้อง" ในความหมายที่มันแก้ของเก่าได้. token ที่ commit ไปแล้วคือ commit ตลอดกาล. โมเดลไม่ย้อนกลับไปแก้ token ตัวที่ 5 หลังจากสร้าง token ตัวที่ 15 แล้ว

ถ้าคุณเห็น AI เขียนออกมาว่า "ร้านนี้อยู่ที่..." แล้วในบรรทัดต่อมาบอกว่า "จริง ๆ ฉันไม่มีข้อมูลร้านนี้" — นั่นไม่ใช่การแก้ตัว เป็นแค่ token ถัดไปที่ถูกสร้างออกมาตามลำดับ

สิ่งที่ตามมาคือ ถ้าคุณเห็น AI "แก้ตัวเอง" กลางคำตอบ แสดงว่า token แรก ๆ พา context ไปทิศหนึ่ง แต่พอ context สะสมมากพอ probability ของ token ที่แก้ทิศก็สูงขึ้น มันไม่ได้ "รู้สึกผิด" — มันแค่กำลังสร้าง token ที่น่าจะมาในตำแหน่งนั้นตามบริบทที่มีอยู่ทั้งหมด

## เมื่อคำตอบหยุดกลางประโยค

บางครั้งคุณขอให้ AI เขียนอะไรสักอย่าง แล้วได้คำตอบที่หยุดกลาง ๆ ดูไม่จบ คุณอาจคิดว่าโมเดล "ขี้เกียจ" หรือ "ลืม" ว่ากำลังทำอะไรอยู่

ไม่ใช่ทั้งสองอย่าง มีสี่สาเหตุที่ LLM หยุดสร้าง token และแต่ละอย่างมีสัญญาณที่แตกต่างกัน:

**1. สัญญาณจบ — โมเดลตัดสินใจเองว่าจบแล้ว**

ใน vocabulary ของโมเดลมี token พิเศษตัวหนึ่งที่มีความหมายว่า "จบแล้ว" นักพัฒนาเรียกมันว่า **EOS token** (ย่อจาก end-of-sequence หรือ "สัญญาณสิ้นสุดลำดับ"). โมเดลเรียนรู้ระหว่าง training ว่าควรสร้าง token นี้เมื่อไหร่ เมื่อสุ่มได้ EOS ระบบหยุดทันที นี่คือการจบที่สมบูรณ์ สังเกตได้จากประโยคสุดท้ายที่จบครบ ไม่กลางคัน

**2. คำห้าม — ผู้พัฒนากำหนดว่าถ้าเจอข้อความนี้ให้หยุด**

สำหรับคนที่ใช้ API หรือสร้างแอปบน AI ผู้พัฒนาสามารถกำหนดลิสต์ของข้อความว่า "ถ้าสร้างออกมาแล้วเจอตัวนี้ให้หยุดเลย" เช่น ถ้าสร้าง chatbot และไม่อยากให้โมเดลพูดเลียนแบบฝั่งผู้ใช้ต่อไปเรื่อย ๆ ก็กำหนดให้หยุดเมื่อเจอข้อความบางแบบ ผู้ใช้ทั่วไปไม่ค่อยเจอสาเหตุนี้โดยตรง

**3. ชน budget ที่กำหนดไว้ — สาเหตุที่พบบ่อยที่สุด**

ทุกการเรียกใช้ LLM มีการตั้ง limit สูงสุดว่าสร้างได้กี่ token เมื่อถึง limit ระบบหยุดทันทีไม่ว่าประโยคจะจบหรือยังไม่จบ

ถ้าคุณเรียกผ่าน API เองและตั้ง max_tokens=100 แล้วขอให้เขียนบทความ 500 คำ คุณจะได้แค่ 1-2 ย่อหน้าแรกแล้วถูกตัด สัญญาณ: ข้อความจบกลางประโยคอย่างกะทันหัน แก้โดยเพิ่มตัวเลข max_tokens

**4. Server หมดเวลา — ใช้เวลานานเกินที่ provider กำหนด**

เกิดไม่บ่อย แต่เกิดได้ในคำตอบที่ยาวมากหรือในช่วงที่ server โหลดหนัก สัญญาณ: การเชื่อมต่อขาด ไม่ใช่แค่ข้อความหยุด

**วิธีแก้เมื่อคำตอบหยุดกลางคัน:**

ถ้าใช้ ChatGPT หรือ Claude โดยตรง (ไม่ใช่ API) ให้ลองพิมพ์ว่า "พูดต่อได้เลย" หรือ "continue" แล้วกด Enter โมเดลจำบริบทของบทสนทนา จะ resume จากจุดที่หยุดได้ แต่อาจมีรอยต่อเล็กน้อย

ถ้าใช้ผ่าน API ให้เพิ่ม `max_tokens` ในการตั้งค่า

### ลองทำดู — ดูขีดจำกัดด้วยตัวเอง

เปิด Claude.ai หรือ ChatGPT แล้วลองขอบางอย่างที่ตอบยาว เช่น:

> "ช่วยเขียนแผนธุรกิจเบื้องต้นสำหรับร้านกาแฟเล็ก ๆ ในชุมชน ให้ละเอียดมาก ๆ ครอบคลุมทุกด้านตั้งแต่ทุนเริ่มต้น การตลาด ไปจนถึงการจัดการพนักงาน"

สังเกตว่าคำตอบจบสมบูรณ์ หรือหยุดกลางคัน ถ้าหยุดกลางคัน ลองพิมพ์ต่อว่า "พูดต่อได้เลย" แล้วดูว่ามัน resume ได้ไหม

สังเกตด้วยว่าคำตอบยาวแค่ไหน แล้วลองเปรียบกับคำถามสั้น ๆ อย่าง "ชื่อดอกไม้ไทย 3 ชนิด" ที่ตอบได้ใน 2-3 บรรทัด ความเร็วในการตอบต่างกันมาก เพราะจำนวน token output ต่างกัน

## Token ภาษาไทยแพงกว่า — และทำไมคุณต้องรู้เรื่องนี้

มีเรื่องหนึ่งที่คนใช้ AI ภาษาไทยควรเข้าใจ และมันเกี่ยวกับวิธีที่ข้อความถูกแบ่งเป็น token

ในขั้นที่ 1 ของกระบวนการ ข้อความถูกตัดเป็น token ก่อนที่โมเดลจะเริ่มทำงาน กระบวนการตัดนี้เรียกว่า **tokenization** และมันไม่ได้ตัดเท่ากันทุกภาษา

**ทำไมภาษาไทยถึงตัดละเอียดกว่า:**

LLM ส่วนใหญ่ที่คนไทยใช้กัน เช่น ChatGPT ของ OpenAI, Claude ของ Anthropic, Gemini ของ Google ถูกสร้างจากข้อมูลภาษาอังกฤษเป็นหลัก ทำให้ระบบตัดคำของมันรู้จักคำภาษาอังกฤษ "เป็นก้อน" ได้ดี คำยอดนิยมอย่าง " the", " and", " is" มักเป็น 1 token ต่อคำ

แต่ภาษาไทยถูกซอยละเอียดกว่ามาก เพราะโมเดลไม่ค่อย "รู้จัก" คำไทยเป็นก้อนใหญ่ คำหนึ่งคำอาจถูกตัดเป็นหลาย token ขึ้นกับว่าระบบตัดคำของรุ่นนั้นเคยเห็นคำนั้นในข้อมูล training มากแค่ไหน

ตัวอย่างที่จับต้องได้:

| ข้อความ | token โดยประมาณ |
|---|---|
| "Hello, how are you?" | ~6 token |
| "สวัสดี เป็นอย่างไรบ้าง" | ~15-25 token |
| "I need help with my project" | ~8 token |
| "ต้องการความช่วยเหลือเรื่องโปรเจกต์" | ~20-35 token |

ความหมายเดียวกัน แต่ภาษาไทยใช้ token มากกว่าเกือบสามเท่า

**ผลกระทบในทางปฏิบัติสามอย่าง:**

ประการแรก **ค่าใช้จ่ายสูงกว่า** ถ้าคุณใช้ API ของ OpenAI, Anthropic, หรือ Google โดยตรง คุณจ่ายตาม token ทั้ง input และ output prompt ภาษาไทยจึงแพงกว่า prompt ภาษาอังกฤษที่มีเนื้อหาเทียบเท่ากันประมาณ 2-3 เท่า

ประการที่สอง **พื้นที่รับข้อความหมดเร็วกว่า** LLM ทุกตัวมีขีดจำกัดว่ารับ token ได้กี่ตัวในครั้งเดียว เรียกว่า context window — คิดว่าเป็น "ความจำระยะสั้น" ที่มีขนาดจำกัด ถ้าคุณมีเอกสารภาษาไทย 10 หน้า อาจกิน token เท่ากับเอกสารภาษาอังกฤษ 25-30 หน้า ถ้าคุณส่งเอกสารยาวให้ AI สรุป อาจเจอปัญหาว่าพื้นที่รับข้อความเต็มก่อนที่คิดไว้ เรื่อง context window เจาะลึกไว้ในบท [Context Windows](/using-ai/context-windows) ที่จะมาถึง

ประการที่สาม **โมเดลเข้าใจภาษาไทยได้ยากกว่าเล็กน้อย** เมื่อคำถูกซอยแบบไม่ค่อยสมเหตุสมผล โมเดลทำงานได้ยากขึ้น ส่งผลต่อคุณภาพการตอบบ้างในบางกรณี เช่น ชื่อเฉพาะภาษาไทยที่ซอยออกมาแปลก ๆ อาจทำให้โมเดลตีความผิดได้

> 💡 **Key insight:** tokenization เป็นเหตุผลที่คุณรู้สึกว่า AI ตอบภาษาไทยช้ากว่าภาษาอังกฤษเล็กน้อย และทำไม prompt ภาษาไทยยาว ๆ บางทีเกินขีดจำกัดก่อนที่คุณคิดว่าจะเกิน สถานการณ์กำลังดีขึ้นเรื่อย ๆ เพราะ Gemini, Claude รุ่นใหม่ ๆ และโมเดลที่ออกแบบมาสำหรับเอเชียมีระบบตัดคำที่ handle ภาษาไทยได้ดีขึ้น แต่ยังเป็นเรื่องที่ควรรู้ในปี 2026

## โมเดลไม่ได้ "วางแผน" ก่อนพิมพ์

นี่คือความเข้าใจผิดที่พบบ่อย และมันส่งผลต่อวิธีที่คุณเขียน prompt

หลายคนคิดว่า LLM "คิดคำตอบทั้งก้อนก่อน" แล้วค่อยพิมพ์ออกมา ราวกับมีคนนั่งเขียนอยู่ในใจก่อน คิดครบแล้วค่อยกด Ctrl+V

ไม่ใช่. LLM ตัดสินใจ **ทีละ token** โดยใช้บริบทที่มีอยู่ ณ ขณะนั้น ไม่มีการมองไปข้างหน้าว่า token อีก 50 ตัวต่อจากนี้ควรเป็นอะไร

ลองนึกภาพนักเขียนที่ห้ามอ่านล่วงหน้าและห้ามลบ ทุกคำที่เขาเขียน ต้องดูทุกอย่างที่เขียนไปแล้ว แล้วเลือกคำต่อไปที่ "น่าจะมาตรงนี้" เขาไม่ได้วางโครงเรื่องทั้งหมดไว้ในหัวก่อนเริ่มเขียน โครงค่อยปรากฏไปพร้อมกับที่เขาเขียน

เปรียบเทียบนี้ใช้ได้ดีแค่จุดเดียว คือ "ตัดสินใจทีละก้าวโดยใช้สิ่งที่มาก่อน" ส่วนที่ต่างคือนักเขียนมีประสบการณ์ ความรู้สึก และความคิดสร้างสรรค์จริง ๆ LLM แค่มีตัวเลขที่บอกว่าคำไหนน่าจะมาต่อจากคำก่อนหน้า

**ผลที่ตามมาสามอย่างที่กระทบการใช้งานของคุณ:**

หนึ่ง ถ้า token แรก ๆ ของคำตอบออกไปผิดทาง โมเดลจะพยายาม "แก้ตัว" ในส่วนที่เหลือ ซึ่งมักทำให้คำตอบทั้งก้อนเพี้ยน แก้โดยการหยุดและ rewrite prompt แทนที่จะรอให้จบแล้วค่อยขอใหม่

สอง ยิ่งคำตอบยาวขึ้น ยิ่งช้าลงและแพงขึ้น เพราะทุก token ต้องผ่านการคำนวณเต็มรอบผ่านทั้งโมเดล ใช้เวลาสร้าง 200 token ยาวกว่า 100 token เกือบสองเท่า นี่ไม่ใช่เพราะเนื้อหา "ยากขึ้น" แต่เพราะมี token มากขึ้น

สาม เทคนิคที่เรียกว่า chain-of-thought ใช้ได้ผลดีมาก เพราะมันบังคับให้ "การวางแผน" ปรากฏเป็น token จริง ๆ ในคำตอบก่อนที่โมเดลจะตอบคำถามหลัก โมเดลอ่าน "การวางแผน" ของตัวเองกลับเข้ามาเป็นบริบท แล้วตอบได้แม่นยำขึ้น เรื่องนี้จะพูดถึงใน [บทเรื่อง prompt engineering](/using-ai/prompt-engineering-basics)

## ทำไมตั้ง temperature=0 แล้วยังได้คำตอบไม่เหมือนกัน

> ถ้าคุณใช้ AI ผ่านหน้าเว็บหรือแอปทั่วไปและไม่ได้แตะ API หัวข้อนี้เป็นเรื่องน่ารู้แต่ไม่ส่งผลต่อการใช้งานประจำวัน ข้ามไปหัวข้อถัดไปได้เลยถ้าต้องการ

ถ้าคุณเคยใช้ API ของ LLM คุณอาจเคยลองตั้ง `temperature=0` เพราะคิดว่าจะได้คำตอบเหมือนเดิมทุกครั้ง แล้วพบว่ายังไม่เหมือนกัน 100%

มีเหตุผลอยู่ — และไม่ได้เกี่ยวกับการ sampling เลย บทเรื่อง [Temperature คืออะไร](/using-ai/temperature-and-determinism) อธิบายสาเหตุเหล่านี้อย่างละเอียด รวมถึงวิธีออกแบบระบบให้รับมือกับความไม่แน่นอนที่เหลืออยู่

## กลไกทั้งหมดในภาพเดียว

ถ้าจะสรุปทุกอย่างที่พูดถึงในบทนี้เป็นภาพเดียว กระบวนการของ LLM ตั้งแต่ข้อความที่คุณพิมพ์ไปจนถึงคำตอบที่คุณเห็น มีหน้าตาแบบนี้:

```
ข้อความที่คุณพิมพ์
    ↓ ตัดเป็น token
แบ่งเป็น token (ภาษาไทยใช้มากกว่าอังกฤษ ~2-3 เท่า)
    ↓ คำนวณโอกาสของทุก token (probability distribution)
รายการโอกาสบน vocabulary ทั้งหมด
    ↓ จั่ว 1 token (sampling)
เลือก token 1 ตัว (มีความสุ่ม — เหตุผลที่คำตอบต่างกัน)
    ↓ แสดงผล (streaming)
ส่งให้คุณเห็นทันที — ไม่รอจนจบ
    ↓ ต่อท้ายบริบท
นำ token ที่ได้ต่อท้ายข้อความ → วนกลับไปขั้นคำนวณ
    ↓ วนซ้ำ
จนเจอสัญญาณจบ (EOS) / คำห้าม / ชน budget / หมดเวลา
    ↓
คำตอบที่คุณเห็น
```

> 💡 **Key insight:** ทุกอย่างที่คุณเห็นใน output — ลำดับคำ น้ำเสียง ความถูกผิด — ล้วนเป็นผลของการตัดสินใจทีละ token โดยใช้ "ความน่าจะเป็น" ที่ได้จาก training ไม่ใช่จาก "ความรู้" ในความหมายที่มนุษย์เข้าใจ LLM ไม่ "รู้" ว่าคำตอบถูกหรือผิด มันแค่รู้ว่าคำไหน "น่าจะมาต่อ"

## กลไกนี้อธิบายปัญหาในชีวิตจริงที่คุณน่าจะเคยเจอ

ก่อนจะไปถึงบทสรุป ลองดูว่ากลไกที่เล่ามาทั้งหมดอธิบายสถานการณ์จริงที่หลายคนเจอได้อย่างไร

**สถานการณ์ที่หนึ่ง: AI ตอบถูกครั้งแรก แต่ตอบต่างครั้งที่สอง**

คุณถาม Claude ว่า "ช่วยแนะนำร้านอาหารแถวอโศกที่เปิดดึกหน่อย" ครั้งแรกได้คำตอบที่ดูดี ครั้งที่สองในบทสนทนาใหม่ได้คำแนะนำต่างกันเลย บางครั้งผิดด้วยซ้ำ

ไม่ใช่ว่า AI "เปลี่ยนใจ" หรือ "ลืม" คำตอบที่ถูก มันแค่จั่วจาก probability distribution ใหม่ทุกครั้ง ชื่อร้านอาหารหลาย ๆ ร้านมีโอกาสใกล้กัน ขึ้นกับว่า token ไหนถูกสุ่มออกมาก่อน คำตอบทั้งก้อนจะออกไปในทิศทางต่างกัน

**สถานการณ์ที่สอง: AI เขียนดีมากในตอนต้น แล้ว "เหนื่อย" ในตอนปลาย**

คุณขอให้ Claude เขียนรายงาน 10 หน้า บรรทัดแรก ๆ ดูดีมาก แต่ยิ่งยาวขึ้น ยิ่งรู้สึกว่าคุณภาพตก บางส่วนซ้ำกัน บางส่วนกลับไม่ตรงประเด็น

กลไกอธิบายได้สองทาง: หนึ่ง โมเดลไม่ได้ "เหนื่อย" แต่ทุก token ที่เพิ่มเข้าไปทำให้บริบทยาวขึ้น และยิ่งบริบทยาว ยิ่งยากที่โมเดลจะ "จำ" สิ่งที่บอกไว้ตั้งแต่ต้นได้อย่างชัดเจน สอง ยิ่งยาวยิ่ง sampling สะสม ถ้า token ตัวหนึ่งออกมาไม่ดีแล้วต่อ ๆ ไปจาก token นั้น ทิศทางก็เพี้ยนสะสม

วิธีแก้: แทนที่จะขอให้เขียนยาวทีเดียว ให้ขอทีละส่วน แล้วตรวจสอบก่อนไปส่วนถัดไป

**สถานการณ์ที่สาม: AI ตอบเร็วมากบางคำถาม ช้ามากบางคำถาม**

"ชื่อประธานาธิบดีสหรัฐคนปัจจุบัน" ตอบภายใน 1-2 วินาที แต่ "ช่วยแปล PDF นี้เป็นภาษาไทย" ใช้เวลานานมาก

ความเร็วขึ้นกับจำนวน token output เป็นหลัก ไม่ใช่ "ความยาก" ของคำถาม คำถามแรกตอบได้ใน 5-10 token คำถามหลังอาจต้องการ 1,000-5,000 token ขึ้นกับขนาด PDF ยิ่งมาก token ยิ่งใช้เวลา เพราะทุก token คือการคำนวณหนึ่งรอบ

นอกจากนี้ถ้า PDF เป็นภาษาไทย token ที่ต้องสร้างก็มากกว่าถ้าแปลเป็นภาษาไทยด้วย เพราะแต่ละคำไทยที่สร้างออกมาใช้ token มากกว่าภาษาอังกฤษ

## ทำความรู้จัก token ให้ลึกขึ้น — อะไรที่ทำให้มันพิเศษ

ก่อนจะจบบทนี้ มีเรื่องหนึ่งเกี่ยวกับ token ที่คุ้มค่ารู้ไว้ เพราะมันอธิบายพฤติกรรมแปลก ๆ ของ AI ที่คุณอาจเคยเจอ

**Token ไม่ใช่ตัวอักษรและไม่ใช่คำ**

ระบบตัดคำของ LLM (tokenizer) ถูก train ให้รู้จัก "ชิ้นส่วนยอดนิยม" ของแต่ละภาษา เหมือนโรงงานตัดผ้าที่ตัดตาม pattern ที่ขายดีที่สุด ไม่ใช่ตัดตามรูปร่างที่สมบูรณ์แบบที่สุด

ผลที่ตามมาคือสิ่งแปลก ๆ เช่น:
- emoji ตัวเดียวอาจกิน 2-4 token
- ชื่อคนไทยที่ไม่ค่อยพบในข้อมูล training เช่น "สุทธิพงษ์" อาจถูกซอยเป็นชิ้นเล็กมาก ทำให้โมเดลอ่านชื่อนี้ "ได้ยาก" กว่า "สมชาย"
- คำในภาษาอื่นที่ไม่ใช่อังกฤษหรือไทย อาจถูกตัดละเอียดมากจนแพงขึ้น

**ทำไมเรื่องนี้กระทบผลลัพธ์?**

เมื่อคำถูกซอยแบบไม่สมเหตุสมผล โมเดลต้องทำงานหนักขึ้นในการ "ประกอบ" ความหมายกลับมา ลองนึกว่าคุณกำลังอ่านหนังสือที่มีการตัดคำแบบสุ่ม เช่น "สวัส|ดี|ครั|บ" แทนที่จะเป็น "สวัสดีครับ" — คุณยังอ่านออกแต่ต้องใช้สมาธิมากขึ้น

สถานการณ์กำลังดีขึ้น tokenizer ของ Gemini และ Claude รุ่นใหม่ ๆ จัดการภาษาไทยได้ดีกว่ารุ่นเก่ามาก แต่ยังเป็นจุดอ่อนที่ควรรู้เอาไว้

**เคล็ดลับเล็ก ๆ ที่ได้ผลจริง:** ถ้าคุณสังเกตว่า AI ตีความชื่อเฉพาะภาษาไทยหรือคำเฉพาะทางผิดบ่อย ลองเขียนคำนั้นซ้ำสองครั้งหรือขึ้นต้นประโยคด้วยคำนั้น ให้โมเดลเห็น token นั้นในบริบทที่ชัดขึ้น บางครั้งช่วยได้มากกว่าที่คิด

## ความเข้าใจผิดที่พบบ่อย — แก้ให้ตรงจุด

ก่อนจะไปถึงบทสรุป มีความเข้าใจผิดสักสองสามอย่างที่เจอบ่อยมากในกลุ่มผู้ใช้ไทย และการแก้ตรงจุดนี้จะช่วยให้คุณไม่เสียเวลากับสมมติฐานที่ผิด

**"AI ตอบผิดเพราะมันไม่ฉลาดพอ"**

ไม่ตรงประเด็น ความผิดพลาดของ LLM ส่วนใหญ่ไม่ใช่เพราะ "ขาดความฉลาด" แต่เพราะ sampling เลือก token ที่ดูสมเหตุสมผลจาก pattern แต่ไม่ได้มีการตรวจสอบความถูกต้องของข้อมูลในตัวกระบวนการ โมเดลไม่รู้ว่าตัวเองผิด มันแค่เลือก token ที่น่าจะมาต่อจากบริบท ถ้า pattern ในข้อมูล training บอกว่า "คำนี้ตามหลังคำนั้นได้" มันก็เลือก — แม้ว่าข้อเท็จจริงจะผิดก็ตาม

**"ถามซ้ำหลาย ๆ ครั้งแล้วเลือกคำตอบที่ดีที่สุด"**

วิธีนี้ใช้ได้จริงในบางกรณี แต่ต้องเข้าใจว่าคำตอบที่ต่างกันไม่ได้หมายความว่ามีคำตอบ "ถูก" หนึ่งอันและ "ผิด" อีกอัน มันแค่เป็น sample ที่ต่างกันจาก distribution เดียวกัน ถ้าต้องการคำตอบที่ดีกว่า การปรับ prompt ให้ชัดขึ้นมักได้ผลมากกว่าการถามซ้ำ

**"ภาษาอังกฤษใช้ได้ดีกว่าไทยเพราะ AI ฉลาดกว่า"**

ไม่ใช่เรื่องความฉลาด แต่เป็นเรื่อง tokenization และสัดส่วนข้อมูล training ที่อธิบายไปแล้ว โมเดลใช้ "สมองเดียวกัน" กับทุกภาษา แต่ภาษาที่มีข้อมูล training เยอะกว่าและ tokenizer รับรู้ได้ดีกว่าจะให้ผลลัพธ์ดีกว่า

### ลองทำดู — เปรียบเทียบ sampling กับ greedy ด้วยตัวเอง

ทดสอบนี้จะช่วยให้คุณเห็นว่า randomness ของ LLM กระทบผลลัพธ์ยังไงในทางปฏิบัติ

เปิด ChatGPT หรือ Claude แล้วลองสองสถานการณ์นี้:

**สถานการณ์หนึ่ง — คำถามที่ตอบได้แบบเดียว:**

> "เมืองหลวงของประเทศไทยชื่อว่าอะไร"

ลองส่งซ้ำ 3 ครั้ง คุณจะพบว่าคำตอบเหมือนกันทุกครั้ง หรือเหมือนกันในส่วนที่สำคัญ นั่นเพราะ "กรุงเทพ" มีโอกาสสูงมากเป็น token แรกของคำตอบ จนแม้แต่การสุ่มก็มักออกมาตัวเดิม

**สถานการณ์สอง — คำถามที่ตอบได้หลายแบบ:**

> "เขียนประโยคเปิดสำหรับนิทานสั้นสัก 1 ประโยค"

ลองส่งซ้ำ 5 ครั้ง สังเกตว่าได้ประโยคต่างกันทุกครั้งหรือเปล่า นั่นเพราะมี token หลายตัวที่เหมาะสมเป็น token แรก ("ครั้งหนึ่ง...", "ในดินแดน...", "มีเด็กหญิง...", ฯลฯ) แต่ละตัวมีโอกาสพอ ๆ กัน

ความต่างระหว่างสองสถานการณ์คือ "ความกว้าง" ของ probability distribution — คำถามแรกมี distribution แคบ (token หนึ่งครอง) คำถามที่สองมี distribution กว้าง (token หลายตัวแข่งกัน)

---

## สามเรื่องที่เปลี่ยนวิธีใช้ AI ของคุณได้ทันที

ความเข้าใจกลไกในบทนี้ไม่ใช่แค่ความรู้สำหรับ trivia ถ้าคุณนำสามเรื่องนี้ไปใช้จริง คุณจะใช้ AI ได้ฉลาดขึ้นตั้งแต่วันนี้

**เรื่องแรก: หยุด generation เมื่อเห็นว่าออกผิดทาง**

เพราะ token ที่ commit แล้วคือ commit ตลอดกาล และโมเดลที่ "เริ่มออกนอกทาง" ตั้งแต่บรรทัดแรก มักจะออกนอกทางต่อไปเรื่อย ๆ ถ้าคุณอ่านสาม-สี่บรรทัดแรกแล้วรู้สึกว่าไม่ใช่ — กด Stop และ rewrite prompt อย่ารอให้จบแล้วค่อยบ่น

**เรื่องที่สอง: prompt เดิมได้คำตอบต่างกันคือปกติ ไม่ใช่ bug**

ถ้าคุณต้องการ output ที่สม่ำเสมอ ให้ระบุให้ชัดใน prompt ว่าต้องการอะไรแน่ ๆ แทนที่จะหวังว่าโมเดลจะ "จำ" ว่าคุณชอบอะไร เพราะมันไม่จำ มันแค่จั่วใหม่ทุกครั้ง

**เรื่องที่สาม: ภาษาไทยกิน context เร็ว — บริหาร prompt ให้กระชับ**

ถ้าคุณต้องส่งเอกสารยาวให้ AI ทำงาน ลองสรุปบางส่วนก่อนแทนที่จะ copy ทั้งหมดมา พื้นที่รับข้อความมีขีดจำกัด และภาษาไทยหมดเร็วกว่าที่คิด

### ลองทำดู — ฝึกอ่านคำตอบ AI แบบ active

เปิด ChatGPT หรือ Claude แล้วลองทำสองอย่างพร้อมกัน:

**ทดสอบที่หนึ่ง — ฝึกหยุดกลางคัน:**

พิมพ์คำขอที่น่าจะได้คำตอบยาว เช่น:

> "อธิบายประวัติศาสตร์กรุงเทพให้ละเอียดตั้งแต่ก่อตั้งจนถึงปัจจุบัน"

อ่านสามบรรทัดแรก แล้วถามตัวเองว่า "ทิศทางถูกต้องไหม?" ถ้าใช่ ปล่อยให้จบ ถ้าไม่ใช่ กด Stop แล้ว rewrite ทันที ฝึกนิสัยนี้สักสองสามครั้ง

**ทดสอบที่สอง — ลองดูว่า prompt ต่างกันเท่าไหร่ถ้าใช้ภาษาต่างกัน:**

ลองส่ง prompt เดียวกันสองภาษาในสองช่องสนทนาใหม่ เช่น:

> ภาษาไทย: "สรุปวิธีทำงานของ ChatGPT ให้ฟังหน่อย"
> ภาษาอังกฤษ: "Summarize how ChatGPT works"

สังเกตความยาวของคำตอบ และสังเกตว่าตัวอักษรเริ่มปรากฏเร็วกว่ากันในแบบไหน ความต่างที่เห็นส่วนหนึ่งมาจาก token count ที่ต่างกัน

## สรุป — สิ่งที่ควรติดอยู่ในหัว

LLM สร้างคำตอบโดยทำสิ่งเดียวซ้ำ ๆ: ดูบริบททั้งหมดที่มี คำนวณว่า token ถัดไปน่าจะเป็นอะไร จั่วจาก probability distribution นั้น ส่งให้คุณเห็น แล้ววนใหม่

จากกลไกนี้ได้ข้อสรุปสามอย่างที่สำคัญ:

หนึ่ง LLM ตัดสินใจ **ทีละ token** ไม่ได้วางแผนทั้งก้อน ทุก token ที่คุณเห็นคือผลของการคำนวณหนึ่งรอบที่ดูบริบททั้งหมดแล้วจั่วจาก probability distribution

สอง **randomness อยู่ใน design** ไม่ใช่ bug sampling ทำให้คำตอบฟังดูเป็นธรรมชาติและหลากหลาย แต่ก็แปลว่าคุณเอา LLM ไปใช้กับงานที่ต้องการความ deterministic แบบ 100% ตรง ๆ ไม่ได้โดยไม่มีการออกแบบเพิ่มเติม

สาม **ภาษาไทยใช้ token มากกว่า** ซึ่งแปลว่าแพงกว่า ช้ากว่า และพื้นที่รับข้อความหมดเร็วกว่าเมื่อเทียบกับภาษาอังกฤษในเนื้อหาที่เทียบเท่ากัน

สิ่งที่น่าทึ่งคือกลไกเดียวกันนี้อธิบายทั้งจุดแข็งและจุดอ่อนของ LLM ได้พร้อมกัน ความสามารถในการตอบได้หลายแบบ เขียนได้ทั้งเชิงสร้างสรรค์และเชิงวิเคราะห์ เกิดจาก sampling ที่มี randomness ในตัว แต่ความไม่แม่นยำในข้อเท็จจริงก็เกิดจากกลไกเดียวกัน — มันเลือก token ที่ "น่าจะมา" โดยไม่ตรวจสอบว่าถูกต้องหรือไม่

เมื่อเข้าใจตรงนี้แล้ว คุณก็เข้าใจแล้วว่า LLM เหมาะกับงานไหนและไม่เหมาะกับงานไหน ไม่ใช่เพราะ "ฉลาดหรือโง่" แต่เพราะกลไกออกแบบมาแบบนี้

ทั้งหมดนี้นำไปสู่คำถามที่สำคัญถัดไป: ถ้า LLM ตัดสินใจทีละ token โดยดูบริบทที่มีอยู่ แล้วคุณควรจัดบริบทนั้นยังไงเพื่อให้ได้คำตอบที่ต้องการ? นั่นคือเรื่องของ prompt engineering — สิ่งที่เราจะพูดถึงในบทถัดไป เมื่อเข้าใจกลไกในบทนี้แล้ว คุณจะเข้าใจว่าทำไม prompt บางแบบได้ผลกว่าอีกแบบ ไม่ใช่เพราะ "เคล็ดลับ" แต่เพราะมันสอดคล้องกับวิธีที่โมเดลทำงาน
