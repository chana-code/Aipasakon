# Context Window คืออะไร — ทำไม AI ถึงลืมต้นบทสนทนา

คุณเคยคุยกับ ChatGPT ยาว ๆ แล้วรู้สึกว่ามัน "ลืม" สิ่งที่บอกไปตอนต้นไหม. บอกว่าชื่ออะไร พอคุยไปอีกหลายรอบ มันเรียกชื่อผิด. บอกบริบทโปรเจกต์ไว้ พอถามทีหลัง มันไม่รู้แล้ว. บอกว่าอย่าตอบเป็นภาษาอังกฤษ พอถึง turn ที่ 30 มันเริ่มสลับภาษาอีกครั้ง.

นี่ไม่ใช่ bug. ไม่ใช่ความตั้งใจของ AI ที่จะเพิกเฉย. และไม่ใช่เพราะ AI โง่ลง.

มันเป็นกลไกพื้นฐานของวิธีที่ LLM ทำงาน — กลไกที่ชื่อว่า **context window**. และเมื่อคุณเข้าใจมัน คุณจะไม่แค่รู้ว่าทำไมมันเกิดขึ้น แต่จะรู้ทันทีว่าต้องทำอะไรเพื่อหลีกเลี่ยงปัญหา.

## กระดานทำงานที่มีขนาดจำกัด

ลองนึกภาพว่า LLM ทำงานบนกระดานไวท์บอร์ดขนาดจำกัด. ทุกอย่างที่โมเดลต้อง "รู้" ในการตอบคำถามหนึ่งครั้ง ต้องถูกเขียนลงบนกระดานนั้นก่อน — system prompt, ประวัติการคุยทั้งหมด, เอกสารที่แนบมา, คำถามที่คุณเพิ่งส่งไป และยังต้องเหลือพื้นที่สำหรับคำตอบที่จะเขียนออกมาอีกด้วย.

ปัญหาคือกระดานนี้มีขนาดจำกัด.

พื้นที่บนกระดานนั้นเรียกว่า **context window** — มันคือปริมาณข้อมูลสูงสุดที่ LLM รับเข้าและประมวลผลได้ในการตอบหนึ่งครั้ง. ขนาดวัดด้วยหน่วย **token** — ชิ้นของข้อความที่โมเดลมองเห็น ประมาณ 1 token ต่อ 1 คำในภาษาอังกฤษ หรือ 1-2 ตัวอักษรในภาษาไทย.

ถ้ายังไม่คุ้นกับคำว่า token สามารถย้อนอ่านบท LLM คืออะไร ที่อธิบายเรื่องนี้ตั้งแต่ต้น ก่อนมาอ่านบทนี้จะเข้าใจได้ชัดขึ้น.

**ตัวเลข context window** ในปี 2026 ของโมเดลหลัก ๆ:

| โมเดล | Context Window |
|-------|---------------|
| Claude (Anthropic) | 200,000 token เป็นมาตรฐาน |
| GPT-4 (OpenAI) | ประมาณ 128,000 token |
| Gemini (Google) | 1,000,000 token เป็นมาตรฐาน บางรุ่นทดลอง 2,000,000 token |

ตัวเลขพวกนี้ฟังดูใหญ่มาก. 200,000 token แปลคร่าว ๆ ได้ว่าประมาณหนังสือเล่มหนาหนึ่งเล่ม. ฟังดูเหมือนไม่มีวันเต็ม.

แต่ในการใช้งานจริง พื้นที่พวกนี้หมดเร็วกว่าที่คิด — และยิ่งใช้ภาษาไทย ก็ยิ่งหมดเร็วขึ้นไปอีก เหตุผลเรื่องนี้จะอธิบายในหัวข้อถัดไป.

> 💡 **Key insight:** context window ไม่ใช่ "ความจำระยะยาว" ของ AI. มันคือ "พื้นที่ทำงานชั่วคราว" ที่รีเซตใหม่ทุกครั้งที่เริ่มแชตใหม่. AI ไม่ได้จำบทสนทนาเก่าจริง ๆ — แอปแชตต่างหากที่ feed ประวัติซ้ำทุกครั้งที่คุณส่งข้อความ. โมเดลไม่มีแนวคิดว่า "คุยกันมาแล้ว 2 ชั่วโมง" — ทุกครั้งมันเห็นข้อความก้อนใหญ่ก้อนหนึ่งแล้วตอบ.

## ทำไม AI ถึง "ลืม" — กลไกจริงที่อยู่เบื้องหลัง

นี่คือสิ่งที่เกิดขึ้นจริงทุกครั้งที่คุณส่งข้อความใน ChatGPT หรือ Claude:

**ขั้นที่ 1:** แอปรวบรวมประวัติการคุยทั้งหมดของแชตนั้น ตั้งแต่ turn แรกจนถึงปัจจุบัน

**ขั้นที่ 2:** นำมารวมกับ system prompt และเอกสารที่แนบมา

**ขั้นที่ 3:** ส่งทั้งหมดเป็น input ให้ LLM — ทุกครั้ง ไม่มีข้อยกเว้น

**ขั้นที่ 4:** LLM อ่าน "เหมือนเห็นครั้งแรก" แล้วสร้างคำตอบ

**ขั้นที่ 5:** แอปบันทึก turn นั้นไว้สำหรับ turn ถัดไป วนซ้ำ

ขั้นตอนนี้วนซ้ำทุกครั้งที่คุณส่งข้อความ. โมเดลไม่ได้มีความจำระหว่างรอบ — มันไม่มีแม้แต่แนวคิดว่า "ฉันคุยกับคนนี้มาแล้วหลายชั่วโมง". ทุกครั้งที่มัน "ตื่น" ขึ้นมา มันเห็นข้อความก้อนใหญ่ก้อนหนึ่ง และมีหน้าที่แค่ตอบสิ่งที่อยู่ตรงปลาย.

ปัญหาเกิดขึ้นเมื่อบทสนทนายาวขึ้นเรื่อย ๆ. ประวัติที่ต้องส่งซ้ำทุก turn ใหญ่ขึ้นเรื่อย ๆ. ถึงจุดหนึ่ง มันเริ่มชนขอบ context window.

เมื่อนั้น แอปต้องเลือก: ตัดต้นบทสนทนาทิ้ง หรือสรุปให้สั้นลง. ไม่ว่าจะเลือกทางไหน ข้อมูลที่บอกไว้ตอนต้นก็หายไปจากกระดาน.

ผลลัพธ์คือ AI เริ่ม "ลืม" — ไม่ใช่เพราะมันโง่ลง แต่เพราะกระดานทำงานเต็มแล้ว.

> 💡 **Key insight:** แอปสมัยใหม่อย่าง ChatGPT และ Claude มี "Memory feature" ที่ช่วยจำข้อมูลข้ามแชตได้. แต่นี่คือระบบแยกต่างหากที่เลือกบางส่วนจาก external storage มาใส่กลับใน context ใหม่ — ไม่ใช่ความจำของโมเดล. ปิดแชตแล้วเปิดใหม่โดยไม่มี memory feature = เริ่มจากศูนย์ทุกครั้ง.

สิ่งที่ได้เรียนรู้จากการเข้าใจกลไกนี้:

- ถ้าอยากให้ AI "จำ" ข้อมูลสำคัญตลอดบทสนทนา ให้ใส่ไว้ใน system prompt ตั้งแต่ต้น ไม่ใช่แค่พูดถึงครั้งเดียวใน turn แรก
- ถ้าบทสนทนายาวเกินไปจนรู้สึกว่า AI เริ่มหลงทาง การเริ่มแชตใหม่และสรุปบริบทใหม่อีกครั้งมักดีกว่าการคุยต่อในแชตเดิม
- ถ้าใช้ AI ทำงานที่ต้องการข้อมูลเยอะ ควรคิดก่อนว่าข้อมูลชิ้นไหนสำคัญจริง ๆ และต้องอยู่ใน context

## ภาษาไทยกิน token เยอะกว่าที่คิด

ภาษาไทยใช้ token มากกว่าภาษาอังกฤษประมาณ 2-3 เท่า ซึ่งส่งผลต่อทั้งค่าใช้จ่ายและคุณภาพ (ดูรายละเอียดและตารางเปรียบเทียบในบท [LLM ตอบคำถามยังไง](/using-ai/how-llms-respond))

ผลในทางปฏิบัติ: เอกสารภาษาไทย 80 หน้าอาจกินเกือบทั้ง context window 200,000 token ของ Claude หมดเลย — คำนวณก่อนส่ง.

### ลองทำดู — ดูตัวเลข token ด้วยตัวเอง

เปิด [platform.openai.com/tokenizer](https://platform.openai.com/tokenizer) ใน browser. ไม่ต้องสมัครสมาชิก เปิดได้เลย.

**ทดลองที่ 1 — เทียบภาษา:**

พิมพ์สองประโยคนี้ทีละอัน แล้วดูจำนวน token ที่แสดง:
1. `"วันนี้อากาศร้อนมาก ฉันอยากกินไอศกรีม"`
2. `"Today is very hot. I want to eat ice cream."`

ความต่างจะเห็นชัด — ประโยคไทยที่ความหมายเดียวกันจะใช้ token มากกว่า 2-3 เท่า.

**ทดลองที่ 2 — เอกสารจริงของคุณ:**

เอาอีเมลงาน 1 ฉบับหรือ LINE message ที่ยาวหน่อยมาวาง. ดูว่ากิน token ไปเท่าไหร่. แล้วลองเปรียบกับ context window ที่โมเดลที่คุณใช้อยู่รองรับ — คุณจะเริ่มรู้สึกว่า context window "จริง ๆ" นั้นใหญ่แค่ไหนสำหรับงานของคุณ.

## ทำไมถึงต้องมีขีดจำกัด — เรื่องของคณิตศาสตร์

Context window ไม่ใช่การตัดสินใจทางธุรกิจที่ว่า "จะให้ผู้ใช้ได้แค่นี้พอ". มันเป็นข้อจำกัดทางวิศวกรรมจริง ๆ ที่มีรากมาจากวิธีทำงานของ LLM.

LLM ทุกตัวในปี 2026 ใช้สถาปัตยกรรมที่เรียกว่า **transformer** ซึ่งมีกลไกสำคัญที่ชื่อว่า **self-attention**. กลไกนี้ทำสิ่งที่น่าทึ่ง: มันให้ทุก token ใน context "มอง" ทุก token อื่นก่อนหน้าตัวเอง เพื่อคำนวณว่าควรให้ความสนใจกับส่วนไหนมากกว่ากัน. นี่คือสิ่งที่ทำให้โมเดลเข้าใจ "ความสัมพันธ์ระหว่างคำ" ได้ดี — รู้ว่า "มัน" ในประโยคหนึ่งหมายถึงอะไร แม้คำที่มัน refer ถึงจะอยู่ห่างออกไป 500 คำ.

แต่มีราคาที่ต้องจ่าย.

ถ้า context มีความยาว n token แต่ละ token ต้องคำนวณความสัมพันธ์กับ token ทั้งหมดที่อยู่ก่อนหน้า. ผลรวมคือ n × n = n² ครั้ง. ในทางคณิตศาสตร์ เรียกว่า complexity แบบ **O(n²)** — แปลว่าถ้า context ยาวขึ้น 2 เท่า งานที่ต้องทำเพิ่มขึ้น 4 เท่า.

ลองดูตัวเลขจริง:

| Context เพิ่มจาก | ยาวขึ้น | Compute เพิ่ม |
|----------------|---------|--------------|
| 4K → 8K token | 2 เท่า | 4 เท่า |
| 4K → 32K token | 8 เท่า | 64 เท่า |
| 4K → 128K token | 32 เท่า | 1,024 เท่า |
| 4K → 1M token | 250 เท่า | 62,500 เท่า |

ความจำ (memory) ที่ต้องใช้ก็เพิ่มในระดับเดียวกัน เพราะต้องเก็บตารางความสัมพันธ์ขนาด n × n ไว้ตลอด. ใน hardware จริง นั่นหมายถึง VRAM ในการ์ดจอของ GPU ที่ต้องใช้มากขึ้นอย่างมีนัยสำคัญ.

นี่คือสาเหตุที่การขยาย context window ไม่ใช่แค่ "กดปุ่มเพิ่ม" — มันต้องการนวัตกรรมทางวิศวกรรมจริง ๆ. โมเดลที่รองรับ context หลักล้าน token ใช้เทคนิคหลายอย่างเพื่อลด complexity ลง เช่น sparse attention (ไม่ต้องมองทุก token เท่ากัน) หรือ sliding window attention (มองแค่ช่วงใกล้ ๆ). รายละเอียดเป็นระดับวิศวกร แต่สิ่งที่ผู้ใช้ควรเข้าใจคือ: **context ที่ใหญ่ขึ้นไม่ได้ฟรี — มีราคาทั้งในแง่ความเร็วและต้นทุน**.

> 💡 **Key insight:** ตัวเลขขนาด context window บนหน้าสินค้าอย่าง "1M token" คือ "เพดานทางวิศวกรรม" — ไม่ใช่การรับประกันว่าโมเดลจะใช้ทุก token ได้คุณภาพเท่ากัน. ในความเป็นจริง โมเดลส่วนใหญ่เริ่ม degrade เมื่อ context ยาวขึ้น โดยเฉพาะตรงกลาง. เรื่องนี้จะอธิบายในหัวข้อถัดไป.

## ข้อมูลตรงกลางหายไปไหน — Lost in the Middle

นี่คือสิ่งที่งานวิจัยค้นพบและเปลี่ยนวิธีที่นักพัฒนา AI ออกแบบ prompt:

แม้โมเดลจะ "อ่าน" context ทั้งหมด แต่มันไม่ได้ใช้ทุกส่วนอย่างเท่าเทียมกัน.

เมื่อให้ LLM ค้นหาข้อมูลเฉพาะในข้อความยาว ๆ งานวิจัย (Liu et al., 2023) พบว่าความแม่นยำสูงสุดเมื่อข้อมูลนั้นอยู่ที่ **ต้น** หรือ **ท้าย** ของ context. แต่ถ้าข้อมูลอยู่ **ตรงกลาง** — ความแม่นยำตกลงอย่างมีนัยสำคัญ. รูปแบบนี้เห็นซ้ำ ๆ ในหลายโมเดล นักวิจัยเรียกมันว่า **"lost in the middle"**.

กราฟประสิทธิภาพเป็นรูปตัว U — สูงที่สองปลาย ต่ำตรงกลาง.

ทำไมถึงเป็นแบบนี้? มีคำอธิบายระดับเทคนิคหลายอย่าง แต่ในระดับที่ผู้ใช้ควรเข้าใจ: attention ไม่ได้แจกความสนใจอย่างเท่าเทียมกันทุก token. ตำแหน่งต้นและท้ายได้ attention มากกว่าตรงกลาง เป็นเหมือนหน่วยความจำชั้นแรกที่ถูกเข้าถึงก่อน.

ผลในทางปฏิบัติของคุณตรงนี้:

**คำสั่งสำคัญไม่ควรอยู่ตรงกลาง prompt**

**Layout ที่มีปัญหา:**
```
[เอกสาร 1: 5,000 token]
[เอกสาร 2: 5,000 token]
[คำสั่งสำคัญ: "ตอบเป็น bullet 3 ข้อ"]  ← อยู่กลาง = เสี่ยงถูกมองข้าม
[เอกสาร 3: 5,000 token]
[เอกสาร 4: 5,000 token]
[คำถาม: "วิเคราะห์เอกสารทั้งหมด"]
```

คำสั่ง "ตอบเป็น bullet 3 ข้อ" อยู่ตรงกลางระหว่างเอกสาร 4 ฉบับ — โมเดลมีโอกาสสูงที่จะมองข้าม และตอบในรูปแบบอื่นที่ไม่ใช่ bullet.

**Layout ที่ดีกว่า:**
```
[คำสั่งสำคัญ: "ตอบเป็น bullet 3 ข้อ จากเอกสารทั้งหมด"]  ← ต้น
[เอกสาร 1: 5,000 token]
[เอกสาร 2: 5,000 token]
[เอกสาร 3: 5,000 token]
[เอกสาร 4: 5,000 token]
[คำถาม: "วิเคราะห์และตอบเป็น bullet 3 ข้อ" ← ทวนคำสั่งที่ท้าย]
```

คำสั่งอยู่ที่ต้นและท้าย — สองตำแหน่งที่โมเดลให้ความสนใจดีที่สุด.

กฎง่าย ๆ ที่จำได้: ถ้ามีคำสั่งหรือข้อมูลที่สำคัญมาก อย่าให้มันถูกฝังอยู่กลางกองเอกสาร — ใส่ที่ต้นก่อน แล้วทวนซ้ำที่ท้าย.

> 💡 **Key insight:** ตัวเลข "1M token context window" บนหน้าโฆษณาไม่ได้รับประกันว่าโมเดลจะ "อ่าน" ทุกส่วนอย่างเท่าเทียม. ในงานจริง ทดสอบด้วย workload ของคุณเองก่อนเชื่อตัวเลข.

### ลองทำดู — ทดสอบ Lost in the Middle ด้วยตัวเอง

เปิด Claude หรือ ChatGPT แล้วทดสอบแบบนี้:

**ขั้นที่ 1:** สร้างข้อมูลหลายชุด — เช่น รายการสินค้าใน catalog ที่มี 4 หมวด แต่ละหมวดมีสินค้า 8-10 รายการ

**ขั้นที่ 2:** ซ่อนข้อมูลพิเศษไว้ตรงกลาง — เช่น ในหมวดที่ 2 ใส่ว่า "**[โปรโมชันพิเศษ: ซื้อ 1 แถม 1 ทุกรายการ]**" แล้วต่อด้วยรายการสินค้าปกติ

**ขั้นที่ 3:** ถามที่ท้าย prompt ว่า "มีโปรโมชันอะไรบ้างในรายการสินค้า?"

สังเกตว่าโมเดลจับข้อมูลที่อยู่กลางได้หรือเปล่า. แล้วลองสลับ — ย้ายข้อมูลโปรโมชันไปไว้ที่ต้นสุดหรือท้ายสุดของ prompt แล้วถามซ้ำ. ความแม่นยำจะต่างกันอย่างชัดเจน.

## Input กับ Output เป็นคนละงบประมาณ

นี่คือความเข้าใจผิดที่พบบ่อยมาก:

"Claude รองรับ 200,000 token แปลว่ามันตอบได้ยาว 200,000 token"

ไม่ใช่.

**Input และ output มีงบประมาณแยกจากกัน**:

- **Input limit** — ทุกอย่างที่ส่งเข้าไป: system prompt + ประวัติแชต + เอกสาร + คำถามของคุณ รวมกันต้องไม่เกินขนาด context window (เช่น 200,000 token)
- **Output limit** — คำตอบที่โมเดลสร้างออกมา มักจำกัดที่ประมาณ 4,000-8,000 token ต่อการตอบหนึ่งครั้ง — แม้โมเดลนั้นรองรับ context ใหญ่กว่ามากก็ตาม

ทำไมถึงจำกัด output ไว้แค่นั้น? มีเหตุผลจริง ๆ:

**เหตุผลที่ 1 — คุณภาพตกเมื่อยาวเกิน:** การสร้างข้อความยาวมาก ๆ ยิ่งยาวยิ่ง drift ออกจากคำสั่งเดิม — โมเดลเริ่มลืมว่ากำลังทำอะไรอยู่ คุณภาพตกลงเรื่อย ๆ

**เหตุผลที่ 2 — เวลาเพิ่มเป็นเส้นตรง:** การสร้าง output เป็น autoregressive — สร้างทีละ token ทีละ token. output 8,000 token ใช้เวลานานกว่า 4,000 token สองเท่า. 50,000 token นานมากจนผู้ใช้ส่วนใหญ่ไม่รอ

**เหตุผลที่ 3 — ผู้ใช้ส่วนใหญ่ไม่ต้องการจริง ๆ:** การตอบ 50,000 token ในครั้งเดียวเป็นกรณีพิเศษมาก ไม่ใช่การใช้งานปกติ

**ผลในทางปฏิบัติ:** ถ้าต้องการให้ AI เขียนเอกสารยาว ๆ เช่น รายงาน 20 หน้า คุณต้องแบ่งงานเป็นส่วน ๆ ขอให้เขียนทีละบท แล้วนำมาประกอบเอง. ไม่สามารถขอครั้งเดียวแล้วได้คำตอบ 20,000 token ออกมาได้.

เทคนิคที่ใช้กันมากในกรณีนี้เรียกว่า **chunked generation** — แบ่งงานเขียนออกเป็น chapter หรือ section ขอทีละส่วน แล้วนำมารวมเอง. ดูแนวทางในบทเรื่อง prompt engineering.

## บทสนทนายาวก็แพงขึ้นด้วย

เรื่องนี้เกี่ยวกับคนที่ใช้ AI ผ่าน API โดยตรง เช่น สร้างแอปของตัวเอง หรือเรียกใช้ผ่านโค้ด. ถ้าคุณใช้แค่หน้าเว็บ ChatGPT หรือ Claude.ai สามารถข้ามไปหัวข้อถัดไปได้เลย — OpenAI และ Anthropic รับค่าใช้จ่ายส่วนนี้ไปแล้ว.

แต่ถ้าคุณสร้างแอปหรือใช้ API อยู่ ตัวเลขนี้สำคัญมาก:

API คิดเงินตาม token ที่ส่งเข้าไปและรับออกมา. ฟังดูตรงไปตรงมา แต่ผลรวมน่าตกใจ:

สมมติแต่ละ turn ผู้ใช้ส่ง 100 token และ AI ตอบ 300 token:

| Turn | Input ที่ส่งจริง | ทำไม |
|------|----------------|------|
| Turn 1 | 100 token | แค่ข้อความแรก |
| Turn 2 | 500 token | 100 (ใหม่) + 400 (turn 1 ทั้งหมด) |
| Turn 10 | ~3,700 token | 100 + 9 turns × 400 |
| Turn 50 | ~20,000 token | 100 + 49 turns × 400 |

**Turn ที่ 50 แพงกว่า Turn ที่ 1 ประมาณ 200 เท่า** — ทั้งที่ผู้ใช้พิมพ์แค่ 100 token เท่าเดิมทุก turn.

เหตุผลคือทุก turn ต้อง re-send ประวัติทั้งหมด. แอปแชตต้องการให้โมเดลรู้บริบทก่อนหน้าทั้งหมด เลยต้องส่งซ้ำทุกครั้ง.

นี่คือสาเหตุที่นักพัฒนา AI จริงจังกับเรื่อง **prompt caching** — เทคนิคที่ cache ส่วนที่ไม่เปลี่ยนแปลง (เช่น system prompt ยาว ๆ) เพื่อไม่ต้องคิดเงินซ้ำ. รายละเอียดอยู่ในบท cost and latency engineering ในระดับ Level 3.

## เมื่อเอกสารใหญ่กว่า Context — 4 กลยุทธ์

บางงานมีข้อมูลที่ใหญ่กว่า context window จะรับได้ — เอกสารกฎหมาย 500 หน้า, database ข้อมูลลูกค้า, คู่มือสินค้าที่มีหลายพันหน้า. ในกรณีพวกนี้ มีกลยุทธ์หลัก 4 แบบ แต่ละแบบมีข้อแลกเปลี่ยนต่างกัน:

### กลยุทธ์ที่ 1 — Summarization: สรุปแล้วเก็บสรุป

สรุปส่วนเก่าของ context ให้สั้นลง แล้วเก็บสรุปแทน raw text. แอปแชตส่วนใหญ่ทำแบบนี้อัตโนมัติเมื่อ context ใกล้เต็ม.

- **ข้อดี:** ง่าย ไม่ต้องออกแบบระบบเพิ่ม — แอปหลายตัวทำให้อัตโนมัติ
- **ข้อเสีย:** เสียรายละเอียด — ข้อมูลที่ถูกสรุปออกไปอาจสำคัญกว่าที่คิด ถ้า AI เลือกสรุปผิดจุด ข้อมูลนั้นหายไปตลอด

เหมาะกับ: งานที่ต้องการ context กว้าง ๆ แต่ไม่ต้องการ detail สูง

### กลยุทธ์ที่ 2 — Chunking: แบ่งแล้วประมวลทีละส่วน

แบ่งเอกสารยาวเป็นชิ้นเล็ก ๆ ประมวลผลทีละชิ้น แล้วรวมผลลัพธ์เอง.

- **ข้อดี:** ทำให้จัดการเอกสารยาวมากได้ โดยไม่ต้องใส่ทั้งหมดพร้อมกัน
- **ข้อเสีย:** ถ้าคำตอบต้องอาศัยความสัมพันธ์ระหว่างหลายส่วนในเอกสาร การแบ่ง chunk ทำให้พลาดได้ — AI เห็นแค่ชิ้นที่กำลังดูอยู่ ไม่เห็นภาพรวม

เหมาะกับ: งาน summarize หรือ extract ที่แต่ละส่วนของเอกสารสามารถประมวลผลแยกกันได้

### กลยุทธ์ที่ 3 — Retrieval (RAG): ดึงเฉพาะส่วนที่เกี่ยวข้อง

เก็บเอกสารใน database พิเศษ. เมื่อมีคำถาม ระบบค้นหาเฉพาะส่วนที่เกี่ยวข้องกับคำถามนั้น แล้วใส่เฉพาะส่วนนั้นเข้าใน context.

- **ข้อดี:** จัดการเอกสารขนาดใหญ่มากได้ ประหยัด token อย่างมาก — context ของแต่ละคำถามเล็กและคมขึ้น
- **ข้อเสีย:** ต้องออกแบบระบบค้นหาที่ดี — ถ้าค้นหาผิด ก็ใส่ข้อมูลผิดเข้าไปใน context และ AI ตอบผิด

นี่คือกลยุทธ์ที่ใช้กันมากที่สุดในระดับองค์กร ชื่อเต็มคือ **Retrieval-Augmented Generation (RAG)** — สถาปัตยกรรมและรายละเอียดการสร้างจริง เจาะลึกไว้ใน Level 3 บท [RAG (Retrieval-Augmented Generation)](/building-with-ai/rag-retrieval-augmented-generation).

### กลยุทธ์ที่ 4 — Incremental Processing: ประมวลทีละหน่วยแล้วส่งสรุปต่อ

ประมวลผลข้อมูลทีละชิ้น ส่ง "สถานะที่บีบอัดแล้ว" ต่อไปยังรอบถัดไป. เหมาะกับงานวิเคราะห์ข้อมูล log หรือ stream ข้อมูลต่อเนื่อง.

- **ข้อดี:** จัดการ stream ข้อมูลได้ ไม่ต้องรอให้ข้อมูลทั้งหมดพร้อมก่อน
- **ข้อเสีย:** ซับซ้อน ต้องออกแบบระบบ — ไม่เหมาะกับผู้ใช้ทั่วไป

ตารางเปรียบเทียบอย่างย่อ:

| กลยุทธ์ | ความซับซ้อน | เหมาะกับ |
|---------|------------|---------|
| Summarization | ต่ำ | บทสนทนายาว |
| Chunking | กลาง | สรุปเอกสาร |
| RAG | สูง | ถาม-ตอบจากเอกสารใหญ่ |
| Incremental | สูงมาก | วิเคราะห์ data stream |

> 💡 **Key insight:** Context window ที่ใหญ่ขึ้นไม่ได้ทำให้กลยุทธ์เหล่านี้หมดความจำเป็น — เพราะ lost in the middle, output budget, และ cost per token ยังอยู่. การคิดเรื่องการจัดข้อมูลใน prompt ยังสำคัญไม่ว่า context window จะใหญ่แค่ไหน.

## อย่าเชื่อตัวเลข Context บน Datasheet

เวลาบริษัท AI ประกาศว่าโมเดลรองรับ "1M context window" วิธีที่พวกเขาทดสอบส่วนใหญ่เรียกว่า **needle in a haystack (NIAH)** — ฝังประโยคหนึ่งประโยคไว้ในข้อความยาว ๆ แล้วถามว่าประโยคนั้นบอกอะไร. ถ้าโมเดลหาเจอ ถือว่า "ใช้ context ยาวได้".

วิธีทดสอบนี้ฟังดูสมเหตุสมผล แต่มีข้อจำกัดสำคัญ:

**ข้อจำกัดที่ 1:** ทดสอบแค่ "ดึงข้อมูลหนึ่งชิ้น" — ไม่ใช่ "เข้าใจความสัมพันธ์ระหว่างหลายส่วน". งานจริงมักต้องการอย่างหลัง.

**ข้อจำกัดที่ 2:** ประโยคที่ฝังเข้าไปมักแปลกและโดดเด่นออกจาก context รอบ ๆ ทำให้โมเดล detect ได้ง่ายกว่างานจริงที่ข้อมูลสำคัญกลมกลืนกับเนื้อหาอื่น.

**ข้อจำกัดที่ 3:** ไม่ทดสอบเมื่อต้องโยง needle หลายอัน เช่น "จาก paragraph ที่ 3 และ paragraph ที่ 47 ข้อสรุปคืออะไร".

งานวิจัยที่ใช้ benchmark ที่ยากกว่า เช่น RULER, InfiniteBench หรือ LongBench ซึ่งทดสอบ reasoning ข้าม context ยาว ๆ พบว่าโมเดลที่อ้างรองรับ 1M token มักเริ่ม degrade อย่างมีนัยสำคัญที่ context เกิน 100,000 token ในงานจริง.

โมเดล "1M context" อาจตอบคำถามจากเอกสาร 100 หน้าได้ดี แต่ถ้าเอกสารยาว 500 หน้า ประสิทธิภาพอาจตกลงมากกว่าที่คิด.

**ข้อสรุปเชิงปฏิบัติ:** ก่อนเชื่อว่าโมเดลจะจัดการ context ยาวของคุณได้ดี ให้ทดสอบกับเนื้อหาจริงของงานคุณเองก่อน. อย่าเชื่อแค่ตัวเลขบนหน้าโปรโมต.

## ความเข้าใจผิดที่พบบ่อยที่สุด

**"AI จำทุกอย่างที่เคยบอกไปแล้ว"**

ไม่ใช่. LLM ไม่มี state ระหว่าง inference. แอปแชต re-send ประวัติทั้งหมดทุก turn — และถ้า context เต็ม ประวัติตอนต้นถูกตัดทิ้ง. ปิด session = เริ่มจากศูนย์.

**"Context window 1M token แปลว่าโมเดลใช้ทั้ง 1M ได้ดีเท่ากัน"**

ไม่ใช่. Lost in the middle ยังอยู่. ตัวเลขคือเพดาน ไม่ใช่การรับประกัน.

**"Output ไม่จำกัด ถ้า context window ใหญ่"**

ไม่ใช่. Input และ output มี budget แยก. โมเดล 200K context มักจำกัด output ที่ 4K-8K token ต่อรอบ.

**"ภาษาไทยใช้ token เท่า ๆ กับภาษาอังกฤษ"**

ไม่ใช่. ภาษาไทยกิน 2-3 เท่า. เอกสาร 80 หน้าภาษาไทยอาจกิน context window ของ Claude หมดเลย.

**"Context window ใหญ่ขึ้นแล้ว ไม่ต้องคิดเรื่อง RAG แล้ว"**

ไม่ใช่. RAG ยังมีประโยชน์เพราะ lost in the middle + cost per token + latency ยังอยู่. Context window ใหญ่และ RAG ทำงานเสริมกัน ไม่ใช่แทนกัน.

---

## สรุป — สิ่งที่คุณควรเอาไปใช้วันนี้

Context window เป็นแนวคิดที่เปลี่ยนวิธีคิดในการใช้ AI ทันทีที่เข้าใจมัน.

**Context window คือพื้นที่ทำงานชั่วคราว ไม่ใช่ความจำระยะยาว.** ทุกอย่างที่ AI ต้อง "รู้" ในการตอบครั้งนั้น ต้องอยู่บนกระดาน — system prompt, ประวัติแชต, เอกสาร, คำถาม และพื้นที่สำหรับคำตอบ. ถ้ากระดานเต็ม ข้อมูลตกหาย.

**ภาษาไทยกิน token 2-3 เท่าของภาษาอังกฤษ.** เอกสาร 80 หน้าอาจกิน context window หมดเลย. คำนวณก่อนส่ง.

**AI ไม่ได้ใช้ทุกส่วนของ context อย่างเท่าเทียมกัน.** คำสั่งสำคัญและข้อมูล key ควรอยู่ที่ต้นหรือท้าย ไม่ใช่กลาง.

**Input และ output มีงบแยก.** โมเดล 200,000 token context ไม่ได้แปลว่าตอบได้ยาว 200,000 token — output ถูกจำกัดที่ 4,000-8,000 token ต่อรอบ. งานเขียนยาวต้องแบ่งทีละส่วน.

**ตัวเลขบน datasheet คือเพดาน ไม่ใช่การรับประกัน.** ทดสอบกับงานจริงของคุณเสมอ.

เมื่อคุณเข้าใจว่า AI ทำงานบนพื้นที่จำกัดแบบนี้ การ prompt ก็เปลี่ยนไป — คุณจะเริ่มคิดถึงว่า "อะไรควรอยู่ตรงไหนบนกระดาน" และ "อะไรสำคัญพอที่จะเสียพื้นที่" มากกว่าแค่ "จะพิมพ์คำถามยังไง".

และนั่นนำไปสู่ประเด็นถัดไป: ใน context window เดียวกัน มีสองส่วนที่แตกต่างกันอย่างสำคัญ — system prompt และ user prompt. สองส่วนนี้ทำงานต่างกัน มีอำนาจต่างกัน และใช้งานต่างกัน. ใครใส่อะไรไว้ที่ไหน ส่งผลต่อคุณภาพของคำตอบโดยตรง. ในบทถัดไปเราจะดูว่ามันคืออะไรและทำไมถึงสำคัญ.
