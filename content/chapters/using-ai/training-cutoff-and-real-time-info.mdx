## พูดสั้นๆ

LLM ทุกตัวมี **training cutoff** — วันที่ข้อมูลที่ใช้ฝึกสิ้นสุด หลังจากนั้นโมเดล "ไม่รู้" ว่ามีอะไรเกิดขึ้น ความรู้ของมันคือ snapshot แช่แข็งที่จุดเวลาหนึ่ง ไม่ใช่ข้อมูลปัจจุบัน. การถามเรื่องเหตุการณ์หลัง cutoff หรือข้อมูลที่เปลี่ยนเร็ว (ราคา ตำแหน่งงาน คนสำคัญ) มักนำไปสู่การปฏิเสธ การบอก "ไม่รู้" หรือที่อันตรายกว่าคือ hallucination. ทางออกหลักคือ tool-augmented LLM (เช่น web search), RAG จากแหล่งข้อมูลที่ update เอง หรือใส่ข้อมูลปัจจุบันเข้าไปใน prompt โดยตรง — แต่ทุกวิธียังต้องการการตรวจสอบจากมนุษย์เสมอ.

LLM ไม่ใช่เครื่องที่ "เรียนรู้ตลอดเวลา". มันถูก train ครั้งเดียว (หรือไม่กี่ครั้ง) บนชุด text ขนาดมหาศาลที่ assemble ไว้ก่อนหน้า เมื่อ train เสร็จ ความรู้ของมันก็ถูก **แช่แข็ง** อยู่ใน parameter — ถ้าไม่ไป fine-tune หรือ retrain ใหม่ มันจะไม่มีทาง "รู้" ว่าหลังวันที่ assemble dataset นั้นเกิดอะไรขึ้น.

วันสุดท้ายที่ข้อมูลใน training set ถูกบันทึก เรียกว่า **training cutoff** หรือ **knowledge cutoff**. ตัวอย่าง:
- GPT-4 รุ่นแรก: cutoff ประมาณ กันยายน 2021
- Claude 3 Opus: cutoff ประมาณ สิงหาคม 2023
- Claude 4 series: cutoff ประมาณ ต้นปี 2025

**สำคัญ:** วัน cutoff ไม่ใช่วันที่โมเดลถูก "ปล่อย" ให้ใช้. โดยทั่วไปมีช่วงเวลาห่างกัน 6-12 เดือนระหว่าง cutoff กับ release เพราะต้องใช้เวลา pretraining + safety review + red-teaming + deployment.

**ผลตามมาในทางปฏิบัติ** เวลาคุณถามอะไรที่เกิดหลัง cutoff โมเดลจะมีพฤติกรรมได้ 3 แบบ:
1. **ปฏิเสธอย่างซื่อ:** "ฉันไม่มีข้อมูลหลังเดือน X ปี Y"
2. **เดาแล้วบอกว่าเดา:** "ฉันไม่แน่ใจ แต่อาจจะ..."
3. **เดาแบบมั่นใจ — นี่คือ hallucination** ที่อันตรายที่สุด เพราะคำตอบฟังดูน่าเชื่อถือ

ระดับความอันตรายเรียงจากน้อยไปมาก: 1 &lt; 2 &lt; 3. ปัญหาคือเราคุมไม่ได้ว่าจะได้แบบไหน — ขึ้นกับ context และโมเดล. นี่คือเหตุผลที่ต้องเข้าใจ cutoff ก่อนจะวางใจคำตอบเรื่องอะไรก็ตามที่ "เปลี่ยนตามเวลา".

### Mental model
นึกถึง LLM เป็น **นักวิชาการที่ถูกขังในห้องสมุดที่ปิดปรับปรุงตั้งแต่วันที่ X**. เขาอ่านทุกเล่มที่อยู่ในห้องนั้นจนขึ้นใจ แต่ไม่มีหนังสือพิมพ์ฉบับวันนี้ ไม่มีโทรศัพท์ ไม่มี internet. ถามอะไรที่อยู่ในหนังสือก่อนวัน X — ตอบได้ดี ถามอะไรหลังจากนั้น — เขาต้องเดาจาก pattern ที่เคยอ่าน ซึ่งบางครั้งเดาถูกเพราะ trend ชัด บางครั้งเดาผิดเพราะโลกพลิก.

**Tool-augmented LLM** เปรียบเทียบได้กับ "นักวิชาการคนเดิม แต่มีผู้ช่วยที่วิ่งไปอ่านหนังสือพิมพ์ข้างนอกแล้วกลับมาสรุปให้". เขายังเป็นคนตอบ — แต่ตอบหลังจากเห็นข้อมูลล่าสุด. ผลลัพธ์ดีขึ้นมาก แต่คุณภาพการตอบยังขึ้นกับคุณภาพของผู้ช่วยและของหนังสือพิมพ์ฉบับนั้น.

## ลึกขึ้นหน่อย

### 2.1 ทำไม training cutoff จึงจำเป็น

Training cutoff ไม่ใช่ข้อบกพร่อง — เป็นข้อบังคับเชิงวิศวกรรมที่หลีกเลี่ยงไม่ได้. กระบวนการสร้าง LLM ตัวหนึ่งใช้เวลา 6-18 เดือน:

1. **Data assembly** (2-6 เดือน) — crawl, filter, deduplicate, decontaminate text จากอินเทอร์เน็ต หนังสือ และ source อื่น ๆ. ต้องตัดข้อมูลที่ละเมิดสิทธิ์ ตัด PII ตัด CSAM ตัด benchmark contamination ฯลฯ. ขั้นนี้ data ต้อง "freeze" เพื่อให้กระบวนการต่อ ๆ ไป reproducible
2. **Pretraining** (1-4 เดือน) — train neural network ขนาดใหญ่บน data ที่ freeze แล้ว ใช้ GPU/TPU cluster ขนาดมหาศาล
3. **Post-training** (1-3 เดือน) — SFT, RLHF/DPO, safety tuning, evaluation
4. **Safety review and red-teaming** (1-3 เดือน) — ทดสอบช่องโหว่ ปรับ guardrail ทำ external audit
5. **Deployment infrastructure** (ขนานกัน) — serving stack, monitoring, scaling

ทุกขั้นต้องการให้ training data **คงที่** — ไม่งั้นต้อง restart รอบ pretraining ทุกครั้งที่อยากเพิ่มข่าวล่าสุด ซึ่งใช้เงินระดับ 10-100 ล้านดอลลาร์ต่อรอบ. ดังนั้น cutoff คือเส้นแบ่งที่ต้องลากจริง ๆ.

### 2.2 ผลกระทบต่อ user ในทางปฏิบัติ

**ประเภทคำถามที่อ่อนไหวต่อ cutoff:**

- **เหตุการณ์หลัง cutoff** — ใครชนะเลือกตั้ง ใครเป็น CEO บริษัท X ตอนนี้ มีสงครามที่ไหน บริษัทไหน IPO. โมเดลไม่รู้
- **ข้อมูลที่เปลี่ยนเร็ว** — ราคาสินค้า ค่าเงิน หุ้น เวอร์ชันของซอฟต์แวร์ API endpoint ที่ใช้ปัจจุบัน
- **ตำแหน่งของคนสำคัญ** — CEO, รัฐมนตรี, ดารา, นักกีฬา. คนเปลี่ยนตำแหน่ง โมเดลยังจำตำแหน่งเก่า
- **เทคโนโลยีที่ evolve เร็ว** — best practice ของ framework, model ใหม่ ๆ, library versions, security advisories
- **กฎหมายและ regulation** — กฎหมายภาษีปีปัจจุบัน, การตีความใหม่, sanction list

**ประเภทคำถามที่ไม่อ่อนไหว** (และเป็นจุดที่ LLM ยังเก่ง):
- ความรู้เชิงหลักการ (คณิตศาสตร์ ฟิสิกส์ ประวัติศาสตร์ที่นานพอ)
- รูปแบบภาษา (ไวยากรณ์ การเขียน การสรุป การแปล)
- pattern ของ code ที่เสถียร (algorithm, design pattern พื้นฐาน)
- คำแนะนำทั่วไปที่ไม่ผูกกับเหตุการณ์ปัจจุบัน

### 2.3 รู้ได้อย่างไรว่า cutoff ของแต่ละโมเดลคือเมื่อไหร่

มี 3 วิธี เรียงจาก "ง่ายแต่ไม่น่าเชื่อ" ไป "ยากแต่แม่นยำ":

**(ก) ถามโมเดลตรง ๆ** เช่น "training cutoff ของคุณคือเมื่อไหร่". วิธีนี้ **เสี่ยง** — โมเดลมักตอบไม่ถูกต้องเกี่ยวกับตัวเอง เพราะ:
- ข้อมูลใน training set อาจไม่มีคำตอบที่แท้จริง
- โมเดลอาจตอบตามที่เห็นใน document ของรุ่นก่อนหน้า
- บางครั้ง provider ฝัง system prompt ที่ทำให้โมเดลตอบ cutoff แตกต่างจากความจริง

**(ข) อ่านเอกสารทางการของ provider** เช่น Anthropic, OpenAI, Google มักประกาศ cutoff ใน model card หรือ API documentation. นี่คือแหล่งที่น่าเชื่อที่สุด

**(ค) Empirical testing** ถามเรื่องเหตุการณ์ที่รู้แน่ ๆ ว่าเกิดเมื่อไหร่ ในเดือนรอบ ๆ ที่สงสัย เช่น:
- "ใครชนะเลือกตั้งสหรัฐปี 2024" → ถ้าโมเดลตอบถูก cutoff ≥ พ.ย. 2024
- "Claude 4 ออกเมื่อไหร่" → ถ้าโมเดลรู้ คือ cutoff ≥ release date ของ Claude 4

วิธีนี้ใช้เมื่อเอกสารไม่ชัดเจน หรืออยากตรวจสอบ.

### 2.4 ปัญหา "Real-time information" และ 3 ทางออก

**Native LLM (ไม่มี tool)** ไม่สามารถเข้าถึงข้อมูลปัจจุบันได้เลย เพราะกลไก inference ของมันคือคำนวณ probability distribution จาก weight ที่แช่แข็ง. ไม่มี hook ไปอินเทอร์เน็ต. ทางออกในระดับสถาปัตยกรรมมี 3 แบบ:

**ทางออก (1): Tool augmentation — เรียก web search หรือ tool อื่น ๆ ระหว่าง inference**
- โมเดลถูก train ให้ "รู้ว่าเมื่อไหร่ควรเรียก tool" และ "tool คืนค่าอะไรมา"
- ตัวอย่าง: Claude ที่มี web search, ChatGPT ที่มี browsing, Gemini ที่ integrate Google search, Perplexity ที่ออกแบบ search-first
- รายละเอียดเชิง implementation ดู [function-calling-and-tool-use](/building-with-ai/function-calling-and-tool-use)

**ทางออก (2): RAG (Retrieval-Augmented Generation)** — มี vector database ของเอกสารที่ update บ่อย ก่อน inference จะ retrieve เอกสารที่เกี่ยวข้องมาใส่ใน context. ใช้กับ domain-specific data ที่เราเป็นเจ้าของ (เอกสารบริษัท ฐานความรู้ภายใน FAQ)
- ตัวอย่างใช้งาน: chatbot ของ HR ที่ใช้นโยบายบริษัทล่าสุด, customer support ที่ใช้ KB ที่อัพเดททุกวัน
- รายละเอียดสถาปัตยกรรม ดู [rag-retrieval-augmented-generation](/building-with-ai/rag-retrieval-augmented-generation) และ paper ดั้งเดิม 

**ทางออก (3): Manual prompt injection** — copy-paste ข้อมูลปัจจุบันเข้าไปใน prompt เอง เช่น วาง spreadsheet ของยอดขายประจำเดือนเข้าไปแล้วถามให้วิเคราะห์
- ดีสำหรับงาน one-off
- ข้อเสีย: ต้อง manual ต้องไม่เกิน context window

ในทางปฏิบัติ tool augmentation (1) คือสิ่งที่ end user ทั่วไปได้ใช้มากที่สุดในปี 2026 — เพราะ provider build-in มาในผลิตภัณฑ์เลย.

### 2.5 Tool-augmented LLM ทำงานได้ดี vs ไม่ดีในสถานการณ์ใด

**ทำงานได้ดี:**
- **Factual lookup ที่มีคำตอบบนเว็บชัดเจน** — ราคาสินค้าจากร้านใหญ่, อุณหภูมิวันนี้, รายงานข่าว
- **Citations** — โมเดลสามารถอ้างถึง URL ต้นฉบับ ทำให้ผู้ใช้ตรวจสอบเองได้
- **คำถามที่ต้องการความสด** — "Apple ประกาศอะไรเมื่อวาน", "ราคาน้ำมันวันนี้"

**ขีดจำกัด:**
- **Latency** — เพิ่ม 3-15 วินาทีต่อ query เพราะต้องค้นและอ่านเว็บ
- **คุณภาพข้อมูลในเว็บ** — โมเดลอ่านอะไรก็เชื่ออันนั้น ถ้า top result เป็น blog ที่ผิด โมเดลก็ตอบผิดตามมา
- **Hallucination ตอน synthesis ยังเหลืออยู่** — แม้ retrieve มาดีแล้ว โมเดลอาจ "ผสม" ข้อมูลใน source ผิด หรือเติม detail ที่ไม่มี
- **เรื่องที่ controversial หรือมีข้อมูลขัดกัน** — โมเดลอาจ blend หลายมุมมองเป็นคำตอบเดียวที่ไม่ตรงกับแหล่งใดเลย
- **คำถามที่ต้องการความเชี่ยวชาญ** — เช่น สรุป paper วิจัยที่ไม่มี abstract ดี, ตีความข้อกฎหมาย: tool ช่วยได้แค่ retrieve ส่วนความเข้าใจยังเป็นปัญหาเดิมของ LLM

ดู survey เรื่อง LLM hallucination ใน  ที่ครอบคลุมว่า retrieval ช่วยลดได้แค่ไหน แต่ไม่ขจัด.

### 2.6 สำหรับ developer: เลือก RAG หรือ web search?

ถ้าคุณกำลังสร้างระบบที่ใช้ LLM ตอบคำถามใน domain ของตัวเอง (ไม่ใช่แค่เป็นผู้ใช้ปลายทาง):

- **ใช้ web search** เมื่อข้อมูลที่ต้องการเป็นความรู้สาธารณะที่มีอยู่บนเว็บ และ source สาธารณะน่าเชื่อพอ
- **ใช้ RAG** เมื่อข้อมูลเป็นของเฉพาะคุณ (เอกสารบริษัท ฐานข้อมูลสินค้า historical conversations) หรือต้องการควบคุมคุณภาพและความ up-to-date ของ source

ในระบบ production จริง มักผสมทั้ง 2 (hybrid retrieval) — รายละเอียดอยู่ใน Level 3.

### 2.7 กฎตรวจสอบ (Verification rule)

**กฎทั่วไป:** time-sensitive information จาก LLM ต้อง verify เสมอ ไม่ว่าจะมี tool หรือไม่.

ตัวอย่าง failure mode ที่พบบ่อยแม้ในระบบ tool-augmented:
- โมเดล cite URL ที่มีอยู่จริง แต่ "อ่านผิด" หรือ "สรุปขัด" กับเนื้อหาในนั้น
- โมเดลเอาข้อมูลจากปี 2022 บนเว็บมาตอบ ทั้งที่ใน 2026 เปลี่ยนแล้ว
- โมเดลผสม 2 บทความ — บทความ A พูดถึงปี 2024 บทความ B พูดถึงปี 2026 — และให้ตัวเลขผสมที่ไม่ตรงปีไหน

ทางปฏิบัติ: คลิก citation จริง อ่านต้นฉบับ ใช้คำตอบของโมเดลเป็น "draft" ไม่ใช่ "ground truth".

### 2.8 บริบทภาษาไทย

ในบริบทไทยมีข้อควรระวังเพิ่มเติม:

- **คุณภาพข้อมูลไทยบนเว็บไม่สม่ำเสมอ** — แหล่งวิชาการในภาษาไทยน้อยกว่าอังกฤษมาก ผลค้นหาจึงมักผสม content คุณภาพหลากหลาย (เว็บข่าว clickbait, ฟอรัม, blog ส่วนตัว)
- **ผลค้นหาปนภาษา** — tool ค้นเรื่องที่ไม่มีในแหล่งไทย จะดึงข้อมูลภาษาอังกฤษมาสรุปเป็นไทย ระดับการแปลและการสรุปคือจุดที่ hallucination เพิ่ม
- **ชื่อเฉพาะภาษาไทย** — ชื่อคน ชื่อบริษัท การสะกดมีหลายแบบ บางครั้ง tool ค้นไม่เจอเพราะสะกดต่างจากที่อยู่ในแหล่งทางการ
- **เหตุการณ์เฉพาะไทย** — การเมือง กฎหมาย ราคาในประเทศ ฯลฯ มี coverage ในแหล่งภาษาอังกฤษน้อย โมเดลอาจดึงข้อมูลที่เก่าหรือไม่ลึกพอ

ในทางปฏิบัติ ถ้าใช้ LLM ค้นเรื่องเฉพาะไทยที่ต้องการความสด แนะนำให้ใช้แหล่งทางการไทย (เว็บราชการ ธนาคารแห่งประเทศไทย ตลาดหลักทรัพย์ฯ) เป็น source โดยตรง แล้วใช้ LLM เป็นเครื่องช่วยสรุปแทน.

## Worked Examples

### Example 1: ถาม CEO ปัจจุบันโดยไม่มี tool
**Prompt:** "ใครเป็น CEO ของ Twitter ตอนนี้"
**โมเดลที่ cutoff = ส.ค. 2023:** อาจตอบ "Elon Musk" — ซึ่งถูกต้องช่วง cutoff แต่ในปี 2026 เปลี่ยนแล้ว
**โมเดลที่ cutoff = ต้น 2025:** อาจตอบ "Linda Yaccarino" — ถูกในช่วง 2023-2025 แต่ตอนนี้อาจเปลี่ยนแล้ว
**บทเรียน:** คำถามดูง่าย แต่คำตอบที่ "ถูกในตอนนั้น" ≠ "ถูกตอนนี้" — ต้องตรวจสอบเสมอ

### Example 2: ตัวอย่าง hallucination จาก cutoff
**Prompt:** "API endpoint ล่าสุดของ OpenAI สำหรับ image generation คืออะไร"
**Failure mode:** โมเดลที่ cutoff เก่าอาจ "เดา" endpoint ที่ฟังดูสมเหตุสมผล เช่น `/v1/images/generate` ทั้งที่จริง endpoint เปลี่ยนไป
**วิธีแก้:** เปิด official docs ตรง ๆ — โมเดลตอบคำถามนี้ไม่น่าเชื่อ ยกเว้นจะ tool-augment

### Example 3: Tool-augmented LLM ตอบดี
**Prompt** (ในระบบที่มี web search): "ราคาทองวันนี้ในไทย"
**Behavior:** โมเดลเรียก web search → ได้ผลจาก ราคาทอง.com หรือ กรุงเทพธุรกิจ → สรุปตัวเลข + cite URL
**ผล:** คำตอบใช้ได้ แต่ผู้ใช้ควรคลิก citation เช็คเอง เพราะราคาเปลี่ยนทุกชั่วโมง

### Example 4: Tool-augmented แต่ยังพลาด
**Prompt:** "ใครชนะการเลือกตั้งครั้งล่าสุดในประเทศ X"
**Behavior:** Tool ค้นเจอบทความเก่า (เพราะ algorithm จัด ranking โดยไม่สนใจวันที่) → โมเดลตอบข้อมูลของเลือกตั้งครั้งก่อนหน้า ทั้งที่ผู้ใช้หมายถึงครั้งล่าสุด
**บทเรียน:** การมี tool ไม่รับประกันว่า retrieved data จะ relevant และ recent ที่สุด

## Common Misconceptions

1. **ความเข้าใจผิด:** Cutoff = วันที่โมเดลออก
   **ความจริง:** Cutoff คือวันที่ training data หยุด มัก **เก่ากว่า** วัน release 6-12 เดือน เพราะต้องใช้เวลา train + safety review

2. **ความเข้าใจผิด:** ถ้ามี web search แล้ว LLM จะตอบเรื่องปัจจุบันถูกหมด
   **ความจริง:** Tool ช่วยลด knowledge gap แต่ไม่ขจัด hallucination ตอน synthesis โมเดลอาจ "อ่านผิด" "ผสมผิด" หรือ "เติมเอง" ยังต้องตรวจสอบ citation จริง

3. **ความเข้าใจผิด:** ถามโมเดลเองว่า cutoff คือเมื่อไหร่ เป็นวิธีที่น่าเชื่อ
   **ความจริง:** โมเดลมักไม่รู้ cutoff ของตัวเองอย่างถูกต้อง เพราะข้อมูลใน training set ไม่ครอบคลุมข้อมูลของตัวมันเอง วิธีที่น่าเชื่อกว่าคืออ่าน documentation ของ provider หรือทดสอบ empirically

4. **ความเข้าใจผิด:** LLM ที่ใหม่กว่ามี cutoff ใหม่กว่าเสมอ
   **ความจริง:** ไม่จำเป็น โมเดลใหม่บางตัวอาจมี cutoff เก่ากว่ารุ่นก่อน เพราะใช้ data preparation pipeline ที่ต่างกัน หรือ provider เลือกหยุดเร็วเพื่อเลี่ยง contamination

5. **ความเข้าใจผิด:** RAG = web search
   **ความจริง:** RAG เป็น framework กว้างกว่า — retrieve เอกสารจาก vector store ที่เราควบคุม (อาจเป็น private docs) web search เป็นกรณีพิเศษหนึ่งที่ "ข้อมูล" คือทั้งอินเทอร์เน็ต. ดู [rag-retrieval-augmented-generation](/building-with-ai/rag-retrieval-augmented-generation)
