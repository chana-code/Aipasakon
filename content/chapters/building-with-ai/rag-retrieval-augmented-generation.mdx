**RAG คือทางออกของปัญหานี้** — เป็นวิธีที่ทำให้ AI ตอบจากข้อมูลของคุณโดยเฉพาะ ไม่ใช่จากความจำที่อาจล้าสมัยหรือไม่ครบถ้วน

## ทำไม LLM ถึงตอบจากข้อมูลของคุณไม่ได้ (ถ้าไม่มี RAG)

ก่อนจะเข้าใจว่า RAG แก้ปัญหาอะไร ต้องเข้าใจก่อนว่า LLM ทั่วไปมีข้อจำกัดอะไรบ้าง

**ข้อจำกัดที่ 1: Knowledge Cutoff**

LLM ทุกตัวถูก train ด้วยข้อมูลที่มีวันหมดอายุ GPT-4o ถูก train ด้วยข้อมูลถึงช่วงต้นปี 2024 Claude ก็มี cutoff ของตัวเอง ถ้าข้อมูลของคุณถูกอัปเดตหลังจากนั้น — ไม่ว่าจะเป็นประกาศ นโยบายใหม่ ราคาสินค้า หรือเหตุการณ์ที่เพิ่งเกิดขึ้น — LLM จะไม่รู้เรื่องนั้นเลย

**ข้อจำกัดที่ 2: Hallucination เมื่อไม่รู้จริง**

เมื่อ LLM ถูกถามเรื่องที่มันไม่มีข้อมูล แทนที่จะพูดว่า "ไม่รู้" มันมักจะสร้างคำตอบที่ฟังดูสมเหตุสมผลแต่ผิด นั่นคือ hallucination ในบริบทธุรกิจ นี่อันตรายมาก ลองนึกภาพ AI ตอบว่า "นโยบายลาป่วยของบริษัทคือ 30 วัน" ทั้งที่จริงคือ 15 วัน

**ข้อจำกัดที่ 3: Context Window จำกัด**

แม้โมเดลรุ่นใหม่จะมี context window ใหญ่ขึ้นมาก แต่มีขีดจำกัดอยู่ดี ยิ่งถ้าเอกสารมีหลายร้อยหน้า การยัดทั้งหมดเข้าไปใน prompt ทุกครั้งก็แพงและช้า ไม่ใช่วิธีที่ scale ได้จริง

**ข้อจำกัดที่ 4: ข้อมูลองค์กรเป็น Private**

ข้อมูลภายในบริษัท — สัญญาลูกค้า เอกสารกฎหมาย ฐานความรู้ Support — ไม่เคยถูกนำไป train LLM เพราะมันเป็นข้อมูลส่วนตัว LLM จึงไม่รู้เรื่องพวกนี้เลยโดยธรรมชาติ

> 💡 **Key insight:** LLM ฉลาดมาก แต่ฉลาดแบบคนที่ไม่เคยอ่านเอกสารของคุณ RAG คือการให้มันเปิดเอกสารที่ถูกต้องก่อนตอบ

## RAG ทำงานยังไง — ภาพรวม 3 ขั้นตอน

RAG ย่อมาจาก **Retrieval-Augmented Generation** ซึ่งอธิบายได้ตรงตัวเลย: เพิ่มความสามารถในการ "ดึงข้อมูล" (Retrieval) เข้าไปใน "การสร้างคำตอบ" (Generation) ของ LLM

ภาพรวมมี 3 ขั้นตอนหลัก:

**ขั้นตอนที่ 1 — Index (เตรียมข้อมูล)**

นี่คือขั้นตอนที่ทำครั้งเดียวตอนตั้งระบบ โดยแปลงเอกสารทั้งหมดให้อยู่ในรูปแบบที่ค้นหาได้ด้วยความหมาย:
- แบ่งเอกสารออกเป็นชิ้นเล็ก ๆ (chunks)
- แปลงแต่ละ chunk เป็นตัวเลขที่แทนความหมาย (embeddings)
- เก็บตัวเลขเหล่านั้นไว้ใน vector database

**ขั้นตอนที่ 2 — Retrieve (ค้นหาส่วนที่เกี่ยวข้อง)**

เมื่อผู้ใช้พิมพ์คำถาม ระบบจะ:
- แปลงคำถามเป็น embedding ด้วยโมเดลเดียวกัน
- ค้นหาใน vector database เพื่อหา chunks ที่มีความหมายใกล้เคียงกับคำถามมากที่สุด
- ดึง chunks นั้นขึ้นมา (ปกติ 3-10 chunks)

**ขั้นตอนที่ 3 — Generate (สร้างคำตอบ)**

ระบบส่งทั้งคำถามและ chunks ที่ดึงมาให้ LLM แล้วบอกว่า "ตอบจากข้อมูลนี้เท่านั้น" LLM จะอ่านบริบทที่ให้มาแล้วสร้างคำตอบที่ถูกต้องและอ้างอิงได้

<!-- IMAGE: ไดอะแกรมแสดง 3 ขั้นตอน: Index (เอกสาร → chunks → embeddings → vector store), Retrieve (คำถาม → embedding → ค้นหา → chunks ที่เกี่ยวข้อง), Generate (chunks + คำถาม → LLM → คำตอบ) -->

เปรียบเทียบให้เห็นภาพ: สมมติมีคนถามว่า "นโยบายลาคลอดของบริษัทคือกี่วัน?" ในระบบ RAG ที่เชื่อมกับคู่มือพนักงาน:
1. คำถามถูกแปลงเป็น embedding
2. ระบบค้นพบว่า chunk หน้า 47 ของคู่มือพูดถึงนโยบายลาคลอดโดยตรง
3. LLM อ่าน chunk นั้นแล้วตอบว่า "ตามนโยบายบริษัท (หน้า 47) ลาคลอดได้ 98 วัน..."

ไม่ใช่ LLM เดาจากความรู้ทั่วไป แต่อ่านจากเอกสารจริงของคุณ

## Chunking — แบ่งเอกสารยังไงให้ดี

ก่อนจะ embed อะไรได้ ต้องแบ่งเอกสารเป็นชิ้นก่อน ขั้นตอนนี้เรียกว่า **chunking** และมันสำคัญมากกว่าที่หลายคนคิด

**ทำไมต้อง Chunk?**

เหตุผลหลักคือ embedding model มี token limit ของตัวเอง และการ embed เอกสารทั้งเล่มในครั้งเดียวก็ไม่มีประโยชน์ เพราะถ้า chunk ใหญ่เกินไป ความหมายของมันจะ "เจือจาง" ลง — เหมือนการสรุปหนังสือทั้งเล่มเป็นประโยคเดียว คุณจะได้ภาพรวมกว้าง ๆ แต่รายละเอียดสำคัญหายไปหมด

และเมื่อ retrieve ก็ต้องการเฉพาะส่วนที่เกี่ยวข้องกับคำถาม ไม่ใช่ทั้งเอกสาร ถ้าส่งเอกสาร 200 หน้าทั้งหมดให้ LLM ทุกครั้ง context window จะเต็ม ค่าใช้จ่ายพุ่ง และ LLM ก็หาข้อมูลสำคัญในกองข้อมูลไม่เจออีก

**วิธี Chunk แบบต่าง ๆ**

*Fixed-size chunking* คือการตัดทุก N ตัวอักษรหรือ N token เท่ากันเป๊ะ ง่ายที่สุด แต่มีปัญหาคือตัดกลางประโยคหรือกลางความคิดได้

```
[ส่วนที่ 1: ตัวอักษรที่ 1-500]
[ส่วนที่ 2: ตัวอักษรที่ 501-1000]
...
```

*Paragraph-based chunking* ตัดตาม paragraph จริง ๆ อ่านได้มีความหมายกว่า แต่ขนาดไม่สม่ำเสมอ บาง paragraph อาจยาวมาก บางอันสั้นมาก

*Header-based chunking* ใช้กับเอกสารที่มีโครงสร้าง เช่น Markdown หรือ HTML ตัดตามหัวข้อ (H1, H2, H3) ได้ chunks ที่มีความหมายสมบูรณ์และรู้ว่าอยู่หัวข้ออะไร

*Recursive chunking* คือวิธีที่นิยมใช้มากที่สุดในปัจจุบัน ลอง split ด้วย separator ตามลำดับ (paragraph → sentence → word) จนกว่าจะได้ขนาดที่ต้องการ

**Overlap คืออะไร ทำไมต้องมี**

สมมติตัด chunk ขนาด 500 token ถ้าตัดชนกันพอดี ข้อมูลที่อยู่ตรงรอยต่ออาจถูกแยกออกจากกันและสูญเสียบริบท

Overlap คือการให้ chunks ที่อยู่ติดกัน "ซ้อนทับ" กันบางส่วน เช่น chunk 1 เป็น token ที่ 1-500, chunk 2 เป็น token ที่ 450-950 (ซ้อน 50 token) วิธีนี้ทำให้มั่นใจได้ว่าข้อมูลที่อยู่ตรงรอยต่อจะไม่หายไป

<!-- IMAGE: ไดอะแกรมแสดง chunk overlap: สองก้อนสี่เหลี่ยมที่มีส่วนสีทับซ้อนกันตรงกลาง -->

**ตัวอย่าง: Chunk คู่มือพนักงาน**

สมมติมีข้อความ:

```
หัวข้อ 3.2: นโยบายการลา

พนักงานมีสิทธิ์ลาพักร้อนได้ 10 วันต่อปี โดยต้องแจ้งล่วงหน้า
ไม่น้อยกว่า 7 วันทำการ ยกเว้นกรณีฉุกเฉิน

การลาป่วยใช้ใบรับรองแพทย์เมื่อลาติดต่อกันเกิน 3 วัน
พนักงานมีสิทธิ์ลาป่วยได้ 30 วันต่อปีโดยได้รับค่าจ้าง
```

ถ้าใช้ header-based chunking จะได้ 1 chunk สำหรับทั้งหัวข้อ 3.2 ซึ่งดีมาก — คำถามเรื่องการลาจะ retrieve chunk นี้ขึ้นมาทั้งหมด พร้อมบริบทครบถ้วน

> 💡 **Key insight:** ขนาด chunk ที่นิยมคือ 256-512 token พร้อม overlap 20-50 token แต่ไม่มีสูตรสำเร็จ — ขึ้นอยู่กับประเภทเอกสารและลักษณะคำถามที่ผู้ใช้จะถาม ต้องทดลอง

## Embedding — เปลี่ยนข้อความเป็นตัวเลข

ทบทวนเร็ว ๆ: embedding คือการแปลงข้อความให้เป็น vector ของตัวเลขที่แทนความหมาย ข้อความที่มีความหมายใกล้กันจะมี vector ที่อยู่ใกล้กัน — กลไกเต็ม ๆ ว่าตัวเลขพวกนี้มาจากไหนและทำงานยังไง ดูได้ในบท [Vector Embeddings](/building-with-ai/embeddings) ตรงนี้เราโฟกัสเฉพาะสิ่งที่ต้องรู้เวลาเอามาใช้ใน RAG:

**ต้องใช้ Embedding Model เดียวกันตลอด**

เมื่อ embed ข้อมูลตอน index ใช้โมเดลอะไร ตอน embed คำถามก็ต้องใช้โมเดลนั้น ถ้าเปลี่ยนโมเดล ตัวเลขที่ได้จะอยู่คนละ "พื้นที่" กัน เปรียบเทียบจะผิดพลาดทั้งหมด

**Embedding Models ที่นิยมใช้**

*OpenAI text-embedding-3-small* — ราคาถูก เร็ว เหมาะสำหรับเริ่มต้น รองรับหลายภาษารวมถึงภาษาไทย dimension 1,536

*OpenAI text-embedding-3-large* — แม่นยำกว่า ราคาแพงกว่า dimension 3,072 เหมาะเมื่อ accuracy สำคัญมาก

*Cohere Embed v3* — เน้นความแม่นยำหลายภาษา มี embedding สำหรับ "search document" และ "search query" แยกกัน ซึ่งช่วยให้ retrieval แม่นยำขึ้น

(เรื่อง dimension — จำนวนตัวเลขใน vector หนึ่งตัว — และผลของมันต่อความละเอียดและความเร็ว อธิบายไว้ในบท [Vector Embeddings](/building-with-ai/embeddings) แล้ว)

## Vector Store — เก็บและค้นหา Embedding

หลังจาก embed เอกสารทั้งหมดแล้ว ต้องมีที่เก็บ vector เหล่านั้นและค้นหาได้อย่างมีประสิทธิภาพ นั่นคือหน้าที่ของ **vector database** (หรือ vector store)

**ทำไม Database ธรรมดาไม่พอ**

ถ้าเก็บ vector ใน SQL database ธรรมดา การค้นหา vector ที่ "ใกล้เคียง" กับ query จะต้องคำนวณระยะห่างกับทุก row ถ้ามี embedding 100,000 ชิ้น ทุกคำถามต้องคำนวณ 100,000 ครั้ง ช้ามาก

Vector database ออกแบบมาเพื่อแก้ปัญหานี้โดยเฉพาะ ใช้โครงสร้างข้อมูลพิเศษที่ทำให้ค้นหา "เพื่อนบ้านที่ใกล้ที่สุด" (Approximate Nearest Neighbor) ได้เร็วมาก แม้จะมี vector หลายล้านตัว

วิธีวัดความ "ใกล้เคียง" ที่ใช้กันมากที่สุดคือ cosine similarity ซึ่งให้ค่าระหว่าง -1 ถึง 1 (1 = ความหมายเหมือนกัน, 0 = ไม่เกี่ยว) — ในทางปฏิบัติ chunk ที่ได้คะแนน 0.8+ กับ query มักจะเกี่ยวข้องและเป็นประโยชน์ กลไกว่า cosine similarity คำนวณยังไงและทำไมถึงใช้มุมแทนระยะห่าง อธิบายไว้ในบท [Vector Embeddings](/building-with-ai/embeddings) แล้ว

**เลือก Vector Database ตัวไหนดี**

ตัวเลือกหลัก ๆ คือ ChromaDB (เริ่มง่าย เหมาะ prototype), Pinecone (managed cloud เหมาะ production), pgvector (ถ้าใช้ Postgres อยู่แล้ว) และตัว open-source อย่าง Weaviate, Qdrant, Milvus การเปรียบเทียบเต็ม ๆ ทั้ง 6 ตัว — ความง่ายในการตั้ง, ราคา, scalability, เหมาะกับงานแบบไหน — อยู่ในบท [Vector Databases](/building-with-ai/vector-databases)

<!-- IMAGE: ตารางเปรียบเทียบ ChromaDB vs Pinecone vs pgvector แกน: ease of setup, cost, scalability, use case -->

> 💡 **Key insight:** เปรียบเทียบ vector database กับห้องสมุดที่จัดหนังสือตาม "ความหมาย" ไม่ใช่ตามตัวอักษร คุณถามว่า "มีเรื่องเกี่ยวกับความรักที่เขียนในฝรั่งเศสยุค 1800 ไหม?" แล้วห้องสมุดนำหนังสือที่ใกล้เคียงมาให้ทันที แทนที่จะต้องค้นหาตามชื่อหรือ ISBN

## ลองทำดู #1: สร้าง RAG Pipeline ง่าย ๆ ด้วย Python

ถึงเวลาลงมือทำ เราจะสร้าง RAG ง่าย ๆ ที่ตอบคำถามจากเอกสารของเราเอง โดยใช้ ChromaDB และ OpenAI

**สิ่งที่ต้องเตรียม**

```bash
pip install chromadb openai
```

และต้องมี OpenAI API key (ดูวิธีได้ที่บท [เรียกใช้ API ครั้งแรก](/building-with-ai/llm-apis))

**Code เต็ม พร้อม comment ภาษาไทย**

```python
import chromadb
from openai import OpenAI

# ตั้งค่า client
openai_client = OpenAI(api_key="your-api-key-here")
chroma_client = chromadb.Client()  # เก็บใน memory สำหรับทดสอบ

# สร้าง collection (เหมือน "ตาราง" ใน database ธรรมดา)
collection = chroma_client.create_collection(name="company_docs")


# ฟังก์ชัน: แปลงข้อความเป็น embedding ด้วย OpenAI
def get_embedding(text):
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",  # โมเดลที่ราคาคุ้มค่าที่สุด
        input=text
    )
    return response.data[0].embedding


# ฟังก์ชัน: แบ่งเอกสารเป็น chunks ง่าย ๆ ตาม paragraph
def chunk_text(text, max_chars=500):
    paragraphs = text.split("\n\n")  # ตัดตาม paragraph
    chunks = []
    current_chunk = ""

    for para in paragraphs:
        # ถ้าเพิ่ม paragraph นี้แล้วยังไม่เกิน max_chars ก็เพิ่มต่อ
        if len(current_chunk) + len(para) < max_chars:
            current_chunk += para + "\n\n"
        else:
            # ถ้าเต็มแล้ว บันทึก chunk ปัจจุบัน แล้วเริ่มใหม่
            if current_chunk:
                chunks.append(current_chunk.strip())
            current_chunk = para + "\n\n"

    if current_chunk:
        chunks.append(current_chunk.strip())

    return chunks


# ========== STEP 1: INDEX ==========
# เอกสารตัวอย่าง (คู่มือพนักงานจำลอง)
documents = [
    """นโยบายการลาพักร้อน
    พนักงานมีสิทธิ์ลาพักร้อนได้ 10 วันต่อปี
    ต้องแจ้งล่วงหน้าไม่น้อยกว่า 7 วันทำการ
    วันลาที่เหลือสามารถสะสมได้ไม่เกิน 5 วัน""",

    """นโยบายการลาป่วย
    พนักงานมีสิทธิ์ลาป่วยได้ 30 วันต่อปีโดยได้รับค่าจ้าง
    กรณีลาป่วยติดต่อกันเกิน 3 วันต้องมีใบรับรองแพทย์
    ไม่สามารถสะสมวันลาป่วยข้ามปีได้""",

    """นโยบายการลาคลอด
    พนักงานหญิงมีสิทธิ์ลาคลอดได้ 98 วัน
    ได้รับค่าจ้างเต็มตลอดระยะเวลาลา
    ต้องแจ้งฝ่าย HR ล่วงหน้าอย่างน้อย 1 เดือน""",
]

# แบ่ง chunk และ embed แล้วเก็บใน ChromaDB
all_chunks = []
all_ids = []
all_embeddings = []

chunk_id = 0
for doc_idx, doc in enumerate(documents):
    chunks = chunk_text(doc)
    for chunk in chunks:
        # embed แต่ละ chunk
        embedding = get_embedding(chunk)

        all_chunks.append(chunk)
        all_ids.append(f"chunk_{chunk_id}")
        all_embeddings.append(embedding)
        chunk_id += 1

# บันทึกทุกอย่างลง ChromaDB
collection.add(
    documents=all_chunks,      # ข้อความต้นฉบับ (เก็บไว้ส่งให้ LLM)
    embeddings=all_embeddings,  # vector ที่ใช้ค้นหา
    ids=all_ids                # ID unique สำหรับแต่ละ chunk
)

print(f"Index เสร็จ: เก็บ {len(all_chunks)} chunks ใน vector store")


# ========== STEP 2 & 3: RETRIEVE + GENERATE ==========
def ask_rag(question):
    # ขั้นตอน 2: Retrieve — ค้นหา chunks ที่เกี่ยวข้อง
    query_embedding = get_embedding(question)

    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=2  # ดึง 2 chunks ที่ใกล้เคียงที่สุด
    )

    # รวม chunks ที่ได้เป็น context
    context = "\n\n".join(results["documents"][0])

    # ขั้นตอน 3: Generate — ให้ LLM ตอบโดยใช้ context ที่ได้
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": """คุณเป็นผู้ช่วยตอบคำถามเกี่ยวกับนโยบายบริษัท
                ตอบจากข้อมูลที่ให้มาเท่านั้น ถ้าไม่มีข้อมูลให้บอกตรง ๆ ว่าไม่ทราบ"""
            },
            {
                "role": "user",
                "content": f"""ข้อมูลจากคู่มือพนักงาน:
{context}

คำถาม: {question}"""
            }
        ]
    )

    answer = response.choices[0].message.content

    # แสดง chunks ที่ใช้ตอบ (เพื่อ transparency)
    print(f"\nคำถาม: {question}")
    print(f"\nตอบ: {answer}")
    print(f"\n[ค้นพบข้อมูลจาก {len(results['documents'][0])} chunks]")

    return answer


# ทดสอบ
ask_rag("ลาป่วยได้กี่วันต่อปี?")
ask_rag("ต้องแจ้งล่วงหน้ากี่วันถ้าจะลาพักร้อน?")
ask_rag("เบี้ยเลี้ยงเดินทางต่างจังหวัดเท่าไหร่?")  # ไม่มีในเอกสาร
```

ลองรัน code นี้ แล้วสังเกตสิ่งที่เกิดขึ้น:
- คำถามแรกและสองจะได้คำตอบที่ถูกต้อง เพราะมีข้อมูลในเอกสาร
- คำถามสามจะได้คำตอบว่า "ไม่มีข้อมูล" เพราะ LLM ได้รับ instruction ให้ตอบจาก context เท่านั้น

## ลองทำดู #2: ปรับปรุง Retrieval Quality

RAG ง่าย ๆ ที่เพิ่งทำนั้นใช้งานได้ แต่ในโลกจริง ยังมีเทคนิคที่ทำให้ดีขึ้นได้มาก มาดู 3 เทคนิคที่ impact สูงที่สุด

**เทคนิค 1: ทดลองเปลี่ยน Chunk Size**

Chunk size เล็กเกินไป = context ไม่พอ ตอบได้ไม่สมบูรณ์
Chunk size ใหญ่เกินไป = ความหมาย "เจือจาง" ค้นหาได้ไม่แม่น

วิธีทดสอบง่าย ๆ: เตรียมชุดคำถาม 10-20 ข้อพร้อมคำตอบที่ถูกต้อง แล้วลองรัน RAG ด้วย chunk size ต่าง ๆ (128, 256, 512 token) แล้วดูว่า chunk size ไหนให้คำตอบถูกมากที่สุด

```python
# ทดสอบ chunk sizes ต่าง ๆ
for chunk_size in [128, 256, 512]:
    # สร้าง collection ใหม่สำหรับแต่ละ chunk size
    # แล้วนับว่าคำตอบถูกกี่ข้อจาก test set ที่เตรียมไว้
    print(f"Chunk size {chunk_size}: accuracy = X%")
```

**เทคนิค 2: เพิ่ม Metadata Filter**

แทนที่จะค้นหาใน documents ทั้งหมด เพิ่ม metadata ให้แต่ละ chunk เพื่อ filter ก่อน

```python
# ตอน index: เพิ่ม metadata
collection.add(
    documents=all_chunks,
    embeddings=all_embeddings,
    ids=all_ids,
    metadatas=[
        {"category": "hr_policy", "department": "all"},  # แต่ละ chunk มี metadata
        {"category": "it_security", "department": "tech"},
        # ...
    ]
)

# ตอน query: filter ก่อนค้นหา
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=3,
    where={"category": "hr_policy"}  # ค้นหาเฉพาะในหมวด HR
)
```

การ filter ด้วย metadata ลดพื้นที่ค้นหาและเพิ่มความแม่นยำ เพราะไม่ต้องแข่งกับ chunks จากหมวดอื่น ๆ ที่ไม่เกี่ยวข้อง

**เทคนิค 3: Re-ranking**

หลังจาก retrieve chunks มาแล้ว ลำดับที่ vector search ส่งมาอาจไม่ใช่ลำดับที่ดีที่สุดเสมอ

Re-ranking คือการนำ chunks ที่ retrieve มา (เช่น top 10) แล้วใช้โมเดลอีกตัวหนึ่งที่เชี่ยวชาญด้านการประเมิน relevance มาจัดลำดับใหม่ ดูเฉพาะ top 3 ที่ผ่านการ re-rank แล้วส่งให้ LLM

```python
# ตัวอย่างใช้ Cohere reranker
import cohere
co = cohere.Client("your-cohere-api-key")

# ดึง top 10 จาก vector search ก่อน
results = collection.query(query_embeddings=[query_embedding], n_results=10)
candidate_chunks = results["documents"][0]

# Re-rank ด้วย Cohere
rerank_results = co.rerank(
    query=question,
    documents=candidate_chunks,
    top_n=3,  # เอาแค่ 3 อันดับแรกหลัง re-rank
    model="rerank-multilingual-v3.0"
)

# ใช้เฉพาะ chunks ที่ผ่านการ re-rank
final_chunks = [r.document for r in rerank_results.results]
```

> 🧪 **ลองทำดู:** เพิ่ม metadata ให้กับ documents ของคุณ เช่น category, date, author แล้วลอง query พร้อม where filter ดูว่าผลลัพธ์แม่นยำขึ้นไหมเมื่อเทียบกับ query โดยไม่มี filter

## ข้อผิดพลาดที่พบบ่อยใน RAG

ถ้าสร้าง RAG แล้วคุณภาพไม่ดีอย่างที่หวัง มักเกิดจากปัญหาเหล่านี้

**ข้อผิดพลาด 1: Chunk ใหญ่หรือเล็กเกินไป**

Chunk เล็กเกินไป (&lt; 100 token): แต่ละ chunk ไม่มีบริบทพอ LLM ได้รับข้อมูลที่กระท่อนกระแท่น ตอบไม่ได้สมบูรณ์

Chunk ใหญ่เกินไป (> 1,000 token): เหมือนส่งบทความทั้งบทให้ LLM เมื่อถามแค่ข้อเดียว — ข้อมูลที่เกี่ยวข้องจริง ๆ จมอยู่ในกองข้อมูลอื่น และ embedding ก็ไม่แม่นพอ

**วิธีแก้**: เริ่มต้นที่ 256-512 token แล้วปรับตามผลทดสอบ

**ข้อผิดพลาด 2: Embedding Model ไม่รองรับภาษาไทยดีพอ**

บาง embedding model ถูก train ด้วยข้อมูลภาษาอังกฤษเป็นหลัก ความสามารถในการเข้าใจความหมายภาษาไทยจึงด้อยกว่า

**วิธีแก้**: ใช้ `text-embedding-3-small` หรือ `text-embedding-3-large` ของ OpenAI ซึ่งรองรับภาษาไทยได้ดี หรือ Cohere Embed v3 ที่เน้น multilingual โดยเฉพาะ

**ข้อผิดพลาด 3: ไม่มี Citation หรือ Source Tracking**

ผู้ใช้ได้รับคำตอบแต่ไม่รู้ว่ามาจากไหน ถ้า AI ตอบผิดก็ตรวจสอบไม่ได้ ความน่าเชื่อถือของระบบลดลง

**วิธีแก้**: เก็บ metadata ของแหล่งที่มาพร้อมกับ chunk และส่งข้อมูลแหล่งที่มาให้ LLM แสดงในคำตอบด้วย

```python
# เก็บ metadata แหล่งที่มา
collection.add(
    documents=chunks,
    metadatas=[{"source": "employee_handbook_v3.pdf", "page": 47}],
    # ...
)

# ใน prompt บอก LLM ให้อ้างอิงแหล่งที่มา
system_prompt = """ตอบคำถามจากเอกสารที่ให้มา
และระบุแหล่งที่มาในรูปแบบ [ที่มา: ชื่อเอกสาร] ท้ายคำตอบเสมอ"""
```

**ข้อผิดพลาด 4: LLM ยังหลอนแม้จะให้ Context**

บางครั้ง LLM จะผสมความรู้ทั่วไปของมันเข้ากับ context ที่ให้ไป ทำให้คำตอบมีส่วนที่ไม่ได้มาจากเอกสารของคุณ

**วิธีแก้**: เข้มงวดกับ system prompt

```python
system_prompt = """คุณตอบได้เฉพาะจากเอกสารที่ให้มาในบริบทเท่านั้น
ห้ามใช้ความรู้ทั่วไปของตัวเองเด็ดขาด
ถ้าเอกสารที่ให้มาไม่มีคำตอบสำหรับคำถามนี้
ให้ตอบว่า "ไม่พบข้อมูลนี้ในเอกสารที่มี" เท่านั้น"""
```

<!-- IMAGE: ตาราง "ปัญหา → สาเหตุ → วิธีแก้" สำหรับ 4 ข้อผิดพลาดที่พบบ่อย -->

> 💡 **Key insight:** RAG ที่ดีไม่ได้เกิดจาก LLM ที่ฉลาดกว่า แต่เกิดจาก retrieval ที่แม่นยำ ถ้า chunks ที่ดึงมาถูกต้อง LLM ก็ตอบได้ดีเกือบทุกครั้ง ปัญหาส่วนใหญ่อยู่ที่ upstream — chunking และ embedding — ไม่ใช่ที่ LLM

## RAG vs Fine-tuning vs Long Context — เมื่อไหร่ใช้อะไร

นอกจาก RAG ยังมีอีกสองวิธีที่คนมักสับสน เรามาดูว่าแต่ละวิธีเหมาะกับสถานการณ์ไหน

**Long Context Window**

โมเดลรุ่นใหม่มี context window ใหญ่มาก เช่น Gemini 2.5 Pro รองรับ 1 ล้าน token ซึ่งเทียบเท่าหนังสือ 700 หน้า บางคนจึงคิดว่า "ยัดทุกอย่างเข้าไปเลยดีกว่า"

แต่มีข้อเสียคือ: (1) แพงมากเพราะคิดค่าตาม token ที่ส่ง ทุกคำถามต้องส่งเอกสารทั้งหมด (2) ช้ากว่า (3) LLM มักมีปัญหา "lost in the middle" — ข้อมูลที่อยู่กลาง context มักถูกมองข้าม (4) ยังไม่แก้ปัญหา private data ที่อัปเดตตลอดเวลา

**Fine-tuning**

Fine-tuning คือการนำ pre-trained model มา train ต่อด้วยข้อมูลของเรา เพื่อให้โมเดลปรับ "บุคลิก" หรือรูปแบบการตอบให้ตรงกับงานของเรา

แต่ fine-tuning ไม่เหมาะสำหรับ "สอนความรู้ใหม่" ให้ LLM เพราะ:
- ข้อมูล fine-tuning ถูก bake เข้าไปในโมเดล ถ้าข้อมูลเปลี่ยน ต้อง fine-tune ใหม่ทั้งหมด
- ราคาแพงและใช้เวลานาน
- โมเดลยังคง hallucinate ถ้าถามนอกเหนือ fine-tuning data

| วิธี | เหมาะกับ | ไม่เหมาะกับ | ต้นทุน |
|------|----------|------------|--------|
| **RAG** | ข้อมูลที่อัปเดตบ่อย, เอกสารองค์กร, ต้องการ citation | ความรู้ที่อยาก bake เข้าโมเดล | ต่ำ-กลาง |
| **Fine-tuning** | ปรับ tone/format, งานที่มี pattern ชัดเจน | ข้อมูลที่เปลี่ยนบ่อย | สูง |
| **Long Context** | เอกสารไม่กี่ชิ้น, วิเคราะห์ครั้งเดียว | ระบบที่ query บ่อยมาก | สูงมาก |

**สรุปง่าย ๆ ว่าเมื่อไหร่ใช้อะไร:**

ใช้ RAG เมื่อ:
- มีเอกสารหรือ knowledge base ที่อัปเดตบ่อย
- ต้องการให้ LLM ตอบจากข้อมูลเฉพาะขององค์กร
- ต้องการ citation และตรวจสอบแหล่งที่มาได้
- ข้อมูลเป็น private และไม่ควร train เข้าโมเดลสาธารณะ

ใช้ Fine-tuning เมื่อ:
- ต้องการปรับ tone/format/style การตอบ
- มี task เฉพาะทางที่ต้องการให้ LLM เชี่ยวชาญมาก ๆ
- ข้อมูลไม่ค่อยเปลี่ยน

ใช้ Long Context เมื่อ:
- มีเอกสารไม่กี่ชิ้นที่ต้องวิเคราะห์ครั้งเดียว
- ไม่ได้ query บ่อย
- ต้องการความสัมพันธ์ระหว่างส่วนต่าง ๆ ในเอกสารที่ chunking อาจทำให้สูญเสียไป

<!-- IMAGE: Decision flowchart: "ข้อมูลอัปเดตบ่อยไหม?" → ใช่ → RAG | "ต้องการปรับ style ไหม?" → ใช่ → Fine-tuning | "เอกสารน้อยชิ้น วิเคราะห์ครั้งเดียว?" → ใช่ → Long Context -->

> 💡 **Key insight:** ในโลกจริง สามวิธีนี้ไม่ได้แข่งกัน บริษัทหลายแห่งใช้ทั้งสามร่วมกัน เช่น fine-tune โมเดลให้ตอบเป็น tone ของแบรนด์ แล้วเชื่อมกับ RAG เพื่อดึงข้อมูลล่าสุด

## Pattern ขั้นสูงที่ควรรู้

เมื่อ RAG พื้นฐานใช้งานได้แล้ว ต่อไปนี้คือ pattern ที่ทำให้ระบบฉลาดขึ้นอีกขั้น

**Hybrid Search: ผสม Vector Search กับ Keyword Search**

Vector search ดีในการเข้าใจความหมาย แต่บางครั้งผู้ใช้ค้นหาด้วย keyword เฉพาะ เช่น ชื่อผลิตภัณฑ์ รหัสสินค้า หรือคำศัพท์เฉพาะทาง

Hybrid search ผสมทั้งสองวิธี:
- ค้นหาด้วย vector (ความหมาย)
- ค้นหาด้วย BM25 (keyword matching แบบคลาสสิก)
- รวมผลลัพธ์ทั้งสองด้วย weighted combination

วิธีนี้ให้ผลดีกว่าในหลายกรณี โดยเฉพาะเอกสารที่มีคำศัพท์เฉพาะทางจำนวนมาก

**Contextual Compression**

แทนที่จะส่ง chunks ทั้งก้อนให้ LLM ใช้ LLM อีกตัวหนึ่ง (หรือตัวเดียวกันแต่เล็กกว่า) กรองเฉพาะส่วนที่เกี่ยวข้องกับคำถามออกจาก chunk ก่อน แล้วจึงส่งส่วนที่กรองแล้วให้ LLM หลักตอบ

ผลคือ LLM หลักได้รับข้อมูลที่ concise ขึ้น โอกาส hallucination ลดลง และประหยัด token ด้วย

**Multi-vector Retrieval**

สร้างหลาย representation สำหรับ document เดียวกัน เช่น:
- Embed ข้อความต้นฉบับ
- Embed สรุปของ chunk
- Embed คำถามที่ chunk นี้น่าจะตอบได้

เมื่อ query มา ค้นหาจาก multiple vector แล้วดึง chunk ต้นฉบับมาให้ LLM

## RAG ในชีวิตจริง — Use Cases ที่ใช้กันทั่วไป

เพื่อให้เห็นภาพว่า RAG ถูกนำไปใช้จริงอย่างไร นี่คือตัวอย่างที่พบบ่อยที่สุด

**Customer Support Bot**

บริษัทเชื่อม RAG กับ knowledge base ของ Support ทั้งหมด เมื่อลูกค้าถาม bot จะหา answer จากบทความ FAQ เอกสารสินค้า และ ticket ที่เคยแก้ไขแล้ว แทนที่จะ hallucinate คำตอบ

ผลคือ resolution rate สูงขึ้น เวลาตอบเร็วขึ้น และ escalation ไป human agent ลดลง

**Legal Document Analysis**

ทนายและ paralegal ใช้ RAG เพื่อค้นหาข้อมูลจากสัญญาหลายร้อยฉบับ แทนที่จะอ่านทีละหน้า ถามว่า "ในสัญญาไหนมีเงื่อนไขเรื่อง force majeure?" แล้วได้คำตอบพร้อม reference ทันที

**Internal Knowledge Assistant**

บริษัทเชื่อม RAG กับ Confluence, Notion, Google Drive ทั้งหมด พนักงานใหม่ถามได้ทุกอย่างโดยไม่ต้องรบกวนเพื่อนร่วมงาน และได้คำตอบที่อัปเดตล่าสุดเสมอ

**Medical Knowledge Base**

โรงพยาบาลและ healthtech บริษัทใช้ RAG เพื่อให้แพทย์ค้นหาข้อมูลทางการแพทย์จาก literature ล่าสุด แนวทางการรักษา และ protocol ของโรงพยาบาล โดยมั่นใจว่าข้อมูลที่ได้มาจากแหล่งที่น่าเชื่อถือ ไม่ใช่ความรู้ทั่วไปของ LLM

## สรุป

RAG เป็นหนึ่งใน pattern ที่สำคัญที่สุดใน AI engineering เพราะมันแก้ปัญหาพื้นฐานของ LLM ได้ตรงจุด: ทำให้ AI ตอบจากข้อมูลของคุณโดยเฉพาะ ไม่ใช่จากความจำทั่วไปที่อาจล้าสมัยหรือผิด

ทบทวนสิ่งที่เรียนรู้ในบทนี้:
- LLM มีข้อจำกัดหลายอย่าง: knowledge cutoff, hallucination, context window, private data
- RAG แก้ปัญหาด้วย 3 ขั้นตอน: Index → Retrieve → Generate
- Chunking ที่ดีคือรากฐานของ RAG ที่ดี ขนาด chunk ส่งผลโดยตรงต่อคุณภาพ
- Embedding model ต้องเข้ากับภาษาของข้อมูล
- Vector store ทำให้ค้นหาด้วยความหมายได้อย่างมีประสิทธิภาพ
- RAG ดีที่สุดสำหรับข้อมูลที่อัปเดตบ่อยและต้องการ citation

เมื่อคุณสร้าง RAG ได้แล้ว ขั้นตอนถัดไปคือ: จะเกิดอะไรขึ้นถ้า AI ไม่ได้แค่ตอบคำถาม แต่ต้องวางแผน ตัดสินใจ และใช้เครื่องมือหลายตัวร่วมกันเพื่อทำงานที่ซับซ้อน?

นั่นคือสิ่งที่ **AI Agents** ทำ — และมันเปลี่ยนวิธีที่เราสร้างระบบ AI ไปอีกระดับ อ่านต่อได้ที่บท [AI Agents คืออะไร](/building-with-ai/agents-and-orchestration)
