คุณพิมพ์ "อาหารไทยอร่อย ๆ ย่านสีลม" ลงไปในช่องค้นหา แล้วระบบแสดงร้าน "ส้มตำนัว" ขึ้นมาเป็นอันดับต้น ๆ — ทั้งที่ในชื่อร้านหรือคำอธิบายไม่มีคำว่า "อร่อย" หรือ "สีลม" เลยสักคำ แล้วคอมพิวเตอร์รู้ได้ยังไงว่าร้านนี้คือสิ่งที่คุณกำลังหา?

หรือลองนึกถึงตอนที่คุณถามช่วย ChatGPT ว่า "มีไอเดียอาหารเย็นอะไรบ้างที่ทำง่ายและดีต่อสุขภาพ" — ChatGPT ตอบถูกทิศทางทันที ทั้งที่คำถามของคุณไม่ได้บอกว่าอาหารประเภทไหน, กี่คน, หรือมีวัตถุดิบอะไรในบ้าน มันเข้าใจ "เจตนา" ไม่ใช่แค่ตัวอักษร

คำตอบคือระบบไม่ได้ค้นหาจาก "คำ" — มันค้นหาจาก "ความหมาย" และเครื่องมือที่ทำให้คอมพิวเตอร์เข้าใจความหมายได้นั้น มีชื่อว่า **Embedding**

บทนี้จะพาคุณเข้าใจว่า embedding คืออะไร ทำงานยังไง และจะเอาไปใช้สร้างระบบ semantic search จริง ๆ ด้วย code ได้ยังไง — ไม่ต้องมีพื้นฐาน data science มาก่อน

## ก่อนจะไปถึง Embedding — ปัญหาของการค้นหาแบบเก่า

ลองนึกถึงวิธีที่ Google ยุคเริ่มต้นทำงาน (และยังคงทำอยู่บางส่วนจนถึงทุกวันนี้) — มันค้นหา "คำ" ที่ตรงกันระหว่างสิ่งที่คุณพิมพ์กับเอกสารที่มีอยู่ในฐานข้อมูล วิธีนี้เรียกว่า **keyword search** หรือ **exact match search**

ฟังดูสมเหตุสมผลใช่ไหม? แต่มันมีปัญหาใหญ่สองอย่างที่แก้ไม่ได้

**ปัญหาที่หนึ่ง: คำเดียวกัน ความหมายต่างกัน**

ลองดูคำว่า "แบงค์" — คุณหมายถึงอะไร?

- ธนาคาร (bank) — "ไปแบงค์กสิกรมา"
- ธนบัตร — "มีแบงค์ร้อยไหม"
- ชื่อคน — "แบงค์โทรมาเมื่อกี้"

ถ้าคุณค้นหาว่า "แบงค์ใกล้ฉัน" แต่ระบบไม่รู้ว่าคุณหมายถึงธนาคารหรือคนชื่อแบงค์ มันจะแสดงผลอะไรออกมาดี? keyword search ทำได้แค่นับว่าคำนั้น "ปรากฏ" ในเอกสารไหนบ้าง ไม่ได้เข้าใจว่าคุณ "ต้องการ" อะไร

ภาษาอังกฤษก็มีปัญหาเดียวกัน คำว่า "bank" อาจหมายถึงธนาคาร, ฝั่งแม่น้ำ, หรือการเลี้ยวของเครื่องบิน คำว่า "bat" อาจเป็นค้างคาวหรือไม้ตีเบสบอล

**ปัญหาที่สอง: ความหมายเหมือนกัน คำต่างกัน**

ลองนึกว่าคุณมีเอกสารเกี่ยวกับ "ความสุข" แต่ผู้ใช้ค้นหาด้วยคำว่า "happy" — keyword search จะไม่เจออะไรเลย เพราะในเอกสารใช้คำว่า "happiness", "joyful", "elated", "content", "delighted"

หรือถ้าคุณค้นหาว่า "วิธีลดน้ำหนัก" แต่บทความที่ดีที่สุดใช้คำว่า "เทคนิคควบคุมน้ำหนัก" — ก็หาไม่เจออีก

นี่คือเหตุผลที่ keyword search มีข้อจำกัดชัดเจน: **มันเข้าใจคำ แต่ไม่เข้าใจความหมาย**

<!-- IMAGE: ตารางเปรียบเทียบ keyword search vs semantic search: keyword search หาได้เฉพาะคำตรงกัน, semantic search หาได้จากความหมายใกล้เคียง -->

> 💡 **Key insight:** ภาษามนุษย์ไม่ได้ทำงานด้วย "คำ" — มันทำงานด้วย "ความหมาย" สองประโยคที่ไม่มีคำร่วมกันเลยอาจหมายความเดียวกัน และสองประโยคที่มีคำเหมือนกันทุกตัวอาจหมายตรงข้ามกัน ("ฉันชอบมาก" vs "ฉันไม่ชอบมาก")

## Embedding คืออะไร — แปลงความหมายเป็นพิกัด

ก่อนจะเข้าใจ embedding ลองคิดถึงสิ่งที่คุณรู้จักดีอย่างหนึ่งก่อน: **พิกัด GPS**

ร้านอาหารทุกร้านในกรุงเทพมีพิกัด GPS — เช่น ร้านส้มตำนัวอาจอยู่ที่ (13.7214, 100.5291) ซึ่งก็คือ latitude 13.7214 และ longitude 100.5291 พิกัดสองตัวเลขนี้บอกตำแหน่งของร้านในโลกจริงได้แม่นยำ

ทีนี้ ถ้าคุณอยู่ที่ถนนสีลมและอยากหาร้านอาหารใกล้ ๆ ระบบทำได้ง่ายมาก: เอาพิกัดของคุณมาเปรียบเทียบกับพิกัดของทุกร้าน แล้วเรียงลำดับตามระยะห่าง ร้านไหนพิกัดใกล้กัน ก็อยู่ใกล้กันในโลกจริง

**Embedding ทำแบบเดียวกันนี้ — แต่กับ "ความหมาย" แทนที่จะเป็น "ตำแหน่ง"**

แทนที่จะเป็น 2 มิติ (latitude, longitude) embedding แปลงข้อความทุกชิ้นเป็น "พิกัดในโลกความหมาย" ที่มีหลายร้อยหรือหลายพัน มิติ ข้อความที่มีความหมายใกล้เคียงกันจะได้พิกัดใกล้กัน ข้อความที่หมายความต่างกันจะได้พิกัดไกลกัน

พิกัดเหล่านี้ในทางเทคนิคเรียกว่า **vector** — ซึ่งก็คือ list of numbers (รายการตัวเลข) นั่นเอง

ตัวอย่างง่าย ๆ ถ้า embedding เป็นแบบ 4 มิติ (ของจริงมีมากกว่านี้มาก) คำว่า "แมว" อาจได้ vector ว่า `[0.82, 0.15, 0.91, 0.33]` และคำว่า "สุนัข" อาจได้ `[0.79, 0.18, 0.88, 0.31]` — ตัวเลขใกล้เคียงกันมาก เพราะทั้งสองคำมีความหมายใกล้กัน (ทั้งคู่เป็นสัตว์เลี้ยง สัตว์ที่คนรัก มีขน ฯลฯ)

แต่คำว่า "รถยนต์" อาจได้ `[0.11, 0.94, 0.22, 0.77]` — ตัวเลขต่างออกไปมาก เพราะความหมายต่างออกไปมาก

<!-- IMAGE: แผนภาพแสดงคำสามคำ (แมว, สุนัข, รถยนต์) พร้อม vector ของแต่ละคำ และแสดงว่าแมวกับสุนัขมีตัวเลขใกล้กัน รถยนต์ต่างออกไป -->

> 💡 **Key insight:** Embedding ไม่ได้ "จำ" ว่าคำไหนเกี่ยวกับอะไร — มันถูก "สอน" โดยการดูข้อความหลายพันล้านประโยคว่าคำไหนมักปรากฏในบริบทเดียวกัน คำที่ปรากฏร่วมกันบ่อย ๆ จะได้ vector ใกล้กัน นั่นคือรากฐานของ "ความหมาย" สำหรับคอมพิวเตอร์

## Vector Space — โลกที่ความหมายมีพิกัด

มาลองจินตนาการ Vector Space ในแบบ 2 มิติก่อน เพื่อให้เห็นภาพชัดขึ้น

สมมติเราเอาคำต่าง ๆ มาวางในระนาบ xy โดยที่แกน x แทน "ความเป็นสิ่งมีชีวิต" (ซ้าย = ไม่มีชีวิต, ขวา = มีชีวิต) และแกน y แทน "ขนาด" (ล่าง = เล็ก, บน = ใหญ่)

ในจินตนาการนี้:
- "มด" อยู่แถวขวาล่าง — มีชีวิต แต่เล็กมาก
- "ช้าง" อยู่แถวขวาบน — มีชีวิต และใหญ่มาก
- "รถบรรทุก" อยู่แถวซ้ายบน — ไม่มีชีวิต แต่ใหญ่
- "ดินสอ" อยู่แถวซ้ายล่าง — ไม่มีชีวิต และเล็ก

ทีนี้ถ้าคุณถาม "สิ่งมีชีวิตตัวใหญ่" ระบบจะมองหาพิกัดที่อยู่บนขวาของระนาบ — และเจอ "ช้าง" ทันที โดยที่ไม่ต้องมีคำว่า "ช้าง" ในคำถามเลย

**แต่ในความเป็นจริง ไม่ใช่ 2 มิติ — มันมี 1,536 มิติ**

OpenAI's text-embedding-3-small สร้าง vector ขนาด 1,536 มิติ หมายความว่าข้อความทุกชิ้นถูกแทนด้วยตัวเลข 1,536 ตัว แต่ละมิติแทน "แง่มุม" ของความหมายที่แตกต่างกัน — บางมิติอาจแทน "ความเป็นสิ่งมีชีวิต" บางมิติแทน "อารมณ์เชิงบวก/ลบ" บางมิติแทน "ความเป็นทางการ" และอีกกว่าพันมิติที่เราไม่รู้ว่าแทนอะไร เพราะโมเดลเรียนรู้มันเองโดยไม่มีคำอธิบาย

ฟังดูซับซ้อน แต่ไม่ต้องกังวล คุณไม่ต้องเข้าใจทุกมิติ — คุณแค่ต้องรู้ว่า "ยิ่งตัวเลขของสอง vector ใกล้กัน ความหมายก็ยิ่งใกล้กัน"

คิดภาพง่าย ๆ แบบนี้: สมมติเราลดให้เหลือแค่ 3 มิติเพื่อดูได้ เราอาจจินตนาการว่าแต่ละคำลอยอยู่ใน "ลูกโลก" สามมิติ คำว่า "กษัตริย์" กับ "ราชินี" ลอยอยู่ใกล้กันในพื้นที่ที่เกี่ยวกับ "ราชวงศ์" คำว่า "หมาป่า" กับ "สุนัข" ลอยอยู่ใกล้กันในพื้นที่ที่เกี่ยวกับ "สัตว์ตระกูลหมา" และกลุ่มเหล่านี้อยู่ห่างจากพื้นที่ "ยานพาหนะ" ที่มี "รถยนต์" "มอเตอร์ไซค์" "เรือ" ลอยอยู่รวมกัน

เมื่อ dimension เพิ่มขึ้นเป็น 1,536 โมเดลสามารถจับความสัมพันธ์ที่ละเอียดอ่อนกว่านั้นมากได้ — เช่น "โกรธ" กับ "หัวร้อน" อยู่ใกล้กัน แต่ "โกรธ" กับ "เศร้า" ก็อยู่ในบริเวณ "อารมณ์ด้านลบ" เหมือนกัน เพียงแต่อยู่คนละส่วนของบริเวณนั้น

<!-- IMAGE: ภาพ 2D scatter plot แสดงคำหลายคำในพื้นที่ 2 มิติ แมว/สุนัข/แมวบ้านอยู่กลุ่มเดียวกัน, รถยนต์/มอเตอร์ไซค์/จักรยานอยู่กลุ่มเดียวกัน, ความสุข/ยินดี/ดีใจอยู่กลุ่มเดียวกัน -->

สิ่งที่น่าทึ่งกว่านั้น — Vector Space ยังมีคุณสมบัติที่เรียกว่า **analogical reasoning** ถ้าคุณเคยได้ยิน vector arithmetic ที่โด่งดัง:

```
king - man + woman ≈ queen
```

หมายความว่าถ้าเอา vector ของคำว่า "king" ลบ vector "man" แล้วบวก vector "woman" ผลลัพธ์จะออกมาใกล้เคียงกับ vector ของคำว่า "queen" มากที่สุด — เป็นหลักฐานว่า vector space นั้น "เข้าใจ" ความสัมพันธ์ระหว่างแนวคิดจริง ๆ ไม่ใช่แค่จำคำ

ทำไมถึงเป็นแบบนี้ได้? เพราะโมเดลเรียนรู้จากการอ่านข้อความหลายพันล้านประโยค มันเห็นซ้ำแล้วซ้ำเล่าว่า "king" และ "queen" มักปรากฏในบริบทเดียวกัน แต่ "king" มักถูกพูดถึงกับ "he/his" ขณะที่ "queen" มักถูกพูดถึงกับ "she/her" — โมเดลเรียนรู้โครงสร้างนี้โดยอัตโนมัติ และโครงสร้างนั้นถูกฝังอยู่ใน vector ทุกตัว

ตัวอย่างที่น่าสนใจอื่น ๆ ของ vector arithmetic:

```
ปารีส - ฝรั่งเศส + ไทย ≈ กรุงเทพ
(เมืองหลวงของประเทศ)

ซูชิ - ญี่ปุ่น + ไทย ≈ ผัดไทย
(อาหารประจำชาติ)

แพทย์ - โรงพยาบาล + โรงเรียน ≈ ครู
(ผู้เชี่ยวชาญในสถาบัน)
```

ผลลัพธ์ไม่ได้แม่น 100% เสมอไป แต่มันแม่นมากพอที่จะแสดงให้เห็นว่า vector space มีโครงสร้างที่มีความหมาย — ไม่ใช่แค่ตัวเลขสุ่ม

## Similarity — วัดความคล้ายยังไง

เมื่อเรามี vector แล้ว คำถามต่อไปคือ "วัดว่าสอง vector ใกล้กันแค่ไหน?" มีสองวิธีหลักที่ใช้กัน

### Cosine Similarity — วัดมุม ไม่ใช่ระยะทาง

วิธีที่ใช้บ่อยที่สุดสำหรับ text embedding คือ **cosine similarity** ซึ่งวัดจาก "มุม" ระหว่าง vector สอง เส้น แทนที่จะวัดระยะห่าง

ลองนึกภาพ vector เป็นลูกศรที่ยิงออกจากจุดกำเนิด (0, 0) ถ้าลูกศรสองอันชี้ไปทิศทางเดียวกัน (มุมน้อย) ความหมายก็ใกล้กัน ถ้าชี้ไปทิศทางตั้งฉากกัน (มุม 90 องศา) ความหมายไม่เกี่ยวกัน ถ้าชี้ไปทิศทางตรงข้าม (มุม 180 องศา) ความหมายตรงข้ามกัน

ค่า cosine similarity วิ่งระหว่าง **-1 ถึง 1**:
- **1.0** — เหมือนกันทุกประการ (ลูกศรชี้ทิศเดียวกันพอดี)
- **0.5-0.9** — คล้ายกัน, เกี่ยวข้องกัน
- **0.0** — ไม่มีความเกี่ยวข้องกัน (ลูกศรตั้งฉาก)
- **-1.0** — ตรงข้ามกันสมบูรณ์

ตัวอย่างค่า cosine similarity ในชีวิตจริง (ค่าประมาณ):
- "แมว" กับ "สุนัข": ~0.85 (สัตว์เลี้ยง คล้ายกันมาก)
- "แมว" กับ "เสือ": ~0.78 (วงศ์เดียวกัน แต่ context ต่างกัน)
- "แมว" กับ "รถยนต์": ~0.12 (ไม่เกี่ยวกัน)
- "ร้อน" กับ "เย็น": อาจติดลบเล็กน้อย (ตรงข้าม)

สูตรคำนวณ cosine similarity มีแค่นี้:

```
cosine_similarity = (A · B) / (|A| × |B|)
```

โดยที่ A · B คือ dot product (คูณทีละตำแหน่งแล้วบวกกัน) และ |A|, |B| คือ magnitude (ขนาด) ของแต่ละ vector

<!-- IMAGE: แผนภาพแสดงสอง vector เป็นลูกศร พร้อมมุมระหว่างกัน และตารางแสดงมุม → ค่า cosine similarity -->

### Euclidean Distance — วัดระยะทาง

อีกวิธีหนึ่งคือ **Euclidean distance** ซึ่งวัดระยะห่างตรง ๆ ระหว่างจุดสองจุดใน space เหมือนไม้บรรทัดในโลก 2 มิติ แค่ขยายไปหลาย dimension ยิ่งระยะทางน้อย ยิ่งคล้ายกัน

สูตรในโลก 2 มิติที่คุณอาจจำได้จากวิชาคณิตศาสตร์:
```
distance = √((x₂-x₁)² + (y₂-y₁)²)
```

ใน 1,536 มิติก็คล้ายกัน เพียงแต่มีพจน์ 1,536 พจน์แทนที่จะเป็น 2 พจน์

สำหรับ text embedding นิยมใช้ cosine similarity มากกว่า เพราะมันไม่สนใจ "ขนาด" ของ vector — ประโยคสั้นและประโยคยาวที่มีความหมายเดียวกันจะยังได้ cosine similarity สูง แม้ว่า vector ของประโยคยาวจะมีค่าสูงกว่าโดยรวม

ลองคิดแบบนี้: ถ้าคุณลากเส้นจากจุดกำเนิดไปหาจุด A และจุด B สองเส้นที่ชี้ทิศเดียวกัน (มุมน้อย) หมายถึงความหมายคล้ายกัน — ไม่ว่าเส้นจะยาวสั้นแค่ไหนก็ตาม Euclidean distance วัดความยาวของเส้นตรงจากจุด A ไปจุด B ซึ่งอาจถูก influence โดยขนาดของ vector ด้วย ไม่ใช่แค่ทิศทาง

ทั้งสองวิธีใช้ได้ในบริบทที่ต่างกัน แต่สำหรับ semantic search และ RAG นิยม cosine similarity เป็นค่า default

> 💡 **Key insight:** Cosine similarity วัด "ทิศทาง" ของความหมาย ไม่ใช่ "ขนาด" ทำให้เหมาะกับการเปรียบเทียบข้อความที่มีความยาวต่างกัน

## Embedding Models — ใครสร้าง Vector ให้เรา

การสร้าง embedding ไม่ใช่อะไรที่เราเขียนเองได้ตั้งแต่ต้น — เราใช้ **embedding model** ที่ถูกเทรนมาแล้วด้วยข้อความมหาศาล ให้มันแปลงข้อความเป็น vector ให้เรา

**Embedding model ถูกสร้างมายังไง?**

โดยย่อ: โมเดลถูกเทรนด้วยข้อความจำนวนมหาศาลจาก internet — บทความ, หนังสือ, เว็บไซต์, บทสนทนา เป้าหมายของการเทรนคือทำให้โมเดล "ทาย" ได้ว่าคำไหนมักอยู่ใกล้คำไหน เช่น ถ้าเห็นประโยค "ฉันพา ___ ไปหาหมอ" โมเดลควรทายได้ว่าช่องว่างนั้นมีโอกาสสูงที่จะเป็น "สุนัข" "แมว" "ลูก" มากกว่า "รถยนต์" หรือ "กาแฟ"

กระบวนการเทรนซ้ำหลายพันล้านครั้งนี้ทำให้โมเดลสร้าง vector ที่สะท้อนโครงสร้างความหมายของภาษาได้ โดยไม่ต้องมีคนไปกำหนดว่า "แมว" และ "สุนัข" เกี่ยวข้องกัน — โมเดลเรียนรู้เองจากการที่เห็นคำสองคำนี้ปรากฏในบริบทคล้ายกันซ้ำแล้วซ้ำเล่า

นั่นคือเหตุผลที่ embedding model ที่เทรนด้วยข้อมูลภาษาไทยน้อยกว่าจะมีประสิทธิภาพต่ำกว่าในภาษาไทย — ถ้าโมเดลเห็นข้อความไทยน้อย มันก็เรียนรู้โครงสร้างความหมายของภาษาไทยได้น้อยกว่า

## ลองทำดู #1: สร้าง Embedding ด้วย OpenAI API

ถึงเวลาลงมือทำจริง เราจะสร้าง embedding จากข้อความจริง แล้วเปรียบเทียบ similarity ระหว่างประโยคต่าง ๆ

ก่อนเริ่มต้องติดตั้ง library:

```bash
pip install openai numpy
```

และต้องมี OpenAI API key (ดูวิธีสมัครได้ที่ [บทเรื่อง API](/building-with-ai/llm-apis))

```python
import openai
import numpy as np

# ตั้งค่า API key
client = openai.OpenAI(api_key="sk-...")  # ใส่ key ของคุณ

def get_embedding(text: str) -> list[float]:
    """แปลงข้อความเป็น vector ด้วย OpenAI embedding"""
    response = client.embeddings.create(
        model="text-embedding-3-small",  # ใช้ model เล็กก่อน
        input=text
    )
    # response.data[0].embedding คือ list of 1,536 ตัวเลข
    return response.data[0].embedding

def cosine_similarity(vec_a: list, vec_b: list) -> float:
    """คำนวณ cosine similarity ระหว่าง vector สองเส้น"""
    a = np.array(vec_a)
    b = np.array(vec_b)
    # dot product หารด้วย product ของ magnitude ทั้งสอง
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# ทดสอบด้วยประโยคต่าง ๆ
sentences = [
    "ฉันชอบกินอาหารไทย",        # ประโยคหลัก
    "ผมโปรดปรานอาหารไทยมาก",    # ความหมายเดียวกัน คำต่างกัน
    "Thai food is my favorite",   # ภาษาอังกฤษ ความหมายเดียวกัน
    "วันนี้อากาศร้อนมาก",         # คนละเรื่องกันเลย
]

# สร้าง embedding ให้ทุกประโยค
print("กำลังสร้าง embeddings...")
embeddings = [get_embedding(s) for s in sentences]
print(f"แต่ละ embedding มีขนาด: {len(embeddings[0])} มิติ")
print()

# เปรียบเทียบประโยคแรกกับทุกประโยคที่เหลือ
reference = sentences[0]
reference_embedding = embeddings[0]

print(f"เปรียบเทียบกับ: '{reference}'")
print("-" * 50)

for i, (sentence, embedding) in enumerate(zip(sentences, embeddings)):
    if i == 0:
        continue  # ข้ามประโยคแรก (เป็นตัวเอง)
    
    similarity = cosine_similarity(reference_embedding, embedding)
    print(f"'{sentence}'")
    print(f"  → similarity: {similarity:.4f}")
    print()
```

```

แต่ละ embedding มีขนาด: 1536 มิติ

เปรียบเทียบกับ: 'ฉันชอบกินอาหารไทย'
--------------------------------------------------
'ผมโปรดปรานอาหารไทยมาก'
  → similarity: 0.9234   ← สูงมาก! ความหมายเดียวกัน

'Thai food is my favorite'
  → similarity: 0.8891   ← สูง! แม้เป็นคนละภาษา

'วันนี้อากาศร้อนมาก'
```

## ลองทำดู #2: Semantic Search ง่าย ๆ

ถึงตรงนี้เราเข้าใจแล้วว่า embedding คืออะไรและ similarity วัดยังไง ตอนนี้มาประกอบร่างเป็น **semantic search** จริง ๆ — ระบบค้นหาที่เข้าใจความหมาย

สมมติเรามีบทความ 10 ชิ้นในฐานข้อมูล และต้องการหาบทความที่เกี่ยวข้องกับคำค้นหาของผู้ใช้มากที่สุด

```python
import openai
import numpy as np

client = openai.OpenAI(api_key="sk-...")

# ฐานข้อมูลบทความ (ตัวอย่าง)
documents = [
    "วิธีทำผัดไทยแบบฉบับต้นตำรับ ใช้เส้นจันท์นุ่ม กุ้งสด ไข่ไก่ ถั่วงอก",
    "เทคนิคออกกำลังกายเพื่อลดน้ำหนักในเวลา 30 วัน",
    "สูตรต้มยำกุ้งน้ำข้น รสจัดจ้าน ใส่เครื่องแกง",
    "วิธีเริ่มต้นลงทุนในตลาดหุ้นสำหรับมือใหม่",
    "อาหารเช้าที่ดีต่อสุขภาพ 10 เมนู ทำง่าย อิ่มนาน",
    "วิธีเก็บเงินให้ได้ล้านแรกภายใน 5 ปี",
    "สูตรแกงเขียวหวานไก่ ใช้กะทิสด หอมฉุย",
    "เทคนิคการออมเงินสำหรับคนเงินเดือนน้อย",
    "ประโยชน์ของการวิ่งเช้า ทำไมนักวิทยาศาสตร์แนะนำ",
    "ข้าวต้มหมูสับ อาหารเช้าไทย ๆ ที่ทำได้ใน 15 นาที",
]

def get_embedding(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def cosine_similarity(a: list, b: list) -> float:
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# STEP 1: สร้าง embedding ให้ทุก document ล่วงหน้า
# ในระบบจริงขั้นตอนนี้ทำครั้งเดียวแล้วเก็บ vector ไว้
print("กำลัง embed documents ทั้งหมด...")
doc_embeddings = [get_embedding(doc) for doc in documents]
print(f"embed แล้ว {len(doc_embeddings)} documents")
print()

def semantic_search(query: str, top_k: int = 3) -> list[dict]:
    """ค้นหา documents ที่เกี่ยวข้องกับ query มากที่สุด"""
    
    # STEP 2: embed คำค้นหาของผู้ใช้
    query_embedding = get_embedding(query)
    
    # STEP 3: คำนวณ similarity กับทุก document
    scores = []
    for i, doc_embedding in enumerate(doc_embeddings):
        score = cosine_similarity(query_embedding, doc_embedding)
        scores.append({"index": i, "score": score, "text": documents[i]})
    
    # STEP 4: เรียงลำดับจาก similarity สูงสุด
    scores.sort(key=lambda x: x["score"], reverse=True)
    
    # คืนค่า top-k ผลลัพธ์
    return scores[:top_k]

# ทดสอบด้วยคำค้นหาต่าง ๆ
queries = [
    "อยากทำอาหารไทย",
    "ประหยัดเงินและรวย",
    "สุขภาพดีด้วยการออกกำลัง",
]

for query in queries:
    print(f"Query: '{query}'")
    print("=" * 50)
    
    results = semantic_search(query, top_k=3)
    for rank, result in enumerate(results, 1):
        print(f"{rank}. [{result['score']:.3f}] {result['text'][:50]}...")
    
    print()
```

```

==================================================

2. [0.798] สูตรต้มยำกุ้งน้ำข้น รสจัดจ้าน ใส่เครื่องแกง...
3. [0.776] สูตรแกงเขียวหวานไก่ ใช้กะทิสด หอมฉุย...

==================================================

2. [0.834] ประโยชน์ของการวิ่งเช้า ทำไมนักวิทยาศาสตร์แนะนำ...
3. [0.612] อาหารเช้าที่ดีต่อสุขภาพ 10 เมนู ทำง่าย อิ่มนาน...
```

สังเกตว่า "อยากทำอาหารไทย" ดึงเฉพาะบทความอาหาร และ "ประหยัดเงินและรวย" ดึงเฉพาะบทความการเงิน — ทั้งที่คำในคำค้นหาไม่ได้ตรงกับคำในบทความทุกตัว ระบบเข้าใจความหมาย ไม่ใช่แค่ตัวอักษร

## Use Cases ที่สำคัญในชีวิตจริง

เมื่อเข้าใจ embedding แล้ว มาดูว่ามันถูกใช้อย่างไรในระบบจริง ๆ ที่คุณอาจใช้งานอยู่ทุกวัน

### 1. Semantic Search

สิ่งที่เราเพิ่งทำในตัวอย่างด้านบน — ค้นหาเอกสารจากความหมาย ไม่ใช่คำ ใช้ใน:
- ระบบค้นหาภายในองค์กร (ค้นหา policy, SOP, เอกสาร) — พนักงานพิมพ์ว่า "ลาป่วยต้องทำยังไง" และระบบหาเอกสาร HR ที่ถูกต้องได้แม้จะเขียนว่า "ขั้นตอนการขอลากิจเนื่องจากสุขภาพ"
- E-commerce search (ลูกค้าพิมพ์ "รองเท้าใส่สบาย" ระบบหาได้แม้สินค้าไม่ใช้คำว่า "สบาย" แต่ใช้คำว่า "ergonomic", "รองรับอุ้งเท้า", "ลดแรงกระแทก")
- Customer support (ค้นหา FAQ ที่ตรงกับปัญหาลูกค้า — "จ่ายเงินแล้วแต่ไม่ได้รับของ" → ระบบหาเจอ FAQ "การติดตามสถานะคำสั่งซื้อ" และ "การคืนเงินเมื่อสินค้าไม่มาส่ง")

### 2. Recommendation System

Netflix, Spotify, Shopee ใช้ embedding เพื่อแนะนำสินค้าหรือเนื้อหา หลักการคือ: ถ้าคุณชอบ item A และ item B มี embedding ใกล้กับ A คุณน่าจะชอบ B ด้วย

ระบบแนะนำหนังสือ: คุณชอบ "Sapiens" → ระบบหา embedding ของ Sapiens → ค้นหาหนังสืออื่นที่ embedding ใกล้เคียง → แนะนำ "Homo Deus", "The Selfish Gene"

นอกจากนี้ยังใช้ embedding ของ "ผู้ใช้" ได้ด้วย — เอา embedding ของทุก item ที่คุณเคยดูหรือซื้อมา "เฉลี่ย" กัน ได้เป็น "โปรไฟล์ของคุณ" ในรูปแบบ vector จากนั้นหา item ที่ embedding ใกล้กับโปรไฟล์ของคุณมากที่สุด นั่นคือ "บ้าน" ของ content-based recommendation อย่างง่าย

### 3. Clustering และ Categorization

เอาข้อความหลายพัน document มา embed ทั้งหมด แล้วใช้ algorithm เช่น k-means จัดกลุ่มโดยอัตโนมัติ — document ที่ embedding ใกล้กันจะถูกจัดอยู่กลุ่มเดียวกัน ใช้สำหรับ:
- จัดหมวดหมู่รีวิวสินค้าโดยอัตโนมัติ — เอา review 50,000 ชิ้นมา embed แล้วรัน clustering พบว่ามี 6 กลุ่มหลัก: คุณภาพสินค้า, ความเร็วการจัดส่ง, บริการลูกค้า, บรรจุภัณฑ์, ราคา, และ การใช้งาน
- แยกประเภท support ticket — email ลูกค้าที่เข้ามาพันฉบับต่อวันถูก categorize โดยอัตโนมัติแล้วส่งต่อไปยังทีมที่รับผิดชอบ
- วิเคราะห์ feedback จากลูกค้าว่าเรื่องไหนถูกพูดถึงบ่อย — ช่วยให้ทีม product รู้ว่า pain point ไหนที่ users บ่นมากที่สุด

### 4. Duplicate Detection

เอา embedding มาเปรียบเทียบ similarity เพื่อหาเนื้อหาซ้ำกัน แม้จะใช้คำต่างกัน ใช้สำหรับ:
- หาบทความที่ถูกตัด-วางมาโดยเปลี่ยนคำบางส่วน — cosine similarity สูงกว่า 0.95 บอกได้ว่าเนื้อหาซ้ำซ้อนกัน
- ตรวจสอบ duplicate question ใน Q&A platform — ถ้ามีคนถาม "จะ cancel subscription ได้ยังไง" และก่อนหน้านี้มีคนถาม "วิธียกเลิกการสมัครสมาชิก" ระบบรู้ว่าเป็นคำถามเดียวกันและแสดงคำตอบเดิม
- หา redundant content ในฐานข้อมูลเอกสาร — ช่วยทีม content รู้ว่ามีบทความหัวข้อเดียวกันที่เขียนซ้ำโดยไม่รู้ตัว

Threshold สำหรับ "ซ้ำ" ขึ้นอยู่กับ use case: งาน academic plagiarism detection อาจใช้ค่า 0.90+ ขณะที่ Q&A deduplication อาจใช้ค่า 0.85+ เพราะคำถามที่ "ใกล้เคียงกันพอ" ก็ควรได้คำตอบเดียวกัน

### 5. RAG (Retrieval-Augmented Generation)

นี่คือ use case ที่ทรงพลังที่สุดในยุค LLM — embedding เป็นรากฐานของ [RAG](/building-with-ai/rag-retrieval-augmented-generation) ซึ่งเป็นเทคนิคที่ทำให้ AI ตอบคำถามจากเอกสารของคุณได้ โดยไม่ต้องเทรน model ใหม่

กระบวนการง่าย ๆ ของ RAG:
1. embed ทุก document ในฐานข้อมูล
2. เมื่อผู้ใช้ถามคำถาม → embed คำถาม
3. หา document ที่ embedding ใกล้คำถามมากที่สุด
4. ส่ง document เหล่านั้นพร้อมคำถามให้ LLM ตอบ

Embedding เป็นขั้นตอน "retrieval" ใน RAG — บทต่อ ๆ ไปจะพูดถึง RAG ในรายละเอียด

**ทำไม embedding ถึงสำคัญสำหรับ RAG?** เพราะมันแก้ปัญหาหลักของ LLM: context window จำกัด คุณไม่สามารถยัดเอกสาร 10,000 หน้าเข้าไปใน prompt ได้ แต่ embedding ช่วยให้คุณดึง "เฉพาะหน้าที่เกี่ยวข้อง" ออกมาส่งให้ AI อ่าน — ทำให้ AI ตอบได้แม่นยำจากข้อมูลของคุณโดยไม่ต้องส่งข้อมูลทั้งหมด

## ข้อจำกัดและ Gotchas ที่ต้องรู้

Embedding มีพลัง แต่ก็มีข้อจำกัดที่คนมักเผลอลืม ถ้าไม่รู้ไว้อาจเจอปัญหาในการใช้งานจริง

### 1. ไม่เก่งเรื่องบริบทยาว ๆ

Embedding model ส่วนใหญ่มี context window จำกัด — text-embedding-3-small รับได้สูงสุด 8,191 token (ประมาณ 6,000 คำ) ถ้าเอกสารยาวกว่านั้น ต้องตัดเป็นชิ้น (chunking) ก่อน

และแม้จะสั้นกว่า 8,191 token embedding ก็บีบอัดข้อมูลทั้งหมดเป็น vector เดียว เอกสาร 10 หน้าและเอกสาร 1 บรรทัดได้ vector ขนาดเดียวกัน (1,536 มิติ) — เอกสารยาวจะสูญเสียรายละเอียดในการบีบอัด

ลองนึกภาพมันเป็นการ "สรุป" ข้อมูลทั้งหมดลงใน 1,536 ตัวเลข — ยิ่งข้อมูลมาก ยิ่งสูญเสียรายละเอียดในการสรุป เหมือนพยายามสรุปนิยาย 500 หน้าในหนึ่งย่อหน้า

วิธีแก้: แบ่งเอกสารยาวเป็น chunk ขนาด 300-500 token แล้ว embed แต่ละ chunk แยกกัน การ overlap chunk (ให้ chunk ถัดไปเริ่มต้นจากตรงกลางของ chunk ก่อน) ยังช่วยให้บริบทข้ามรอยต่อไม่หายไปด้วย

### 2. ภาษาไทยทำได้ไม่สุด

OpenAI embedding models ถูกเทรนด้วยข้อมูลภาษาอังกฤษเป็นหลัก ภาษาไทยมีอยู่บ้างแต่น้อยกว่ามาก ผลคือ:
- คำศัพท์เฉพาะทางภาษาไทย อาจได้ embedding ที่ไม่แม่นยำ
- สำนวนไทย อาจถูก embed ผิด context
- คำทับศัพท์และคำภาษาถิ่นมักมีปัญหา

ถ้างานต้องการภาษาไทยแม่นยำ ลองใช้ Cohere embed-multilingual-v3.0 หรือ open-source multilingual models แทน

### 3. เปลี่ยน Model = Re-embed ทั้งหมด

นี่คือ gotcha ที่แพงมากถ้าไม่รู้ไว้ — ถ้าคุณ embed document 100,000 ชิ้นด้วย model A แล้วเปลี่ยนมาใช้ model B คุณต้อง embed ทั้งหมดใหม่ตั้งแต่ต้น

เพราะ vector ของ model A และ model B อยู่ใน "space" ที่ต่างกันสิ้นเชิง ไม่สามารถนำมาเปรียบเทียบข้ามกันได้ — เหมือนพิกัด GPS และพิกัดแผนที่เก่าใช้ระบบอ้างอิงต่างกัน

**วิธีป้องกัน:** เลือก model ให้ชัวร์ก่อน เริ่มเก็บ embedding จริงเมื่อแน่ใจแล้ว และบันทึกว่าใช้ model อะไร version อะไรเสมอ

### 4. ขนาด Vector vs ค่าใช้จ่ายจัดเก็บ

Vector ขนาด 1,536 มิติ ต่อ document หนึ่งชิ้นใช้พื้นที่ประมาณ 6 KB (float32) ฟังดูน้อย แต่ถ้ามี 1 ล้าน document:

```
1,000,000 documents × 6 KB = 6 GB
```

6 GB ไม่เยอะมากสำหรับ storage ทั่วไป แต่ถ้าต้อง load ทั้งหมดเข้า RAM เพื่อค้นหา มันจะช้ามาก และถ้ามี 100 ล้าน document ก็กลายเป็น 600 GB — นั่นคือเหตุผลที่ระบบ production ต้องใช้ **Vector Database** (บทถัดไป) แทนการเก็บใน list ธรรมดา

### 5. Embedding ไม่ใช่ความจริงสัมบูรณ์

Embedding เป็นการ "เดา" ความหมายโดยอิงจากข้อมูลที่ model เห็นระหว่างเทรน ถ้าคำหรือแนวคิดนั้นไม่มีในข้อมูลเทรน หรือหายากมาก embedding อาจผิดได้

ตัวอย่าง: คำศัพท์เฉพาะทางในอุตสาหกรรมเฉพาะ, ชื่อผลิตภัณฑ์ใหม่ ๆ, คำสแลงล่าสุด — อาจได้ embedding ที่ไม่ตรงความหมายที่คุณคาดหวัง

ถ้าคุณทำงานในอุตสาหกรรมที่มีคำเฉพาะมาก เช่น กฎหมาย, การแพทย์, วิศวกรรม — ควรทดสอบ embedding กับชุด query จริงก่อนเชื่อมันเต็มที่ บางครั้งการ fine-tune embedding model กับข้อมูล domain เฉพาะช่วยได้ แต่นั่นเป็นเรื่องขั้นสูงที่ต้องใช้ทรัพยากรมากกว่า

<!-- IMAGE: infographic แสดงข้อจำกัด 5 ข้อ พร้อมวิธีแก้แต่ละข้อแบบย่อ -->

> 💡 **Key insight:** Embedding ทรงพลังมากสำหรับข้อความทั่วไปในภาษาหลัก แต่ยิ่ง domain เฉพาะ ภาษาน้อย หรือเอกสารยาว ยิ่งต้องระวังและทดสอบมากขึ้น อย่า assume ว่ามันทำงานได้ดีโดยไม่วัดผลจริง

## เมื่อไหร่ควรใช้ Embedding และเมื่อไหร่ไม่ควร

ก่อนจบบท มาพูดถึงสิ่งที่หนังสือหรือคอร์สออนไลน์มักข้าม — embedding ไม่ใช่คำตอบสำหรับทุกปัญหา

**ใช้ Embedding เมื่อ:**
- ต้องการค้นหาจาก "ความหมาย" ไม่ใช่ "คำ"
- ผู้ใช้พิมพ์คำหลายรูปแบบเพื่อความหมายเดียวกัน
- ข้อมูลมีหลายภาษาและต้องค้นหาข้ามภาษา
- ต้องการ cluster หรือจัดหมวดหมู่ข้อความโดยอัตโนมัติ
- สร้างระบบ recommendation หรือ RAG

**ไม่จำเป็นต้องใช้ Embedding เมื่อ:**
- ค้นหา SKU สินค้า, รหัสลูกค้า, หรือ ID — keyword หรือ exact match เร็วกว่าและแม่นกว่า
- Query แบบ structured เช่น "หาสินค้าราคาต่ำกว่า 500 บาท หมวดอาหาร" — SQL ทำได้ดีกว่า
- ข้อมูลน้อยมาก (ต่ำกว่า 100 รายการ) — keyword search เพียงพอและ setup ง่ายกว่า
- ต้องการผลลัพธ์ที่ deterministic 100% — embedding อาจให้ผลต่างกันเล็กน้อยเมื่อ model update

กฎง่าย ๆ: ถ้าปัญหาของคุณแก้ได้ด้วย `WHERE name LIKE '%keyword%'` ก็ไม่ต้องใช้ embedding ใช้ embedding เมื่อ "คำ" ไม่พอ และคุณต้องการ "ความหมาย"

## สรุป — จากตัวอักษรสู่ความหมาย

มาสรุปสิ่งที่เรียนรู้ทั้งหมดในบทนี้:

**Embedding คืออะไร** — เทคโนโลยีที่แปลงข้อความ (หรือรูปภาพ เสียง ฯลฯ) เป็น vector ซึ่งก็คือ list of numbers ที่แทน "ตำแหน่ง" ในโลกของความหมาย

**ทำไมต้องใช้** — เพราะ keyword search เข้าใจแค่ตัวอักษร แต่ embedding เข้าใจความหมาย ทำให้ค้นหาได้แม้ใช้คำต่างกัน ภาษาต่างกัน หรือพูดถึงแนวคิดเดียวกันในหลายวิธี

**Vector Space** — ข้อความที่มีความหมายใกล้กันได้ vector ใกล้กันใน space ความใกล้วัดได้ด้วย cosine similarity (0 = ไม่เกี่ยว, 1 = เหมือนกัน) และ vector arithmetic เปิดให้ทำ analogy ข้ามแนวคิดได้

**Embedding Models** — OpenAI text-embedding-3-small เหมาะสำหรับเริ่มต้น Cohere embed-multilingual-v3.0 เหมาะถ้าต้องการภาษาไทยดีกว่า open-source models เหมาะถ้าต้องการ privacy หรือต้นทุนต่ำ

**ลองทำได้เลย** — เรียก OpenAI Embedding API ด้วยโค้ดไม่กี่บรรทัด ได้ vector กลับมา คำนวณ cosine similarity สร้าง semantic search อย่างง่ายได้ในชั่วโมงเดียว

**ข้อจำกัดสำคัญ** — ไม่ดีกับข้อความยาวมาก (ต้องทำ chunking), ภาษาไทยยังมีข้อจำกัดกับ OpenAI models, เปลี่ยน model ต้อง re-embed ทั้งหมด, และ vector ขนาดใหญ่ต้องการระบบจัดเก็บเฉพาะ

<!-- IMAGE: mind map สรุปแนวคิดหลักในบท: embedding → vector → similarity → use cases → ข้อจำกัด -->

และนั่นนำมาสู่คำถามสำคัญ — เราสร้าง vector ได้แล้ว แต่ถ้ามีเอกสารหลายล้านชิ้น เราจะเก็บ vector เหล่านั้นที่ไหน และจะค้นหา "vector ที่ใกล้ที่สุด" จากล้านชิ้นได้อย่างรวดเร็วในเสี้ยววินาทีได้ยังไง?

การลูปเปรียบเทียบ vector ทีละชิ้นแบบที่เราทำในตัวอย่างนั้นทำงานได้ดีกับ 10 หรือ 100 document แต่เมื่อข้อมูลเป็นล้านชิ้น การลูปเปรียบเทียบทุกคู่จะใช้เวลาเป็นนาที — ซึ่งช้าเกินไปสำหรับ search ที่ผู้ใช้รอได้ไม่เกินเสี้ยววินาที

นั่นคือปัญหาที่ **Vector Database** ถูกสร้างมาเพื่อแก้โดยเฉพาะ มันใช้ algorithm พิเศษที่ชื่อว่า Approximate Nearest Neighbor (ANN) ซึ่งสามารถหา "vector ที่ใกล้ที่สุด" จากหลายล้านรายการได้ในเวลาน้อยกว่า 100 millisecond บทถัดไปจะพาคุณไปรู้จักกับ Pinecone, Weaviate, และ pgvector — ฐานข้อมูลที่คิดมาสำหรับค้นหา vector โดยเฉพาะ และทำให้ semantic search ทำงานได้จริงในระดับ production
