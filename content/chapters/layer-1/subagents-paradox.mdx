# Sub-agent: ตัวช่วยที่ย้อนแย้ง — ช่วยให้คิดสะอาดขึ้น แต่ไม่ฟรี และไม่ใช่ค่าตั้งต้น

คุณอาจเคยอ่านเจอคำแนะนำทำนองนี้ — "อยากให้ AI ทำงานดีขึ้น ลองให้มันแยกงานหนัก ๆ ไปทำใน sub-agent สิ จะช่วยประหยัด token ด้วย". ฟังดูดี เหมือนของฟรีที่ได้ทั้งคุณภาพและประหยัด.

แต่พออ่านต่ออีกแหล่ง คุณกลับเจอคนบอกว่า sub-agent นี่แหละตัวเผา token — เพราะมันเปิด context ของตัวเองขึ้นมาใหม่ทั้งกอง. แล้วถ้ามันดีขนาดที่เขาว่ากันจริง ทำไมเครื่องมือ AI ถึงไม่เปิดมันเป็นค่าตั้งต้นให้ทุกงานไปเลย.

สองเรื่องนี้ฟังดูขัดกันเอง. แต่จริง ๆ มันจริงทั้งคู่ — และพอคุณเห็นว่าทำไมมันถึงจริงทั้งคู่ได้ คุณจะอ่านคำแนะนำเรื่อง sub-agent ออกว่าอันไหนเชื่อถือได้ อันไหนขายฝัน.

## ก่อนอื่น sub-agent คืออะไรกันแน่

จำเรื่องจากบทก่อน ๆ ใน Act นี้ได้ไหม — ตัว AI ที่คุณใช้จริง ๆ ไม่ใช่โมเดลเปล่า ๆ แต่เป็น harness ที่ห่อโมเดลไว้ (เรื่อง harness อยู่ในบท "โมเดลไม่ใช่ผลิตภัณฑ์"). และทุกครั้งที่มันจะตอบ มันต้องอ่านกองข้อความทั้งหมดที่เรียกว่า context ใหม่ตั้งแต่ต้น เพราะมันไม่มีความจำ (เรื่องนี้อยู่ในบท "AI ไม่ได้จำคุณ").

sub-agent คือเรื่องที่ต่อยอดจากตรงนั้น. มันคือ AI อีกตัวหนึ่ง — มี context เป็นของตัวเอง แยกขาดจากตัวหลัก — ที่ตัวหลัก *เรียกขึ้นมา* ให้ไปทำงานหนึ่งงานที่รก ๆ เสร็จแล้วส่งผลกลับมา.

จุดที่ต้องเน้นคือคำว่า "context แยก". sub-agent ไม่ได้เห็นบทสนทนาทั้งหมดที่คุณคุยกับตัวหลักมา. มันเริ่มจากกระดาษเปล่า รับโจทย์เฉพาะหนึ่งงาน ทำงานนั้นในพื้นที่ของมันเอง แล้วส่งกลับมาแค่ *สรุปสั้น ๆ* ไม่ใช่ทุกอย่างที่มันเจอระหว่างทาง.

> sub-agent นี่ต่างจาก "ทีม AI หลายตัว" (swarm) ที่เราคุยกันในบทก่อนนะ. swarm คือหลาย agent ทำงานเป็นทีมขนานกัน — บทก่อนชี้ว่ามันมักช้ากว่าแพงกว่า. ส่วน sub-agent ในบทนี้คือตัวหลักหนึ่งตัว เรียกลูกมือมาทำงานรกชิ้นเดียวแล้วจบ คนละเรื่องกัน.

## ทำไมต้องแยก context ออกไป — ปัญหามันอยู่ตรง "โต๊ะรก"

ลองนึกถึงสถานการณ์จริง. คุณให้ AI ช่วยเขียนรายงานสรุปจากเอกสาร 50 ไฟล์. ถ้าตัวหลักลุยอ่านทั้ง 50 ไฟล์เองในบทสนทนาเดียว เนื้อหาทั้งหมดของ 50 ไฟล์นั้นจะถูกยัดเข้าไปกองอยู่ใน context ของมัน.

ผลคืออะไร. จากบทก่อนเรารู้แล้วว่า context มีพื้นที่จำกัด — ยาวเกินเมื่อไหร่ ของเก่าจะถูกตัดทิ้ง แล้วมันก็เริ่ม "ลืม". พอ context อัดแน่นไปด้วยข้อความดิบจาก 50 ไฟล์ สองอย่างเกิดขึ้น: หนึ่ง โจทย์ตั้งต้นกับคำสั่งสำคัญของคุณเสี่ยงโดนดันหลุดกระดาน. สอง ยิ่งมีข้อความรก ๆ เยอะ โมเดลยิ่งมีโอกาสหยิบรายละเอียดผิด ๆ มาปนจนตอบมั่ว — อาการที่เรียกว่า hallucination.

นี่คือเหตุผลที่การแยกไปทำใน sub-agent ช่วยได้จริง. งานอ่าน 50 ไฟล์ไปเกิดในพื้นที่ของ sub-agent — โต๊ะรกอยู่ที่นั่น ไม่ใช่ที่โต๊ะหลัก. ตัวหลักได้รับกลับมาแค่ "สรุป 1 หน้า" โต๊ะของมันเลยยังโล่ง ยังเห็นโจทย์ตั้งต้นชัด ยังจำบริบทงานได้ และมีของรก ๆ ให้หยิบผิดน้อยลง.

> 💡 **ใจความสำคัญ:** ของจริงที่ sub-agent ให้คุณไม่ใช่ "ประหยัด token" แต่คือ **context หลักที่สะอาด** — ซึ่งแปลว่าลืมน้อยลงและหลอนน้อยลง. มันแลก token เพิ่ม เพื่อแลกกับคุณภาพและความน่าเชื่อถือ.

## แล้วเรื่อง "ประหยัด token" มันมาจากไหน — และมันพลาดตรงไหน

ที่คนพูดว่าประหยัด เขามองแค่ฝั่งเดียว — ฝั่งตัวหลัก. จริงอยู่ว่า context ของตัวหลักไม่บวมขึ้นเพราะของรก ๆ ไม่ได้เข้ามากองที่นั่น. ถ้าวัดเฉพาะ token ที่ตัวหลักต้องอ่านซ้ำในแต่ละตา มันน้อยลงจริง.

แต่นั่นคือการนับไม่ครบ. งานอ่าน 50 ไฟล์ไม่ได้หายไปไหน — มันยังต้องเกิดขึ้น แค่ไปเกิดในพื้นที่ของ sub-agent. token ที่ใช้อ่าน 50 ไฟล์นั้น *ยังถูกจ่ายอยู่ดี* แค่จ่ายในบัญชีของ sub-agent แทน.

ซ้ำร้าย sub-agent มีต้นทุนตั้งต้นของมันเอง. การจุดมันขึ้นมาหนึ่งตัว ต้องป้อนคำสั่ง ป้อนโจทย์ ป้อนบริบทที่จำเป็นให้มันก่อน — นั่นคือ token ก้อนหนึ่งที่ไม่มีถ้าทำในตัวหลักรวด ๆ. เสร็จแล้วสรุปที่มันส่งกลับมาก็เป็น token ที่ตัวหลักต้องอ่านอีก.

เพราะฉะนั้นถ้านับ token *รวมทั้งระบบ* — ตัวหลัก บวก sub-agent — หลายครั้งมันกลับ *เพิ่มขึ้น* ไม่ใช่ลดลง. คุณจ่ายค่าจุดเครื่องใหม่ บวกค่าทำงานรกที่ยังต้องทำอยู่ดี บวกค่าส่งสรุปไปมา.

## เปรียบเทียบให้เห็นภาพ

ลองคิดว่าตัวหลักคือ *ผู้จัดการร้าน* ที่กำลังยืนรับออเดอร์อยู่หน้าร้าน หัวเต็มไปด้วยรายละเอียดลูกค้าตรงหน้า. แล้วมีคนถามว่าของในโกดังเหลือเท่าไหร่.

ผู้จัดการที่ดีจะไม่ทิ้งเคาน์เตอร์เดินเข้าไปนับสต็อกเอง — เพราะกว่าจะนับเสร็จเดินกลับมา เขาลืมออเดอร์หน้าร้านไปครึ่งหนึ่งแล้ว. เขาเลยส่งลูกน้องเข้าโกดังไปนับ แล้วลูกน้องเดินกลับมาบอกแค่ "ของหมด 3 อย่าง". ผู้จัดการได้คำตอบที่ต้องใช้ โดยหัวยังอยู่กับงานหน้าร้าน. โต๊ะสะอาด.

แต่สังเกตว่ามัน *ไม่ฟรี*. ร้านต้องจ้างลูกน้องเพิ่มอีกแรง. งานนับสต็อกก็ยังต้องนับอยู่ดี ไม่ได้หายไป — แค่เปลี่ยนคนนับ. คุณจ่ายค่าแรงเพิ่ม เพื่อแลกกับการที่ผู้จัดการไม่เสียสมาธิ.

จุดที่การเปรียบเทียบนี้พัง: ลูกน้องคนจริงทำงานไปเรียนรู้ไป ครั้งหน้านับสต็อกเร็วขึ้นเพราะจำชั้นวางได้. แต่ sub-agent เริ่มจากศูนย์ทุกครั้ง — มันไม่มีความจำสะสม (เหตุผลเดียวกับที่ AI ไม่จำคุณข้ามบทสนทนา). เรียกใหม่ครั้งหน้าก็ต้องป้อนโจทย์ใหม่หมด เสียต้นทุนตั้งต้นซ้ำทุกรอบ.

## เพราะงี้มันถึงไม่ใช่ค่าตั้งต้น

ทีนี้คำถามแรกของคุณก็ตอบตัวเองได้แล้ว — ถ้า sub-agent ดี ทำไมไม่เปิดเป็นค่าตั้งต้นทุกงาน.

เพราะมันเป็น *การแลก* ไม่ใช่ของฟรี. มันคุ้มเฉพาะเมื่องานรกก้อนนั้นใหญ่พอที่จะทำให้ thread หลักเลอะจริง ๆ — อ่านไฟล์เป็นกอง ค้นข้อมูลยาว ๆ ที่ได้ผลดิบมาเยอะ ลุยงานที่จะคายข้อความขยะออกมามหาศาล. งานพวกนี้ ความสะอาดของ context หลักที่ได้กลับมา คุ้มค่า token ที่จ่ายเพิ่ม.

แต่ถ้าเป็นงานเล็ก ๆ ถามตอบสั้น ๆ — การจุด sub-agent ขึ้นมามีแต่จะเพิ่มต้นทุนตั้งต้นและความช้าโดยไม่ได้อะไรกลับมา เพราะโต๊ะหลักมันก็ไม่ได้จะรกอยู่แล้ว. เปิดเป็นค่าตั้งต้นให้ทุกงานจึงเป็นการจ่ายแพงขึ้นเปล่า ๆ ในงานส่วนใหญ่.

### ลองทำดู

ครั้งหน้าที่คุณต้องให้ AI สรุปจากเอกสารหรือข้อมูลกองโต ลองสังเกตอาการ "โต๊ะรก" ด้วยตัวเอง. เปิดแชตใหม่ แล้วแปะเนื้อหายาว ๆ หลาย ๆ ก้อนต่อกันลงไป — เอกสารหลายชิ้นในแชตเดียว. คุยต่อไปเรื่อย ๆ สักพัก แล้วย้อนถามมันถึงคำสั่งหรือโจทย์ที่คุณบอกไว้ตอนเปิดแชต.

ถ้ามันเริ่มตอบเพี้ยน หยิบรายละเอียดผิดก้อนมาตอบ หรือลืมโจทย์เดิม — นั่นคือคุณเพิ่งเห็นกับตาว่า "โต๊ะรก" หน้าตาเป็นยังไง. และคุณจะเข้าใจทันทีว่าทำไมการยกงานก้อนรก ๆ นั้นออกไปทำที่อื่น แล้วรับกลับมาแค่สรุป ถึงช่วยให้คำตอบนิ่งขึ้น — แม้จะต้องจ่ายเพิ่มก็ตาม.

## สรุป paradox นี้ในประโยคเดียว

sub-agent ไม่ใช่ปุ่ม "ประหยัด" และไม่ใช่ปุ่ม "เก่งขึ้นฟรี ๆ". มันคือการ *ยอมจ่าย token เพิ่ม เพื่อซื้อ context หลักที่สะอาด* — ซึ่งแปลเป็นลืมน้อยลงและหลอนน้อยลง. เป็นการแลก ไม่ใช่ของแถม. ดังนั้นมันจึงเป็นทางเลือกที่หยิบมาใช้ตอนงานรกพอจะคุ้ม ไม่ใช่ค่าตั้งต้นของทุกงาน.

พอคุณอ่าน trade-off นี้ออก คำแนะนำเรื่อง sub-agent บนเน็ตก็แยกแยะได้ทันที — คนที่บอกว่า "ใช้เถอะ ประหยัด token" กำลังนับไม่ครบ. คนที่บอกว่า "ใช้เมื่องานหลักจะเลอะ เพื่อให้คำตอบเชื่อถือได้ขึ้น" กำลังพูดถูก.

ส่วนเรื่อง *เมื่อไหร่กันแน่ที่ควรเรียก sub-agent ขึ้นมา และตั้งค่ามันยังไง* — นั่นเป็นเรื่องลงมือทำจริงที่เราจะเจาะในโมดูล "ใช้ AI เป็นระบบงานของคุณ" ทีหลัง. ตอนนี้คุณมีของที่สำคัญกว่าแล้ว: คุณอ่าน paradox มันออก และจะไม่โดนคำว่า "ประหยัด token" หลอกอีก.
