# โมเดลมีเป็นสิบ — เลือกตัวไหน และตัวไหนภาษาไทยไม่เพี้ยน

คุณเปิดเมนูเลือกโมเดลใน ChatGPT หรือ Claude แล้วเจอรายชื่อยาวเป็นพืด. Opus, Sonnet, Haiku. GPT ตัวนั้นตัวนี้. Gemini อีกหลายรุ่น. แต่ละตัวมีตัวเลขรุ่นห้อยท้ายที่ดูคล้ายกันไปหมด. แล้วคุณก็คิดในใจว่า — เอาตัวที่ฟังดูใหญ่สุด เก่งสุดไว้ก่อนสิ จะได้ไม่ต้องเดา จะได้ไม่พลาด.

ฟังดูสมเหตุสมผล. แต่พอใช้ไปสักพักคุณอาจเริ่มเอะใจ. ตัวใหญ่สุดมันช้า — ถามอะไรง่าย ๆ ก็ต้องนั่งรอ. แล้วถ้าเป็นการเรียกผ่าน API บิลก็พุ่งเร็วกว่าที่คิด. แถมพอเป็นงานภาษาไทย บางตัวที่ขึ้นชื่อว่าเก่งสุดในโลก ดันแปลไทยออกมาแข็ง ๆ เพี้ยน ๆ อ่านแล้วรู้เลยว่าเครื่องเขียน.

คำถามจริง ๆ เลยไม่ใช่ "ตัวไหนเก่งสุด". มันคือสองคำถามที่ซ้อนกันอยู่ — *งานนี้ต้องใช้ตัวเก่งแค่ไหน* และ *ตัวไหนพูดไทยได้ไม่เพี้ยน*. และนี่คือสองแกนคนละเรื่องกันเลย.

## ทำไมไม่มี "โมเดลที่ดีที่สุด" ตัวเดียว

เริ่มจากความเข้าใจผิดที่อยู่เบื้องหลังคำว่า "เอาตัวใหญ่สุดไว้ก่อน" ก่อน.

โมเดลแต่ละค่ายมักออกมาเป็น *ชุด* — ตัวเล็ก ตัวกลาง ตัวใหญ่. ฝั่ง Claude เรียง Haiku (เล็ก) → Sonnet (กลาง) → Opus (ใหญ่). ค่ายอื่นก็มีโครงคล้าย ๆ กัน แค่ใช้ชื่อต่างไป. และของพวกนี้ไม่ได้เรียงเป็น *บันไดความดี* ที่ขั้นบนสุดดีกว่าทุกอย่าง. มันเรียงตามการแลกเปลี่ยนสามอย่างที่ขยับพร้อมกัน: ยิ่งตัวใหญ่ ยิ่ง *เก่งขึ้น* แต่ก็ *ช้าลง* และ *แพงต่อการใช้งานมากขึ้น* ด้วย.

ตรงนี้คือจุดที่คนเข้าใจผิดบ่อยที่สุด. ตัวใหญ่ไม่ได้แปลว่า "ดีกว่า" เฉย ๆ. มันแปลว่า "เก่งกว่า แต่แลกมาด้วยเวลาและเงินที่มากกว่า". ถ้างานของคุณไม่ได้ต้องใช้ความเก่งระดับนั้น คุณก็แค่จ่ายแพงขึ้นและรอนานขึ้นเพื่อคำตอบที่ดีพอ ๆ กัน.

ลองนึกถึงเรื่องเงินก่อน. แต่ละครั้งที่คุณใช้โมเดล คุณจ่ายตามปริมาณข้อความเข้า-ออก (เราเรียกหน่วยนี้ว่า token — เรื่องนี้บทค่าใช้จ่ายจะเจาะให้ชัด). โมเดลตัวใหญ่คิดค่าต่อ token แพงกว่าตัวเล็กหลายเท่า. เพราะฉะนั้นการเอาตัวใหญ่สุดไปตอบทุกอย่าง — รวมถึงงานจิ๊บจ๊อยอย่าง "ช่วยสรุปย่อหน้านี้ให้หน่อย" — ก็เหมือนเรียกรถตู้มาส่งของชิ้นเดียวข้ามซอย. ได้อยู่ แต่เปลืองเปล่า ๆ.

## วิธีเลือกจริง ๆ: จับคู่ "ความยากของงาน" กับ "ชั้นของโมเดล"

พอเลิกมองว่ามันเป็นบันได แล้วมองเป็นการแลกเปลี่ยน วิธีเลือกก็ง่ายขึ้นเยอะ. แทนที่จะถามว่า "ตัวไหนเก่งสุด" ให้ถามว่า "งานตรงหน้าตอนนี้ยากแค่ไหน".

วิธีที่ใช้ได้จริงคือ *ตั้งตัวกลางเป็นค่าเริ่มต้น แล้วค่อยขยับขึ้นลงตามงาน*. ตัวกลาง (เช่น Sonnet หรือรุ่นเทียบเท่าของค่ายอื่น) ทำงานในชีวิตประจำวันได้เกือบหมด — ร่างอีเมล สรุปเอกสาร ตอบคำถามทั่วไป ระดมไอเดีย. คุณอยู่กับตัวกลางเป็นหลัก แล้ว:

- **ขยับลงไปตัวเล็ก** เมื่องานง่ายและทำซ้ำเยอะ ๆ — จัดรูปแบบข้อความ แปลงตาราง ดึงข้อมูลออกจากข้อความ. งานพวกนี้ตัวเล็กทำได้สบาย เร็วกว่า และถูกกว่ามาก.
- **ขยับขึ้นไปตัวใหญ่** เมื่อเจองานที่ยากจริง ๆ — วิเคราะห์ปัญหาหลายชั้น เขียนงานยาวที่ต้องลื่นและแม่น ไล่ตรรกะที่ซับซ้อน. ตรงนี้แหละที่ความเก่งที่แพงกว่าคุ้มกับเงินที่จ่ายเพิ่ม.

สังเกตว่าทั้งหมดนี้คุณไม่ต้องท่อง spec ของแต่ละรุ่นเลย. ไม่ต้องจำว่าตัวไหนมีพารามิเตอร์เท่าไหร่ รองรับ context ยาวแค่ไหน. คุณแค่ต้องรู้สึกได้ว่า "งานนี้ระดับไหน" แล้วจับคู่กับชั้น. ตัวเลขสเปกพวกนั้นเปลี่ยนทุกไม่กี่เดือนอยู่แล้ว — ยึดหลักการเลือกไว้จะอยู่ได้นานกว่าการจำรุ่น.

มีอีกปุ่มหนึ่งที่คุณจะเจอ คือ "โหมดคิด" หรือ reasoning mode (ตัวที่ขึ้นว่า "กำลังคิด..." แล้วใช้เวลานานกว่าปกติ). อย่ามองมันเป็น "โมเดลที่อยู่สูงกว่าที่ต้องใช้ตลอด" — มองเป็น *ความสามารถที่เปิดเมื่อเจองานยากที่ต้องคิดเป็นขั้น* เช่นเลขหลายชั้นหรือตรรกะที่พันกัน. งานทั่วไปอย่างร่างอีเมลหรือถามตอบสั้น ๆ เปิดโหมดนี้มีแต่ทำให้ช้าและแพงขึ้นเปล่า ๆ. (รายละเอียดว่าโหมดคิดทำอะไรข้างใน อยู่ในบทเรื่องโหมดคิดก่อนตอบ.)

## เปรียบเทียบให้เห็นภาพ: เลือกพาหนะตามงาน ไม่ใช่ตามขนาด

คิดง่าย ๆ ว่าเลือกโมเดลก็เหมือนเลือกพาหนะออกไปทำธุระ. คุณไม่ได้เลือกคันที่ใหญ่ที่สุดทุกครั้ง — คุณเลือกตามว่าจะไปไกลแค่ไหนและขนของหนักแค่ไหน. ไปซื้อของหน้าปากซอยก็มอเตอร์ไซค์. ไปทำงานปกติก็รถเก๋ง. ขนของย้ายบ้านทั้งหลังถึงค่อยเรียกรถตู้. การเรียกรถตู้ไปซื้อไข่ฟองเดียวไม่ได้ทำให้คุณได้ไข่ดีขึ้น — แค่เปลืองน้ำมันและหาที่จอดยาก.

จุดที่การเปรียบเทียบนี้พัง: รถยิ่งใหญ่ยิ่งช้าเสมอ ทั้งออกตัวและเข้าซอย. แต่โมเดลไม่ได้เป็นแบบนั้นเป๊ะ — บางงานโมเดลใหญ่ก็ตอบได้เร็วพอ ๆ กับตัวเล็ก. แกนที่ต่างชัดที่สุดของโมเดลจริง ๆ ไม่ใช่ความเร็ว แต่คือ *ค่าใช้จ่ายต่อ token* ที่ตัวใหญ่แพงกว่าหลายเท่าทุกครั้งที่เรียกใช้. เพราะงั้นเวลาตัดสินใจ ให้คิดเรื่องเงินเป็นหลัก ความเร็วเป็นรอง.

> 💡 **ใจความสำคัญ:** ไม่มีโมเดล "ที่ดีที่สุด" มีแต่โมเดลที่ "เหมาะกับงานตรงหน้า". ตั้งตัวกลางเป็นค่าเริ่มต้น ขยับขึ้นเมื่อยากจริง ขยับลงเมื่อง่าย — แทนการคว้าตัวใหญ่สุด (แพงสุด) ทุกครั้ง.

## แกนที่สอง ที่ฝรั่งไม่เคยต้องคิด: ภาษาไทย

ทีนี้มาถึงเรื่องที่สำคัญกับคุณเป็นพิเศษ และเป็นเรื่องที่คนเขียนรีวิวโมเดลภาษาอังกฤษแทบไม่เคยพูดถึง.

ความ "เก่ง" ของโมเดลที่เราคุยกันมาทั้งหมด — เก่งคิด เก่งวิเคราะห์ เก่งเขียน — เขาวัดกันบนงานภาษาอังกฤษเป็นหลัก. แต่ *ความลื่นของภาษาไทย* เป็นคนละแกนกันเลย. โมเดลที่ทำคะแนนเหตุผลสูงลิ่วในเทสต์ภาษาอังกฤษ อาจเขียนไทยออกมาแข็ง ๆ ใช้คำแปลก ๆ วางรูปประโยคแบบฝรั่งแปลไทย. ในขณะที่อีกตัวที่ "เก่ง" น้อยกว่าในกระดาษ อาจเขียนไทยลื่นเป็นธรรมชาติกว่ามาก.

ทำไมถึงเป็นแบบนั้น. ส่วนหนึ่งเพราะข้อมูลภาษาไทยเป็นแค่เศษเสี้ยวเล็ก ๆ ของกองข้อความมหาศาลที่โมเดลอ่านมาตอนฝึก ซึ่งส่วนใหญ่เป็นภาษาอังกฤษ. และอย่างที่เคยเล่าไปว่าตัวตัดคำ (tokenizer) ของโมเดลส่วนใหญ่ซอยภาษาไทยละเอียดยิบกว่าอังกฤษมาก — ราว 3 ถึง 5 เท่า — ซึ่งนอกจากจะทำให้งานไทยแพงและเต็มเร็วกว่าแล้ว ยังทำให้โมเดล "อ่าน" ไทยได้ยากกว่าด้วย. แต่ละค่ายฝึกกับข้อมูลไทยไม่เท่ากันและซอยคำไม่เหมือนกัน ผลคือความลื่นไทยของแต่ละตัวต่างกันชัดเจน.

แปลว่าเวลาเลือกโมเดลสำหรับงานไทย คุณต้องชั่งสองแกนแยกกัน — *งานนี้ยากแค่ไหน* (เลือกชั้น) และ *ตัวไหนพูดไทยเพี้ยนน้อยสุด* (เลือกค่าย). สองเรื่องนี้ไม่จำเป็นต้องชี้ไปที่ตัวเดียวกัน.

## อย่าเชื่อใคร — ทดเอง

คุณอาจเคยได้ยินคนพูดว่า "Gemini เขียนไทยดีสุด" หรือเชียร์ค่ายนั้นค่ายนี้. เก็บไว้เป็นข้อสังเกตได้ แต่อย่าถือเป็นกฎตายตัว. เหตุผลมีสองข้อ. หนึ่ง — โมเดลออกรุ่นใหม่กันแทบทุกเดือน ตัวที่ไทยดีสุดวันนี้อาจโดนแซงเดือนหน้า. สอง — "ไทยดี" ของงานแต่ละแบบไม่เหมือนกัน. โมเดลที่เขียนคอนเทนต์การตลาดไทยลื่น อาจสรุปเอกสารราชการไทยได้งั้น ๆ. งานของคุณเป็นตัวตัดสิน ไม่ใช่รีวิวของคนอื่น.

ข่าวดีคือคุณทดเองได้ง่ายมาก และใช้เวลาไม่ถึงห้านาที. นี่คือเครื่องมือที่เชื่อถือได้กว่ารีวิวทุกอัน.

### ลองทำดู

หยิบงานไทยจริงที่คุณทำบ่อย — เช่นย่อหน้าโปรโมตสินค้า อีเมลถึงลูกค้า หรือสรุปประชุมสั้น ๆ. เขียน prompt เดียวกันเป๊ะ ๆ แล้วส่งให้โมเดลสามตัวจากคนละค่าย (เช่นตัวกลางของ ChatGPT, Claude, และ Gemini). จากนั้นอ่านคำตอบทั้งสามเรียงกัน แล้วถามตัวเองว่า — อันไหนอ่านแล้วเหมือนคนไทยเขียน อันไหนอ่านแล้วสะดุดเพราะคำแปลก ๆ หรือรูปประโยคแบบแปลมา.

คุณจะเห็นความต่างทันทีด้วยตาตัวเอง โดยไม่ต้องดูสเปกหรือคะแนนเทสต์ใด ๆ. เก็บผลนี้ไว้ใช้กับงานไทยประเภทนั้นไปเลย แล้วค่อยทดใหม่เมื่อมีรุ่นใหม่ออกมา. นี่คือ "การทดลองชิมไทย" ของคุณเอง — และมันเชื่อถือได้กว่าคำบอกเล่าใคร ๆ เพราะมันใช้งานจริงของคุณเป็นเกณฑ์.

## สรุป: สองแกน หนึ่งนิสัย

กลับมาที่ความสับสนตอนเปิดบท. เมนูยาวเหยียดนั้นไม่ได้น่ากลัวอย่างที่คิด เพราะคุณไม่ได้ต้องเลือก "ตัวที่ดีที่สุด" — ไม่มีตัวนั้นอยู่จริง. คุณแค่ชั่งสองแกน: งานนี้ยากแค่ไหน (เลือกชั้นเล็ก-กลาง-ใหญ่ตามนั้น โดยตั้งตัวกลางเป็นค่าเริ่มต้น) และงานไทยของคุณตัวไหนทำได้ลื่นสุด (ทดเอง ไม่เชื่อรีวิว). การคว้าตัวใหญ่สุดทุกครั้งไม่ได้ทำให้คุณได้งานดีขึ้น — แค่ช้าลงและจ่ายแพงขึ้น.

เลือกโมเดลถูกตัวแล้ว ยังมีอีกเรื่องที่ขยับผลลัพธ์ได้แรงพอ ๆ กันแต่คนมองข้าม — *วิธีที่คุณยื่นเนื้อหาให้มัน*. เวลาคุณมีเอกสารยาว ๆ หรือตารางสักอันให้ AI ช่วยอ่าน คุณจะก๊อปวางลงช่องแชต หรือแนบเป็นไฟล์ไปเลย. หลายคนคิดว่าสองทางนี้เหมือนกันเพราะ "ก็เนื้อหาเดียวกัน" — แต่จริง ๆ แล้วสิ่งที่ AI ทำกับมันได้ต่างกันมากกว่าที่คิด. บทต่อไปจะอธิบายว่าทำไม.
